# 7.14.7 Tool and Details `msg_idx` validation (CLI replay)

Status: **proposed**

Checklist: `.cursor/rules/CODING_STANDARDS.md` — *Checklist for all plans*. Workflow: `docs/guide-to-writing-plans.md`.

## Purpose

- **`Tool.msg_idx`** is set from **`Details.msg_idx`** in **`Details.build_run_queue()`** only — one value shared by all **`Tool`** rows for that detail.
- **`Details.msg_idx`** is updated from **`response.message.idx`** in exactly **two** places on **`Details`** (both LLM **`send()`** returns):
  1. **`refine()`** — refinement reply (**before** execution queue).
  2. **`run_post_exec()`** — post-exec synthesis (**after** tool runs).
- **Executor** (**`Tool.run()`**): LLM **`send()`** for the review pass does **not** assign **`msg_idx`** from that response today; tools keep the idx copied at **`build_run_queue`** (normally refinement).
- **Do not** design fixes or validation around **“last returned”** replay cursor APIs (**`last_return_ix`**, **`last_returned_*`**, mirrored fields). The **`send()`** path that yields **`Response.Chat`** must attach the transcript **`idx`** for **that** returned row onto **`response.message`** so **`Details`** picks up the correct value through **`response.message.idx`** — same iteration, same **`msg`**, no secondary lookup.

## Scope

| In scope | Out of scope |
| -------- | ------------ |
| **`ReplayChat.send()`** wiring **`response.message.idx`** from the replay **`msg`** | Cursor-based “last returned” accessors |
| **`oc-test-skill-agent`** replay + logs | GTK **`ProgressView`** |
| Clarify refine / executor / post_exec | Persist **`idx`** in JSON |

## Acceptance criteria

- Plan explicitly rejects **last-returned** / replay-index mirrors as the mechanism for **`msg_idx`** correctness.
- **`ReplayChat.send()`** assigns **`response.message.idx`** from the **`Message`** row actually returned in that **`send()`** (the **`msg`** already in scope before building **`Response.Chat`**), so **`Details`** assignments need no separate replay lookup.
- Optional temporary **`GLib.debug`** may assert **`response.message.idx == msg.idx`** immediately before return from **`ReplayChat.send()`** (prove wiring only).

---

## 1. Three LLM sends; two update `Details.msg_idx`

| Stage | Where | Updates `Details.msg_idx`? | Updates `Tool.msg_idx`? |
| ----- | ----- | ---------------------------- | ------------------------- |
| Refinement | **`Details.refine()`** | **Yes** (`send` → **`response.message.idx`**) | Via **`build_run_queue`** → **`Tool`**. |
| Executor | **`Tool.run()`** | **No** | **No** — keeps refinement idx. |
| Post-exec | **`Details.run_post_exec()`** | **Yes** | N/A |

---

## 2. Root cause (replay)

**`Message`** assigns **`idx`** in **`construct`** from a monotonic counter. **`ReplayChat.send()`** builds **`response.message = new Message("assistant", msg.content)`**, so **`response.message.idx`** is a **new** id unless **`ReplayChat`** overwrites **`response.message.idx`** from **`msg.idx`** for the replay row returned in that call.

---

## 3. Concrete code proposals

Apply in order. **Rejected:** any **`last_returned_*`** accessor or validation read off **`last_return_ix`** — not listed below.

### 3.1 `libollmchat/Call/ReplayChat.vala` — inside **`send()`**, after **`response.message = new Message(...)`**

#### Keep (anchor — surrounding lines unchanged except insertion)

```vala
				this.messages = messages;
				var response = new Response.Chat(this.connection, this);
				response.message = new Message("assistant", msg.content);
				GLib.debug("replay response.message.idx=%d", response.message.idx);
				response.done = true;
				return response;
```

#### Replace with

```vala
				this.messages = messages;
				var response = new Response.Chat(this.connection, this);
				response.message = new Message("assistant", msg.content);
				response.message.idx = msg.idx;
				GLib.debug(
					"replay response.message.idx=%d transcript_msg.idx=%d",
					response.message.idx,
					msg.idx);
				response.done = true;
				return response;
```

**Effect:** **`Details`** lines **`this.msg_idx = response.message.idx`** (refine / post_exec) receive the **same** **`idx`** as the replay transcript row **`msg`** for that **`send()`**, without touching **`last_return_ix`** for logic.

### 3.2 `liboccoder/Task/Details.vala`

**No code change** for this plan — **`refine`** / **`run_post_exec`** assignments stay as-is once **`ReplayChat`** matches **`response.message.idx`** to **`msg.idx`**.

### 3.3 `liboccoder/Task/Tool.vala` — executor (**optional**, separate plan)

Only if product wants **`Tool`** rows to scroll to the **executor** LLM reply: after **`send()`** in **`run()`**, assign **`this.msg_idx`** from **`response.message.idx`** (again valid once **§3.1** lands). Not required for “fix wrong refinement idx” bug.

---

## 4. Interpret

- Once **`ReplayChat`** copies **`msg.idx`** onto **`response.message`**, **`Details`** lines **`this.msg_idx = response.message.idx`** align with the transcript row for that **`send()`** without parallel state.
