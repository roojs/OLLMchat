# 2.10.4 — `ollmfilesd` file service (out-of-process file + index operations)

## Status

**PROPOSAL** — for review. No code yet.

**API:** [`2.10.4.1`](2.10.4.1-ollmfilesd-rpc-api.md) (overview + vector). **Per object (client+server, includes today→wire maps):** [2.10.4.3](2.10.4.3-daemon-and-rpcclient.md) → [2.10.4.4](2.10.4.4-projectmanager.md) → [2.10.4.5](2.10.4.5-folder.md) → [2.10.4.6](2.10.4.6-file.md). **Layout:** [`2.10.4.0`](2.10.4.0-libocfiles-v2-and-rpc-layout.md).

## Overview

Move **all file-system and file-database work** out of `ollmchat` into a headless daemon **`ollmfilesd`**. The existing codebase keeps calling **`libocfiles`** APIs unchanged; **`libocfiles` is rewritten as an RPC client** that relays to the daemon. Server implementation lives inside the daemon (copied/moved from current `libocfiles` + vector scan code).

**🔷** User direction:
- Daemon owns filesystem scan, DB, disk read/write, semantic indexing, and (later) file watching.
- **No WAL migration required to start** — daemon is the **only** SQLite reader/writer; UI never opens `files.sqlite`.
- **Separate** filesystem and semantic queues (not forced into one unified queue); optional threading inside daemon can be tested on/off.
- **Debug mode** — run `ollmfilesd` in foreground without detaching to exercise the API.
- Logs to **`~/.cache/ollmchat/ollmfilesd.debug.log`** (same pattern as app debug logging).

Aligns with [`docs/plans/4.2.3-background-sync-file-watcher.md`](4.2.3-background-sync-file-watcher.md) (watcher in daemon, last phase).

## Problem (why change)

| Layer | Today | Pain |
| ----- | ----- | ---- |
| Filesystem scan | UI main thread — `Folder.read_dir()` | Minutes of GTK main-loop work |
| Semantic index | UI worker thread — `BackgroundScan` | Stops when app exits; shared DB/FAISS/Ollama |
| DB | In-memory SQLite + backup in UI process | `db_mutex` contention |
| Duplication | `worker_project_manager` in background scan | Two copies of project state |

## Architecture

```
┌─────────────────────────────────────────────────────────────┐
│  ollmchat, ollmchat-cli, tools, agents                      │
│                                                             │
│  libocfiles  (client — same public API as today)            │
│       │ RPC                                                 │
└───────┼─────────────────────────────────────────────────────┘
        │  JSON-RPC / NDJSON
        ▼
┌─────────────────────────────────────────────────────────────┐
│  ollmfilesd                                                 │
│                                                             │
│  Server: ProjectManager, Folder.read_dir, Indexer, Git, DB  │
│  ~/.local/share/ollmchat/files.sqlite  (single writer)      │
│  FAISS index files                                          │
│  ollmfilesd.sock + ollmfilesd.pid                          │
└─────────────────────────────────────────────────────────────┘
```

### Library split

| Piece | Role |
| ----- | ---- |
| **`ollmfilesd`** | Executable; hosts server `MainLoop`, socket, moved implementation |
| **`libocfiles` (client)** | Types + API surface unchanged for callers; methods become RPC stubs + local caches |
| **Server sources** | **⏳** Initially copied into daemon tree from current `libocfiles` / `BackgroundScan`; may later split `libocfiles-server` if build needs it |

**🔷** Goal: `ollmapp/Window.vala`, tools, and agents still use `OLLMfiles.ProjectManager` — they do not know about RPC except via `libocfiles` internals.

**🔷** RPC: JSON-RPC **`method`** = `{Object}.{method}`; flat **`params`** for arguments only (see **[`2.10.4.1`](2.10.4.1-ollmfilesd-rpc-api.md)**). Wire rows are **`filebase`** (`FileBase` / DB fields).

### UI memory model — lean client, fat daemon

**🔷** Today the UI process holds a **large in-memory mirror** of project state: full `Folder.children` hierarchy, `ProjectFiles` flat list + `child_map` / `folder_map` / `all_files`, `ProjectManager.file_cache`, `tree_cache`, and a **second** copy in `BackgroundScan.worker_project_manager`. Most of that exists to support scanning and DB sync that will live in **`ollmfilesd`** only.

**🔷** After cutover, the UI should **not** reload the whole project tree into RAM on every `project_changed`. Honest use in the app today is narrower:

| UI / tool need | What it actually uses | Keep in UI memory? |
| -------------- | --------------------- | ------------------ |
| Project picker (settings) | `ProjectList` — path, label, `is_active` | **Yes** — small list of project `Folder` rows |
| File dropdown / recent files | `ProjectFiles` as `ListModel` (`ProjectFile` wrappers) | **Partial** — flat slice or lazy page, not full `all_files` |
| Active editor tab | `active_file` + `File.buffer` | **Yes** — open files only |
| Approvals banner | `review_files` subset | **Yes** — small derived list |
| Path resolve (`get_file_from_active_project`) | `child_map` lookup | **💩** On-demand RPC or tiny path→id cache |
| Agent tools / codebase search | Paths, sometimes one `File` | **RPC** — fetch row when tool runs |
| Full directory tree widget | `Folder.children` recursive tree | **🚫** Not a primary UI surface today — do not hydrate whole tree in v1 client |

**Daemon** holds the canonical graph: `read_dir`, DB, indexes, `file_cache`, scan queues.

**Client (`libocfiles`)** holds:

- **Open-session state** — projects list, active project path, active/open files + buffers.
- **UI-facing caches** — optional flat `ProjectFiles` built from last `get_project_snapshot` or incremental `project_changed` deltas (**not** a full rescan of `children`).
- **Lazy `File` / `Folder` objects** — materialize from `filebase` JSON only when a widget or tool touches that path; evict when project deactivates.

**🚫** v1 client should **not** call `load_files_from_db` + `update_from(project)` to rebuild entire `children` trees on every notification — that pattern was for in-process DB sharing and is the main RAM multiplier.

**⏳** `get_project_snapshot` may ship as **flat `filebase[]`** first (enough for dropdown/search); nested `children` on the wire only if a future tree UI needs it — daemon can still keep hierarchy internally.

**ℹ️** Types (`File`, `Folder`, `ProjectFile`) stay for Gtk bindings and tool APIs; **count** of live instances in the UI process should drop from “whole project” to “what’s on screen + open tabs”.

### What stays in the UI process

| Stays local | Why |
| ----------- | --- |
| **Editor buffers** for open tabs | Typing latency; content via `read_file` / `write_file` RPC |
| **Small Gtk models** | Project list, file dropdown slice, `review_files` |
| **Gtk** — tabs, banners | Presentation only |
| **Chat / agents / history** | Not file-service scope |

### What moves to the daemon

| Moves | Notes |
| ----- | ----- |
| **All disk read/write for project files** | **🔷** Avoid watcher seeing a UI write then also getting an RPC “saved” message |
| **SQLite** — all reads and writes | No cross-process DB; negates WAL-as-prerequisite |
| Filesystem scan (`read_dir`) | Off UI thread entirely |
| Semantic index (`BackgroundScan` / `Indexer`) | Same process as scan; separate queues |
| `GitProvider` for scan-time git | **🔷** Main use today is scanning; blame/diff on open file **⏳** not required v1 |
| File watcher | Final phase |

## IPC

- **🔷** v1: Unix socket + JSON-RPC 2.0 — API in **[`2.10.4.1`](2.10.4.1-ollmfilesd-rpc-api.md)** including **`vector`** (`libocvector` search/index/metadata). **`Daemon`** RPC is only **`hello`** + **`shutdown`**; migration and DB setup are daemon startup, not client calls.
- **🚫** MCP / D-Bus as primary channel for v1.

---

## Queues and threading

**🔷** Keep **two pipelines** (filesystem scan vs semantic index), same logical design as today — both run inside `ollmfilesd`, not in the UI.

- Existing “wait until filesystem scan idle” coordination can remain **inside the daemon** if still needed.
- **🚫** Do not require a single unified queue up front.

**💩** Threading inside daemon is **optional** and togglable for testing:
- Default may be **single-threaded `MainLoop`** — UI lag was the main problem; that is gone once work is out-of-process.
- RPC `read_file` / `write_file` must remain responsive while long scans run — if single-thread blocks, add worker thread or idle chunking (measure in test phase).

### Phase 5 — Test daemon

- Toggle threading on/off if implemented.

### Phase 6 — UI cutover to `V2/`

- Point app at **`libocfiles/V2/*`** (not in-place edits of old sources); keep public API.
- `ollmchat` / CLI spawn or connect to `ollmfilesd`.
- Remove in-process `BackgroundScan` and main-thread `read_dir` from app binary.
- VectorScanBanner via `scan_update` notifications.
- **Lean client memory:** stop full-tree hydrate; snapshot/delta into flat `ProjectFiles` + open files only (see **UI memory model**).

### Phase 7 — File watcher

- `Gio.FileMonitor` in daemon ([`4.2.3`](4.2.3-background-sync-file-watcher.md)).
- Internal writes from `write_file` must not double-trigger scan (generation counter or suppress flag).

## Resolved decisions

| Topic | Decision |
| ----- | -------- |
| Daemon name | **`ollmfilesd`** |
| DB concurrency | **Daemon-only** SQLite — no WAL prerequisite for v1 |
| Disk I/O | **Daemon** — `read_file` / `write_file` RPC |
| Client shape | **`libocfiles`** becomes RPC client; callers unchanged |
| Unified queue | **No** — separate filesystem + semantic queues |
| WAL migration | **Not required to start** |

## Summary

**🔷** **`ollmfilesd`** owns files, DB, indexing, disk I/O, and the **full in-memory project graph**; the UI keeps **thin state** (open buffers, list models, lazy path lookups).

**🔷** **`libocfiles`** stays the public API but becomes an RPC client that does **not** mirror the whole project in RAM.

**🔷** API-first delivery: design in **2.10.4.1**, then daemon, copy server code, test without UI, then swap client library, then file watcher.

**⏳** Threading and snapshot shape are tuned during daemon test phase — not blockers for starting without WAL.
