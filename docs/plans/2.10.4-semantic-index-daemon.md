# 2.10.4 — File service daemon (out-of-process file + index operations)

## Status

**PROPOSAL** — for review. No code yet.

## Overview

Move **all heavy file-system work** out of `ollmchat` into a **separate headless executable** that owns scanning, persistence, and indexing. The GTK app becomes an **RPC client**: it displays state, edits open buffers, and sends commands — it does not walk directories, queue semantic scans, or contend on shared DB mutexes with background work.

**🔷** User direction: not just semantic indexing — the background process should be responsible for **all file operations** (filesystem scan, project sync, vector indexing); the main app talks to it over RPC.

This subsumes the original semantic-index-only proposal and aligns with the unimplemented file-watcher plan (`docs/plans/4.2.3-background-sync-file-watcher.md`).

---

## Problem (why change)

### Current model — everything lives in the UI process

| Layer | Where it runs today | Pain |
| ----- | ------------------- | ---- |
| **Filesystem scan** | Main thread — `Folder.read_dir()` via idle recursion (`ProjectManager.activate_project`) | 449 subdirs, minutes of main-loop work; `pm.scanning` blocks semantic queue |
| **Semantic index** | Worker thread — `BackgroundScan` + `Indexing.Indexer` | Stops when app exits; shares SQLite/FAISS/Ollama with UI |
| **DB persistence** | Both threads — in-memory SQLite + coalesced `backupDB()` | Global `db_mutex`; full DB copy to disk |
| **Project tree load** | Main thread — `load_files_from_db()` | Reload storms; stale tree vs background writes |

All of this is orchestrated by `OLLMfiles.ProjectManager` inside `ollmchat` (`ollmapp/Window.vala`, tools, CLI).

### Why it is slow and fragile

| Issue | Evidence |
| ----- | -------- |
| **Work stops when the app exits** | No overnight filesystem sync or semantic indexing |
| **Main-thread filesystem scan** | Bug investigation: ~3 min folder walk overlapping semantic work; hundreds of `backupDB` calls on main context |
| **LLM analysis dominates semantic throughput** | ~80–190 Ollama calls per PHP file, ~2 min/file (`docs/bugs/done/2026-05-15-FIXED-background-scan-ui-sluggish.md`) |
| **Cross-thread / cross-subsystem contention** | Shared in-memory SQLite, FAISS mutex, semantic scan waits on `pm.scanning` |
| **Duplicated project state** | BackgroundScan keeps its own `worker_project_manager` mirroring main `ProjectManager` |

### What a file-service daemon fixes

| Fixed by daemon | Still needs separate work |
| --------------- | ------------------------- |
| Main-thread directory walks move off UI | Reduce LLM calls per file (`should_skip_llm`, batching) |
| Semantic + filesystem scan in one queue/process | Scope of scanned paths (stray dirs) |
| Scanning continues when UI is closed | FAISS concurrent read while writer updates |
| Single writer to file metadata DB | Editor buffer ↔ disk conflict policy |
| One daemon shared by UI + CLI + future watcher | |

| **D-Bus session service** | Upgrade path — activation, introspection |

## Proposed architecture

```
┌──────────────────────────────┐         JSON-RPC           ┌─────────────────────────────┐
│  ollmchat (GTK UI)           │ ◄──── + notifications ───► │  ollmfilesd (headless)      │
│                              │                            │                             │
│  - chat / agents             │                            │  - ProjectManager (server)  │
│  - project tree display      │                            │  - Folder.read_dir          │
│  - file buffers (open tabs)  │                            │  - filesystem scan queue    │
│  - ProjectManagerClient      │                            │  - semantic index queue     │
│    (RPC proxy)               │                            │  - SQLite writer            │
│  - tool read via cache/DB    │                            │  - FAISS writer             │
│  - VectorScanBanner          │                            │  - file watcher (later)     │
└──────────────┬───────────────┘                            └──────────────┬──────────────┘
               │ read + hot metadata                                         │ all heavy writes
               ▼                                                             ▼
        ~/.local/share/ollmchat/files.sqlite (WAL)                  faiss index files
```

### Executable: `ollmfilesd` (working name)

**💩** Broader than `ollmindexd` — owns **files + index**, not just vectors. Confirm name.

- Headless — no GTK
- Links `libocfiles`, `libocvector`, `libollmchat` (Ollama client), `libocsqlite`
- **Single** `ProjectManager` instance — no `worker_project_manager` duplicate
- Owns `Folder.read_dir`, `BackgroundScan` queue, `Indexing.Indexer`
- Socket: `~/.local/share/ollmchat/ollmfilesd.sock` + `ollmfilesd.pid`
- Config from same data dir as app

### What stays in the UI process

| Stays local | Why |
| ----------- | --- |
| **Editor buffers** (`BufferProvider`) | Hot in-memory text; user typing latency |
| **Active file read/write to disk** | Save path must feel instant; notify daemon after save |
| **Gtk widgets** — tree view, tabs, banners | Presentation only |
| **Chat / agents / history** | Not file-service scope |
| **Git blame/diff on open file** | **💩** optional — may stay UI-local via `GitProvider` for v1; daemon owns bulk scan-time git |

### What moves to the daemon

| Moves to daemon | Replaces |
| --------------- | -------- |
| `activate_project` → filesystem scan | Main-thread `read_dir` in `ProjectManager.activate_project` |
| Project list load/save | `load_projects_from_db` server-side |
| Semantic scan queue | `BackgroundScan` thread |
| Bulk `filebase` row updates from scan/index | Worker + main DB writes during scan |
| Delete/rename reconciliation | `DeleteManager` server-side (**⏳** phase) |
| File watcher (`Gio.FileMonitor`) | Plan 4.2.3 — natural home in daemon |

### UI client: `ProjectManagerClient`

**💩** Thin RPC wrapper implementing the subset of `ProjectManager` the UI and tools need:

- Same **signals** where possible (`active_project_changed`, `file_contents_changed` → forwarded from daemon notifications)
- **No** local `read_dir`, **no** `BackgroundScan`, **no** direct `backupDB` for scan-driven writes
- Opens SQLite **read-only** (or read-write for UI-only columns — see below)
- Spawns/connects to daemon on startup (GitLive-style)

Tools (`ReadFile`, `CodebaseSearch`, `RunCommand`, agents) continue to receive a `ProjectManager`-like object — implemented as the client proxy, not the server.

## Database concurrency

**🔷** Prerequisite: migrate from in-memory-primary SQLite to **WAL on-disk** shared file.

| Writer | Typical operations |
| ------ | ------------------ |
| **Daemon** | Scan results, `last_modified`, `last_vector_scan`, element metadata, deletes |
| **UI** | **💩** High-frequency UI metadata only (`cursor`, `scroll`, `last_viewed`) — direct short WAL transactions OR batched `update_metadata` RPC |

**💩** Split rationale: sending cursor position over RPC on every keystroke is wrong; daemon owns **filesystem truth**, UI owns **view state** on open files.

---

## Wire protocol (sketch)

JSON-RPC 2.0 over Unix stream socket. NDJSON lines; daemon pushes **notifications**.

### Methods (client → daemon)

#### Project / filesystem

| Method | Purpose |
| ------ | ------- |
| `get_status` | Daemon health, queues, uptime |
| `list_projects` | All projects from DB |
| `activate_project` | `{ project_path }` — load from DB, enqueue filesystem scan if stale |
| `scan_project` | Force filesystem rescan |
| `scan_file` | `{ project_path, file_path }` — mtime check + filesystem + semantic if needed |
| `get_project_snapshot` | `{ project_path, since_revision? }` — tree summary for UI hydrate |
| `file_saved` | `{ file_path, mtime }` — user saved in editor; daemon reconciles + enqueues semantic scan |

#### Semantic index

| Method | Purpose |
| ------ | ------- |
| `index_project` | Enqueue all stale files for vector indexing |
| `index_file` | Enqueue one file |
| `pause_index` / `resume_index` | Toggle semantic queue (banner click) |

#### Lifecycle

| Method | Purpose |
| ------ | ------- |
| `shutdown` | Graceful stop (optional — daemon may survive UI quit) |

### Notifications (daemon → client)

| Event | Payload | UI action |
| ----- | ------- | --------- |
| `filesystem_scan_update` | `{ project_path, current_path, progress? }` | Optional progress UI |
| `filesystem_scan_done` | `{ project_path }` | Refresh tree |
| `scan_update` | `{ queue_size, current_file }` | `VectorScanBanner` |
| `project_changed` | `{ project_path, revision }` | Debounced `load_files_from_db` / snapshot fetch |
| `database_saved` | `{}` | Debounced tree reload (plan 2.10.2) |
| `indexer_state` | `{ state }` | Banner / status icon |
| `daemon_ready` | `{ protocol: 1 }` | Client handshake |

**💩** `project_changed.revision` monotonic counter replaces polling `needs_reload()` across processes.

## Unified queue (filesystem + semantic)

**💩** Today filesystem scan and semantic scan are separate pipelines that block each other (`pm.scanning` vs `BackgroundScan` wait loop). In the daemon, **one coordinator** owns ordering:

1. **Filesystem scan first** for a project/path (discover mtimes, new/deleted files)
2. **Semantic index** for files where `last_vector_scan < mtime`
3. Fair scheduling across projects (**⏳** open question: single global queue vs per-project)

No cross-process `scanning` map — the UI never walks the filesystem.

## Lifecycle

| Mode | Behaviour |
| ---- | --------- |
| **On demand** | UI starts `ollmfilesd` if socket missing; daemon persists after UI exit |
| **User service** | Optional systemd user unit — scan/index at login |
| **Single instance** | Second UI connects as RPC client; CLI one-shot same socket |

**💩** Default: daemon **survives UI quit** — overnight indexing and filesystem sync continue.

## Phased delivery

### Phase 0 — Prerequisites

- **⏳** SQLite WAL migration (shared on-disk `files.sqlite`)
- **⏳** In-process quick wins until daemon lands: coalesced backup (**✅** done), pause/resume (2.10.2), LLM skip rules
- **⏳** Baseline metrics: scan duration, index files/hour, UI stall count

### Phase 1 — Daemon skeleton + RPC shell

- Meson target `ollmfilesd`
- Unix socket + JSON-RPC: `get_status`, `shutdown`, handshake
- `ProjectManager` server instance inside daemon
- Manual RPC test client

### Phase 2 — Filesystem scan in daemon

- Move `activate_project` / `read_dir` orchestration to daemon
- RPC: `activate_project`, `scan_project`, `filesystem_scan_*` notifications
- UI: `ProjectManagerClient` — no local `read_dir`
- **Removes main-thread folder walk** — primary UI responsiveness win

### Phase 3 — Semantic index in daemon

- Move `BackgroundScan` / `Indexer` queue into daemon (same process as Phase 2 — no second daemon)
- RPC: `index_project`, `index_file`, pause/resume
- Wire `VectorScanBanner`, `file_saved` → daemon
- Delete in-process `BackgroundScan`

### Phase 4 — UI integration + tool proxy

- Tools/agents use `ProjectManagerClient`
- `project_changed` / debounced snapshot reload
- `--disable-indexer` disables semantic RPC; **💩** consider `--disable-files-daemon` for offline/test mode with read-only DB

### Phase 5 — File watcher

- `Gio.FileMonitor` in daemon (plan 4.2.3)
- Incremental `scan_file` on external changes — no full `read_dir` on every event

### Phase 6 — Hardening + cleanup

- PID/zombie handling, systemd unit, protocol doc
- Remove duplicate code paths; `oc-vector-index` CLI uses same daemon or batch mode
- Delete server-side code from UI binary where dead

## Acceptance criteria (v1 = through Phase 4)

- Opening a project sends `activate_project` RPC; **no** `read_dir` on UI main thread.
- Filesystem + semantic progress visible in UI via notifications.
- Saving a file in editor: local disk write + `file_saved` RPC; daemon re-indexes.
- Closing UI does not stop daemon scan/index work.
- WAL SQLite: no cross-process in-memory DB.
- Daemon crash does not kill `ollmchat`; UI shows disconnected + restart offer.
- `ollmchat-cli` can use same daemon or documented fallback.

## Summary

**🔷** The right split is a **file service daemon**, not a semantic-index sidecar: filesystem scan, DB sync, and vector indexing all belong in a headless process that can run 24/7.

**🔷** The GTK app becomes an RPC client with local editor buffers — it displays project state and sends commands; it does not walk directories or run `BackgroundScan`.

**⏳** LLM-efficiency and FAISS read-while-write remain separate optimizations, but moving **all** heavy file work off the main thread is the architectural fix for UI sluggishness and scan/index coupling.
