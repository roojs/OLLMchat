# 6.3 Reference links: work around LLM using relative paths

**Purpose:** The prompts tell the LLM to use **absolute paths only** for file references, but the LLM often outputs **relative paths** anyway (e.g. `src/main.vala`, `docs/README.md`). Rather than keep failing validation or resolving content incorrectly, we **work around** this: the code **secretly accepts** relative paths, resolves them against the active project root, and proceeds. We do **not** change the prompt to encourage relative paths; we just make the system tolerate them so we can move forward.

**Checklist (see `.cursor/rules/CODING_STANDARDS.md`):** Nullable types avoided (use empty string); no redundant null checks; no `??` (use ternary); `this.` and `GLib.` prefix; inline braces for control flow; no one-use temporaries; link object holds parsed state so we branch once.

---

## 1. Problem

- **Current rule:** Prompts say: use **absolute path** for file references; **do not use relative paths**. Validation in `Details.validate_references()` only accepts `#anchor`, `http(s)://` URLs, or **absolute** file paths (and for absolute paths, checks that the file exists).
- **Observed:** The LLM still frequently outputs relative paths. That causes validation errors ("Invalid reference target … Use only: #anchor, http(s) URL, or absolute file path") or, when validation is bypassed, wrong behaviour (relative path treated as anchor, so no file content is injected).
- **Desired behaviour:** **Secretly accept** relative paths in code: at validation time, resolve them against the project root and check the resolved file exists; at usage time, resolve the same way and inject content. Prompts can stay as-is (we still say "use absolute path"); the implementation just handles relative paths when they appear.

---

## 2. Current behaviour

### 2.1 Validation (`Details.validate_references()`)

- **Location:** `liboccoder/Task/Details.vala`; called from `ResultParser.parse_task_list()` after `fill_names()`.
- **Logic:** For each reference `href`:
  - `#anchor` → task-output and document anchors validated (including section ordering per 6.2).
  - `http://` or `https://` → accepted (TODO validate later).
  - **Absolute path** (`GLib.Path.is_absolute(href)`) → `GLib.File.new_for_path(href).query_exists()`; if not, append "file does not exist".
  - **Anything else** → error: "Use only: #anchor, http(s) URL, or absolute file path (must exist)."
- So relative paths are **rejected** today.

### 2.2 Usage (`Details.reference_contents()`)

- **Location:** `liboccoder/Task/Details.vala`; used when building refinement and executor prompts.
- **Logic:** For each reference:
  - If `link.href` is **not** absolute → treated as anchor (or project-description); content from `runner.reference_content(link.href)`.
  - If **absolute** → `project_manager.get_file_from_active_project(link.href)` (or `File.new_fake` if null); then `header_file(..., found)`.
- **Project manager:** `get_file_from_active_project(string file_path)` looks up `active_project.project_files.child_map.get(file_path)`. The map is keyed by **full absolute path** of files in the project (see `ProjectFiles` / `update_from_recursive`). So lookup expects an absolute path that matches a known project file path.
- So relative paths are never resolved to absolute here; they fall into the "non-absolute" branch and are wrongly treated as anchors (e.g. `runner.reference_content("src/main.vala")`), which will not return file content.

### 2.3 Prompts

- **task_creation_initial.md** (and **task_list_iteration.md**): "Path: absolute path plus `#anchor`. Do **not** use relative paths."

## 3. Proposed solution

### 3.1 Behaviour (workaround)

- **Accept in code:** When we see a reference that is not `#anchor`, not `http(s)://`, and not an absolute path, **treat it as a relative file path** and resolve it against the active project root. We do not change what we tell the LLM; we just accept relative paths when they appear.
- **Validation:** For that relative file path:
  - Require `runner.sr_factory.project_manager.active_project != null` (otherwise we cannot resolve; treat as error or skip path validation and fail at usage).
  - Resolve: `resolved = GLib.Path.build_filename(active_project.path, href)` (or a small helper that normalises `..` and `.` if desired).
  - Ensure the resolved path is **under** the project root (no escaping via `..`).
  - Check that the resolved path exists and is a file (e.g. `GLib.File.new_for_path(resolved).query_exists()`).
- **Path vs anchor:** If `href` contains `#` (e.g. `path/to/file#section`), treat the **path part** as the file path and the rest as anchor; validate only the path part. (Anchors are for future use, e.g. GFM/AST section injection; validation need not validate the anchor.)
- **Error messages:** Keep existing style; for invalid relative path use e.g. "Invalid reference target \"…\": file does not exist (resolved from project folder)." or "… path is outside project folder."

### 3.2 Usage (reference_contents and lookup)

- **Resolve relative to absolute:** Before calling `get_file_from_active_project`, if the href is a file path (absolute or relative) and contains `#`, split into path and fragment. If the path is relative, resolve: `resolved_path = GLib.Path.build_filename(active_project.path, path)` (and optionally normalise). If the path is already absolute, use as-is (or ensure it’s under project for consistency).
- **Lookup:** Call `get_file_from_active_project(resolved_path)` with the **path only** (no `#anchor`). The project file map is keyed by full path; including `#` would prevent a match. So we must strip the fragment for lookup in all cases (current code may already pass href with `#` and get null; this change makes resolution explicit).
- **Fake file:** If lookup returns null, `File.new_fake(..., resolved_path)` so that paths are stored consistently (absolute).

### 3.4 Edge cases

- **No active project:** If `active_project == null`, relative file paths cannot be resolved. Options: (a) treat relative path as invalid in validation and error; (b) accept in validation but fail at usage. Prefer (a) so the task list is rejected early with a clear message.
- **Path traversal:** Reject `../` that escapes the project root (after resolution, check `resolved.has_prefix(project_root + "/")` or equivalent).
- **Anchor-only in path:** If href is `#section` it’s already handled as anchor. If href is `path#anchor`, path may be relative; resolve path only.
- **Empty path:** Not an error. An empty path (after stripping fragment) means the reference is **anchor-only** (e.g. `#project-description`, `#task-1-results`). Those are valid and are already handled by the anchor branch; we simply never call file resolution for them. With a structured link object (scheme / path / hash), empty path is just "this link is an anchor".

### 3.5 Link object: add scheme, path, hash, is_relative

Make the parsed structure part of the link so we branch on it once and avoid re-parsing href:

- **Where:** `Markdown.Document.Format` (libocmarkdown) currently has `href` and `title`. Add:
  - **scheme** — URI scheme: `"file"` for file refs and anchors (default), or `"http"`, `"https"`, etc. for URLs. We use `"file"` by default so we never use `""` as scheme.
  - **path** — path part of the reference (before `#`). For `#section` this is `""`; for `path/to/file` or `path/to/file#section` this is `path/to/file`; for URLs the path component if needed for resolution.
  - **hash** — fragment after `#`, or `""` when there is no fragment.
  - **is_relative** — true when scheme is `"file"` and the path is not absolute (e.g. `src/main.vala`). False for anchor-only (path empty), URLs, and absolute paths.

- **Method `abspath(string to_path)`:** Returns an absolute file path for this link **only when scheme is `"file"`**. If scheme is not `"file"` (e.g. `http`, `https`), returns `this.path` as-is. When scheme is `"file"`: if `path` is already absolute, return `path`; if relative, return the merge of `to_path` and `path` via `GLib.File.resolve_relative_path`. So callers can do `link.abspath(project.path)` for file refs.

- **When to set:** Parse `href` once when the link is created. For `#anchor`, set scheme = `"file"`, path = `""`, hash = `anchor`, is_relative = false. For `path#anchor` or `path`, split on first `#`; path_part gets scheme via `Uri.parse_scheme` and if null use `"file"`; path = path_part, hash = rest; **is_relative** = (scheme == `"file"`) && !`GLib.Path.is_absolute(path)`. We never use `""` as scheme; default is `"file"`.

- **Use in task flow:** Branch on `link.scheme` and `link.path`. Anchor-only: `link.path == ""` → handle as anchor. URL: `link.scheme == "http"` or `"https"` → `reference_content(link.href)`. File: `link.scheme == "file"` and `link.path != ""` → use `link.abspath(project.path)`; then validate / fetch content. Empty path is never an error; it denotes a valid anchor link.

## 4. Implementation notes

### 4.1 Resolution: use link object and Format.abspath (no separate helper)

**Redundant:** A separate `resolve_file_href()` (or similar) in Details is no longer needed. The link object (`Format`) has scheme, path, hash, is_relative and **abspath(string to_path)**. In `validate_references()` and `reference_contents()`, branch on `link.path` / `link.scheme`; for file refs call `link.abspath(project.path)` to get the resolved absolute path. No duplicate parsing or resolution logic in Details.

### 4.4 Using GLib for scheme detection and path/URL resolution

GLib provides APIs that can replace ad-hoc string checks and path logic:

- **Scheme detection:** `GLib.Uri.parse_scheme(uri)` (since 2.16) returns the scheme or `null` if there is no scheme. We use `"file"` as the default (for anchors and bare paths); when parse_scheme returns null we set scheme = `"file"`. So we never use `""` as scheme. URLs (http/https) are handed off to `reference_content`; anchors (`#...`) are detected by path == "" and have scheme = `"file"`.
- **“Under project root” check:** `GLib.File.has_prefix(File prefix)` returns true if the file is under the given prefix (or equal for some implementations; Valadoc says a file is not a prefix of itself, so use `resolved_file.has_prefix(base_file) || resolved_file.equal(base_file)` if we want to allow the project root itself). This replaces manual string prefix comparison and is more robust (path normalization, separators).
- **Relative path/URL resolution:** For relative paths we already use `File.resolve_relative_path(relative_path)`, which is the right API. Alternatively, **URL-style** resolution can be done with `GLib.Uri.resolve_relative(base_uri_string, uri_string, UriFlags.NONE)` (since 2.66): build a base file URI with `File.new_for_path(project.path).get_uri()`, resolve the path part (e.g. `docs/readme.md`) against it, then `File.new_for_uri(resolved_uri).get_path()` to get the path. That unifies “relative reference” handling with standard URI resolution; for our case `File.resolve_relative_path` is simpler and sufficient.

Recommended: use **Uri.parse_scheme()** for URL vs file detection, and **File.has_prefix()** (and optionally **equal()**) for the “under project” check; keep **File.resolve_relative_path()** for resolving relative paths.

Replace `on_a` so that when creating the LINK Format we also set scheme, path, hash, is_relative from href:

**Remove:**

```vala
		public override void on_a(bool is_start, string href, string title, bool is_reference)
		{
			this.on_inline(is_start ? new Format(FormatType.LINK) { 
				href = href, title = title, is_reference = is_reference } : null);
		}
```

**Replace with:**

```vala
		public override void on_a(bool is_start, string href, string title, bool is_reference)
		{
			if (!is_start) {
				this.on_inline(null);
				return;
			}
			var link_fmt = new Format(FormatType.LINK) {
				href = href,
				title = title,
				is_reference = is_reference
			};
			if (href.has_prefix("#")) {
				link_fmt.scheme = "file";
				link_fmt.path = "";
				link_fmt.hash = href.substring(1);
				link_fmt.is_relative = false;
				this.on_inline(link_fmt);
				return;
			}
			var hash_idx = href.index_of_char('#');
			var path_part = hash_idx >= 0 ? href.substring(0, hash_idx) : href;
			link_fmt.hash = hash_idx >= 0 ? href.substring(hash_idx + 1) : "";
			path_part = path_part.strip();
			var scheme = GLib.Uri.parse_scheme(path_part);
			link_fmt.scheme = scheme != null ? scheme : "file";
			link_fmt.path = path_part;
			link_fmt.is_relative = (link_fmt.scheme == "file") && !GLib.Path.is_absolute(path_part);
			this.on_inline(link_fmt);
		}
```

Note: `scheme` is used twice (assign to link_fmt.scheme and in is_relative condition) so the local is justified.

**Method `abspath(string to_path)`** on `Format` — only for file refs (scheme `"file"`):

```vala
/**
 * Absolute path for this link when it is a file reference (scheme "file").
 * Returns path if already absolute, otherwise resolves path relative to to_path.
 * When scheme is not "file" or path is empty, returns this.path as-is.
 */
public string abspath(string to_path)
{
	if (this.scheme != "file" || this.path == "" || GLib.Path.is_absolute(this.path)) {
		return this.path;
	}
	var base_file = GLib.File.new_for_path(to_path);
	var resolved = base_file.resolve_relative_path(this.path);
	var result = resolved.get_path();
	return result != null ? result : "";
}
```

Callers can use `link.abspath(project.path)` to get the resolved path for file refs; for non-file refs the method returns `this.path` as-is and the caller handles anchor/URL separately.

	{
		foreach (var link in this.reference_targets) {
			var href = link.href;
			if (link.path == "") {
				var anchor = link.hash;
				if (anchor.has_suffix("-results")) {
					var name_slug = anchor.substring(0, anchor.length - "-results".length);
					if (name_slug == "" || !this.runner.task_list.slugs.has_key(name_slug)) {

					}

					continue;
				}

					continue;
				}

				continue;
			}
			if (link.scheme == "http" || link.scheme == "https") {
				continue;
			}
			if (link.path != "" && link.scheme == "file") {

				string resolved_path;

						continue;

				} else {
					resolved_path = link.path;
				}

				}
				var resolved_file = GLib.File.new_for_path(resolved_path);
