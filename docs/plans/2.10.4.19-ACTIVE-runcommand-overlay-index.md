## Purpose

- **🔷** **`RunCommandV2`** must **not** call **`fetch_files`** before bubble exec — whole-project index prefetch is wrong and heavy.
- **🔷** Overlay **`Scan`** still needs to know if a path is **already indexed** (created vs modified vs removed).
- **🔷** Fix with **small, targeted** lookups — **overlay paths only** — **🚫** not bulk **`fetch_files`**.

---

## Problem

**Today in `RunCommandV2/Request.vala`:**

```vala
yield project.fetch_files();
```

- Return value **discarded**.
- Side effect: fills **`file_cache`** from every row returned.
- Shipping never did this — client already had **`project_files`** in RAM.

**Why it was added:** **`FileVerification.indexed_file_type`** reads **`file_cache`** only (sync). Without cache, overlay scan sees **`UNKNOWN`** for every path.

**🚫** Bulk **`fetch_files`** is the wrong fix — especially once dropdown uses paged list ([**B-10**](2.10.4.9-ACTIVE-v2-caller-cutover.md)).

---

## Agreed direction (🔷)

1. **Remove** prefetch from **`liboctools/RunCommandV2/Request.vala`**.
2. **Fix overlay classification** without loading the whole project:
   - **`Scan`** only touches paths under the bubble overlay upper layer — **O(changed paths)**, not **O(project)**.
   - On cache miss for an overlay path: **`yield project.fetch_file(path)`** (or equivalent exists check) **before** created/modified/removed — **only paths the scan visits**.
   - **`removed()`** already falls back to **`fetch_file`** on cache miss.
3. **🚫** no per-path RPC storm — batch or cache results **within one bubble run**; **🚫** not one RPC per file in the project.

**⏳** Optional later: daemon **`File.exists_in_index(path)`** or small batch RPC — only if **`fetch_file`** per overlay path is too heavy in practice.
