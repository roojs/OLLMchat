		{
			if (!this.idx_to_widget.has_key(idx)) {

					"scroll_to_idx missing idx=%d registered=%u",
					idx,

### Issue 7 — CLI replay: are ProgressItem ids right?

- **Question:** On CLI replay, does each **`ProgressItem`** get the **`msg_idx`** we expect, or are we only seeing “every tool has the same id” because **`Details.msg_idx`** was set wrong (replay uses **`response.message.idx`**, not the transcript **`msg.idx`**)?
- **Same id on every tool is normal** if **`Details.msg_idx`** is one value — **`build_run_queue`** copies **`Details.msg_idx`** onto each **`Tool`**. The bug is **wrong shared parent id**, not “tools should differ.”
- **Harness:**

```vala
		yield this.runner.replay(this.session.messages);
```

- **How to read it (no row dump):**
  - Use existing **`GLib.debug`** in **`ReplayChat.send`** (`replay transcript … msg.idx=` vs `replay response.message.idx=`) and **`Details`** refine/post_exec (`refine msg_idx=`, `post_exec msg_idx=`).
  - **Expected:** for that step, **`Details.msg_idx`** should match the **transcript** row **`msg.idx`** returned by **`ReplayChat`** for that **`send()`**, not the synthetic **`response.message.idx`**.
  - Confirm **`Tool.msg_idx`** equals **`Details.msg_idx`** in the debugger at **`build_run_queue`** if needed — one Detail is enough; no loop over all progress rows.
