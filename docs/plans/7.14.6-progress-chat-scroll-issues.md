# 7.14.6 — Progress tree → chat scroll: separate issues

- One issue per slice; order: validation → data → UI chrome → scrolling
- Avoid mixing fixes in a single large change

## Issue 1 — `msg_idx` often `-1`

- Symptom
  - Debug: `select_row scroll msg_idx=-1` for runner rows, nested rows, replay
- Spec (golden — do not reinterpret)
  - `docs/plans/7.14.6-continue-replay-as-new.md`
    - `ProgressItem.msg_idx` = last transcript `Message.idx` for that row’s step
    - Mutable as the step gains messages
    - Continue-from-here: cut after that message; pairs with `scroll_to_idx`
    - See `ProgressList` / `Runner` §§6–13, `session.add_message`, `idx`
  - `docs/plans/7.14.6-progress-tree-click-scroll.md`
    - `ProgressItem.msg_idx` ↔ `Message.idx` ↔ `ChatView.scroll_to_idx`
    - Scroll only when `msg_idx >= 0`; `< 0` → no scroll
- Cause (code vs spec)
  - `ProgressList` uses last session message for `Details.msg_idx` — valid only if every line gets `Message.idx` via `session.add_message` (not raw `messages.add`)
  - Some `ProgressRunner` `on_replay` branches skip `msg_idx = m.idx` where §13 applies
  - `Tool` rows may get no `msg_idx` after `Details.build_run_queue` — see §9 / `ProgressItem`
  - Any path that skips `session.add_message` breaks `idx` and `msg_idx` (Dependencies in continue-replay doc)
- Plan
  - Audit `Runner`, `ProgressList`, `Details`, `Tool` vs §8–§13 and Dependencies
  - Fix missing `msg_idx` and wrong `add_message` usage; change golden docs only if product rule changes
  - Replay: every `ProgressRunner` from `on_replay` matches §13 (`msg_idx = m.idx` on driving `Message`)
  - Tools: `Tool` gets `msg_idx` per §9 when row is navigable
- Acceptance
  - Navigable rows: `msg_idx >= 0` matches session `Message.idx`
  - Unset only where spec allows
  - `Window.scroll_to_message` keeps `idx < 0` guard per progress-tree doc

## Issue 2 — `scroll_to_idx` / `registered=0`

- Symptom
  - Log: `scroll_to_idx missing idx=… registered=0`
  - Selection works; no widget for that `Message.idx`
- Cause
  - Transcript lives in `MarkdownGtk.Render` (buffers + boxes)
  - `idx_to_widget` helps only if something registers per displayed `idx`
  - No registration → `scroll_to_idx` no-ops
- Spec
  - `7.14.6-continue-replay-as-new.md` §14
    - Public `idx_to_widget`
    - `set(m.idx, w)` when `m.idx >= 0` at render sites
    - `clear()` clears map
    - `add_widget_frame` / `widgets` strong refs unchanged
- Plan
  - Register anchor widgets at render sites per §14
  - Clones for `append_complete_assistant_message` / display must copy `m.idx` from session `Message`
- Acceptance
  - After session load, map has entries for rendered indices
  - `scroll_to_idx` no-ops for indices that never map (waiting / chrome) per §14 discussion

## Issue 3 — Pick hits column chrome, no `TreeListRow`

- Symptom
  - `pick type=GtkColumnViewCellWidget`; walk ends at `ColumnView` without row
  - Or chain lacks `Gtk.TreeExpander` / stage `progress-row`
- Cause
  - `pick()` returns `GtkColumnViewCellWidget` / internal wrappers
  - Parent-only walk may never reach expander or stage label
- Plan (choose one track)
  - Option A: subtree DFS from picked widget → `Gtk.TreeExpander` → `get_list_row()`; or stage label `progress-row`
  - Option B: GTK API mapping point → `TreeListRow` without relying on widget tree (verify per GTK version)
- Acceptance
  - Title padding, stage column, expander resolve to same `TreeListRow` where intended

## Issue 4 — Title column: expander vs `set_data`

- Symptom / hygiene
  - Duplicating `progress-row` on many widgets drifts from truth
- Plan
  - Title: `Gtk.TreeExpander` — `set_list_row` / `get_list_row()`
  - Stage: no expander — keep `progress-row` on stage `Gtk.Label` only
- Acceptance
  - Title hit-testing uses `get_list_row()` when expander is in chain
  - Stage uses label `progress-row`

## Issue 5 — Viewport scroll vs focus only

- Symptom
  - With an anchor, `grab_focus()` may not move `Gtk.ScrolledWindow` enough
- Cause
  - Focus ≠ scroll; need `vadjustment` or viewport `scroll_to`
- Plan
  - After `idx_to_widget[idx]`, scroll so anchor’s vertical bounds intersect viewport
    - e.g. `compute_bounds` vs scroll child + `vadjustment.value`, or `GtkViewport.scroll_to` if stack supports it
  - Respect `programmatic_scroll_in_progress` / autoscroll pause (product: pause vs force)
- Acceptance
  - Progress selection puts matching transcript on screen, not only focus

## Suggested order

- 1 — Issue 1 (`msg_idx`)
  - No benefit scrolling if `scroll_to_message` gets `-1`
- 2 — Issue 2 (`idx_to_widget`, `idx`)
  - Gives `scroll_to_idx` targets
- 3 — Issue 5 (viewport scroll)
  - Makes jump visible once indices work
- 4 — Issue 3 (pick / chrome)
  - Widens hit area after scroll works
- 5 — Issue 4 (expander API)
  - Cleanup / consistency
