# 7.14.6 — Progress tree → chat scroll: separate issues and plan

Treat each issue below as its **own** slice of work. Fixing them in one large change led to tangled behaviour; address **validation → data → UI chrome → scrolling** in order.

---

## Issue 1 — `msg_idx` is often `-1` on progress rows

**Symptom:** Debug shows `select_row scroll msg_idx=-1` for many rows (runner rows, nested rows, replay).

**Cause (layers):**

1. **`ProgressList`** sets `Details.msg_idx` from the **last** session message at append time — not necessarily the message that “owns” that row for navigation.
2. **`ProgressRunner`** instances created during **replay** (e.g. LIST phase) historically omitted `msg_idx` while other phases set it from the replay `Message`.
3. **`Tool`** rows under **`Details`** did not inherit **`Details.msg_idx`** when the tool queue was built, so child rows stayed `-1`.

**Plan:**

1. Decide the **single rule**: each visible progress row should carry the **`Message.idx`** of the transcript line users expect when jumping from that row (often the latest UI-visible agent/task message for that slice — document the rule).
2. **Replay:** When constructing **`ProgressRunner`** / **`Details`** / **`Tool`** from replay, assign **`msg_idx`** from the **`Message`** that drives that slice (`content-stream`, `agent-stage`, etc.) consistently with live runs.
3. **Live:** When **`Details`** / **`Tool`** are appended from **`ProgressList`**, set **`msg_idx`** using that rule, not only “last message in session”.
4. **Acceptance:** Clicking any row that should map to chat shows **`msg_idx >= 0`** in debug where product intent says it should; rows that are intentionally non-navigable stay `-1` and **do not** call **`scroll_to_message`**.

---

## Issue 2 — `ChatView.scroll_to_idx` has nothing to scroll to (`registered=0`)

**Symptom:** Logs like `scroll_to_idx missing idx=%90 registered=0` — selection works but **no widget** is registered for that **`Message.idx`**.

**Cause:**

The transcript is rendered mostly inside **`MarkdownGtk.Render`** (text buffers + boxes). **`idx_to_widget`** is only useful if something **registers** a stable **`Gtk.Widget`** (or scroll target) **when each indexed message is displayed**. Until that exists, **`scroll_to_idx`** will always no-op for real sessions.

**Plan:**

1. **Define registration site(s):** For each UI-visible message with **`idx >= 0`**, when the chat first **renders** that message (history restore, **`message_created`**, tool line, etc.), record **`idx_to_widget[idx] = anchor`**.
2. **Preserve `idx` through the UI pipeline:** Any path that builds a **new** **`Message`** for display (e.g. cloning `"assistant"` from `"ui"`) must copy **`m.idx`** from the session message, or **`scroll_to_idx`** will look up the wrong key.
3. **Clear on `ChatView.clear()`:** Registration map must reset with the transcript.
4. **Acceptance:** After loading a session, **`registered`** is non-zero for indices that were rendered; **`scroll_to_idx(idx)`** finds the key for a **`msg_idx`** that came from Issue 1.

## Issue 3 — Clicks on column chrome miss `TreeListRow` (CellWidget / empty padding)

**Symptom:** `pick type=GtkColumnViewCellWidget` then **walk** stops at **`ColumnView`** with no row; or hits internal widgets without **`Gtk.TreeExpander`** / **`progress-row`** on the picked widget.

**Cause:**

**`pick()`** returns internal wrappers (**`GtkColumnViewCellWidget`**, row widgets). Walking **only parents** may never reach the **`Gtk.TreeExpander`** or the stage **`Gtk.Label`** that holds row data.

**Plan (pick one approach and stick to it):**

1. **Subtree search:** From the picked widget, **depth-first** search descendants for **`Gtk.TreeExpander`** and use **`get_list_row()`**, or for the stage label’s **`progress-row`** data (same as title semantics).
2. **Alternatively:** Hit-test **row + column** via selection/list APIs if GTK exposes a stable way to map coordinates → **`TreeListRow`** without relying on widget tree (evaluate API per GTK version).
3. **Acceptance:** Clicking **title padding**, **stage column**, and **expander** all resolve to the same **`TreeListRow`** as far as product intent allows.

---

## Issue 4 — Title column: use GTK row binding, not duplicate `set_data` where avoidable

**Symptom / hygiene:** Copying **`progress-row`** onto multiple widgets is easy to get out of sync.

**Plan:**

- **`Gtk.TreeExpander`:** Prefer **`set_list_row` / `get_list_row()`** for the title column; keep **`progress-row`** only on widgets that have **no** expander (e.g. stage **`Gtk.Label`**).
- **Acceptance:** Title picks resolve via **`get_list_row()`** where **`Gtk.TreeExpander`** is in the chain; stage column still resolves via **`progress-row`** on the label.

---

## Issue 5 — Scrolling the transcript, not only focus

**Symptom:** Even with a registered anchor, **`grab_focus()`** may not move the **`Gtk.ScrolledWindow`** viewport enough for long transcripts.

**Cause:** Focus and scroll are separate; **`vadjustment`** must be updated or **`scroll_to`**-style API used so the anchor is **visible**.

**Plan:**

1. After resolving **`idx_to_widget[idx]`**, scroll so the anchor’s **vertical bounds** intersect the viewport (e.g. **`compute_bounds`** relative to the scroll child + **`vadjustment.value`**, or **`GtkViewport.scroll_to`** if available on the stack).
2. Respect **`programmatic_scroll_in_progress`** / autoscroll pause rules so scroll-to-message does not fight user “scrolled up” behaviour (product decision: pause vs force scroll).
3. **Acceptance:** Selecting a progress row scrolls the chat so the corresponding message is **on screen**, not only focused.

---

## Suggested order of work

| Order | Issue | Why first |
|------|--------|-----------|
| 1 | Issue 1 (`msg_idx`) | No point scrolling if **`scroll_to_message`** receives **`-1`**. |
| 2 | Issue 2 (`idx_to_widget` + **`idx`**) | Ensures **`scroll_to_idx`** has targets. |
| 3 | Issue 5 (viewport scroll) | Makes navigation visible once indices resolve. |
| 4 | Issue 3 (pick / chrome) | Improves hit area once scroll works. |
| 5 | Issue 4 (expander API) | Cleanup / consistency. |
