## Purpose

- Wire **`ProgressItem.msg_idx`** → **`Message.idx`** → **`ChatView.scroll_to_idx`** so progress tree navigation can scroll the transcript
- Fix **`registered=0`** by **`idx_to_widget`** at explicit transcript boundaries only

| **`idx_to_widget`** in **`ChatView`** + **targeted** **`idx`** on **`ChatWidget.on_message_created`** (**`content-stream`** / **`content-non-stream`** only — §4) | **`append_assistant_chunk`** streaming anchor — separate plan |

## Acceptance criteria

- **`msg_idx`:** navigable progress rows use **`msg_idx >= 0`** when session **`Message.idx`** exists per **`7.14.6-continue-replay-as-new.md`**; **`Tool.msg_idx`** comes from **`Details.build_run_queue`** (**`this.msg_idx`** at **`Tool`** creation), not a **`ProgressList`** post-assign loop
- **`scroll_to_idx`:** after restore, **`idx_to_widget.has_key(idx)`** for assistant **`idx`** rendered via **`append_complete_assistant_message`** (**§5**) once §14 registration lands; **`append_tool_message`** **`idx`** (**§6**) **⏳** until an approved buffer/widget strategy exists
- **`scroll_to_message`:** still no-ops for **`idx < 0`** (**`Window`** guard)

### §14 alignment — **`idx_to_widget`** maps **existing** rows (**no** synthetic anchor **`Gtk.Box`** per **`idx`**)

- **🚫** **Rejected:** Appending an extra empty **`Gtk.Box`** (**`height_request = 1`**) per message solely so **`scroll_to_idx`** has a widget. That contradicts **`docs/plans/7.14.6-continue-replay-as-new.md`** §14: **`idx_to_widget.set(m.idx, w)`** refers to **real** transcript UI (**strong ref** to the widget that represents that message), not spacer rows.
- **🔷** **Approved direction (plan-only until fences are approved and applied):**
  - **`append_complete_assistant_message`:** Immediately after **`clear_waiting_indicator()`**, record **`tail_before = text_view_box.get_last_child()`**. After **`finalize_assistant_message_direct()`**, if **`message.idx >= 0`**, set **`idx_to_widget[message.idx]`** to **`tail_before == null ? text_view_box.get_first_child() : tail_before.get_next_sibling()`** (first **new** box child added by the markdown renderer for this message). **One** helper; **no** extra **`append`** for scroll-only widgets.
  - **`append_tool_message`:** Inserts into the shared **`Gtk.TextBuffer`** — typically **no** new **`text_view_box`** child per line. **⏳** **`idx_to_widget`** for tool **`idx`** needs a **separate** approved approach (**`Gtk.TextMark`**, **`TextChildAnchor`**, or caller **`add_widget_frame`** + **`idx`**) — **do not** duplicate the §14 empty-box hack here.

### 3. **`Tool.msg_idx`** — canonical assignment (no **`ProgressList`** patch)

**🚫** Do **not** add **`foreach (var t in d.tools()) { t.msg_idx = d.msg_idx; }`** (or similar) after **`d.msg_idx = …`** in **`ProgressList`**. That copies **`Details`** onto **`Tool`** rows after the fact.

**🔷** **`Tool.msg_idx`** is set **only** where each **`Tool`** is constructed — **`Details.build_run_queue`** (**§2**): **`ex.msg_idx = this.msg_idx`**, **`lone.msg_idx = this.msg_idx`**. **`Details.msg_idx`** must already be correct **before** **`build_run_queue`** runs so **`this.msg_idx`** at append time matches the session message.

**ℹ️** If ordering leaves tools built while **`Details.msg_idx`** is still wrong, fix **call order** or re-enter **`build_run_queue`** — do not tail-sync in **`ProgressList`**.
