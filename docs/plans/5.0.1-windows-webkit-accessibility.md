## Purpose

- **🔷** Bring **Windows** up to the same **accessibility** contract as the Linux **5.0** spike: dump tree → markdown, fill/submit, **`press N`**.
- **🔷** Expect work in **`webview2-gtk`** (and/or WebView2 APIs) — Snappr’s Windows path today is browse/CF/cookies, **not** a11y control.
- **🔷** Prove via **`libocwebkit/examples/oc-test-webkit`** on Windows once the backend exists.
- **🔷** When **[`5.0.5`](5.0.5-webkit-download.md)** lands: same **basic download** contract as Linux — `ChatPermission`, save under the Windows user **Downloads** folder, WebView2 download API / progress (not Linux-only).
- **⏳** Do **not** block the Linux Phase 0 spike or Linux chat wiring on this plan.

---

## Why separate from 5.0

- **🔷** **5.0** initial spike is **Linux-only** (WebKitGTK a11y).
- **🔷** Windows needs library/API changes that are out of scope for the first theory-check.
- **ℹ️** Load/CF/cookie patterns from Snappr remain useful later; a11y is the missing piece.

---

## Likely work (backlog)

- **⏳** **🔷** Spike: what WebView2 can expose for name/role/value/URL and invoke (UIA / other).
- **⏳** **🔷** Extend **`webview2-gtk`** so Vala can dump and activate nodes (same markdown contract as Linux).
- **⏳** **🔷** `libocwebkit` Windows backend behind the same **`Browser` / `BrowserStack`** + tool API.
- **⏳** **🔷** Run **`oc-test-webkit`** recipes on Windows (fetch / search / press).
- **⏳** **🔷** CF handoff already partly exists in Snappr on Windows — wire through once a11y works.
- **⏳** **🔷** Browser **download** parity with **[`5.0.5`](5.0.5-webkit-download.md):** `ChatPermission` + Windows user Downloads folder + WebView2 download / progress.
