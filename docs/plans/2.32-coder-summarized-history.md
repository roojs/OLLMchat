# 2.32. Coding Assistant — shared summarized history

Status: proposed

> **Note:** `docs/plans/1.0-summary.md` is not updated until this plan is done
> and archived.

ℹ️ Plan checklist: `docs/guide-to-writing-plans.md` — Checklist for plans.
Proposed Vala code must follow `docs/coding-standards.md`. Plan layout and
implementation workflow: `docs/guide-to-writing-plans.md`.

ℹ️ Builds on **`docs/plans/2.31-just-ask-summary-history.md`** (Chatter agent,
`summary` role, `session_fetch`, summarizer contract).

## Purpose

- 🔷 Share Chatter-style **summarized conversation history** with the original
  **Coding Assistant** (`OLLMcoder.Agent` / `code-assistant`).
  - Same `summary` transcript role, hash links, and background summarizer.
  - Same outbound rule: only messages **since the latest `summary`**, plus
    system prompt.
- 🔷 **Move** `Chatter.Summarizer` to **`OLLMchat.Agent.Summarizer`** so both
  agents compose one shared class.
- 🔷 Add **`create_summary()`** on `Agent.Base` (shared summary-reset scan).
- 🔷 Coder keeps **`AgentFactory.system_message()`** (first round — full rules +
  context) and **`generate_user_prompt()`**.
  - **Second round** (when `active_summary` is set): append
    **`coder_followup.md`** via `Prompt.Template.system_fill()` — same pattern
    as Chatter's `chatter_followup.md`, not hardcoded strings in Vala.
  - **Stop persisting** `system` rows to `session.messages` (outbound only).
- ⏳ **`Chatter.Agent`** calls the Base scan helper (no behaviour change).
- ⏳ Reuse existing **`chatter_summary.md`** as-is for the background
  summarizer (same file Chatter uses today).
- ⏳ Manual verification (see **Verification**).

## Architecture

- **`OLLMchat.Agent.Summarizer`** — moved from `Chatter/`; finds
  **`user_sent_index`** inside **`run()`** (agents do not pass it).
- **`OLLMchat.Agent.SummaryOutbound`** — return type for `create_summary()`.
- **`OLLMchat.Agent.Base.create_summary()`** — returns `SummaryOutbound`.
- **`OLLMchat.Chatter.Factory`** — **unchanged** (`load_prompt()` stays as-is).
- **`OLLMchat.Agent.Factory`** — **unchanged** (no new prompt loaders).
- **`OLLMcoder.Agent`** — spawns **`Summarizer`** at end of `send_async()` (no
  field); second-round system tail from **`coder_followup.md`** (§6).

## Current behaviour

- ℹ️ **`OLLMcoder.Agent.send_async()`** persists a new `system` row each turn
  and sends the **full** API transcript.
- ℹ️ **`Chatter.Summarizer`** lives under `Chatter/` and loads the summary
  prompt via `(Chatter.Factory) this.factory`.load_prompt().
- ℹ️ Plan 2.31 🚫 on `Agent.Base` is **superseded** only for
  `create_summary()` — default `send_async()` unchanged.

## Proposed behaviour

- 🔷 Coder: background **`Summarizer.run()`** after each completed turn (same
  as Chatter).
- 🔷 Coder follow-up: **`create_summary()`** outbound body +
  `system_message()` + **`coder_followup.md`** when `active_summary` is set.
- 🔷 Coder first turn (no summary yet): `system_message()` only — no follow-up
  template (same effective body as today, minus persisted system rows).
- 🚫 No refactors to **`Agent.Factory`** or **`Chatter.Factory`** prompt
  loading.
- 🚫 No hardcoded summary / `session_fetch` prose in Vala — that lives in
  **`coder_followup.md`**.

## Phase tasks

- ⏳ **Phase 1** — Move `Summarizer`, add `create_summary()`, update
  `Chatter.Agent`, `meson.build`.
- ⏳ **Phase 2** — Add **`coder_followup.md`**; update `OLLMcoder.Agent`.
- ⏳ **Phase 3** — Manual verification.

## Verification

- ⏳ **Coder first turn** — full history; no summary in system; summary after
  reply.
- ⏳ **Coder follow-up** — pre-summary messages omitted; system is
  `system_message()` + filled **`coder_followup.md`**; `session_fetch` works.
- ⏳ **Coder system rows** — no new `system` messages in `session.messages`.
- ⏳ **Chatter regression** — unchanged outbound assembly and summarizer.
- ⏳ **Just Ask regression** — full transcript; no summaries.

### 2. `libollmchat/Agent/Base.vala` — `create_summary()`

**Why:** Shared summary-reset scan for Chatter and Coder outbound assembly.

**Where:** `Base` class, after `fill_model()`, before `send_async()`.

**Depends on:** none.

#### Add — struct + method after `fill_model()`, before `send_async()`

**Add** — `SummaryOutbound` struct in `Agent` namespace (above `Base` class
in the same file).

```vala
	/**
	 * API messages since the latest summary boundary, plus that summary row.
	 */
	public struct SummaryOutbound
	{
		public Gee.ArrayList<Message> messages {
			get; set; default = new Gee.ArrayList<Message>();
		}
		public Message? active_summary { get; set; default = null; }
	}
```

**Add** — method on `Base`:

```vala
		/**
		 * Collect API messages since the latest summary boundary.
		 *
		 * Each summary message clears messages and becomes active_summary.
		 * Only user, assistant, and tool roles are collected.
		 *
		 * @return outbound slice for the main chat request
		 */
		protected SummaryOutbound create_summary()
		{
			var result = new SummaryOutbound();
			foreach (var msg in this.session.messages) {
				if (msg.role == "summary") {
					result.messages.clear();
					result.active_summary = msg;
					continue;
				}
				if (msg.role != "user"
					&& msg.role != "assistant"
					&& msg.role != "tool") {
					continue;
				}
				result.messages.add(msg);
			}
			return result;
		}
```
