# 6.6 Write executor: do not treat `next_section` body headings as unexpected top-level sections

**Status:** DONE (implemented in `liboccoder/Task/ResultParser.vala` — `exec_extract`, `next_section` → **`break`** after successful `WriteChange`.)

**Note on numbering:** `docs/plans/done/6.5-DONE-write-plan-fragment-links-and-absolute-paths.md` already uses **6.5** (completed). This plan is the next item in the same **6.x write executor** series.

---

## Purpose

Allow **write_file** executor output that uses `**## Change details`** with `**output_mode: next_section**` to include a **full markdown plan** after the key/value list (`#` title, many `##` sections, etc.) **without** `ResultParser.exec_extract` reporting spurious **“unexpected top-level section”** errors for every heading in that body.

---

## Problem summary

1. For write skills, `exec_extract` walks `document.header_list` and rejects any top-level heading whose slug is not `result-summary` and does not start with `change-details`.
2. The markdown renderer registers **every** heading whose **parent is the root document** in `header_list`.
3. For `**next_section`**, `WriteChange.from_header` takes the file body from the **sibling chain** after the Change details bullet list (`list_block.next()` … `n.next()` until end). That body is **legitimate write payload** (often a plan with many `#` / `##` headings).
4. Those payload headings are **still** root-level blocks in the **same** parsed executor document, so they appear in `header_list` alongside `result-summary` and `change-details`.
5. The validator incorrectly treats them as **extra executor sections** (as if the model had added forbidden `## Status` next to `## Result summary`), and emits one error per slug (`status`, `purpose`, `phase-1-foundation`, …).

This breaks **plan_create**-style output that follows the skill: `**## Result summary` → `## Change details` (list) → plan markdown**.

## Proposed solution (behaviour)

**Only** edit `exec_extract` — **no new methods**, **no** `**Gee.HashSet**`, **no** walking the `**next_section**` body to collect slugs.

**What `next_section` means here:** `**WriteChange.from_header**` already decided where the structured part ends (bullet list + keys) and where the **file content** begins (everything after that list in the raw markdown). That content is **not** more executor scaffolding to validate — it is **opaque payload** for the write tool. So **`exec_extract`** should **stop treating the rest of the document as a sequence of executor sections** once Change details has been parsed successfully and `**output_mode**` is **`next_section`**.

**Mechanics:** After `**ex.writes.add (wc)**`, if `**wc.output_mode.strip ().down () == "next_section"`**, **`break`** out of the **`foreach`** over **`this.document.header_list`**. Do **not** keep iterating and classifying later top-level headings as “unexpected sections” — those slugs are inside the file body, not unknown `##` sections for the runner.

**Why not a flag + `continue`?** Same outcome for the usual shape, but **`break`** matches the contract: **parsing / section discovery for the executor response stops**; we are not “allowlisting” headings one by one.

**Invert** the `**WriteChange**` error branch: `**if (wc.issues != "")**` → append + `**continue**`; then `**ex.writes.add (wc)**`; then **`if (next_section) break;`**.

**Non–`next_section`** Change details (fenced / replace / etc.): do **not** **`break`** — loop continues so today’s rules still apply to any further top-level headings (rare).

**Safe / unchanged:** stray `##` **before** `## Change details` in `**header_list`** order still hits the non–**`change-details`** branch before any **`break`** → still errors. Path 2 unchanged. `**WriteChange.from_header**` still runs **once** for that Change details block.

			foreach (var slug in this.document.header_list) {
				if (slug == "result-summary") {

## Edge cases (optional one-line code comment)

- **Multiple `## Change details` blocks:** **`break`** after the first successful **`next_section`** means a second top-level **`## Change details`** later in **`header_list`** is never visited by this loop. The usual contract is **one** Change details + **`next_section`** + body; multi-write / second CD is out of scope unless product revisits.

## Tests

- **Manual / fixture:** Executor markdown matching **plan_create** shape: Result summary + Change details (`next_section`) + plan with multiple `##` headings → `**exec_extract` returns true** and `**issues`** empty for the section check.
- **Regression:** Stray top-level `##` **before** `## Change details` → still errors (**`break`** not reached). Plan headings **after** Change details → loop has already **exited** on **`break`** → never misclassified.
- **Automated:** If there is or will be a small `ResultParser` / `oc-test` path for executor samples, add one golden string for `next_section` plan output.
