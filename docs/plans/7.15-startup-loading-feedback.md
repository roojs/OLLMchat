# 7.15 Startup loading feedback (spinner vs blank main pane)

**Status:** proposed — analysis and **proposed** changes only (**⏳** no implementation in this task)

**Layout:** `docs/guide-to-writing-plans.md` — **Concrete code proposals**, **Keep** / **Remove** / **Replace with** / **Add**; **Discussion style (emoji prefixes)** for non-fence sections.

**Checklist:** `.cursor/rules/CODING_STANDARDS.md` — **Checklist for all plans**

---

## Purpose

- **⚠️** On startup, the app verifies the LLM connection and loads / verifies models (including **required** tool models and optional **pull**). Today the **CheckingConnectionDialog** spinner appears only during **`config.check_connections()`**. After it closes, **long-running work** continues with **no dedicated loading UI**, while the **main chat area is still empty** until **`initialize_client()`** runs.
- **⏳** Goal: define changes so users **always see clear feedback** (reuse / extend the **existing spinner dialog**, and/or address the **empty `OverlaySplitView` content** window) during the full **`Initialize.run()`** path — without mixing in unrelated refactors.

## Current behaviour

- **ℹ️** **`Window` ctor** builds chrome (**`Adw.ToolbarView`**, header, banners) and **`Adw.OverlaySplitView`**, but **`split_view.content` is assigned only inside `initialize_client()`** when **`ChatWidget`** / **`WindowPane`** are created (`ollmapp/Window.vala` — single assignment to **`this.split_view.content`**).
- **✅** **`Initialize.run()`** shows **`CheckingConnectionDialog`**, yields **`config.check_connections()`** ( **`Call.Models`** per connection, short timeout ), then **closes the dialog** before **`initialize_model()`** and **`ensure_required_models()`** (`ollmapp/Initialize.vala`).
- **⏳** **`initialize_model()`** calls **`ConnectionModels.refresh()`** → **`connection.load_models()`** — can trigger **many** HTTP calls (**list** + **`show`** per uncached model) (`libollmchat/Settings/ConnectionModels.vala`, **`Connection.load_models()`**).
- **⏳** **`ModelUsage.verify_model()`** also calls **`connection_obj.load_models()`** (`libollmchat/Settings/ModelUsage.vala`) — potential **duplicate** work during init; still **no** spinner.
- **⏳** **`ensure_required_models()`** may **`settings_dialog.show_dialog.begin("tools")`** and **`pull_manager.start_pull`** + **`wait_for_pull`** — progress may appear **inside Settings**, not in the **main** window; the **main pane can remain empty** throughout.
- **ℹ️** **Sample logs:** connection attempts, **`Pull failed … Unacceptable TLS certificate`** — failure returns through **`Initialize.run`** to **`show_settings`**; user may still have seen a **long blank** phase **before** the error dialog.

---

## Proposed behaviour

- **⚠️** While **`Initialize.run()`** has not finished successfully, the user should see **either** a **modal loading dialog** (spinner + short **phase** text) **or** a **non-empty** main content placeholder — not a **bare** split view with only chrome.
- **💩** Optional: **phase strings** — e.g. “Checking connections…”, “Loading model list…”, “Preparing required models…”, “Downloading model…” — updated **in place** on one dialog instance to avoid flicker.
- **💩** When opening **Settings** for tools/pull: **close** or **hide** the startup spinner first so **two** modal layers do not fight for focus (**ℹ️** **`Adw.Dialog`** stacking).

---

## Concrete code proposals

**ℹ️** Pick **one** primary approach ( **A** recommended ); combine with **B** only if product wants visible **inline** loading instead of modal.

### 1. `ollmapp/SettingsDialog/CheckingConnectionDialog.vala` — optional phase text API

**Rationale:** Today the label is fixed (“Verifying connection to server…”). Init needs **updatable** copy without rebuilding the widget tree.

#### Keep

```vala
		public CheckingConnectionDialog(Gtk.Window parent)
		{
			Object(
				title: "Checking Connection",
				parent: parent
			);
```

#### Add

- **⏳** Store the **`Gtk.Label`** in a **private field** (e.g. **`status_label`**) so it can be updated after construction.
- **⏳** Add a **public** method **`void set_status(string message)`** (or property with setter) that sets **`status_label.label = message`** — **⚠️** naming is a **proposal**; align with existing style (**`this.`**, docblock if public).

**👎** Do **not** add multiple parallel dialog classes for each phase.

---

### 2. `ollmapp/Initialize.vala` — cover full init loop with one dialog instance

#### Keep

```vala
		public async bool run(OLLMchat.Settings.Config2 config)
		{
			// Loop until both connection and model succeed
			while (true) {
				// Check all connections and get first working one
				var checking_dialog = new SettingsDialog.CheckingConnectionDialog(this.window);
				checking_dialog.show_dialog();
				yield config.check_connections();
				checking_dialog.hide_dialog();
```

#### Replace with (conceptual — verbatim when implementing)

- **⏳** After **`show_dialog()`**, call **`set_status`** (once API exists) for **“Checking connections…”** before **`check_connections()`** (or rely on dialog default for first phase).
- **⏳** **Remove** the **`hide_dialog()`` immediately after **`check_connections()`**.
- **⏳** Before **`yield this.initialize_model(config, working_conn)`**, set status to **“Loading models…”** (or similar).
- **⏳** Before **`yield this.ensure_required_models(config)`**, set status to **“Checking required models…”**.
- **⏳** **Inside `ensure_required_models`**, immediately **before** **`settings_dialog.show_dialog.begin("tools")`**, call **`checking_dialog.hide_dialog()`** (if still visible) so **Settings** is not obscured — **⚠️** requires **`checking_dialog`** to be **passed in** or **`ensure_required_models`** to receive a **`CheckingConnectionDialog?`** / closure — **design choice** for implementer (**ℹ️** avoid **new methods** on **`Initialize`** unless you **inline** the hide at the call site in **`run()`** by splitting **`ensure_required_models`** body with **callbacks** — **💩** user may prefer a **small** parameter over a **large** refactor).

- **⏳** **`hide_dialog()`** only after **`initialize_model`** **and** **`ensure_required_models`** succeed **and** after **`history_manager`** creation / **`ensure_model_usage`** block — i.e. at the **current** successful exit path before **`break`**, or on **continue** / **error** paths with **finally**-style discipline so the dialog never leaks.

**ℹ️** **Edge case:** **`show_settings()`** / **`initialize_model`** failure paths must **close** the spinner so the user is not stuck behind an invisible modal.

### 3. `ollmapp/Window.vala` — optional: inline placeholder (`OverlaySplitView`)

**Alternative B** if modals are rejected.

#### Add (after **`this.split_view = new Adw.OverlaySplitView();`** in ctor)

- **⏳** Create a **`Gtk.Box`** with **`Gtk.Spinner`** + **`Gtk.Label`** (“Starting…”) and assign **`this.split_view.content = placeholder`** **temporarily**.
- **⏳** In **`initialize_client()`**, **before** **`this.split_view.content = this.window_pane`**, no-op if placeholder already replaced — **or** unconditionally set **`window_pane`** (replaces placeholder).

**💩** **Risk:** duplicate visual noise if **`CheckingConnectionDialog`** **also** shows — prefer **either** modal **or** placeholder unless **explicitly** combined (e.g. placeholder **only** when dialog is hidden).

### 4. `ollmapp/SettingsDialog/MainDialog.vala` — consolidation note

- **ℹ️** **`MainDialog`** declares **`checking_connection_dialog`** (`ollmapp/SettingsDialog/MainDialog.vala`) — **Initialize** currently constructs a **separate** **`CheckingConnectionDialog`**. **💩** Optional follow-up: **reuse** one instance from **`MainDialog`** to avoid **two** implementations diverging — **out of scope** unless **7.15** implementer chooses it for consistency.

## Risks and testing notes

- **⚠️** **`Adw.Dialog.present`** is **modal** to the window — keeping it open **too long** blocks interaction with **Settings**; **hide** before **`settings_dialog`** opens for **pull**.
- **ℹ️** **Manual test:** start with **Ollama stopped** (connection failures), **slow** network, **missing** required embedding model (triggers **pull**), and **TLS failure** — confirm **spinner** / **placeholder** appears for **entire** pre-chat phase and disappears on **success** or **error dialog**.
