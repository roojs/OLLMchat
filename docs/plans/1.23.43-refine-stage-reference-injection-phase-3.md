## Coding standards checklist

Copied from **`.cursor/rules/CODING_STANDARDS.md`** — “Checklist for all plans”. If a bullet conflicts with **§ Conventions** in the [parent plan](1.23.40-refine-stage-reference-injection.md), **the standards file wins**.

- **Nullable types**: Design new APIs/properties to avoid nullable types where possible (default objects / flags).
- **Null checks**: Avoid generic null checks; only where the design explicitly requires null (see standards for the narrow exceptions).
- **String interpolation**: No `@"..."` except multi-line usage/help/documentation; use `+` concatenation otherwise.
- **Temporary variables**: No one-use temporaries; no trivial aliases (allowed exception: long property chains **4+** steps).
- **Brace placement**: Line breaks before `{` for namespace/class/method; **inline** `{` for `if`/`for`/`while`/`switch`; never a one-line `if` with body on the same line.
- **`this.` prefix**: Use for instance member access in new/modified Vala.
- **GLib prefix & `using`**: Fully qualify `GLib.*`; do not add `using` for new code.
- **Property initialization**: Prefer `get; set; default = …` (or field defaults), not constructor assignment for those defaults.
- **Line length & breaking**: Break long method calls, concatenations, docblocks, and comments (docblock/comment line length per full standards).
- **StringBuilder**: Avoid `GLib.StringBuilder` unless building strings in loops with **hundreds** of iterations; use `+` and `string.joinv()`.
- **String building in loops**: Never accumulate with `+=` inside a loop; use `string.nfill()` / `replace()`, `string.joinv()`, etc.
- **ArrayList for strings**: Do not use `Gee.ArrayList<string>` only to join; use `string[]` with **`string[] name = {}`** only for empty init (not `new string[0]`).
- **Loops and nesting**: In loops, use **`continue`** to skip cases and keep the body flat; avoid **`else`** and nested **`if`** inside loops.
- **Character looping**: Do not iterate characters unless there is truly no alternative; prefer string methods and `GLib.Regex`.
- **File info try/catch**: Wrap `query_info()`-style calls only when file existence is unknown — not right after a successful read of the same path.
- **Try/catch scope**: Wrap only the minimal expression(s) that throw; keep setup outside `try`.
- **Underscore prefix**: No leading `_` on variables, fields, or properties.
- **`get_*` methods**: Prefer properties or verb-less/action names over `get_foo()` (see standards for exceptions).
- **Docblocks**: Multiline `/** … */` with `@param` / `@return` where relevant; not one-liners for real behavior.
- **Debug / warning messages**: No timestamps or monotonic time in `GLib.debug()` / `GLib.warning()`; no class/method prefixes in the message (file/line come from the log).
- **Gee.HashMap**: Use `.set()` / `.get()` / `.has_key()` / `.unset()` — not `[]`.
- **Gee.ArrayList**: Use `.get()` / `.set()` / `.add()` / `.remove_at()` — not `[]`.
- **Reducing nesting**: Prefer early `return` / `continue`; avoid `else` where flow stays clear; prefer `switch` over long `if` / `else if` chains when editing.
- **SQL table aliases**: Not applicable to this plan’s fences; if you add SQL elsewhere, no table aliases (full names only).
- **Plan Vala fences**: Do **not** use comment-only placeholders (`/* … */` or empty `return "";`) where real logic belongs — either **complete, compilable Vala** in the fence or **prose-only** steps (no fake stub block).

## Proposed code changes — abbreviation metadata (file / section size)

**Goal:** When `MarkdownPhase.REFINEMENT` abbreviates linked content, append **line-count metadata** after the preview so the refiner can judge relevance (see **Task reference contents — abbreviation vs execution** above). Replace the bare **`_(abbreviated)_`** footer with a single readable convention.

**Suggested footer text (markdown, one line):**

```text
_(Abbreviated: first 10 lines shown; 227 more lines omitted; 237 lines total.)_
```

Adjust wording if you prefer shorter copy; keep **total line count** and **omitted count** (or “N more lines”) explicit.

**Where abbreviation happens today**

| Location | Behaviour |
|----------|-----------|
| `liboccoder/Task/Details.vala` — `header_file` | `file.get_contents(10)` when `stage == REFINEMENT`; if `file.get_line_count() > 10`, append `_(abbreviated)_`. |
| `liboccoder/Skill/Runner.vala` — `reference_content` | Task link with hash: `section_md.split("\n")`, first 10 lines; user-request anchor: same. If `lines.length > 10`, append `_(abbreviated)_`. |

**Constant:** The number **`10`** is duplicated. Introduce one named constant (e.g. `REFINEMENT_REFERENCE_PREVIEW_LINES = 10`) in a single place both call sites can use — either a small `private const int` in each file (acceptable if no shared header yet) or a shared internal in `liboccoder` (prefer one definition per **CODING_STANDARDS** / project norms).

### `Details.vala` — `header_file`

After `content` is built for refinement, when `file.get_line_count() > preview_lines`:

- Let `total = file.get_line_count()`.
- Let `omitted = total - preview_lines` (guard `omitted > 0`).
- Append `\n\n` + footer string including `total` and `omitted` (and optionally `preview_lines` as “shown”), instead of only `\n\n_(abbreviated)_\n`.

Conceptual shape:

```vala
const int REFINEMENT_REFERENCE_PREVIEW_LINES = 10;
// ...
var content = stage == MarkdownPhase.REFINEMENT
	? file.get_contents(REFINEMENT_REFERENCE_PREVIEW_LINES)
	: file.get_contents(0);
if (stage == MarkdownPhase.REFINEMENT) {
	int total = file.get_line_count();
	if (total > REFINEMENT_REFERENCE_PREVIEW_LINES) {
		int omitted = total - REFINEMENT_REFERENCE_PREVIEW_LINES;
		content += "\n\n_(Abbreviated: first " + REFINEMENT_REFERENCE_PREVIEW_LINES.to_string()
			+ " lines shown; " + omitted.to_string() + " more lines omitted; "
			+ total.to_string() + " lines total.)_\n";
	}
}
```

Use project string style (`+` vs `@"..."` per **CODING_STANDARDS**). If `get_line_count()` can be 0 with content edge cases, keep the existing guards.

### `Runner.vala` — `reference_content`

Two refinement branches truncate to **10** lines (task section and user-request anchor). Replace the return that ends with `_(abbreviated)_` with the same numeric idea:

- `int total = lines.length;` (or compute from split — ensure it matches logical “lines” as today).
- `int shown = int.min(REFINEMENT_REFERENCE_PREVIEW_LINES, total);` — if you always take `[0:10]`, `omitted = total - 10` when `total > 10`.

Conceptual shape for the task-section branch:

```vala
const int REFINEMENT_REFERENCE_PREVIEW_LINES = 10;
// ...
string[] lines = section_md.split("\n");
if (lines.length <= REFINEMENT_REFERENCE_PREVIEW_LINES) {
	return section_md;
}
int total = lines.length;
int omitted = total - REFINEMENT_REFERENCE_PREVIEW_LINES;
return string.joinv("\n", lines[0:REFINEMENT_REFERENCE_PREVIEW_LINES])
	+ "\n\n_(Abbreviated: first " + REFINEMENT_REFERENCE_PREVIEW_LINES.to_string()
	+ " lines shown; " + omitted.to_string() + " more lines omitted; "
	+ total.to_string() + " lines total.)_\n";
```

Mirror the same pattern for the `link.path == ""` anchor branch (lines ~182–187 today).

**Task link, hash empty:** The `result-summary` branch (refinement path ~151–157) wraps full inner markdown in a fence — **not** the same 10-line chop; **no change** unless you intentionally abbreviate that path too.

### Tests / behaviour checks

- File with **≤ 10** lines: no abbreviation footer.
- File with **> 10** lines: footer shows **total** and **omitted** consistent with `get_contents(preview_lines)`.
- Task section with **≤ 10** lines: full `section_md`, no footer.
- Split `"\n"` edge cases (trailing newline producing an extra empty element): confirm `lines.length` matches intended line count or normalize before counting.
