## Coding standards checklist

Copied from **`.cursor/rules/CODING_STANDARDS.md`** — “Checklist for all plans”. If a bullet conflicts with **§ Conventions** in the [parent plan](1.23.40-refine-stage-reference-injection.md) § Conventions, **the standards file wins**.

- **Nullable types**: Design new APIs/properties to avoid nullable types where possible (default objects / flags).
- **Null checks**: Avoid generic null checks; only where the design explicitly requires null (see standards for the narrow exceptions).
- **String interpolation**: No `@"..."` except multi-line usage/help/documentation; use `+` concatenation otherwise.
- **Temporary variables**: No one-use temporaries; no trivial aliases (allowed exception: long property chains **4+** steps).
- **Brace placement**: Line breaks before `{` for namespace/class/method; **inline** `{` for `if`/`for`/`while`/`switch`; never a one-line `if` with body on the same line.
- **`this.` prefix**: Use for instance member access in new/modified Vala.
- **GLib prefix & `using`**: Fully qualify `GLib.*`; do not add `using` for new code.
- **Property initialization**: Prefer `get; set; default = …` (or field defaults), not constructor assignment for those defaults.
- **Line length & breaking**: Break long method calls, concatenations, docblocks, and comments (docblock/comment line length per full standards).
- **StringBuilder**: Avoid `GLib.StringBuilder` unless building strings in loops with **hundreds** of iterations; use `+` and `string.joinv()`.
- **String building in loops**: Never accumulate with `+=` inside a loop; use `string.nfill()` / `replace()`, `string.joinv()`, etc.
- **ArrayList for strings**: Do not use `Gee.ArrayList<string>` only to join; use `string[]` with **`string[] name = {}`** only for empty init (not `new string[0]`).
- **Loops and nesting**: In loops, use **`continue`** to skip cases and keep the body flat; avoid **`else`** and nested **`if`** inside loops.
- **Character looping**: Do not iterate characters unless there is truly no alternative; prefer string methods and `GLib.Regex`.
- **File info try/catch**: Wrap `query_info()`-style calls only when file existence is unknown — not right after a successful read of the same path.
- **Try/catch scope**: Wrap only the minimal expression(s) that throw; keep setup outside `try`.
- **Underscore prefix**: No leading `_` on variables, fields, or properties.
- **`get_*` methods**: Prefer properties or verb-less/action names over `get_foo()` (see standards for exceptions).
- **Docblocks**: Multiline `/** … */` with `@param` / `@return` where relevant; not one-liners for real behavior.
- **Debug / warning messages**: No timestamps or monotonic time in `GLib.debug()` / `GLib.warning()`; no class/method prefixes in the message (file/line come from the log).
- **Gee.HashMap**: Use `.set()` / `.get()` / `.has_key()` / `.unset()` — not `[]`.
- **Gee.ArrayList**: Use `.get()` / `.set()` / `.add()` / `.remove_at()` — not `[]`.
- **Reducing nesting**: Prefer early `return` / `continue`; avoid `else` where flow stays clear; prefer `switch` over long `if` / `else if` chains when editing.
- **SQL table aliases**: Not applicable to this plan’s fences; if you add SQL elsewhere, no table aliases (full names only).
- **Plan Vala fences**: Do **not** use comment-only placeholders (`/* … */` or empty `return "";`) where real logic belongs — either **complete, compilable Vala** in the fence or **prose-only** steps (no fake stub block).

## Implementation

### Phase 3

#### Bundling the new prompt (`gresources` + Meson)

**Meson:** Prompts under **`resources/skill-prompts/`** are **not** listed per file in **`meson.build`**. **`gnome.compile_resources`** already passes **`../resources/gresources.xml`** and **`source_dir`** entries that include **`../resources/skill-prompts`** (see **`libollmchat/meson.build`** and **`examples/meson.build`**). **Adding a new `*.md` prompt** requires **only**: (1) create the file under **`resources/skill-prompts/`**, (2) add **`<file>…</file>`** under **`<gresource prefix="/skill-prompts">`** in **`resources/gresources.xml`**. **No** Meson edits **unless** you add files under a **new** directory that is not already in **`source_dir`**.

**Order:** Create **`task_refinement_references.md`** → **Add** the **`gresources.xml`** entry → **then** apply the **`refinement_prompt()`** **`Replace`** below so **`PromptTemplate.template(…)`** never names a resource that is not yet bundled.

**Add** (`resources/gresources.xml`) — new `<file>` next to the other **`skill-prompts`** files inside **`<gresource prefix="/skill-prompts">`**.

```xml
<!-- resources/gresources.xml inside <gresource prefix="/skill-prompts"> -->
<file>task_refinement_references.md</file>
```

**Replace** (`liboccoder/Task/Details.refinement_prompt()`) — after the resource exists, add template name selection using **`definition.tools`** inline (**§ Definition.tools**): **`task_refinement.md`** when the skill lists at least one tool **and** **`!definition.tools.contains("write_file")`**; else **`task_refinement_references.md`**. **`tpl.fill(7, …)`** unchanged from Phase 1.

```vala
// Details.vala — full replacement of refinement_prompt() — after Phase 3 bundles task_refinement_references.md
public OLLMcoder.Skill.PromptTemplate refinement_prompt() throws GLib.Error
{
	var definition = this.skill_manager.fetch(this);
	var use_tool_refinement = definition.tools.size > 0 && !definition.tools.contains("write_file");
	var tpl_name = use_tool_refinement ? "task_refinement.md" : "task_refinement_references.md";
	var tpl = OLLMcoder.Skill.PromptTemplate.template(tpl_name);
	tpl.system_fill(0);
	var completed_md = this.runner.completed.to_markdown(MarkdownPhase.REFINE_COMPLETED);
	tpl.fill(7,
		"issues", tpl.header_raw("Issues with the current call", this.result_parser.issues),
		"task_data", tpl.header_raw("Task", this.to_markdown(MarkdownPhase.REFINEMENT)),
		"environment", this.runner.env(),
		"project_description", (this.runner.sr_factory.project_manager.active_project == null ?
			"" : this.runner.sr_factory.project_manager.active_project.project_description()),
		"task_reference_contents", this.reference_contents(MarkdownPhase.REFINEMENT),
		"skill_details", definition.refine,
		"completed_task_list", (completed_md == "" ? "" :
			"## Completed tasks (so far)\n\n" + completed_md));
	return tpl;
}
```

**Prompt.** New `task_refinement_references.md`.

- **`## Task`** = same rules as `task_refinement.md` (**Key** + value; **References** = flat link line).
- Extra keys only: **Shared references**, **Examination references** (same link syntax; omit if empty).
- **Expected output** copy: single-run vs split per **§ Reference refinement → runner**.
- **Write-file skills** (skill `tools:` is write-only per **§ Definition.tools**): this wrapper’s body **does not** describe **`## Tool Calls`** — the model is not instructed to emit fenced JSON tool calls in refinement; **`extract_refinement`** must not require them (**§ Phase 4**).

**Samples (canonical `## Task` shape):**

**A — single run (References only, no Examination line):**

```markdown
## Task

- **What is needed** Compare error handling in the two modules.
- **Skill** analyze_code
- **References** [Handler](/abs/proj/src/ErrorHandler.vala) [Call site](/abs/proj/src/Main.vala)
- **Expected output** Short comparison across both files: differences in error handling and call patterns.

```

**B — split runs (Shared + Examination lines; no References line in refine output):**

```markdown
## Task

- **What is needed** Analyze each failing test file separately.
- **Skill** analyze_code
- **Expected output** For the file in **References** this run: a tight summation — what fails, key assertions, and how it relates to the shared fixture; do not discuss other examination files.
- **Shared references** [Fixture setup](/abs/proj/tests/fixture.vala)
- **Examination references** [Test A](/abs/proj/tests/test_a.vala) [Test B](/abs/proj/tests/test_b.vala)

```

*(Runner: run 1 sees **References** → `[Test A](…)` only + shared; run 2 sees **References** → `[Test B](…)` only + shared.)*
