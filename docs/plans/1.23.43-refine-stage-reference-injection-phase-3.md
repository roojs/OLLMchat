## Coding standards checklist

Copied from `**.cursor/rules/CODING_STANDARDS.md`** — “Checklist for all plans”. If a bullet conflicts with **§ Conventions** in the [parent plan](1.23.40-refine-stage-reference-injection.md), **the standards file wins**.

- **Nullable types**: Design new APIs/properties to avoid nullable types where possible (default objects / flags).
- **Null checks**: Avoid generic null checks; only where the design explicitly requires null (see standards for the narrow exceptions).
- **String interpolation**: No `@"..."` except multi-line usage/help/documentation; use `+` concatenation otherwise.
- **Temporary variables**: No one-use temporaries; no trivial aliases (allowed exception: long property chains **4+** steps).
- **Brace placement**: Line breaks before `{` for namespace/class/method; **inline** `{` for `if`/`for`/`while`/`switch`; never a one-line `if` with body on the same line.
- `**this.` prefix**: Use for instance member access in new/modified Vala.
- **GLib prefix & `using`**: Fully qualify `GLib.*`; do not add `using` for new code.
- **Property initialization**: Prefer `get; set; default = …` (or field defaults), not constructor assignment for those defaults.
- **Line length & breaking**: Break long method calls, concatenations, docblocks, and comments (docblock/comment line length per full standards).
- **StringBuilder**: Avoid `GLib.StringBuilder` unless building strings in loops with **hundreds** of iterations; use `+` and `string.joinv()`.
- **String building in loops**: Never accumulate with `+=` inside a loop; use `string.nfill()` / `replace()`, `string.joinv()`, etc.
- **ArrayList for strings**: Do not use `Gee.ArrayList<string>` only to join; use `string[]` with `**string[] name = {}`** only for empty init (not `new string[0]`).
- **Loops and nesting**: In loops, use `**continue`** to skip cases and keep the body flat; avoid `**else**` and nested `**if**` inside loops.
- **Character looping**: Do not iterate characters unless there is truly no alternative; prefer string methods and `GLib.Regex`.
- **File info try/catch**: Wrap `query_info()`-style calls only when file existence is unknown — not right after a successful read of the same path.
- **Try/catch scope**: Wrap only the minimal expression(s) that throw; keep setup outside `try`.
- **Underscore prefix**: No leading `_` on variables, fields, or properties.
- `**get_*` methods**: Prefer properties or verb-less/action names over `get_foo()` (see standards for exceptions).
- **Docblocks**: Multiline `/** … */` with `@param` / `@return` where relevant; not one-liners for real behavior.
- **Debug / warning messages**: No timestamps or monotonic time in `GLib.debug()` / `GLib.warning()`; no class/method prefixes in the message (file/line come from the log).
- **Gee.HashMap**: Use `.set()` / `.get()` / `.has_key()` / `.unset()` — not `[]`.
- **Gee.ArrayList**: Use `.get()` / `.set()` / `.add()` / `.remove_at()` — not `[]`.
- **Reducing nesting**: Prefer early `return` / `continue`; avoid `else` where flow stays clear; prefer `switch` over long `if` / `else if` chains when editing.
- **SQL table aliases**: Not applicable to this plan’s fences; if you add SQL elsewhere, no table aliases (full names only).
- **Plan Vala fences**: Do **not** use comment-only placeholders (`/* … */` or empty `return "";`) where real logic belongs — either **complete, compilable Vala** in the fence or **prose-only** steps (no fake stub block).

#### Task reference contents — abbreviation vs execution

When the app builds the **refinement** prompt for a task, it fills in **Task reference contents**: the resolved text for each link the user put in that task (files, prior tasks, pieces of the user request, and so on). The rules below are about **that** refinement prompt. **Execution** (when the skill actually runs) uses **full** content for those same links unless noted otherwise.

**File paths** — If a link points at a file and the file is longer than 20 lines, the refinement prompt shows the **first 20 lines** and then a line stating how many lines the full file has. Shorter files are shown in full.

**Links to another task’s output (`task://…`)** — For refinement, the prompt does **not** dump a long section from that task (and does not use the “first 20 lines + abbreviated” pattern for task bodies). It shows only the **Result summary** block from that task’s output, wrapped in a markdown code fence. Validation may still require a `#section` on the link when the task list asks for it; the **text** injected for refinement is still the summary, not that section’s body.

**A section of the current user request (`#anchor` on the request document)** — If that section is long, refinement uses the same idea as files: **first 20 lines**, then the line that states the full line count.

**Execution** — Full file content, full task sections, full anchor text, and whatever else the runner resolves for the run — **not** the shortened refinement previews above.

**Task creation and task-list iteration** — Those steps use other prompts (environment, project, open files, skill list). They do **not** fill **Task reference contents** the way refinement does.

See **Code changes — abbreviation metadata** at the end of this plan for where this is implemented in Vala.

## Prompt: `task_refinement_references.md`

**Aim.** **Single purpose:** tell the refiner to **organize** the coarse task into a concise `**## Task`** list: **What is needed**, **Skill**, **Expected output**, and precursor links under **Shared references** and **Examination references**. Decisions must be **deliberate** — grounded in **Refinement** (skill document, under **## Skill Details** / `{skill_details}`) and in the runner-filled sections (**What you receive**, **Task reference contents**, `{tool_instructions}`, etc.). **Short rules** cover link formats and shared vs examination (parent **§ Reference refinement → runner**). **Skill-first split:** When the skill's **Refinement** subsection tells you where links go — e.g. **everything in Shared references** and **nothing in Examination references**, or **no Shared references** and **everything in Examination references** — **follow the skill**; that overrides the default balance in the next sentence. **Default balance** (when the skill is silent or only gives general guidance): **Shared references** should hold only what **every** run truly needs (fixtures, shared modules, cross-cutting context). **Examination references** should hold the per-run slice so each execution gets **enough** local context. Dumping nearly everything into **Shared references** overloads a single run; putting nearly everything into **Examination references** starves runs of shared context — aim between those extremes. Output is **only** that list — concise lines, **no** extra prose. See **[1.23.44](1.23.44-refine-stage-reference-injection-phase-4.md) §6b** for validation of refined YAML.

## Code changes — abbreviation metadata (file / section size)

### Stage: where `reference_contents` runs (REFINEMENT shaping)

`**MarkdownPhase stage**` is passed into `**Details.link_content(link, stage)**` from every caller that resolves bodies. `**REFINEMENT**` uses **bounded** precursors: `**file:`** and user-request `**#anchor**` may use **20-line preview + footer**; `**task:`** uses **result-summary only** (no section-body abbreviation). Other stages get **full** text (or stage-specific rules below).


| Flow                                          | Injects `{task_reference_contents}` / calls `reference_contents`?                                                                                                         | Typical `stage` into `link_content` | Refinement precursor                                                         |
| --------------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | ----------------------------------- | ---------------------------------------------------------------------------- |
| **Refinement** (`Details.refinement_prompt`)  | **Yes** — `reference_contents(REFINEMENT)`                                                                                                                                | `**REFINEMENT`**                    | **Files / anchors:** preview + footer when long; `**task://`:** summary only |
| **Task creation / task list iteration**       | **No** — templates use env, project, open file, skill catalog via `**Runner.task_creation_prompt`** / iteration fill; **not** the per-task `**reference_contents`** block | *(N/A for that slot)*               | **N/A**                                                                      |
| **Execution** (`Tool` precursor / exam links) | Per-link via `**link_content(..., EXECUTION)*`*                                                                                                                           | `**EXECUTION**`                     | **Full** file bodies, **full** task sections, **full** anchors               |


So: `**Runner.reference_content`** and `**Details.header_file**` both **branch on `stage`**. `**REFINEMENT**` task links are **summary-only**; `**REFINEMENT`** file/anchor links use the **abbreviation** fences in `**header_file`** / anchor branch; **non-** `**REFINEMENT`** paths return full content (or empty for http outside execution).

---

**Goal:** When `MarkdownPhase.REFINEMENT` shows a **preview** of long **file** or **user-request anchor** bodies, append a **single markdown line** after the preview so the refiner knows how large the full body is (see **Task reference contents — abbreviation vs execution** above). `**task:`** links in **REFINEMENT** do **not** use that preview — they show `**result-summary`** only. The shipped footer replaces the former `**_(abbreviated)_**` marker with:

```markdown
**This has been abbreviated.** The full content has 237 lines.
```

**N** in the shipped prompt is the integer **237** (or whatever `get_line_count()` / `lines.length` yields). No separate “omitted” count in the footer — only **total lines**.

**Where abbreviation / summary is implemented**


| Location                                                                         | Behaviour                                                                                                                                                                                   |
| -------------------------------------------------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `liboccoder/Task/Details.vala` — `header_file`                                   | `file.get_contents(20)` when `stage == REFINEMENT`; if `file.get_line_count() > 20`, append the `****This has been abbreviated.** …`** line with **N** = `get_line_count()`.                |
| `liboccoder/Skill/Runner.vala` — `reference_content` — `**task:`**               | `**stage == REFINEMENT`:** `**result-summary`** heading only, nested markdown fence — **no** section-body truncation; `**stage != REFINEMENT`:** full task doc or section by hash as today. |
| `liboccoder/Skill/Runner.vala` — `reference_content` — `**path == ""`** (anchor) | `**stage == REFINEMENT`:** first **20** lines or full; if truncated, append footer with **N** = `lines.length`.                                                                             |


**Preview limit:** **20** lines — inlined at each call site (no shared constant).

### `Details.vala` — `header_file` (full method)

**keep** — Docblock, `**content`** ternary (**REFINEMENT** → first 20 lines; else full file), fence choice, `**return`** with `**line**` + fenced `**content**`. **add** — middle block only: `**REFINEMENT`** + line count **> 20** → append abbreviation footer (no trivial alias for `**get_line_count()`**).

```vala
	/**
	 * Line + fenced body; content and language from file.
	 * Use for reference file path. Exception: we do output the header
	 * (and empty block if needed) when content is empty.
	 *
	 * @param line heading or label line
	 * @param file file to read content and language from
	 * @return fenced block with line + content
	 */
	internal string header_file(string line, OLLMfiles.File file, MarkdownPhase stage)
	{
		var content = stage == MarkdownPhase.REFINEMENT
			? file.get_contents(20)
			: file.get_contents(0);
		if (stage == MarkdownPhase.REFINEMENT) {
			if (file.get_line_count() > 20) {
				content += "\n\n**This has been abbreviated.** The full content has "
					+ file.get_line_count().to_string() + " lines.\n";
			}
		}
		var fence = (content.index_of("\n```") >= 0 || content.has_prefix("```")) ? "~~~~" : "```";
		return line + "\n\n"
			+ fence
			+ (file.language != "" ? file.language + "\n" : "\n")
			+ content + "\n"
			+ fence + "\n\n";
	}
```

### `Runner.vala` — `reference_content` (full method)

**keep** — Method signature through closing `**}`**: docblock, debug line, `**task:**` slug resolution, `**hash == ""**` non-**REFINEMENT** path (`**return doc.to_markdown()`**), `**section_md**` / `**anchor_md**` when `**stage != REFINEMENT**` (**full** markdown), `**http(s)`** branch, `**GLib.error**`. **add** — `**REFINEMENT`-only** shaping: (1) `**task:`** `**hash == ""**`: `**result-summary**` nested fence (not the 20-line chop). (2) `**task:**` with section hash: `**if (stage == REFINEMENT)**` after `**section_md**` — 20-line preview + footer when `**lines.length > 20**`. (3) `**path == ""**`: same pattern after `**anchor_md**`.

```vala
		/**
		 * Resolve non-file reference content for task refs, document
		 * anchor, or http (deferred). Do not add validation here;
		 * validation is done in Details.validate_references(). Call only
		 * for links that validation has accepted; never return "".
		 *
		 * @param link the reference link (scheme, path, href, hash
		 *        already parsed)
		 * @param stage refinement uses abbreviated / result-summary-scoped markdown; other stages match prior behavior
		 * @return resolved content for the link
		 */
		public string reference_content(Markdown.Document.Format link, OLLMcoder.Task.MarkdownPhase stage)
		{
			GLib.debug("reference_content: link scheme=%s path=%s href=%s hash=%s",
				 link.scheme, link.path, link.href, link.hash);
			if (link.scheme == "task") {
				var slug = link.path.has_suffix(".md") ?
					link.path.substring(0, link.path.length - 3) : link.path;
				if (!this.completed.slugs.has_key(slug) && !this.pending.slugs.has_key(slug)) {
					GLib.error("reference_content: no task for slug=%s", slug);
				}
				var task = this.completed.slugs.has_key(slug) ?
					this.completed.slugs.get(slug) : this.pending.slugs.get(slug);
				var doc = task.task_output_document;
				if (link.hash == "") {
					if (stage == OLLMcoder.Task.MarkdownPhase.REFINEMENT) {
						if (!doc.headings.has_key("result-summary")) {
							return "";
						}
						var inner = doc.headings.get("result-summary").to_markdown_with_content();
						var fence = (inner.index_of("\n```") >= 0 || inner.has_prefix("```")) ? "~~~~" : "```";
						return fence + "markdown\n" + inner + "\n" + fence + "\n";
					}
					return doc.to_markdown();
				}
				if (!doc.headings.has_key(link.hash)) {
					GLib.error("reference_content: task section missing slug=%s hash=%s",
						slug, link.hash);
				}
				var section_md = doc.headings.get(link.hash).to_markdown_with_content();
				if (stage == OLLMcoder.Task.MarkdownPhase.REFINEMENT) {
					string[] lines = section_md.split("\n");
					if (lines.length <= 20) {
						return section_md;
					}
					return string.joinv("\n", lines[0:20])
						+ "\n\n**This has been abbreviated.** The full content has "
						+ lines.length.to_string() + " lines.\n";
				}
				return section_md;
			}
			if (link.path == "") {
				var anchor = link.hash;
				var block = this.user_request.headings.get(anchor);
				GLib.debug("reference_content: anchor href=%s user_request=%p headings.has_key(%s)=%s block=%p",
					link.href, this.user_request, anchor,
					this.user_request != null ? this.user_request.headings.has_key(anchor).to_string() : "n/a", block);
				var anchor_md = block.to_markdown_with_content();
				if (stage == OLLMcoder.Task.MarkdownPhase.REFINEMENT) {
					string[] lines = anchor_md.split("\n");
					if (lines.length <= 20) {
						return anchor_md;
					}
					return string.joinv("\n", lines[0:20])
						+ "\n\n**This has been abbreviated.** The full content has "
						+ lines.length.to_string() + " lines.\n";
				}
				return anchor_md;
			}
			if (link.scheme == "http" || link.scheme == "https") {
				if (stage != OLLMcoder.Task.MarkdownPhase.EXECUTION) {
					return "";
				}
				var key = link.href != "" ? link.href : link.path;
				if (key == "" || !this.http_cache.has_key(key)) {
					return "";
				}
				return this.http_cache.get(key);
			}
			GLib.error("reference_content: unsupported link scheme or path (href=%s)", link.href);
		}
```

Use project string style (`+` vs `@"..."` per **CODING_STANDARDS**). When the plan and **HEAD** diverge, treat the repository as source of truth and refresh these fences.

### Tests / behaviour checks

- `**stage == EXECUTION`** (or any non-**REFINEMENT** path used for full precursors): **no** abbreviation footer; file bodies use `**get_contents(0)`**; task/anchor bodies return **full** markdown.
- File with **≤ 20** lines (**REFINEMENT**): no abbreviation footer.
- File with **> 20** lines (**REFINEMENT**): footer is `****This has been abbreviated.** The full content has N lines.`** with **N** = `get_line_count()`, consistent with `get_contents(20)`.
- Task section with **≤ 20** lines (**REFINEMENT**): full `section_md`, no footer.
- Split `"\n"` edge cases (trailing newline producing an extra empty element): confirm `lines.length` matches intended line count or normalize before counting.
