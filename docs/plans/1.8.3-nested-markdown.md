# Plan 1.8.3: Nested Markdown (fenced ` ```markdown ` blocks)

**Coding standards:** All code in this plan must follow `.cursor/rules/CODING_STANDARDS.md`. In particular: no `@"..."` string interpolation (use concatenation); use `this.` for instance members; fully-qualified `GLib.*`; brace style (line break for namespace/class/method, inline for control structures); no one-use temporaries; docblocks multiline with `@param`/`@return`. See the "Checklist for all plans" in that file before implementation.

- **Nullable types / null checks:** Avoid nullable types; no generic null checks unless design requires.
- **String interpolation:** Avoid `@"..."`; use `+` concatenation (exception: multi-line help/docs).
- **Temporaries:** No one-use temporaries; no trivial aliases (except 4+ step chains).
- **Braces:** Line break for namespace/class/method; inline for if/for/while; no one-line if body.
- **this. / GLib.:** Use `this.` for instance members; use `GLib.*` fully qualified; no `using`.
- **Docblocks:** Multiline `/** ... */` with `@param`/`@return`; break long lines (≤72 in comments).
- **Line length:** Break long method calls and concatenations for readability.
- **StringBuilder / loops:** No StringBuilder unless hundreds of iterations; no string build in loops; use `string.joinv()` or `+`; no character looping (use string methods or regex).
- **ArrayList:** Use `string[]` and `string[] name = {}` for string arrays; avoid `Gee.ArrayList<string>` for join-only.
- **Loops:** Use `continue` to skip; avoid `else` and deep nesting.
- **Underscore / get_*:** No leading `_` on names; prefer properties or verb-less names over `get_*()`.

## Summary

When the Markdown renderer receives a **fence block with type `markdown`**, it should:

1. **Display** the inner content as rendered Markdown (nested renderer) inside a frame, not only as raw source in a SourceView.
2. **Support** the info-string format: **language** then optional **space + description** (e.g. ` ```markdown ` or ` ```markdown vala snippet `). The frame header should show **description** when present, otherwise **language**; currently it shows neither.
3. **Provide** a view-source toggle (e.g. `<` / `>` or expand/collapse) on the frame so the user can switch between rendered view and SourceView.

Phase 3 replaces the current UI path that displays the **request/body message** (e.g. tool call body or user message body sent to the LLM) so that this content uses the new nested Markdown descender where applicable.

## Context

- **Current behaviour:** A fenced code block (including ` ```markdown `) is rendered by the GTK renderer as a single **SourceView** (`libocmarkdowngtk/RenderSourceView.vala`). The block’s info string is passed as `lang` from the parser through `Render.on_code_block(is_start, lang)` and used for syntax highlighting and frame title; **language/description are not yet parsed or displayed correctly** (e.g. frame shows neither).
- **Parser:** Fenced code info string is read in `BlockMap.peek()` as a single trimmed string `block_lang` and passed via `do_block(..., block_lang, ...)` to the renderer. No splitting of “language” vs “description” yet.
- **Document model:** `Block` has `lang` (single string). No separate `description` field yet.
- **Display of tool and user content:** Tool messages are shown via `ChatView.append_tool_message()`: content is converted with `PangoRender.toPango()` and inserted into the **renderer’s current buffer** (same box as assistant content, but without using the renderer’s markdown add/flush pipeline). User messages are added as a separate Frame (header + `TextView`) to the same box, with `buffer.text = text` (plain text). Phase 3 will switch the relevant body/request content to use the full Markdown widget (and nested descender) instead.

### Frame styling (`resources/frame.css`)

**`resources/frame.css`** provides Bootstrap-style frame themes via CSS variables. Base class **`.oc-frame`** holds structure (border, shadow, radius); theme classes set `--frame-bg`, `--frame-border`, `--frame-shadow`, `--frame-text-emphasis`. All chat frames that wrap user/tool/error/preview content should use **`.oc-frame`** plus exactly one theme class.

| Theme class | Use case |
|-------------|----------|
| **`.oc-frame-primary`** | User content at start (user messages). |
| **`.oc-frame-success`** | Tool output (tool call results / tool message body). |
| **`.oc-frame-danger`** | Errors. |
| **`.oc-frame-info`** | What we send (e.g. execute preview / request body shown before or with the call). |

- **User messages:** Frame = `oc-frame` + `oc-frame-primary`.
- **Tool messages:** When rendered as a frame (Phase 3), frame = `oc-frame` + `oc-frame-success`.
- **Errors:** Frame = `oc-frame` + `oc-frame-danger`.
- **Execute / request preview:** Frame = `oc-frame` + `oc-frame-info`.

Ensure **`frame.css`** is loaded with the other app CSS (e.g. in ChatView and any test that shows frames: add `"frame.css"` to the `css_files` array and load from resource `/ollmchat/frame.css`).

**Info string format (do not deviate):** `` {type}.{oc-frame-{theme}} {title} `` — e.g. `text.oc-frame-primary You said:`, `markdown.oc-frame-success Read file`, `text.oc-frame-danger Error`. **type** = `text` | `markdown` | `vala`. **oc-frame-{theme}** = `oc-frame-primary` | `oc-frame-success` | `oc-frame-danger` | `oc-frame-info` | `oc-frame-warning`. **title** = header text. RenderSourceView: split on first space; part before = type + "." + class; add that class to frame; part after = title.

### Fence length: avoid break-out (parser stays 3-only; sender chooses fence or indents)

**Current:** `BlockMap` supports **exactly 3** backticks or tildes for fenced code. Inner content that contains a line that looks like a closing fence would break out.

**Chosen approach — sender-side only (no parser change):** BlockMap stays 3-only. Message.fenced(header, body) picks fence in order: ``` then ~~~ then ````; else indents the whole body (4 spaces per line) and uses ```. No variable-length support in the parser.

**Alternative (deferred):** Support variable-length fences (3+) in BlockMap: consume the full run of backticks/tildes on the opening line, set `fence_open` to that run, require the closing line to match exactly.

### Fence strategy: automatic vs assumed

- **Automatic** — When we **know the content beforehand** (e.g. user message, tool result, error, request body): inspect the content and choose the appropriate fence. If the content has no line that could close a 3-backtick block (no bare ` ``` ` at line start), use **3 backticks**. If it does, use **4 backticks** (or more). Worst case: **adjust the content** so it is safe for 3 backticks — e.g. add a space at the start of any line that would otherwise be a closing fence (so ` ``` ` becomes `  ``` ` and no longer matches). Then we can use 3 backticks and keep the fence minimal.
- **Assumed** — When we have **unknown content streaming in** (e.g. Phase 4 thinking): we cannot inspect ahead, so **assume the worst** and always use **4 backticks**. That way any ` ``` ` in the stream cannot close the block.

Use **automatic** for user, tool, error, and execute/request preview when the full content is available before we emit. Use **assumed** (4 backticks) in Phase 4 for thinking and any other streamed or unknown content.

## Phase 1: Language (and description) visible in code blocks ✅ DONE

**Goal:** Ensure the code-block info string is parsed and shown correctly (language, and when present description), and add tests so we can validate before adding nested rendering.

### 1.1 Parser: info string = language + optional description

- **Location:** `libocmarkdown/BlockMap.vala` (where `block_lang` is set from the fence line), and optionally the document model if we want a separate description field.
- **Behaviour:** After the fence (e.g. ` ``` `), the info string is everything up to the newline. Split it into:
  - **Language:** first token (e.g. `markdown`, `vala`).
  - **Description:** remainder after first space, trimmed (e.g. `vala snippet` for ` ```markdown vala snippet `).
- **API:** Either:
  - Keep a single `lang` string and a new `description` string in the pipeline (BlockMap → Parser.do_block → renderer), or
  - Pass a single “info” string and let the renderer split (language vs description) for display. Decision: prefer **parser splits** and passes two values so document model and all renderers can use them consistently.
- **Document model:** If we add a `description` property to `Block` (e.g. in `libocmarkdown/document/Block.vala`), document render and GTK render can show “description if present, else language” in the frame header. Alternatively, keep only `lang` and encode “language description” in one string for now and split only in the GTK renderer for display; minimal change is **split in renderer** for Phase 1, then optionally move to parser/document in Phase 2.

**Practical Phase 1 choice:** Keep parser as-is (single `lang` string = full info string). **Consumers** (e.g. GTK RenderSourceView, or any other renderer that needs a label) do the splitting: split `lang` on the first space: **language** = first token, **description** = rest trimmed. Frame header text = **description if non-empty, else language**. This fixes “currently displays neither” without parser changes. If tests show the full info string is not reaching the renderer (e.g. empty `lang`), fix the parser path so `block_lang` is passed through correctly.

**Implemented:** `RenderSourceView` splits the info string in the constructor: language = first token (used for GtkSource syntax highlighting), description = rest after first space; header shows description when present, else language. Header label has tooltip with full text when ellipsized.

### 1.2 Test files

- **Single test file:** `tests/markdown/fencing-markdown.md` contains:
  - A fenced block with **language only** (` ```markdown `) and inner markdown (heading, list, emphasis).
  - A fenced block with **language + description** (` ```markdown vala snippet `) and inner content.
- Use this file in the Markdown **viewer** test (e.g. `examples/oc-test-gtkmd.vala`).

### 1.3 Viewer test and assertions

- In the test harness that runs the GTK Markdown viewer (e.g. `oc-test-gtkmd` or a small test that instantiates `MarkdownGtk.Render` and feeds parsed content):
  - Load `tests/markdown/fencing-markdown.md`.
  - **Assert** that the code-block frame headers show:
    - First block (language only): **markdown** (language).
    - Second block (language + description): **vala snippet** (description).
- If the header is empty, trace back: ensure the parser passes the info string through `do_block` → `on_code_block` → `RenderSourceView` and that the header label is set from the split (description else language).

### 1.4 Parser tests (if needed)

- If the info string is not reaching the renderer (e.g. `lang` is empty), run the **markdown parser test** (e.g. block/markdown tester) with:
  - A fence line with `markdown` only.
  - A fence line with `markdown description here`.
- Ensure the document model or parser output includes the full info string so the GTK renderer can split and display it.

**Phase 1 exit criteria:** Code block frame shows **language** when only language is given, and **description** when “language space description” is given; tests added and passing. **Done:** consumer split in RenderSourceView, test file `fencing-markdown.md`, header label + tooltip.

## Phase 2: Nested Markdown rendering for ` ```markdown ` blocks ✅ DONE

**Goal:** When the fenced block type is **markdown**, use a **stack** of two views: one **Markdown renderer widget** (rendered content) and one **SourceView** (raw source), with a frame and a **view-source toggle** (e.g. `<` / `>` or expand/collapse). Frame title remains description if present, else language.

**Implemented:** `RenderSourceView` always builds stack (rendered_box + source page), view-source toggle, and nested `MarkdownGtk.Render` for markdown blocks; `code_language.down() == "markdown"` controls toggle visibility and default stack page. Content is streamed to the nested renderer in `add_code_text()` and flushed in `end_code_block()`. Toggle switches stack visible child and triggers resize; frame resizes to rendered content (capped) after flush.

### 2.1 Detection and widget choice

- In **RenderSourceView** (created from `Render.on_code_block(is_start, lang)`):
  - **Always** create the same structure: stack (rendered box + SourceView), view-source toggle. No wrapper/branch on “is markdown” for building widgets.
  - **Split** the info string (Phase 1) into **language** and **description**. Store **language** lowercase: `this.code_language = language.down()` (use original for header/syntax as today). Then: if `this.code_language == "markdown"` set `view_source_toggle.visible = true`, `stack.visible_child_name = "rendered"`; else set `view_source_toggle.visible = false`, `stack.visible_child_name = "source"`. No separate `is_markdown_block` — use `code_language == "markdown"`.

### 2.2 Stack and frame layout

- **Container:** Same frame as today. **Always** build the same layout (no branch).
- **Stack:** Two layers, always created:
  1. **Rendered:** A box that will hold a **MarkdownGtk.Render** for the inner markdown (populated in `end_code_block()` when markdown).
  2. **Source:** The **SourceView** with the raw inner text.
- **Toggle:** A button on the frame header (e.g. “View source” / “View rendered”) that switches the visible child of the stack. Always created; for non-markdown set `visible = false`. Default visible page: “rendered” when markdown, “source” otherwise.

### 2.3 Data flow

- When the parser feeds **code text** for a ` ```markdown ` block:
  - Accumulate as today.
  - On **code block end:** push the accumulated text to:
    - The SourceView (set buffer text).
    - The nested Markdown renderer (e.g. `add_start`; `add(accumulated_text)`; `flush`).
- Nested markdown can in theory nest further if inner content contains ` ```markdown `; no cap in Phase 2. A one-level cap can be added later (e.g. constructor param on Render) if needed.

### 2.4 Frame header

- **Title:** Description if present, else language (same as Phase 1).
- **Buttons:** Copy, expand/collapse (if kept), and **view-source toggle** (e.g. `<` / `>`) — **only for markdown blocks**; for non-markdown, hide the toggle (don’t add it or set `visible = false`).

**Phase 2 exit criteria:** Fenced ` ```markdown ` blocks show rendered Markdown in a frame with a working view-source toggle; other fenced blocks still use SourceView only. **Done:** stack + toggle in RenderSourceView, streaming to nested renderer, flush on end_code_block, resize after flush.

## Phase 3: Use nested Markdown and frame styling per output type

**Goal:** The frame appears **because the parser sees a fence**. Use **standard UI calls only** — no special methods that build frames or insert markup by hand. The **sender** (tool call, send message, etc.) builds the fenced content (e.g. `send_message("ue", "```{type} {tool description} output\n....\n```")`); the renderer just renders it. **Do not** have RenderSourceView branch on `code_language` to add `oc-frame` / `oc-frame-primary` etc.; frame theme is not applied in the renderer.

**Unified flow:**
- Every output type (Phase 3: user, tool, error, preview) is emitted as **fenced content**: opening fence (3 or 4+ backticks) + info string + newline + body + newline + closing fence. Fence length: **automatic** when content is known (inspect and use 3 backticks if safe, else 4 or adjust content); **assumed** (4 backticks) in Phase 4 for thinking/streaming. See "Fence strategy: automatic vs assumed" in Context.
- One pipeline: `renderer.start(); renderer.add(fenced_content); renderer.flush();` (and scroll etc.). Parser sees the fence → code block → RenderSourceView creates the frame. Frame theme (primary/success/danger/info) and header label come from the **info string**; RenderSourceView maps that to `frame.css` classes.

**Prerequisite:** Parser support for **4+ backticks** (see “Fence length” in Context).

**Fence:** `Message.fence(string content)` returns `"````"` if content would close a 3-backtick fence, else `"```"`. **`Message.fenced(string header, string body)`** builds the full fenced block: `Message.fence(body) + " " + header + "\n" + body + "\n" + Message.fence(body) + "\n"`. **`Message.fenced_assumed(string header, string body)`** same but uses fixed fence `"````"` (for streaming/unknown content, e.g. Phase 4 thinking).

**user-sent:** Keep role **"user-sent"** to store the original user message (history, title from first user message). Set **is_ui_visible = false** for "user-sent" so the UI does not display it. Add a separate **Message("ui", Message.fenced(header, body))** for display; UI displays role "ui".

### Phase 3 in short

**What we're changing:** Today user/tool/error/preview (and thinking) are shown in different ways (plain text in a Frame, "### title" + ``` type ```, Pango markup, or chunked into the markdown pipeline). Phase 3 changes user/tool/error/preview so they are sent as a **single fenced block** (e.g. `` ``` text.oc-frame-primary You said:\n`` + body + ``\n``` ``) and the UI **only** does `renderer.add(message.content)`. **Thinking is out of scope for Phase 3** (see Phase 4).

**Why:** One pipeline for all of them; frames and themes come from the parser (info string → RenderSourceView → `oc-frame-*` from `frame.css`). No special-case UI code per type.

**Where:** Senders: `Session.send()` (user), `RequestBase.send_ui()` (tool), `ChatWidget.handle_error` / session (error), callers that add_message (preview). Display: `ChatView` when it handles role "ui" (and we remove append_error); all paths end in the existing "ui" handler. Thinking is Phase 4.

### Block 1: User message

**What:** Session today creates one message `Message("user-sent", plain text)` and the UI builds a Frame + TextView for it. We keep adding `Message("user-sent", ...)` to `messages` for history/title only — we do *not* call `message_added` for it, so the UI never sees it. We add a second message `Message("ui", Message.fenced(...))` and call `message_added` only for that; the UI displays the “You said” block via the shared renderer pipeline.

**Why:** User text is then just another fenced block; same layout and theme as other blocks, no custom Frame code.

**Where:** `libollmchat/History/Session.vala` in `send()` when `message.role == "user"`; `libollmchatgtk/ChatView.vala` `append_user_message()` and the path that runs when role is "ui" in `on_message_created` / `load_messages`.

**Call flow (user enters message, presses Send → how we get to the renderer)**

- User types in ChatInput and presses Send (or Ctrl+Enter).
  - **ChatBar** (or ChatInput) emits `send_requested` / `send_clicked(text)`.
- **ChatWidget**
  - `on_send_clicked(text)` runs: clears input, stores `last_sent_text`, calls `send_chat_request.begin(text)`.
  - `send_chat_request(text)`:
    - Builds `user_message = new Message("user", text)`.
    - Calls `yield manager.send(manager.session, user_message, cancellable)`.
- **Manager** (`libollmchat/History/Manager.vala`)
  - `send(session, user_message, cancellable)` delegates to `yield session.send(user_message, cancellable)`.
- **Session** (`libollmchat/History/Session.vala`), inside `send(message, cancellable)` when `message.role == "user"`:
  - **Today:** creates `user_sent_msg = new Message("user-sent", message.content)`, adds to `messages`, calls `manager.message_added(user_sent_msg, this)` so the UI displays it.
  - **After:** adds `Message("user-sent", message.content)` to `messages` only (for history/logging); does *not* call `message_added` for it — the UI never sees it. Creates `ui_msg = new Message("ui", Message.fenced(...))`, adds to `messages`, calls `manager.message_added(ui_msg, this)` so the UI displays the “You said” block.
  - (Then continues to `agent.send_async(message, ...)` for the actual LLM request.)
- **Manager** emits the `message_added` signal with that message and session.
- **ChatWidget** has `manager.message_added.connect(on_message_created)`.
  - `on_message_created(m, session)` runs:
    - **Today:** when `m.role == "user-sent"` → `chat_view.append_user_message(m.content, session)`.
    - **After change:** when `m.role == "ui"` (for the “You said” block) → the branch that does `renderer.start(); renderer.add(m.content); renderer.flush(); scroll_to_bottom();` (see “ChatView — display” below).
- **ChatView**
  - **Today:** `append_user_message(text, session)` builds a Frame + TextView and sets `user_text_view.buffer.text = text` — that’s the “before” code.
  - **After:** We never call `append_user_message` for user text. The “You said” block is shown via the **ui** message; the handler for that message runs `renderer.start(); renderer.add(text); renderer.flush(); scroll_to_bottom();` — that’s the “after” code. So the pipeline is: Send → Session creates "ui" message with fenced content → `message_added` → `on_message_created` (role `"ui"`) → renderer.add(fenced content).

**Session — send path**

**Before** (exact code today):

| Before | After |
|--------|--------|

| — | `ui_msg` is new: the "ui" message whose content is the fenced block, added to `messages` and passed to `message_added`. |

**ChatView — display**

- **What it’s acting on:** The need to show the user’s “I said this” text in the chat. Today that’s a "user-sent" message with plain text; after the change it’s a "ui" message with fenced content (same text, wrapped in a fence).

**Before**

- **Trigger:** `ChatWidget.on_message_created(m, session)` with `m.role == "user-sent"` → calls `chat_view.append_user_message(m.content, session)`.
- **Acting on:** `m.content` = plain user text.
- **Where:** Inside `ChatView.append_user_message(string text, ...)` — we build a Frame + TextView and show that text.

Exact code (inside `append_user_message`):
```vala

user_text_view.buffer.text = text;
this.add_widget_frame(user_frame);
```

**After**

- We **remove** the code that handled "user-sent": the `case "user-sent":` branch in `on_message_created` (and in `load_messages`) that calls `append_user_message`, and the body of `append_user_message` that builds the Frame + TextView (or the whole method if it’s only used for this).
- The “You said” block is sent from Session as a **"ui"** message with fenced content. ## Phase 4: Thinking in a framed box

**Scope:** Display assistant "thinking" as a **fenced block** (framed box) in the chat, using **oc-frame-info** styling, injected into the assistant response stream in the correct order.

**Constraints:**
- **Stream injection:** Thinking must be **injected into the assistant response stream** (correct order relative to content), not sent as a separate "ui" message.
- **Frame style:** Use **oc-frame-info** (not oc-frame-warning).
- **Fence length:** Does not matter: content is sent directly into the frame via the API, not parsed as markdown, so the frame cannot be escaped by backticks in the thinking text.

**Design — child frame API and stream flow:**

**Verified:** On session restore, thinking is delivered the same way: `append_complete_assistant_message()` is called with `message.thinking` set (for stored `think-stream` messages the thinking text is in `msg.content` and passed as the message’s thinking field; for `content-stream` it is in `msg.thinking`). So both live streaming and restore use the same path — bypass the standard renderer and push thinking into a dedicated frame.

1. **Make the child frame publicly available**  
   Expose an API so the code that captures the stream (e.g. ChatView or the component that handles `stream_chunk`) can **directly create a child frame** and obtain a reference to it. The renderer (or RenderSourceView) must allow opening a frame programmatically (not only via the parser seeing a fence in `add()`).

2. **Open the thinking frame when thinking starts**  
   When the first thinking chunk arrives, the stream handler opens a child frame with the standard info string (e.g. `` ```markdown.oc-frame-info Thinking... ``). Fence length (3 vs 4 backticks) does not matter because we stream text into the frame via the API — the frame cannot be escaped. The handler receives a **reference** (handle) to that frame or to a writer that feeds it.

3. **Pass all thinking text into that frame**  
   For each subsequent chunk with `is_thinking == true`, pass the incoming text into the frame — **similar to on text** / `process_add_text_direct` / `add_code_text`: the same kind of streaming-into-block behaviour as for content, but targeted at the thinking child frame.

4. **Hold a reference**  
   The stream handler keeps a reference to the open thinking frame (or its writer) for the duration of the thinking span.

5. **Close the frame when thinking stops**  
   When thinking ends (e.g. `is_thinking` becomes false or the stream completes), the handler calls **close** on the referenced frame. That finalizes the fenced block (e.g. end_code_block / flush) so the frame is complete and no longer accepts input.

**Summary:** The stream-capture path opens a child frame with `` ```markdown.oc-frame-info Thinking... ``, streams all thinking text into it, holds a reference, and closes it when thinking stops. No separate "ui" message; the frame is created and fed directly from the stream handler so thinking appears in the right place in the assistant response.

Add a reference to the current thinking frame. When entering the thinking block, open the frame; when sending text in thinking mode, send to the frame; when leaving thinking, close the frame.

**New field (with other private fields):**

**In `start_block_direct(bool is_thinking)` — when starting a thinking block, open the child frame:**

```vala
private void start_block_direct(bool is_thinking)
{
    switch (this.content_state) {
        case ContentState.THINKING:
        case ContentState.CONTENT:
            this.renderer.start();
            if (is_thinking) {
                this.renderer.on_code_block(true, "markdown.oc-frame-info Thinking...");
                this.thinking_frame = this.renderer.childview;
                this.last_chunk_start = 0;
                return;
            }
            // ... existing content styling (style_state, default_state, last_chunk_start) ...
            return;
        case ContentState.NONE:
            return;
    }
}
```

**In `process_add_text_direct(string text, bool is_thinking)` — when in THINKING state, send text to the frame instead of the renderer:**

```vala
private void process_add_text_direct(string text, bool is_thinking)
{
    this.last_line += text;

    switch (this.content_state) {
        case ContentState.THINKING:
            if (this.thinking_frame != null) {
                this.thinking_frame.add_code_text(text);
            }
            return;
        case ContentState.CONTENT:
            this.renderer.add(text);
            return;
        case ContentState.NONE:
            this.content_state = is_thinking ? ContentState.THINKING : ContentState.CONTENT;
            this.start_block_direct(is_thinking);
            if (this.thinking_frame != null) {
                this.thinking_frame.add_code_text(text);
            } else {
                this.renderer.add(text);
            }
            return;
    }
}
```

**In `process_new_line_thinking_direct(bool is_thinking)` — send newline to the frame when in thinking:**

```vala
private void process_new_line_thinking_direct(bool is_thinking)
{
    if (!is_thinking) {
        if (this.thinking_frame != null) {
            this.thinking_frame.add_code_text("\n");
            this.renderer.on_code_block(false, "");
            this.thinking_frame = null;
        }
        this.renderer.add("\n");
        this.end_block_direct(this.is_thinking);
        return;
    }
    if (this.thinking_frame != null) {
        this.thinking_frame.add_code_text("\n");
    } else {
        this.renderer.add("\n");
    }
}
```

**In `end_block_direct(bool is_thinking)` — when ending a thinking block, close the frame (do not flush the renderer; thinking went to the frame):**

```vala
private void end_block_direct(bool is_thinking)
{
    switch (this.content_state) {
        case ContentState.THINKING:
            if (this.thinking_frame != null) {
                if (this.last_line.length > 0) {
                    this.thinking_frame.add_code_text("\n");
                }
                this.renderer.on_code_block(false, "");
                this.thinking_frame = null;
            }
            this.last_line = "";
            this.content_state = ContentState.NONE;
            return;
        case ContentState.CONTENT:
            if (this.last_line.length > 0) {
                this.renderer.add("\n");
            }
            this.renderer.flush();
            this.renderer.end_block();
            this.renderer.default_state = null;
            this.last_line = "";
            this.content_state = ContentState.NONE;
            return;
        case ContentState.NONE:
            return;
    }
}
```

**In `finalize_assistant_message_direct()` — ensure thinking frame is closed when the message ends:**

```vala
// At start of finalize_assistant_message_direct(), after the content_state check:
if (this.thinking_frame != null) {
    this.renderer.on_code_block(false, "");
    this.thinking_frame = null;
}
```

**Session restore — same path, no separate branch**

When restoring a session, thinking content reaches the same code path and must use the same frame behaviour (bypass the standard renderer, create the thinking frame, push content into it).

- **Source:** `ChatWidget.load_messages()` and `on_message_created()` handle stored messages. For role **`think-stream`** the thinking text is in `msg.content`; the code builds `new Message("assistant", "", msg.content)` so `message.thinking == msg.content`. For **`content-stream`** / **`content-non-stream`** it uses `new Message("assistant", msg.content, msg.thinking)` so `message.thinking` is already set.
- **Display:** Both paths call `chat_view.append_complete_assistant_message(stream_msg, session)`. That method already does: if `message.thinking != ""`, call `process_new_chunk_direct(message.thinking, true)` then `end_block_direct(true)`.
- **Conclusion:** Restore passes the full thinking string in one go through `process_new_chunk_direct(..., true)`. That splits on newlines and drives the same state machine: first line triggers `start_block_direct(true)` (which opens the child frame), subsequent lines and newlines go to `thinking_frame.add_code_text(...)`, and the explicit `end_block_direct(true)` closes the frame. So no separate restore logic is needed — we must not send restored thinking through the standard renderer; the same frame open → push into frame → close path handles both streaming and restore.
