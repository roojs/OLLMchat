# 1.23.28 DONE — Agent bugs (task list retry / replay mismatch)

**Parent:** 1.23.x (skills agent). **Source:** Live run vs replay divergence; task list create reports stale/wrong issues on retry; replay cannot reproduce the bug.

**Status: DONE.**

**Resolved —** Root cause was stale `streaming_response` state across `send()` calls (§2.3). **Fix:** In `Call.Chat.send()`, reinitialize the streaming accumulator each send—replace `streaming_response` with a new `Response.Chat` (full reset of `done`, chunks, etc.), not only `streaming_response.message`. That aligns live parsing with what is streamed/saved; try-2+ validation and replay should match.

**Coding standards:** Any code added for debugging or fixes must follow `.cursor/rules/CODING_STANDARDS.md`.

**Plan focus (historical):** Analysis and debugging strategies. The streaming-message reinit fix is implemented in `libollmchat/Call/Chat.vala`; §1.3 / replay mismatch should no longer occur for that reason.

## 1. Observed behaviour

### 1.1 Session example

- **History file:** `~/.local/share/ollmchat/history/2026/03/17/21-29-19.json`
- **Flow:** Task list create with retries (skill-runner, task creation stage).

### 1.2 Try 1 (correct validation)

- **Content (message index 4):** First LLM output includes a bad reference: `[CODING_STANDARDS.md](/.cursor/rules/CODING_STANDARDS.md)`.
- **Validation:** Correctly reports e.g.  
`Invalid reference target "/.cursor/rules/CODING_STANDARDS.md": file does not exist (resolved from project folder).`
- **UI:** "Task list had issues (retrying)" with that message. Behaviour is correct.

### 1.3 Try 2+ (incorrect — same error despite fixed output) — **fixed**

- **Was:** Live run showed stale validation errors on retry because the Runner sometimes parsed wrong/stale accumulated content while the session stored the correct stream. **Fix:** New `Response.Chat` per `send()` so the next response’s first chunk does not see `done == true` from the previous call and `message.content` matches the current stream.
- **Content (message index 10):** Second LLM output in the JSON is **corrected**: no `/.cursor` reference; references are e.g. `[README.md](/home/alan/gitlive/OLLMchat/README.md)`; task names like "Research Task List Generation", "Research Coder Task Flow", etc. No "Research 1" or "Task 6"/"Task 7" in the markdown.
- **Live run behaviour:** After try 2 (and 3, 4), the UI **still** shows the same class of errors:
  - "Task section 1 Research 1 (References): Invalid reference target /.cursor/rules/CODING_STANDARDS.md..."
  - "Task section 4 Task 6 references skill , which is not in the available skills list." (and Task 7).
- So the **saved** content for try 2 is correct, but the **live** run reported issues as if it had parsed the **first** (bad) output again, or a mix of old and new.

### 1.4 Replay

- **Command:** Replay the same session (e.g. `--replay` with that JSON).
- **Result:** Replay fails with:  
`Replay fatal: array index 10: live run had issues here but replay did not.`
- **Meaning:** Replay feeds the **saved** content at index 10 (the corrected task list) to the parser; the parser finds **no** issues. The replay logic infers from the saved message list that at index 10 the "live" run had issues (because the next "ui" message says "Task list had issues (retrying)"); so it compares "live had issues" vs "replay had no issues" and exits.
- **Conclusion:** Replay cannot reproduce the bug because it parses the **same** content that was saved (correct). The bug is that during the **live** run, the code path that decided "we have issues" for the **second** response was not using that same content (or was using stale/wrong state).

### 1.5 Task list iteration: redoing original first task

- **Observed:** At the end of the task list iteration stage, the flow appears to **redo the first task from the original plan** instead of working from the **new tasks** produced by the iteration.
- **Expected:** After `run_task_list_iteration()` the runner should refine and run the **first step of the new list** (the list returned by the LLM from the iteration prompt). The next round should not repeat the original plan’s first task.
- **Code path (for reference):** `Runner.handle_task_list()`: after a step completes we call `run_task_list_iteration()` which sets `this.pending = new List(this)` and parses the iteration response into it; on success we then call `this.pending.refine()` and `this.pending.run_step()`, which use `this.pending.steps.get(0)` (the new list’s first step). So by code, we *do* replace the list and run the new first step.
- **Possible causes (to investigate):** (1) Same class of bug as §1.3 — **wrong content** passed to the iteration parser (e.g. we parse stale/previous response so the “new” list is actually the old one or malformed). (2) Iteration prompt or LLM output keeps the same first task (product/prompt issue). (3) Some other reference (e.g. UI or logging) still showing the old task even though the runner is on the new list. (4) Iteration response not being used correctly (e.g. parse_task_list_iteration builds a list that is then overwritten or not used).
- **Next step:** Add the same “Debug parsed content” pattern for the iteration response in `run_task_list_iteration()` (after we have `response`, before `ResultParser(this, response)` and `parse_task_list_iteration()`) so we can see in the session log what list was actually parsed after each iteration. Then confirm whether the runner is parsing the intended new list or something else.

## 2. Root cause hypotheses

### 2.1 Wrong content passed to the parser on retry (most likely)

- **Hypothesis:** On try 2, the string passed to `ResultParser(this, response)` (and thus to `parse_task_list()`) is **not** the second LLM response. It could be:
  - The first response (try 1) still in a buffer or return value.
  - A concatenation or wrong accumulation from streaming.
  - A different code path returning a different `response` than the one that was streamed to the session.
- **Why it would match observation:** Session (and JSON) receives the **stream** that is shown to the user (correct content). The Runner receives `response = response_obj.message.content` from `chat_call.send()`. If that `response` is wrong (e.g. from a previous call or uncleared buffer), the parser would validate the wrong text and report issues that do not match the saved content.

### 2.2 Parser or validation using shared state

- **Hypothesis:** The parser or `validate_references()` uses state that is not reset between retries (e.g. `runner.pending` from a previous parse, or a cached list of tasks/slugs). So even with correct `response` text, validation could still see old references or task names.
- **Why it would match observation:** Issues would refer to "Research 1", "Task 6", "Task 7", or "/.cursor" even when the **document** built from the current response does not contain them, because the validator is reading from somewhere else.
- **Note:** Code inspection shows `this.pending = new OLLMcoder.Task.List(this)` and a new `ResultParser(this, response)` each loop iteration; `parse_task_list()` builds from `this.document` (built from `response`). So this is less likely unless some other shared state (e.g. project path, file existence cache, or runner-global list) influences validation.

### 2.3 Session vs Runner response divergence (mitigated: fresh Response per send)

- **Hypothesis:** The stream that updates the **session** (and thus the saved JSON) is not the same as the stream that updates the **Response** object whose `message.content` is returned to the Runner. For example:
  - Chat has no `agent` set in the skill path, so the session never gets the stream; session might get content from somewhere else (e.g. a different callback or a final "assistant" message).
  - Or the Response is reused and not fully reset between `send()` calls (e.g. `message.content` not cleared or merged with previous).
- **Root cause (confirmed):** The live `Call.Chat` was reusing a single `streaming_response` and only resetting `streaming_response.message` at the start of each `send()`. That left `done`, `is_first_chunk`, `thinking`, `new_content`, etc. from the previous response, so the first chunk of the next request could see `response.done == true` and trigger `finalize_streaming()` too early (wrong session state) and/or leave wrong `message.content` for the Runner vs what was streamed.
- **Fix applied:** At the start of each `send()`, replace `streaming_response` with a new `Response.Chat(connection, this)` (same idea as ReplayChat returning a new Response each time). The Chat instance (connection, model, agent, tools) stays the same—only the Response that accumulates the stream is reinitialized.

## 3. Replay: how "run had issues" is inferred

- **File:** `libollmchat/Call/ReplayChat.vala`
- **Constructor:** Scans `replay_messages`. For each message index `i`, if the message is a content message (`content-stream`, `content-non-stream`, `assistant`) with non-empty content, it sets `state = i`. When it later sees a `ui` or `ui-warning` message that contains "had issues" (or "Task list had issues" or Executor failed), it does `run_issues.add(state)`.
- **Effect:** The **last** content index before each "had issues" UI message is recorded as "the run had issues after this content". So for the session above, after the content at index 10 we have a UI at index 14 saying "Task list had issues (retrying)" → index 10 is added to `run_issues`. Replay then expects that when it returns the content at index 10 and calls `report_replay_outcome(parser.issues)`, the parser should report issues. When the parser reports **no** issues (because the content at 10 is correct), we get "live run had issues here but replay did not."
- **Implication:** The heuristic is consistent with the saved log: the **live** run did show "had issues" after the second response. The bug is that the live run was wrong to report issues for that second response (or it was reporting issues for the wrong content).

## 4. Debugging strategies

### 4.1 Show parsed content in session log and UI (implemented)

- **Approach:** In `Runner.send_async()` (task list create loop), after getting `response` from `send()` and before creating the parser, add a UI message with the content that will be passed to the parser. Use `add_message(ui, Message.fenced("markdown.debug Debug parsed content", content))` so it appears both in the session JSON and visually in the chat.
- **Implementation:** Same pattern in two places:
  - **Task list create:** `liboccoder/Skill/Runner.vala`: after `replay_step("task_list_parse", response)`, add a fenced UI message with info string `"markdown.debug Debug parsed content"` and body = `response`. This is the exact string passed to `ResultParser(this, response)`.
  - **Refinement:** `liboccoder/Task/Details.vala`: in `refine()`, after getting `response_text` from `send()` and before creating `ResultParser(this.runner, response_text)`, add the same fenced UI message with body = `response_text`. Same bug (wrong/stale content validated) can occur on refinement retries.
- **Use:** On a live run (or replay), compare the "Debug parsed content" block for try 2 with the preceding assistant content-stream. If they differ, the Runner is parsing different content than what was streamed/saved.

### 4.2 Assert equality: response vs last session content

- After the Runner gets `response` from `send()`, if the session is available and was just updated by the stream, compare `response` to the last "content-stream" (or equivalent) message content in the session. If they differ, the stream path that updates the session is not the same as the one that sets `response_obj.message.content`.

### 4.3 Trace streaming_response and session in one run

- In the **live** run, add temporary logging:
  - In `Call.Chat.process_streaming_chunk`: when calling `agent.handle_stream_chunk(new_text, ...)`, log a fingerprint of `new_text` and of `response.message.content` so far.
  - In `Session.handle_stream_chunk`: when appending to `current_stream_message`, log a fingerprint of the full content of that message.
- After the second response completes, compare: (a) the last `response.message.content` in Chat, (b) the last content-stream message in the session. They should be identical; if not, the divergence is in the streaming path.

### 4.4 Replay with run_issues relaxed (diagnostic only)

- Temporarily change replay so that when "live had issues but replay did not" at index 10, instead of exiting, log the replayed content and the fact that replay found no issues, then **continue**. Run to completion and see if the rest of the session replay is consistent. This does not fix the bug but confirms whether the only mismatch is at that index (wrong content parsed live) vs multiple divergences.

### 4.5 Check agent and Chat wiring for the skill runner

- Verify which `Chat` instance the Runner uses and whether it has `agent` set when `send()` is called. If `agent` is null, the session would not receive `handle_stream_chunk` from this Chat; then the session’s content for the task list would have to come from somewhere else (e.g. another agent or a fallback path). Trace where the session’s content-stream messages for the skill run are actually appended.

### 4.6 Task list iteration: debug parsed content and confirm which list is used (implemented)

- **Goal:** Confirm whether the iteration stage is parsing the correct LLM response and thus replacing `pending` with the intended new list (§1.5).
- **Implementation:** In `Runner.run_task_list_iteration()`, after we have `response` and before creating the new List and parser, add the same debug UI message: `add_message(ui, Message.fenced("markdown.debug Debug parsed content", response))` The iteration response is now captured in the session log and UI so we can compare it with the preceding assistant content-stream and see if we are parsing the right response (or stale/wrong content). Optionally log or display a one-line summary of `this.pending.steps.size` and the first step’s task name(s) after a successful parse, so we can confirm the runner is about to run the new first step and not the old one.
