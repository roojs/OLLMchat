# 7.14.1.3 — Thinking frame collapsed while streaming

**Status:** done (archived) — implemented; optional verify with a thinking model + tool round-trip

**Pointer:** `.cursor/rules/CODING_STANDARDS.md` — Checklist for all plans

**Parent / index:** `docs/plans/done/7.14-DONE-task-progress-tree-ui.md`, `docs/plans/done/7.14.1-2-DONE-task-progress.md`

**Related:** `docs/plans/done/7.14.2.4-DONE-loading-history-performance-change-bugs.md`

---

## Purpose

- **`RenderSourceView`** (**`.collapsed-on-done`**) needs **`is_streaming`** true when the thinking fence opens so the body reveals during live streaming (**7.14.2.4** ctor gate).

---

## Scope

| In scope | Out of scope |
| -------- | ------------ |
| **`libollmchatgtk/ChatView.vala`** — **`append_assistant_chunk`**, **`clear_waiting_indicator`**, optional **`initialize_assistant_message`** | **`RenderSourceView.vala`** chevron / **`body_revealer`** polish (**7.14.2.4**) |
| Order and ownership of **`is_assistant_message`** / **`renderer.is_streaming`** on first chunk after waiting | Session restore-only hacks (**`Session.vala`**) unless QA shows a gap after A+B+C |

---

## Acceptance criteria

- After “waiting…” clears, **first thinking tokens** open the thinking fence **expanded** until that fence completes.

- Tool round-trip: assistant stream **after tools** still initializes streaming state correctly.

- **`ChatView.clear()`** / new chat: no additional reset logic required for this bug (**ℹ️** see **What this is not**).

---

## Evidence (ordering)

- **`clear_waiting_indicator(response)`** ran **before** **`initialize_assistant_message`** and set **`is_assistant_message = true`**, so **`initialize_assistant_message`** was skipped on the first chunk **`renderer.is_streaming`** never set before **`on_code_block`**.

---

## What this is **not** (review focus)

- **ℹ️** Not “missing reset on new chat”: **`ChatView.clear()`** already sets **`is_assistant_message = false`** and clears renderer / waiting (**`libollmchatgtk/ChatView.vala`** **`clear()`**).

- **ℹ️** The bug is **per-send order** inside **`append_assistant_chunk`**, not **`switch_to_session`** failing to wipe state.

- **ℹ️** **Stale `is_assistant_message`** without finalize is a **different** hypothesis than the ordering issue above.

---

## Concrete code proposals

Hunks are **Remove** / **Replace with** / **Add** from the tree; verify surrounding lines before applying.

Implement **§1 + §2** together; **§3** is optional (**belt-and-brackets** for **`role == "assistant"`** on the wire).

### 1. `libollmchatgtk/ChatView.vala` — `append_assistant_chunk` (reorder)

#### Keep

```vala
		public void append_assistant_chunk(string new_text, OLLMchat.Response.Chat response)
		{

```

#### Remove

```vala
			if (this.is_waiting) {
				this.clear_waiting_indicator(response);
			}

			if (!this.is_assistant_message) {
				this.initialize_assistant_message(response);
			}
```

#### Replace with

```vala
			if (!this.is_assistant_message) {
				this.initialize_assistant_message(response);
			}

			if (this.is_waiting) {
				this.clear_waiting_indicator(response);
			}
```

#### Keep

```vala
			// Process the incoming new_text chunk directly
			if (new_text.length > 0) {
				this.process_new_chunk(new_text, response);
			}
```

```vala

```

