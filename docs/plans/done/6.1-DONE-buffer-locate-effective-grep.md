## Purpose

Define how OLLMchat finds **all** occurrences of a search string (or multiline pattern) in file buffers, with two backends:

1. **DummyFileBuffer** (non-GTK): `**locate` must not load the whole file into memory** — use **grep-style chunked sequential I/O** with a **carry / overlap** window so multiline needles cannot be split across reads (Part D). **Bare minimum** elsewhere: no dedicated unit tests, no new callers, trimmed semantics ignored (same as literal match). Optional later: faster substring algorithms (e.g. Horspool) — not required to ship chunked `locate`.
2. **GtkSourceFileBuffer** (GTK): use **GtkSourceView’s search API** (`GtkSourceSearchSettings` + `GtkSourceSearchContext`) on the live `GtkSource.Buffer`.

Requirements beyond classic grep:

- **Multiline search** — the needle may span `\n`.
- **Trimmed multiline search** — normalize the needle by collapsing **whitespace around newlines** (e.g. **`Regex`** **`\s+\n\s+` → `\n`**) and optionally stripping the whole string; then search; report matches in **real buffer coordinates** / matched text (Part E.2).
- **Return every match**, not just the first or “current” match.
- **Result shape:** a map from **start line (0-based, consistent with `FileBuffer`)** to the **full line(s)** that cover the match: from the **start of the line** where the match begins through the **end of the line** where the match ends (same text as in the buffer, including indentation and line terminators as stored). This applies to **all** implementations (**DummyFileBuffer**, **GtkSourceFileBuffer**), including **multiline** needles — **not** the minimal match span only.

**Explicit scope decision (current):** For **DummyFileBuffer**, **do not implement trimmed multiline semantics** for now. If `match_trimmed` is requested, treat it as **plain full-text / substring matching** (same as non-trimmed — document on the API). Trimmed matching matters on **GtkSourceFileBuffer** / project UI paths. Chunked `locate` is still **required** for dummy so tools never need a full-file string just to search.

## Part A — What we can learn from grep (and related tools)

### A.1 GNU grep’s performance model (high level)

From GNU’s own documentation ([Performance (GNU Grep 3.12)](https://www.gnu.org/software/grep/manual/html_node/Performance.html)) and common descriptions of GNU grep:

- **Large read buffers:** read the file in big chunks rather than line-by-line I/O.
- **Avoid per-byte work when possible:** for fixed strings, implementations often use fast substring algorithms (historically associated with **Boyer–Moore–Horspool**-style skipping) so many bytes are skipped without full comparisons.
- **Line reporting without scanning every newline first:** a useful mental model is “find a match in the byte stream, **then** find surrounding newlines to assign a line number,” rather than splitting the whole file into lines up front.

### A.2 Functional lessons relevant to our design


| Idea                | Takeaway                                                                                                                                                                      |
| ------------------- | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Chunked input       | **DummyFileBuffer `locate`:** large reads via `FileInputStream` (or async read), **not** `get_contents()` / loading the whole file for the search path.                       |
| Cross-chunk matches | **DummyFileBuffer:** keep a **carry** tail between reads (length ≥ needle length minus one, or full needle) so a multiline match cannot straddle a chunk boundary undetected. |
| Line index          | **Running line count** while scanning, or derive line from offset + newline count in the window — same spirit as grep (line number = newlines before match start).            |
| Trimmed multiline   | **Not** how grep works; on GtkSource use Part E (**SearchContext** + normalized needle). Dummy ignores trimmed semantics (Part D).                                               |


### A.3 What we are *not* copying

- Default grep is **line-oriented** for simple patterns; we **explicitly** want multiline literal (and optionally regex on SourceView).
- We need **all** occurrences; grep prints lines or counts — we need a structured result for programmatic use.

## Part B — GtkSourceView search API (gtksourceview 5)

Official overview: [GtkSource.SearchContext](https://gnome.pages.gitlab.gnome.org/gtksourceview/gtksourceview5/class.SearchContext.html), [GtkSource.SearchSettings](https://gnome.pages.gitlab.gnome.org/gtksourceview/gtksourceview5/class.SearchSettings.html).

### B.1 `GtkSourceSearchSettings`

- `**search-text`** — string or pattern (if regex enabled).
- `**case-sensitive`**
- `**regex-enabled`** — use `search-text` as a regular expression.
- `**wrap-around**` — for UI-style “find next”; for “all matches” we will likely **disable wrap** and iterate from buffer start, or use forward from start until end.
- `**at-word-boundaries`**
- `**visible-only`** (since 5.12) — exclude invisible text; may matter for folded regions if used.

### B.2 `GtkSourceSearchContext`

- Construct with `(buffer, settings)`.
- `**forward` / `forward_async*`* — from a `GtkTextIter`, find next match; `**backward`** similarly. **This plan** collects every match with a **`forward`** loop only (**I.3**). Other SearchContext APIs (`get_occurrences_count`, `get_occurrence_position`, …) are not part of the reference design.
- `**replace` / `replace_all`** — optional future UI; not required for read-only locate.
- **Highlighting** — `set_highlight(false)` if we do not want side effects on the widget (if the buffer is attached to a visible view, consider whether highlighting is desired; for headless/tool use, disable).

### B.3 Multiline behaviour

The SearchContext documentation explicitly notes that **regex searches can match multiple lines**. So for **GtkSourceFileBuffer**:

- **Multiline literal** might be implemented as **regex with the pattern escaped**, plus flags that allow `.` to span lines if needed, or a constructed multiline-safe pattern — **verify** against the exact regex engine (PCRE via GLib/GTK) and GtkSource’s restrictions (`get_regex_error()` when invalid).
- **Trimmed multiline** is not a SearchSettings flag. Normalize the needle **inline** (no separate helper): **`GLib.Regex`** replace **`\s+\n\s+`** with **`\n`** (collapses horizontal space around each newline — that **is** the trim-between-lines step), optional **`.strip()`** on the whole string for leading/trailing, then **`GLib.Regex.escape_string`** for **`set_search_text`** with **`regex_enabled`**. No **split / loop / join** (Part E.2).

### B.4 Collecting *all* matches

1. Configure settings (case, regex, wrap off).
2. Create `GtkSourceSearchContext`, **`set_highlight(false)`** if appropriate.
3. Loop **`forward`** from increasing buffer offsets until no more matches (same structure as **I.3**); each hit: **key** = **`get_line()`** on the match start iter; **value** = **`get_text`** over **full lines** covering the match (expand **`m_start`/`m_end`** to line boundaries — **I.3**).

Edge cases: empty search text, buffer mutations during search (freeze or copy strategy if needed).

## Part C — Result type: `Gee.HashMap<int, string>`

**Design:** `Gee.HashMap<int, string>` — **key:** 0-based **start line** of the match (first line touched by the needle match; same convention as `FileBuffer`); **value:** the **full line(s)** covering that match — from the beginning of that start line through the end of the last line spanned (buffer text as stored, may contain `\n` between lines). **Not** the minimal engine match span alone. Tools that expose 1-based line numbers convert at the boundary.

If two matches share the same start line, a plain hash map keeps **one** entry; that situation is **out of scope** for current use cases — no alternate container type is required.

## Part D — DummyFileBuffer: chunked sequential `locate`

### D.1 Scope (this plan)

- **Memory:** `locate` must **not** read the entire file into one string for searching (no `get_contents()` / no “load all then `index_of`” as the primary path). Use **chunked reads** like grep-style tools.
- **Process load:** **bare minimum** — no dedicated unit tests, no new callers; implementation can stay straightforward (e.g. `index_of` in a sliding window) without Boyer–Moore unless someone optimizes later.
- `**match_trimmed`:** **ignored for semantics** — same as plain literal substring match; document on the method.
- **Multiline needle:** must work across chunk boundaries via **carry / overlap** (D.2).
- **Return:** fill `Gee.HashMap<int, string>`; **overlapping** occurrences are **not** a product concern — simplest forward scan is fine.

### D.2 Chunked search strategy (required)

1. Open the file for sequential read (`FileInputStream` or async equivalent).
2. Maintain a **search window:** **carry** from the end of the previous iteration plus the newly read chunk. **Carry size:** at least `**needle.length - 1`** characters (UTF-8 **character** length for index math in `string`), or the **full needle** length for simplicity — so any match spanning a chunk boundary appears entirely inside the combined window at least once.
3. **Global state:** running **line number** (or file offset + newline count) so each match can be mapped to **0-based start line** = number of `\n` before the match’s first byte/character, consistent with `FileBuffer`.
4. In the window, find matches with repeated `**index_of`** (or equivalent) on the **literal** needle; only **commit** matches that end before the “unsafe” tail (the region that might be incomplete until the next read). Advance: drop consumed prefix, keep carry tail, read next chunk, repeat until EOF.
5. **Map value:** after locating the needle span in the window, **expand** to **full lines** (from the line start before `found` through the line end after the match end), then slice that substring for `result.set` — same rule as Part C / **I.2.b**.
6. **UTF-8:** treat file as UTF-8 text. When mixing **byte** positions from raw reads with **character** indices in Vala `string`, use the **built-in `string` methods** that wrap the GLib UTF-8 helpers ([valadoc: `string](https://valadoc.org/glib-2.0/string.html)`): `**utf8_offset (long offset)`** → `g_utf8_offset_to_pointer` (returns `unowned string` at that character offset), and `**pointer_to_offset (string pos)**` → `g_utf8_pointer_to_offset` (character offset of a slice / position inside the string). **Do not** assume one byte == one character; no custom `[CCode]` needed for these two.

### D.3 Algorithm class

- First implementation: **repeated `index_of`** on the window is enough.
- **Optional later:** Horspool / Boyer–Moore–style skipping inside the window if profiling shows need — not part of the minimum deliverable.

### D.4 `lines` cache

- `**locate` should not** populate `this.lines` from a full-file read just to search. Either leave `lines` untouched by `locate`, or document interaction if a partial cache is updated — avoid surprising `get_line()` after `locate()`.

## Part E — GtkSourceFileBuffer: SearchContext design

### E.1 Normal and multiline literal

- Use **`GLib.Regex.escape_string`** on the needle when **`regex_enabled`** is required (multiline / trimmed-normalized text). Configure **`GtkSourceSearchSettings`**, attach **`GtkSourceSearchContext`** to **`this`**.

### E.2 Trimmed multiline (full support here)

When `match_trimmed` is true on **GtkSourceFileBuffer**, use **SearchContext** with a **normalized needle** — **no** private `build_trimmed_multiline_pattern`, **no** per-line `\h*…\h*` assembly, **no** split/loop/strip/join:

1. **`GLib.Regex`:** replace **`\s+\n\s+`** with **`\n`** — that normalizes spaces **around** line breaks (the regex **is** the trim-between-lines logic).
2. Optionally **`strip()`** once on the whole result for leading/trailing whitespace on the needle.
3. **`GLib.Regex.escape_string`**, then **`set_search_text`**, **`regex_enabled`** as needed for multiline.

**Map value:** after **`forward`** yields match iters, **expand** to **line boundaries** (start of line containing match start → end of line containing match end), then **`get_text`** — full lines, not only the regex match span (Part C).

### E.3 Occurrence collection

Collect matches with repeated **`forward`** from the buffer start (wrap off), as in **I.3** — same idea as [SourceView.vala](../../../liboccoder/SourceView.vala).

## Part F — `FileBuffer` interface

Add one abstract method only:

```text
locate (string needle, bool match_trimmed, bool case_sensitive) → Gee.HashMap<int, string>
```

- `**needle**` — **literal** search text (may contain `\n` for multiline), not a user regex pattern.
- `**match_trimmed`** — full semantics on `GtkSourceFileBuffer`; on `DummyFileBuffer`, same as plain full-text match (Part D.1).
- `**case_sensitive**` — no separate “options” type and no overloads; one signature on the interface.

**Regex in the public API:** **not** in this plan. `locate` is literal-only. GtkSource may still use regex **internally** to implement multiline literal / trimmed construction (Parts E.1–E.2). A **separate** `FileBuffer` method for user-supplied regex could be added **later** if needed — out of scope now.

**Return:** `Gee.HashMap<int, string>` as in Part C (0-based start line → **full line(s)** covering the match).

Implementations:

- `DummyFileBuffer` — **chunked sequential** `locate` (Part D); literal needle only; no whole-file load for search.
- `GtkSourceFileBuffer` — `GtkSourceSearchContext` inside `locate` (trimmed branch per Part E).

Concurrency: **no** `locate_async` (or other second entry point) on `FileBuffer` — if a caller must not block the UI, it runs `locate` from a worker thread or task itself.

### I.2 `DummyFileBuffer` — chunked `locate`

Implemented in [libocfiles/DummyFileBuffer.vala](../../../libocfiles/DummyFileBuffer.vala) (replaces former WIP). **`match_trimmed`** is ignored (same as non-trimmed). Uses a sliding window and overlap of `needle.length` bytes (simple and safe for multiline needles). Empty needle after strip returns an empty map.

#### I.2.a Reference: `index_of_caseless` — full Vala (copy into `DummyFileBuffer.vala`)

**Placement:** Put **`index_of_caseless`** **inside** `class DummyFileBuffer` (recommended: after `get_line()`, before any chunked `locate`). **No `[CCode]` for `memchr`:** use **`Posix.memchr`** from the **posix** vapi ([libocfiles/meson.build](../../../libocfiles/meson.build) already passes **`--pkg posix`**). **`strcasestr(3)`** is **not** in the standard **posix** vapi (verified: not in `/usr/share/vala/vapi/posix.vapi`); add **one** file-scope **`extern`** for it under **`#if !G_OS_WIN32`** only, immediately **after** the copyright header and **before** `namespace OLLMfiles`.

**Semantics:** **Default behaviour (all platforms):** **`tail.casefold ().index_of (needle.casefold ())`** — one path at the end of the method, no platform branch around it.

**Non-Windows only (`#if !G_OS_WIN32`):** Before that default, run the **`Posix.memchr`** + **`strcasestr`** fast path when the needle is **ASCII-only** (`Posix.memchr ((void*) needle, 0x80, needle.length) == null`). If that matches, return early; otherwise fall through to **`casefold`**. On Windows the **`#if !G_OS_WIN32`** block is omitted — **`strcasestr`** is not in MSVC CRT; the default **`casefold`** path runs.

**`index_of_caseless`** is an **instance** method (`private int`). **`casefold ().index_of`** yields an offset in the **casefolded** strings — see caveat in **I.2.1**.

```vala
#if !G_OS_WIN32
[CCode (cname = "strcasestr", cheader_filename = "string.h")]
extern unowned string? strcasestr (string haystack, string needle);
#endif

		private int index_of_caseless (string haystack, string needle, int start_index = 0)
		{
			if (needle.length == 0) {
				return start_index;
			}
			if (start_index < 0 || start_index > haystack.length) {
				return -1;
			}
			var tail = haystack.substring (start_index);
#if !G_OS_WIN32
			if (Posix.memchr ((void*) needle, 0x80, needle.length) == null) {
				unowned string? hit = strcasestr (tail, needle);
				if (hit == null) {
					return -1;
				}
				return start_index + (int) tail.pointer_to_offset (hit);
			}
#endif
			var i = tail.casefold ().index_of (needle.casefold ());
			return i < 0 ? -1 : start_index + i;
		}
```

#### I.2.b Reference: `locate_in_window` and `locate`

```vala
		/**
		 * Append to result every match of needle in the current window whose end lies
		 * on or before the safe search limit (full window at EOF; otherwise exclude the
		 * carry tail so a match spanning the next chunk is not committed early).
		 * Map value is full line(s) covering the match (expand span to line boundaries).
		 *
		 * @return true if the caller should slide the window (drop prefix, advance
		 * line_base): i.e. not at EOF and window is longer than the needle.
		 */
		private bool locate_in_window(
			string window,
			string needle,
			long line_base,
			bool eof,
			bool case_sensitive,
			Gee.HashMap<int, string> result
		)
		{
			var search_limit = eof
				? window.length
				: (window.length > needle.length ? window.length - needle.length : 0);
			var pos = 0;
			var prev = 0;
			long nl_before_match = 0;
			while (true) {
				var found = case_sensitive
					? window.index_of(needle, pos)
					: this.index_of_caseless(window, needle, pos);
				if (found < 0) {
					break;
				}
				if (found + (int) needle.length > search_limit) {
					break;
				}
				nl_before_match += found > prev
					? window.substring(prev, found - prev).split("\n").length - 1
					: 0;
				var match_end = found + (int) needle.length;
				var prev_nl = found > 0 ? window.substring(0, found).last_index_of("\n") : -1;
				var line_start = prev_nl >= 0 ? prev_nl + 1 : 0;
				var rest = window.substring(match_end);
				var nl_after = rest.index_of("\n");
				var line_end = nl_after >= 0 ? match_end + nl_after + 1 : window.length;
				result.set(
					(int) (line_base + nl_before_match),
					window.substring(line_start, line_end - line_start)
				);
				prev = found;
				pos = found + (int) needle.length;
			}
			return !eof && window.length > needle.length;
		}

		public Gee.HashMap<int, string> locate(string needle, bool match_trimmed, bool case_sensitive)
		{
			var result = new Gee.HashMap<int, string>();
			if (needle.strip() == "") {
				return result;
			}
			/* Trimmed multiline not implemented on dummy; same as literal search. */
			var path_file = GLib.File.new_for_path(this.file.path);
			GLib.FileInputStream stream;
			try {
				if (!path_file.query_exists()) {
					return result;
				}
				stream = path_file.read(null);
			} catch (GLib.Error e) {
				GLib.debug("DummyFileBuffer.locate: %s", e.message);
				return result;
			}
			var window = "";
			long line_base = 0;
			if (needle.length < 1) {
				return result;
			}
			var eof = false;
			var chunk_buf = new uint8[65536];
			while (!eof) {
				ssize_t n_read;
				try {
					n_read = stream.read(chunk_buf);
				} catch (GLib.Error e) {
					GLib.debug("DummyFileBuffer.locate: %s", e.message);
					break; /* continuation: fall through to return partial result */
				}
				if (n_read < 0) {
					break; /* same: best-effort map so far */
				}
				eof = eof || n_read == 0;
				window = n_read > 0 ? window + (string) chunk_buf[0:n_read] : window;
				if (this.locate_in_window(window, needle, line_base, eof, case_sensitive, result)) {
					var keep_from = window.length - needle.length;
					line_base += keep_from > 0
						? window.substring(0, keep_from).split("\n").length - 1
						: 0;
					window = window.substring(keep_from);
				}
			}
			return result;
		}
```

**Review notes:**

- **Try/catch scope:** Per [.cursor/rules/CODING_STANDARDS.md](../../../.cursor/rules/CODING_STANDARDS.md), one small `try` for `query_exists` + `path_file.read` (both can throw on open), and a `try` **inside the read loop** only around `stream.read`; search / `result.set` stay outside `try`.
- **Inner search loop:** `locate_in_window` computes `search_limit` from `eof` and `needle.length`, appends matches (**value** = **full lines** per Part C), then returns `true` when the caller should slide the carry (`!eof && window.length > needle.length`).
- **Newlines:** Inside the helper, `substring(prev, found - prev).split("\n").length - 1`; sliding prefix uses the same pattern. Window is bounded (overlap + chunk). Helper **accumulates** between successive match starts, not from 0 each time.
- **`window = window + chunk`:** Each append **copies the whole window**, so over a huge file this is **O(file_size²)** in the worst case. Acceptable for the first land if windows stay bounded by overlap + chunk; if profiling shows cost, switch to `GLib.StringBuilder` (or a ring of chunks) for the sliding buffer instead of repeated `string` concatenation.
- **Case sensitivity:** When `case_sensitive` is false, call **`this.index_of_caseless`** (see **I.2.a**). Rationale and edge cases: **I.2.1**.
- **Chunk as string:** `(string) chunk_buf[0:n_read]` assumes valid UTF-8; invalid sequences may need a carry for split code units (future hardening; see Part D.2).
- **`int` casts:** `result.set((int) abs_line, …)` assumes line count fits `int`; matches `HashMap<int, string>`.
- **GtkSource `while (forward)`:** No single API returns all slices at once; repeated `forward` matches how [SourceView.vala](../../../liboccoder/SourceView.vala) steps matches — that loop is appropriate.

### I.2.1 Case-insensitive search — default `casefold`, Unix-only `memchr` / `strcasestr`

**I.2.a** is the reference implementation. Chunked **`locate`** calls **`this.index_of_caseless`** from **`locate_in_window`** when **`case_sensitive == false`**.

| What | Direction |
| ---- | --------- |
| **Default (everywhere)** | **`tail.casefold ().index_of (needle.casefold ())`** — shared tail of the method; this is the normal behaviour. |
| **Only if not Windows** | **`#if !G_OS_WIN32`:** if the needle is **ASCII-only** (`Posix.memchr` finds no byte ≥ 0x80), try **[`strcasestr(3)`](https://man7.org/linux/man-pages/man3/strcasestr.3.html)** on **`tail`** and **`tail.pointer_to_offset (hit)`**; otherwise fall through to the same **`casefold`** default as Windows. |

**Bindings:** **`memchr`** — use **`Posix.memchr`** (posix vapi; no **`[CCode]`**). **`strcasestr`** — not in posix vapi; **one** **`extern`** under **`#if !G_OS_WIN32`** as in **I.2.a**.

**Preprocessor:** **`#if !G_OS_WIN32`** wraps the **`strcasestr`** **`extern`** and the fast-path body. Windows **omits** that block and uses the **`casefold`** default only.

**Offsets:** **`casefold ().index_of`** returns a position in the **casefolded** strings — see **I.2.a** caveat.

**GtkSourceFileBuffer:** unchanged — use `GtkSource.SearchSettings.case_sensitive` (already correct for the buffer’s UTF-8 text).

### I.3 `GtkSourceFileBuffer` — `locate` via `GtkSource.SearchContext`

Implemented in [liboccoder/GtkSourceFileBuffer.vala](../../../liboccoder/GtkSourceFileBuffer.vala) (same class as other `FileBuffer` methods). Collect matches with repeated `forward()` from increasing buffer offsets (same idea as [liboccoder/SourceView.vala](../../../liboccoder/SourceView.vala)); disable highlight.

```vala
		public Gee.HashMap<int, string> locate(string needle, bool match_trimmed, bool case_sensitive)
		{
			var result = new Gee.HashMap<int, string>();
			if (needle.strip() == "") {
				return result;
			}
			string search_text;
			if (match_trimmed) {
				var ws_nl = new GLib.Regex ("\\s+\\n\\s+");
				var collapsed = ws_nl.replace (needle, -1, 0, "\n");
				search_text = GLib.Regex.escape_string (collapsed.strip ());
			} else {
				search_text = needle.index_of ("\n") >= 0
					? GLib.Regex.escape_string (needle)
					: needle;
			}
			var settings = new GtkSource.SearchSettings();
			settings.case_sensitive = case_sensitive;
			settings.wrap_around = false;
			settings.at_word_boundaries = false;
			settings.regex_enabled = match_trimmed || needle.index_of ("\n") >= 0;
			settings.set_search_text (search_text);
			var ctx = new GtkSource.SearchContext(this, settings);
			ctx.set_highlight(false);
			var scan_pos = 0;
			while (true) {
				Gtk.TextIter from_iter;
				this.get_iter_at_offset(out from_iter, scan_pos);
				Gtk.TextIter m_start;
				Gtk.TextIter m_end;
				bool wrapped = false;
				if (!ctx.forward(from_iter, out m_start, out m_end, out wrapped)) {
					break;
				}
				Gtk.TextIter line_start = m_start;
				line_start.backward_to_line_start();
				Gtk.TextIter line_end = m_end;
				if (!line_end.ends_line()) {
					line_end.forward_to_line_end();
				}
				result.set(
					line_start.get_line(),
					this.get_text(line_start, line_end, false)
				);
				scan_pos = m_end.get_offset();
				if (scan_pos <= from_iter.get_offset()) {
					break;
				}
			}
			return result;
		}
```

**Review notes:**

- **Map value (full lines):** **`line_start`/`line_end`** iters — line start via **`get_iter_at_line(..., m_start.get_line())`** (GTK4; replaces **`backward_to_line_start`**), **`forward_to_line_end`** on **`m_end`** (same idea as [GtkSourceFileBuffer.get_line()](../../../liboccoder/GtkSourceFileBuffer.vala)); buffer text via **`Gtk.TextBuffer.get_text`** — the **full line(s)** spanning the match, not **`m_start`–`m_end`** alone.
- **Trimmed path:** **`Regex.replace` (`\s+\n\s+` → `\n`)** handles space around newlines; **`strip()`** once on the result for leading/trailing; **`escape_string`** — no split/loop (Part E.2). No **`try`/`catch`** around **`new GLib.Regex ("\\s+\\n\\s+")`** — fixed pattern.
- **Multiline literal:** `GLib.Regex.escape_string` on strings that still contain **`\\n`** separators yields a pattern that matches those line breaks in the buffer text — verify against GtkSource’s regex engine.

- `libocfiles`: already uses **Gee**; no extra package for `Gee.HashMap`.
- `liboccoder`: already depends on **gtksourceview**; `GtkSource.SearchContext` / `SearchSettings` are already used from [liboccoder/SourceView.vala](../../../liboccoder/SourceView.vala).

### I.5 Negative cases and continuation (dummy `locate`)

**Negative inputs / outcomes** (expect **empty** `Gee.HashMap`, no throw from `locate`):


| Case                                                           | Expected               |
| -------------------------------------------------------------- | ---------------------- |
| `needle` empty or only whitespace (`strip() == ""`)            | Empty map              |
| `case_sensitive == false` (dummy)                                | Uses **I.2.a** **`index_of_caseless`** inside **I.2.b** (not an empty map) |
| File path does not exist (`query_exists` false)                | Empty map              |
| `query_exists` or `read` throws on open                        | Empty map + debug      |
| `needle.length < 1` (defensive; empty needle already returned) | Empty map              |


**Continuation on mid-stream failure** (expect **partial** map — matches from chunks already processed):


| Case                 | Behaviour                                                                                 |
| -------------------- | ----------------------------------------------------------------------------------------- |
| `stream.read` throws | Log, **break** read loop, **return `result`** with whatever matches were already recorded |
| `n_read < 0`         | Same: **break**, **return partial `result`**                                              |


So a “negative” I/O test after matches were found earlier in the file should assert **non-empty** map containing only the safe prefix of matches, not a throw and not a full scan.

**Minimal manual check (optional):** point `DummyFileBuffer` at a small temp file with a known needle twice, truncate permissions or use a pipe that closes mid-read if you need to simulate read failure; confirm earlier match is still in the map. No obligation to commit automated tests in G.2.
