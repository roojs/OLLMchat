# 6.1 Plan: Show refinement/iteration stream in UI (Skill Runner)

## Status

**DONE** — This plan has been completed. Refinement/iteration stream is shown in the UI via the session-based chunk display (stream_chunk includes session; ChatWidget shows chunks when session matches manager.session).

## What this is about

When the user runs the Skill Runner (task creation flow):

1. The first LLM response (the task list) streams to the UI and finishes; the user sees the content and the "total duration..." summary.
2. The runner then runs refinement (and later iteration) and sends more LLM requests. The backend is still sending and receiving (debug shows traffic).
3. None of that refinement or iteration output appears in the chat UI.

This plan is about fixing (2)→(3): why those later streams never show, and what to change so they do.

---

## Why refinement never appears: the code that causes it

Every stream chunk from the backend eventually reaches the same UI handler. If that handler **returns without drawing the chunk**, the chunk never appears on screen. Below, "drop" means exactly that: the handler returns early and never calls `append_assistant_chunk`, so the chunk is discarded for display.

### 1. The UI handler that draws (or drops) each chunk

**File:** `libollmchatgtk/ChatWidget.vala`

```vala
private void on_stream_chunk_handler(string new_text, bool is_thinking, OLLMchat.Response.Chat response)
{
    // Check if streaming is still active (might have been stopped)
    if (!this.manager.session.is_running) {
        return;
    }

    // Process chunk (even if done, there might be final text to process)
    if (new_text.length > 0) {
        this.chat_view.append_assistant_chunk(new_text, response);
    }
    // ... rest of handler
}
```

So: **if `manager.session.is_running` is false, the handler returns immediately and never calls `append_assistant_chunk`.** That is the "drop": the chunk is ignored for display.

### 2. Where `is_running` is set to false

**File:** `libollmchat/History/Session.vala`, in `finalize_streaming()`:

```vala
private void finalize_streaming(Response.Chat response)
{
    // ... add end-stream, done, metrics messages ...

    // Set running state to false when response is done
    this.is_running = false;
    GLib.debug("Stopping running");

    this.save_async.begin();
    // ...
}
```

`finalize_streaming()` is called from `Session.handle_stream_chunk()` when `response.done` is true — i.e. when the **first** response (the task list) finishes. So as soon as that first response completes, `session.is_running` is set to false.

### 3. What happens to later chunks (refinement / iteration)

- Refinement and iteration use the **same** session; their stream chunks go through the same path and end up in the same handler: `ChatWidget.on_stream_chunk_handler`.
- By the time those chunks arrive, the first response has already completed, so `finalize_streaming()` has already run and **`session.is_running` is already false**.
- So when the handler runs for a refinement (or iteration) chunk, it hits:

  `if (!this.manager.session.is_running) { return; }`

  and **returns without drawing**. Every refinement/iteration chunk is dropped in this way, so the user never sees any of that output.

So the problem is not that the stream isn't connected; it's that the handler is written to ignore all chunks when `is_running` is false, and the first response's completion sets that to false before any refinement or iteration chunks are processed.

---

## Why doesn't setting `is_running = true` again fix it?

The Runner already sets `this.session.is_running = true` at the start of `handle_task_list()` (before refinement runs). In theory that should fix it. From the code we can establish the following.

### Is the session the same?

**Yes.** The Runner is created by `Session` in `ensure_agent_handler()` via `factory.create_agent(this)` — so `Runner.session` is the `Session` that created it. That `Session` is the one we're in when we send (we haven't switched). `manager.session` is only changed in `Manager.switch_to_session()` (and initial setup in `new_session` / EmptySession conversion). During the skill runner flow we do not call `switch_to_session()`, so `manager.session` is still that same session. So `Runner.session` and `manager.session` are the same object. Details are created with `runner.session` (ResultParser), so refinement streams also use that same session.

### Ordering: do we set `is_running = true` before refinement chunks can arrive?

**Yes.** We set `is_running = true` at the start of `handle_task_list()` before we yield to `task_list.refine()`. The refinement request is only sent inside `Details.refine()` when we `yield chat_call.send()`. So refinement response chunks cannot arrive before we have set `is_running = true`.

### Does anything set `is_running = false` in between?

**No.** The only places that set `is_running = false` are:

- `Session.finalize_streaming()` (line 272) — already ran when the first response completed.
- `SessionBase.add_message()` when `message.role == "done"` (line 360) — the "done" message for the first response is added via `this.messages.add(done_msg)` and `manager.message_added()`, not via `add_message()`, so this path is not triggered.
- `Session.cancel_current_request()` (line 502) — we are not cancelling during refinement.

So nothing sets `is_running = false` between the start of `handle_task_list()` and the arrival of refinement chunks.

### What we know

- Session identity and ordering from the code say refinement chunks should be drawn: same session, `is_running` set true before refinement, nothing clears it in between.
- In practice refinement does **not** show, so at least one of these is wrong in the real run:
  - Refinement chunks never reach `Session.handle_stream_chunk` (wrong agent/path or stream not wired for task Chat).
  - They reach it but `is_active` is false, so we never call `base.handle_stream_chunk` and never emit `manager.stream_chunk`.
  - We do emit but `ChatWidget.on_stream_chunk_handler` is never called (e.g. connection or manager mismatch).
  - The handler is called but `manager.session.is_running` is false, so we return without drawing.
  - The handler is called, we pass the check and call `append_assistant_chunk`, but the view doesn't show it (e.g. wrong buffer, or new stream not starting a new message).

We need to find which of these actually happens.

## Investigation: find where the chain breaks

Add targeted debug (or breakpoints) and run the Skill Runner until refinement runs. Then check in order:

1. **Do refinement chunks reach the session?**  
   In `Session.handle_stream_chunk()` (e.g. at the top): log or break when called, and when it's for refinement (e.g. after the first response has already finalized). If this never fires for refinement, the stream from the task's `Chat` is not reaching this session (e.g. wrong agent/session on the Details' Chat).

2. **When they reach the session, do we relay?**  
   In `Session.handle_stream_chunk()` at `if (this.is_active) base.handle_stream_chunk(...)`: log `is_active` and whether we enter the block. If we reach this for refinement but `is_active` is false, that's the cause (something deactivated the session).

3. **Does the ChatWidget handler run for refinement?**  
   In `ChatWidget.on_stream_chunk_handler`: log or break when called (and e.g. first few chars of `new_text` or a "refinement" hint). If this never runs for refinement, we're not emitting to the manager the ChatWidget is connected to, or the connection is wrong.

4. **When the handler runs, what is `is_running`?**  
   In the same handler, log `this.manager.session.is_running` and whether we hit the early return. If we run for refinement but `is_running` is false, that's the cause (session or timing).

5. **When we pass the check, does the view get the chunk?**  
   In the same handler after the `is_running` check, log before `append_assistant_chunk`. If we reach here for refinement but still see nothing on screen, the bug is in ChatView / renderer (e.g. not starting a new message or drawing to a hidden buffer).

### Debug result (confirmed)

With the minimal debug in place, refinement stream logs show:

- **Session.handle_stream_chunk:** `is_active=true`, `current_stream_null=false` — session is active and we relay every chunk.
- **ChatWidget.on_stream_chunk_handler:** `is_running=false` on every refinement chunk — handler runs but returns immediately and never draws.

So the chain breaks in the ChatWidget: refinement chunks are relayed from the session, but `manager.session.is_running` is false when the handler runs, so every chunk is dropped. Either (1) the session we set `is_running = true` on in the Runner is not the same object as `manager.session` when the handler runs, or (2) something sets `is_running` back to false after we set it true and before refinement chunks arrive. The proposed fix (pass session in the signal, show when `session == manager.session`) avoids relying on `is_running` and will show refinement regardless.

## Proposed fix (libollmchat / libollmchatgtk)

**Idea:** Decide whether to show a chunk based on **which session** it belongs to (the one we're currently displaying), not on `is_running`. Then refinement/iteration chunks for the current session can be shown even though `is_running` was set false when the first response finished.

1. **Manager** (`libollmchat/History/Manager.vala`)  
   - Change the signal to include the session that the chunk belongs to:  
     `stream_chunk(SessionBase? session, string new_text, bool is_thinking, Response.Chat response)`  
   - So the UI can show the chunk when `session == manager.session` and ignore when it's for another session.

2. **SessionBase** (`libollmchat/History/SessionBase.vala`)  
   - In `handle_stream_chunk`, pass the session when relaying:  
     `this.manager.stream_chunk(this, new_text, is_thinking, response);`

3. **ChatWidget** (`libollmchatgtk/ChatWidget.vala`)  
   - Change the handler to take the new first argument:  
     `on_stream_chunk_handler(SessionBase? session, string new_text, bool is_thinking, Response.Chat response)`  
   - Only draw when the chunk is for the session we're displaying, e.g.:  
     - If `session != null && session != this.manager.session` → return (wrong session).  
     - If `session == null` and we want to keep old behaviour for callers that don't pass session: only draw when `this.manager.session.is_running`.  
     - Otherwise call `append_assistant_chunk` (and the rest of the existing logic) so the chunk is shown.

4. **ollmchat-cli** (`ollmapp/ollmchat-cli.vala`)  
   - Update the `stream_chunk` connection to the new 4-argument signature; the CLI can ignore the `session` argument.

---

## Scope

- Changes only in libollmchat, libollmchatgtk, and ollmapp (CLI) as above.
- No change to when or where `is_running` is set; only to how the UI decides whether to show a given chunk.
