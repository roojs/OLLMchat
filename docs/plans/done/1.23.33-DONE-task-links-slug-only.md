# Plan 1.23.33: Task links — optional `#fragment` (`task://slug.md`)

**Parent:** 1.23.x (skills agent / task runner).  
**Context:** Discussion 2026-03 — LLMs often emit wrong GFM-style `#section` fragments on `task://…` references; question whether requiring fragments is worth the validation churn.

**Coding standards:** `.cursor/rules/CODING_STANDARDS.md`.

## Status

**DONE** — Completed 2026-03. See **§ As implemented (append-only)** at the end for what shipped; sections above are **historical** (original plan and options).

---

## Goal (original plan)

Reduce **task list / refinement / post-exec** failures caused by **incorrect `task://slug.md#fragment`** while keeping behaviour predictable for executors and precursors.

**Preferred direction (recommended in discussion):** treat **`task://slug.md`** (no hash) as **“inject the full canonical task output”** (`task_output_document`), and treat **`task://slug.md#section`** as **optional** — when present and valid, inject **only that section** (current behaviour); when absent, fall back to full document.

Alternative (stricter product choice): **forbid** fragments entirely for cross-task refs and **always** inject full document (simpler rules, higher token cost).

## Current behaviour (baseline — historical)

- **`Runner.reference_content()`** (`liboccoder/Skill/Runner.vala`): for `scheme == task`, historically resolved **only** via `task_output_document.headings.get(link.hash)` — empty or wrong hash failed (`headings.has_key` / `GLib.error`).
- **`Details.validate_link_task()`** (`liboccoder/Task/Details.vala`): **REFINEMENT** and **POST_EXEC** required non-empty `link.hash` and a matching heading on the target document; **LIST** with `step_index == 0` also required a fragment (when validation continued past slug check). **LIST** with `step_index != 0` returned early after slug existence — fragments not enforced there today.

See **[docs/task-and-skills-flow.md](../../task-and-skills-flow.md)** for the evolving description of task links.

## Design options (historical)

### A — Optional fragment with full-document fallback (recommended in discussion)

| Case | Resolution |
|------|------------|
| `task://slug.md` (empty hash) | `task_output_document.to_markdown()` (or equivalent full body) |
| `task://slug.md#section` | If heading exists → that section only; if **invalid** fragment → either **error** (strict) or **fallback to full doc** (lenient) — **decide explicitly** |

**Validation:** Allow empty hash for cross-task references in **LIST** and **REFINEMENT** (and **POST_EXEC** for links to **other** tasks). Still validate slug, step ordering, completed-only rules for refinement.

**Pros:** Fewer LLM failures; keeps token savings when fragment is correct.  
**Cons:** Must define invalid-fragment policy; full-doc injection can **bloat** precursor (large research/codebase tasks).

### B — Slug only, no fragments for cross-task refs

Always inject full `task_output_document`; remove fragment requirement and optionally reject any `#` on `task://` for References.

**Pros:** Simplest mental model for task creators.  
**Cons:** Largest prompts; overlap with **completed task list** summaries unless templates are trimmed.

### C — Post-exec self-links (historical question)

**Cross-task** References: optional or no fragment per A/B.  
**Within** post-exec synthesis output, **`task://this-task.md#section`** in **Result summary** may still be desired so iteration UI lists **discoverable sections** — **decide** whether to keep requiring fragments **only** for self-navigation links, or move to another mechanism (e.g. section list only in structured output).

## Implementation checklist (original — historical)

1. **`Runner.reference_content`** — `task` branch: if `link.hash == ""`, return full document content for resolved task; else keep section lookup (and define behaviour when hash non-empty but missing).
2. **`Details.validate_link_task`** — relax empty-hash rules per chosen option; keep slug / ordering / completed-task rules; split **self** vs **other** task if post-exec keeps stricter rules.
3. **Prompts** — `task_creation_initial.md`, `task_list_iteration.md`, `task_refinement.md`, `task_execution.md`, `task_post_exec.md`: document new rules, examples, and retry-issue wording.
4. **Docs** — `docs/skills-format.md`, `docs/task-and-skills-flow.md` (Task links section).
5. **Tests** — `tests/markdown/task-parse-test*.md` and any Vala tests touching validation / resolution.

## Trade-offs to accept or mitigate (historical)

- **Token / context growth** when using slug-only links on large outputs.
- **Redundancy** with **`completed_task_list`** injection if both carry full bodies — consider tightening completed-task markdown to summaries-only if full refs become common.
- **Refinement “extract links from Detail”** — larger precursor may or may not reduce need for extracted file links; no change required unless product wants it.

## As implemented (append-only — do not remove sections above)

**Date:** 2026-03.

**Product direction taken:** Closer to **option B** for **LLM-authored** content: prompts and **`docs/skills-format.md`** teach **`task://{slug}.md`** with the URL **ending at `.md`** (full document injected). Deliberately **avoid** teaching optional fragment forms in prompts (they encouraged bad fragments).

**Code (summary):**

- **`Runner.reference_content`:** empty `hash` → full `task_output_document.to_markdown()`; non-empty `hash` → one section (still supported for legacy/tests/malformed output).
- **`Details.validate_link_task`:** fragment optional for cross-task refs; if non-empty, section must exist; **post-exec** rejects **`task://`** links to **this** task (model uses `##` headings + prose for self-reference).
- **Completed-task listing** in **`Details.to_markdown`:** “Sections in this output” uses plain titles, not `header_links()` (avoids echoing `task://…#…` into iteration context).
- **`post_exec_prompt`:** `system_fill(0)` — removed `task_link_base` fill when **`task_post_exec.md`** was rewritten.
- A short-lived helper **`document_section_titles_hint`** was added then **removed**; validation messages use a fixed hint to use `` `task://{slug}.md` `` with nothing after `.md` instead of enumerating sections.

**Prompts updated:** `task_creation_initial.md`, `task_list_iteration.md`, `task_refinement.md`, `task_execution.md`, `task_post_exec.md` — task URLs stop at **`.md`**; file **`/path#anchor`** rules unchanged.

**Docs updated:** `docs/skills-format.md`, `docs/task-and-skills-flow.md`.

This section records **what changed** relative to the historical plan above; it does **not** replace the original reasoning or options.
