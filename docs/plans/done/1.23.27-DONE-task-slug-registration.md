## Split register vs validate

- **Current:** `register_slugs` does both registration and validation → makes things complicated.
- **Register:** just register (slugs only). **Uniqueness:** if slug clashes, rename the task (e.g. suffix -2, -3) so it’s unique, then register.
- **Validate:** split out into its own step. Document **what it validates** (see below).
- Keep `register_slugs` as “register only”; split validate elsewhere. (In plan for now.)

## What does validate validate?

- **Currently in register_slugs:** Name uniqueness (completed vs pending) — goes away if we drop names. **ResultParser.validate_task(t):** task_data no Output, required keys, only valid keys. **Details.validate_references():** each ref (task://, #anchor, file, http) must be resolvable. When we split: validate = task_data shape + refs; register = slugs only, no overlap check.
- **Validation must use runner.pending and runner.completed**, not this.list. Ref resolution and any list-based checks run against the runner's pending and completed lists, not the step's list.

### 2. Step: register only (slugs), check completed and pending

**Step.vala** — `register_slugs`: set `step_index`, then for each child:

- **Not empty and not in completed and not in pending:** `this.list.slugs.set(s, t)`, continue (next task).
- **Empty:** set name to skill name + (list.slugs.size + 1). Set that name in task_data. Then `this.list.slugs.set(t.slug(), t)`.
- **Not empty (clash):** append the suffix (list.slugs.size + 1) to get unique slug. Set name in task_data so slug matches. Then `this.list.slugs.set(t.slug(), t)`.

Uniqueness is checked against **runner.completed.slugs** and **runner.pending.slugs**, not just this.list. Name is always set in task_data before registering so `t.slug()` is correct. **Step.list** is public (`list { get; set; }`) so we can set `step.list = completed` when moving the step to completed (§4).

```vala
/**
 * Register this step's tasks in list.slugs (and set step_index).
 * If slug non-empty and not in completed/pending, set and continue.
 * Empty: name = skill + (list.slugs.size + 1); set name; then slugs.set(t.slug(), t).
 * Clash: append (list.slugs.size + 1); set name; then slugs.set(t.slug(), t).
 *
 * @param step_index index of this step in the list (set on each child)
 */
public void register_slugs(int step_index)
{
	var completed = this.list.runner.completed;
	var pending = this.list.runner.pending;
	var uid = completed.slugs.size + pending.slugs.size;
	foreach (var t in this.children) {
		t.step_index = step_index;
		var s = t.slug();
		if (s != "" && !completed.slugs.has_key(s) && !pending.slugs.has_key(s)) {
			this.list.slugs.set(s, t);
			continue;
		}
		var uid = completed.slugs.size + pending.slugs.size + 1;
		if (s == "") {
			var skill = t.task_data.has_key("Skill") ?
				t.task_data.get("Skill").to_markdown().strip() : "";
			s = (skill != "" ? skill : "Task") + " " + uid.to_string();
		} else {
			s = s + "-" + uid.to_string();
		}
		var b = new Markdown.Document.Block(Markdown.FormatType.PARAGRAPH);
		b.adopt(new Markdown.Document.Format.from_text(s));
		t.task_data.set("Name", b);
		this.list.slugs.set(t.slug(), t);
	}
}
```

Call sites: **ResultParser.parse_task_list()** and **parse_task_list_iteration()** — register (`step.register_slugs(i)` per step), then validate (validate_task + validate_references). See §3 below.

### 3. Register, then validate

**ResultParser** — register first, then validate. That's it.

1. Build steps (parse_step, steps.add) and fill_names as now.
2. **One loop** over steps: for each step, register then validate its children.

```vala
for (var i = 0; i < steps.size; i++) {
	var step = steps.get(i);
	step.register_slugs(i);
	foreach (var t in step.children) {
		this.validate_task(t);
		t.validate_references();
		if (t.issues != "") {
			this.issues += "\n" + "Task (References): " + t.issues;
		}
	}
}
```

Validation must use **runner.pending** and **runner.completed** (not this.list) for ref resolution.

Same pattern in both `parse_task_list()` and `parse_task_list_iteration()`. **Details.validate_references** must resolve task refs against `this.runner.completed` and `this.runner.pending` only, not the step's list.

### 4. When a step completes: register on completed

We still need completed tasks in `completed.slugs`. Options (from earlier in plan):

- **A** Repoint step: `step.list = completed`, then `step.register_slugs(completed.steps.size - 1)` so registration writes into `completed.slugs`. (Requires `Step.list` to have a setter or a `repoint_list(completed)`.)
- **B** Inline: after `completed.steps.add(step)`, loop `step.children` and `completed.slugs.set(t.slug(), t)`.

Proposed code for **B** (inline, no new method, no repoint):

**List.vala** (both places where we do `completed.steps.add(step)`):

Completed tasks already have names and slugs (we registered and validated them on pending). Repoint the step to completed and copy into completed.slugs. **Step.list** must be public (e.g. `list { get; set; }`) so we can set it here.

```vala
this.runner.completed.steps.add(step);
step.list = this.runner.completed;
foreach (var t in step.children) {
	this.runner.completed.slugs.set(t.slug(), t);
}
this.steps.remove_at(0);
yield this.runner.run_task_list_iteration();
```

## Initial task list

**Parse — what is created when**

- Per task-section: `ResultParser.parse_step(section)`:
  - `new Step(pending)`
  - Per list block: `new Details(...)`, `step.children.add(task)`
  - Return step (not on list yet)
- `pending.steps.add(step)` for each step
- `pending.fill_names()`
- For each step: `Step.register_slugs(i, parser)`

**When slugs/names registered**

- Only in `Step.register_slugs()`
- After all steps are on pending → only **pending** gets `names` / `slugs`
- Per child: uniqueness (completed, then pending), then `pending.names.set`, `pending.slugs.set`

**Why**

- One pass after full parse; duplicate check across sections and vs completed

## When a step completes

**Current**

- `completed.steps.add(step)` only
- No write to `completed.names` or `completed.slugs`

**Gap**

- Validation/resolution use `completed.slugs.get(slug)` / `pending.slugs.get(slug)`
- Completed tasks never in `completed.slugs` → refs to completed tasks fail

---

## Would calling Step.register_slugs on the completed step work?

- No. register_slugs uses this.list (the step's list).
- Step is created with list = pending and never changes.
- After completed.steps.add(step), step.list is still pending — would register into pending, not completed.

---

## Option: repoint step.list to completed

- When moving: completed.steps.add(step) then set step.list = completed (need setter or e.g. step.repoint_list(completed); Step has `list { get; private set; }`).
- Then Step.register_slugs(...) would write into completed.names / completed.slugs.
- Downside: step now "belongs" to two lists (in completed.steps but step.list could be completed); or we consider it moved and list is the owner. Consistent if we always repoint when moving.
