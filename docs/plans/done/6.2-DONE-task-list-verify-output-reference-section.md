# 6.2 Task list verification: output references must be from an earlier section

**Status: DONE**

**Purpose:** Detect and reject the case where the LLM generates two (or more) tasks in the **same** task section and one task references the **Output** of another task in that same section. That is invalid because tasks within a section run together (concurrent or same step); the referenced task's output is not available until the section has completed. If a task needs another task's Output, it must be in a **later** section.

Includes concrete code proposals for Option A (§3.2.1).

---

## 1. Problem

- **Structure:** The task list has **sections** (e.g. "Task section 1", "Task section 2"). Each section is a `Step`; each `Step` has one or more tasks (`Details`). Sections run **sequentially**; tasks **within** a section run together (single task sequentially, or multiple tasks concurrently).
- **Output references:** A task can list in **References** a link to another task's output, e.g. `#slug-results` (e.g. `#research-1-results`). The runner resolves that to the prior task's Result summary / output when building the refinement and executor prompts.
- **Observed bug:** The LLM sometimes puts two tasks in the **same** section and has the second task reference the first task's Output. At run time, when that section runs, both tasks run in the same step — the second task cannot yet have the first task's output. So the reference is invalid.
- **Desired rule:** A task may reference `#X-results` (output of task X) **only if** task X is in a **strictly earlier** section (lower step index). Same-section or later-section output reference → validation error.

---

## 2. Current behaviour

- **Where validation runs:** In `ResultParser.parse_task_list()`, after `fill_names()`, we loop over every step and every task and call `t.validate_references()`.
- **What `Details.validate_references()` does today:** For each reference with `href` like `#...-results`, it strips `-results` to get a slug and checks `runner.task_list.has_slug(name_slug)`. So we only check that **some** task with that slug exists. We do **not** check that the referenced task is in an **earlier** section.
- **Gap:** There is no check that the task owning the reference is in a **later** step than the task identified by the slug. So same-section output references are currently accepted and only fail later when content is resolved (or behave incorrectly).

---

## 3. Proposed solution

### 3.1 Rule to enforce

- For any reference of the form `#<slug>-results`:
  - If there is no task with that slug → keep current error: "no task for …".
  - If there **is** a task with that slug:
    - Let **my_step_index** = index of the step that contains **this** task (the one that has the reference).
    - Let **ref_step_index** = index of the step that contains the task with that slug.
    - If **ref_step_index >= my_step_index** (same section or later) → **new error**: e.g. "Reference target \"#X-results\" refers to a task in the same or a later section. A task may only reference the Output of a task from an **earlier** section. Move this task to a later section or remove this reference."

### 3.2 Where to implement

- **Option A (recommended):** Implement inside the existing reference-validation flow. **Store section/order on tasks when the list is built** and maintain a **slug → task** map on List so lookups are O(1), no looping.
  - **Details** gets a property `step_index` (int, 0-based; -1 if unknown). Set it when the task list is built (parser assigns it when iterating steps).
  - **List** has a **public** property **slugs** (e.g. `Gee.HashMap<string,Details>`: slug → task). Callers use the map directly — no `has_slug` / `step_index_for_slug` methods; check `task_list.slugs.has_key(slug)` and get the task with `task_list.slugs.get(slug)` (then use `task.step_index`).
  - **ResultParser.parse_task_list():** When iterating steps (with index `i`), set `t.step_index = i`, then `task_list.slugs.set(t.slug(), t)` when `t.slug() != ""`, then call `t.validate_references()` (no argument).
  - **Details.validate_references():** No parameter. When handling `#...-results`, check `task_list.slugs.has_key(name_slug)`; if not present, error. If present, get the task with `task_list.slugs.get(name_slug)` and compare `ref_task.step_index` to `this.step_index`. If `ref_task.step_index >= this.step_index`, append the new issue and `continue`.

- **Option B:** Add a separate pass on **List** that only checks output-reference section ordering (e.g. `List.validate_output_reference_sections()`) and appends to `parser.issues`. This keeps step indices inside List and avoids changing `validate_references` signature. Details would not need to know its step index; List would iterate steps and tasks and for each task's `#X-results` references, look up the step of X and compare. Slightly more logic in List, no change to Details API.

Either option is valid; A reuses the existing "per-task validate_references" and keeps all reference rules in one place; B keeps Details unaware of step structure.

### 3.2.1 Concrete code proposals (Option A)

**1. Details — add `step_index` (store section when list is built)**

**File:** `liboccoder/Task/Details.vala`

Add a property so each task knows its section; the parser sets it when building the list (see step 2).

```vala
	/**
	 * Step index (0-based) of the section this task belongs to, or -1 if not set.
	 * Set when the task list is built (e.g. in ResultParser.parse_task_list) so we don't
	 * re-derive section from position later.
	 */
	public int step_index { get; set; default = -1; }
```

**2. ResultParser — set step_index when iterating, then validate_references()**

**File:** `liboccoder/Task/ResultParser.vala`

When iterating steps, assign each task's `step_index` then call `validate_references()` (no argument). Replace the reference-validation loop in `parse_task_list()` with:

```vala
		this.runner.task_list.fill_names();
		var steps = this.runner.task_list.steps;
		for (var i = 0; i < steps.size; i++) {
			var step = steps.get(i);
			foreach (var t in step.children) {
				t.step_index = i;
				if (t.slug() != "") {
					this.runner.task_list.slugs.set(t.slug(), t);
				}
				t.validate_references();
				if (t.issues != "") {
					this.issues += "\n" + "Task (References): " + t.issues;
				}
			}
		}
		if (this.issues != "") {
			this.runner.task_list = null;
		}
	}
```

**3. List — public property `slugs` (slug → task); callers use the map directly**

**File:** `liboccoder/Task/List.vala`

Add a **public** property **slugs** (the map). No setter method and no `has_slug` / `step_index_for_slug` — callers use the hashmap directly: `task_list.slugs.set(slug, t)` when building, `task_list.slugs.has_key(slug)` to check, `task_list.slugs.get(slug)` to get the task (then `.step_index`). Replace the current **has_slug** implementation (the loop) with a check against the map, or remove **has_slug** and have all callers use **slugs.has_key()** and **slugs.get()**.

```vala
	/** Slug → task (Details). Populated when the list is built; callers use .set(), .has_key(), .get() directly. */
	public Gee.HashMap<string,Details> slugs { get; private set; default = new Gee.HashMap<string,Details>(); }
```

**4. Details — validate_references() and section-order check (no parameter)**

**File:** `liboccoder/Task/Details.vala`

Keep signature `validate_references()` (no parameter). In the `#...-results` branch, use **task_list.slugs** directly: check `task_list.slugs.has_key(name_slug)`; get the task with `task_list.slugs.get(name_slug)` and use `ref_task.step_index` for the section-order check.

Replace:

```vala
				if (anchor.has_suffix("-results")) {
					var name_slug = anchor.substring(0, anchor.length - "-results".length);
					if (!this.runner.task_list.has_slug(name_slug)) {
						this.issues += "\n" + "Invalid reference target \"" +
							 href + "\": no task for \"" + name_slug + "\".";
					}
					continue;
				}
```

with:

```vala
				if (anchor.has_suffix("-results")) {
					var name_slug = anchor.substring(0, anchor.length - "-results".length);
					if (name_slug == "" || !this.runner.task_list.slugs.has_key(name_slug)) {
						this.issues += "\n" + "Invalid reference target \"" +
							 href + "\": no task for \"" + name_slug + "\".";
						continue;
					}
					var ref_task = this.runner.task_list.slugs.get(name_slug);
					if (this.step_index >= 0 && ref_task.step_index >= 0 && ref_task.step_index >= this.step_index) {
						this.issues += "\n" + "Reference target \"" + href +
							"\" refers to a task in the same or a later section. " +
							"A task may only reference the Output of a task from an earlier section. " +
							"Move this task to a later section or remove this reference.";
					}
					continue;
				}
```

### 3.3 Error message (for LLM retry)

- Message should:
  - Identify the **reference** (e.g. `#X-results`).
  - State that it refers to a task in the same or a later section.
  - State the rule: only tasks from an **earlier** section may be referenced as Output.
  - Suggest remediation: move this task to a **later** section, or remove the reference.

Example (can be tuned for tone/length):

- `"Reference target \"#research-1-results\" refers to a task in the same or a later section. A task may only reference the Output of a task from an earlier section. Move this task to a later section or remove this reference."`

### 3.5 Update task list creation description (tell the LLM the rule)

Update the task list creation prompts so the LLM is explicitly told: **if a task references another task's Output** (`#slug-results`), that task **must be in a later section** — do not put both tasks in the same section. This reduces invalid task lists and retries.

**Files to update:** `resources/skill-prompts/task_creation_initial.md`, `resources/skill-prompts/task_list_iteration.md`.

**1. task_creation_initial.md**

- In the **Output format** section, in the paragraph that describes **Tasks** (line ~63), after the sentence about "Sections run sequentially" and "Within a section you can have multiple tasks", add:

  **Text to add:**  
  `If a task needs the **Output** of another task (e.g. references \`#research-1-results\`), put the **consumer** task in a **later** task section than the one that produces that output. Do **not** put both tasks in the same section — the output is not available until the producer's section has completed.`

- In **Reference link types**, under **Task output** (line ~79), add a sentence after the existing task-output description:

  **Text to add:**  
  `A task that references another task's output (e.g. \`#research-1-results\`) must be in a **later** task section; the producer and consumer cannot be in the same section.`

**2. task_list_iteration.md**

- In **When adding new tasks - RAPIR and ordering** (line ~27), after "put tasks that depend on prior outputs in a later section", add:

  **Text to add:**  
  `If a new task references another task's output (\`#slug-results\`), it must be in a **later** section than that task; do not put the consumer and producer in the same section.`

- In **Reference link types**, under **Task output** (line ~65), add the same rule:

  **Text to add:**  
  `A task that references another task's output must be in a **later** task section than the producer; they cannot be in the same section.`

**Summary:** Add the same rule in both prompts (output-format / ordering and reference link types) so the LLM sees it when structuring sections and when using task-output references.

## 4. Summary

| Item | Proposal |
|------|----------|
| **Rule** | A task may reference `#X-results` only if task X is in a strictly earlier section (step index &lt; this task's step index). |
| **Where** | In the existing task-list verification path, when we verify references (e.g. in or alongside `Details.validate_references()`). |
| **List** | **Public** property **slugs** (slug → task). Callers use the map directly: `.set(slug, t)` when building, `.has_key(slug)` and `.get(slug)` for lookups; no `has_slug` / `step_index_for_slug`. |
| **Store section on tasks** | When building the list, set `t.step_index = i` for each task (parser); `validate_references()` uses `this.step_index` (no parameter). |
| **Error text** | Clear message that the reference targets the same or a later section and that the task must be moved to a later section or the reference removed. |
| **Task list creation description** | Update `task_creation_initial.md` and `task_list_iteration.md` so the LLM is told the rule: a task that references another task's Output must be in a **later** section (§3.5). |

Implementing this will make same-section output reference an explicit validation error and allow the existing retry loop (e.g. with `parser.issues`) to ask the LLM for a corrected task list (e.g. move the second task to "Task section 2" or remove the reference).
