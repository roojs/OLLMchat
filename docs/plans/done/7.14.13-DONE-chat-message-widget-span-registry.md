- One canonical code proposal shape per `docs/guide-to-writing-plans.md`.

## Purpose

- 🔷 Replace `ChatView.idx_widgets` tail with `RenderBox` (`by_id`, `mark()`); `Message` carries only `idx_first` / `idx_last` (inclusive widget-row span), not a third scalar `idx`.
- 🔷 `RenderBox` is Gtk-only; no `OLLMchat.Message` inside it. `ChatView` calls `this.render_box.mark()` at assistant span start (Part 1c); `ChatWidget` assigns `idx_first` / `idx_last` inline from `this.chat_view.render_box`.
- 🔷 `ProgressItem`: `msg_idx_to_string()` holds Idx string logic on `this`; the extended getter `msg_idx_txt` only calls that method (not on `Message`, not `static`). `ProgressRunner` / `Details` / `Tool` delete their local `msg_idx_txt` getters.
- 🚫 No GObject signals on `MarkdownGtk.Render` for registration.
- ℹ️ Related: `docs/plans/done/7.14.6-DONE-progress-chat-scroll-issues.md` (archived).

## Acceptance criteria

- 🔷 `RenderBox`: no `libollmchat` imports; public `by_id`; `append` via **`new void append`** when GTK exposes no virtual `append` (see **§2**); `mark()`; no `append_register`.
- 🔷 `RenderBox.append`: first exactly one line `base.append(new Gtk.Label(this.by_id.size.to_string()));` (id row **before** **`child`**), then `base.append(child)` and `this.by_id.add(child)`; **no** new locals, **no** helpers, **no** label in **`by_id`**.
- 🔷 `ChatView`: no `idx_widgets`; `public OLLMchatGtk.RenderBox render_box { get; private set; }` (ctor + each `clear()`); `scroll_to_idx` uses `render_box.by_id`. Call `this.render_box.mark()` at assistant span start (see Part 1c), not via extra `ChatView` methods.
- 🔷 Replay and progress Idx column documented in fenced hunks below.

## Current behaviour

- ℹ️ `idx_widgets` public list; tail-dedupe on `append_assistant_chunk` / `append_complete_assistant_message` / `append_tool_message`.
- ℹ️ Returns `idx_widgets.size - 1` (tail index); that scalar becomes `Message.idx_last` only (`idx_first` comes from `RenderBox.first_id` at `mark()`).
- ℹ️ `scroll_to_idx` indexes `idx_widgets`.
- ℹ️ `MarkdownGtk.Render(this.text_view_box)`.

## Proposed behaviour

- ℹ️ `ChatView` ctor builds the first `RenderBox`, appends it under `text_view_box`, passes it to `MarkdownGtk.Render`. After `clear()` tears down `text_view_box` children, allocate a new `RenderBox`, append it, and set `this.renderer.box` — same wiring as ctor; no in-place emptying of the old box.
- ℹ️ `mark()` then appends; `ChatWidget` sets `idx_first` from `this.chat_view.render_box.first_id` and `idx_last` from each `append_*` return (tail row), and re-syncs both after finalize where fenced.
- ℹ️ `scroll_to_idx(int)` still takes one row index; callers use `message.idx_last` as the scroll anchor (same row `append_*` used to return as `idx`). `ProgressView` / `Window.scroll_to_message` pass `message.idx_last` (see §9).
- ℹ️ `ProgressView` Idx column (`msg_idx_txt`): `first-last` when both `message.idx_first` and `message.idx_last` are ≥ 0 (single digit when equal), else `—`. `assign_message` watches `notify["idx-last"]` only (`idx_first` is written before `idx_last` each update); disconnect when both ≥ 0.
- ℹ️ Idx label text comes from `ProgressItem.msg_idx_to_string()` only; the extended getter `msg_idx_txt` must be `return this.msg_idx_to_string();` with no Idx logic in the property body.

```vala
namespace OLLMchatGtk
{
	/**
	 * Vertical Gtk.Box whose {@link append} registers each child in {@link by_id} order.
	 * Serves as the live markdown target from {@link ChatView}.
	 */
	public class RenderBox : Gtk.Box
	{
		/** Append order for scroll / id queries; updated only from {@link append}. */
		public Gee.ArrayList<Gtk.Widget> by_id { get; private set; default = new Gee.ArrayList<Gtk.Widget>(); }

		/** Start index of the current span; set by {@link mark}. */
		public int first_id { get; private set; default = 0; }

		/** Last assigned id, or 0 when {@link by_id} is empty. */
		public int last_id {
			get {
				return this.by_id.size > 0 ? this.by_id.size - 1 : 0;
			}
		}

		public RenderBox()
		{
			Object(orientation: Gtk.Orientation.VERTICAL, spacing: 0);
		}

		/**
		 * Append {@link child} to the hierarchy and register it in {@link by_id}.
		 * One leading {@link Gtk.Label} whose text is the row id **`child`** will get (**`by_id.size`** before **`child`** is registered); then **`child`** (no new locals).
		 *
		 * @param child widget to append
		 */
		public new void append(Gtk.Widget child)
		{
			base.append(new Gtk.Label(this.by_id.size.to_string()));
			base.append(child);
			this.by_id.add(child);
		}

		/**
		 * Set {@link first_id} to the next index that {@link append} will assign.
		 */
		public void mark()
		{
			this.first_id = this.by_id.size;
		}
	}
}
```

	/** Idx column string from `this.message` span (`first–last`, or —). */
	public virtual string msg_idx_to_string()

- ℹ️ `Gee.ArrayList.size` is `int`; keep `first_id` / `last_id` as `int` (same as `Message.idx_first` / `idx_last`) so no `uint` casts are needed around `by_id.size` or assignments.
- ℹ️ In `append_assistant_chunk`, `append_complete_assistant_message`, `append_tool_message`, and `clear()`, keep existing commented `idx_map` `GLib.debug` blocks; only drop `idx_widgets` mutation and rewire size reads to `this.render_box.by_id` where the fenced hunks show.
- ℹ️ `clear()` already unparents every `text_view_box` child; follow that with a new `RenderBox` and `this.renderer.box = this.render_box` (Part 5b).
- ℹ️ `ProgressItem.assign_message`: watch `notify["idx-last"]` only; `idx_first` is written before `idx_last` on each `ChatWidget` update. After `this.message = m`, notify `msg_idx_txt`. Extra `GLib.debug` for rebind goes only inside the existing `if (this.message != null && this.idx_notify_id != 0)` (real disconnect path, not debug-only branching — see `docs/coding-standards.md`).
- ℹ️ `ProgressItem.msg_idx_to_string()`: default virtual instance method on `ProgressItem`. Extended getter `msg_idx_txt` only forwards to it for `notify` / Gtk.
