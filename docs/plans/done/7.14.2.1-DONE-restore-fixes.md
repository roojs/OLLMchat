# 7.14.2.1 — Restore → `ProgressView` (replay + tree + titles)

**Status:** **failed in practice (2026-04-14)** — The **concrete code** from this plan is largely **in the tree** (`ProgressView` **`Gtk.TreeListModel`** + **`Gtk.TreeExpander`**, **`Details.title`** from **`task_data["name"]`**, **`Runner.on_replay` → `progress`** calls). **Acceptance is not met** when **restoring a Skills session**: the strip is **too short** (~two lines visible); the **Stage** column often reads **blank**; **expand works** but the row **collapses again on its own** shortly after expanding. **Working hypothesis:** **`Details` / `ProgressRunner` `status`** is **not** driven to **`PhaseEnum.COMPLETED`** (and related **`status_str`**) during replay the way live execution does, so **`ProgressList.clear_pending`** / tail logic **drops or churns** rows that are not anchored as **completed** — which would also explain **unstable** tree state. ## Known issues — restore session (reported)

1. **Strip height:** **`ProgressView`** viewport is **too small** — only **~2 lines** of the **`ColumnView`** are usable. **Interim fix:** increase **`ScrolledWindow.set_min_content_height`** to roughly **three times** the current value (**96 → ~288**), or otherwise allocate vertical space so multiple task rows are visible without fighting the parent layout.
2. **Blank Stage:** The **status / Stage** column often appears **empty** after restore (Pango markup or **`status_str`** source). Verify **`PhaseEnum`** / **`Details.status`** / **`ProgressRunner`** phase on hydrated objects during **`on_replay`** vs live.
3. **Expand then auto-collapse:** **Nested rows expand** correctly, but the expander **closes by itself** shortly after — consistent with **model churn** (row removed/reinserted or **`items_changed`** on the **`TreeListModel`** parent) rather than a pure **`Gtk.TreeExpander`** bug.
4. **Hypothesis — completion / list retention:** **`ProgressList.clear_pending`** stops at the first **`PhaseEnum.COMPLETED`** row from the **tail**; non-completed tail rows are removed. If replay **never sets** prior work to **`COMPLETED`** (or **`ProgressRunner`** phase) the way live **`add_completed`** / **`List` does, **`rebuild()`** / **`clear_pending`** + **`add_pending`** may **strip** history or **replace** **`Details`** references, which would **collapse** the tree and make **Stage** look wrong. **Next step:** trace **`on_replay`** → ensure replayed **`Details`** and **`ProgressRunner`** rows have **`status`** / **`phase`** matching transcript progress before each **`progress.rebuild()`** / **`add_pending`**; compare to **`ProgressList`** invariants in **`ProgressList.vala`**.

## TreeListModel + TreeExpander — notes

**Goal:** Wrap **`Runner.progress`** in **`Gtk.TreeListModel`**; **`passthrough = false`** → **`ListItem.item`** is **`Gtk.TreeListRow`**; **`TreeExpander`** on the **Title** column only; **Stage** column binds **`status_str`** via **`row.get_item()`**. **Child model:** **`ProgressItem.children`** when **`get_n_items() > 0`**, else **`null`**.

**GTK4:** [Gtk.TreeListModel](https://docs.gtk.org/gtk4/class.TreeListModel.html), [Gtk.TreeExpander](https://docs.gtk.org/gtk4/class.TreeExpander.html), [Gtk.TreeListRow](https://docs.gtk.org/gtk4/class.TreeListRow.html).

**Source:** **`liboccoder/Task/ProgressView.vala`** — inline **`TreeListModel`** create func in **`ProgressView()`** and **`set_runner`** (no shared helper).
