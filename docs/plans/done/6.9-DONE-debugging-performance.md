### 4.1. `libocmarkdowngtk/State.vala` — `add_text` without `parent.update_ranges_from`

**Follow-up to 4:** Proposal **4** ruled out “all cost is in **`on_text` → `add_text`**” as a whole; **4.1** isolates the **ancestor walk** (`update_ranges_from`) after insert + tag apply.

**Hypothesis:** **`update_ranges_from`** walks ancestors and reapplies tags every insert (see comment in **`Parser.handle_line_break`** about deep stacks).

#### Remove

```vala

			// Update parent ranges and apply parent tags
			if (this.parent != null) {
				this.parent.update_ranges_from(this);
			}
		}
```

#### Replace with

```vala
			buf.move_mark(this.end, start_iter);

			// docs/plans/6.9 proposal 4.1: A/B parent walk off — restore if block to re-enable nested tag ranges
			// Update parent ranges and apply parent tags
			// if (this.parent != null) {
			// 	this.parent.update_ranges_from(this);
			// }
		}
```

#### Interpretation

- Replay **much faster**: **ancestor tag / mark updates** are a major cost; consider structural fixes in **`State`** / fewer walks per chunk (separate from **`6.8`**).
- **Little change**: cost is elsewhere (plain insert/apply_tag, parser, **`CODE_TEXT`**, etc.).

#### Result

```text
confirmed — full-range re-apply per ancestor was dominant; see **Fix** below
```

#### Fix (implemented)

**Cause:** Each insert called `apply_tag` on the **entire** span `[ancestor.start, ancestor.end]` for every ancestor. As the paragraph grew, that became **O(depth × paragraph length)** work **per chunk**, on top of the leaf insert.

**Change:** In **`libocmarkdowngtk/State.vala`** — `update_ranges_from` now tags only the **new segment** `[old_end, new_end]` (where `old_end` / `new_end` are the ancestor’s end mark before and after the child’s insertion) and then moves the ancestor’s end mark. Earlier inserts already tagged `[ancestor.start, old_end]`, so nested styles stay correct without re-walking the full paragraph.

**Alternatives considered:** Idle-coalescing parent updates (adds latency and complexity); flattening the style model (large refactor). Incremental tagging preserves the existing `State` tree and Gtk tag model.
