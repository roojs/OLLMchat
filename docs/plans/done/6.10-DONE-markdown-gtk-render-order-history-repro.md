# 6.10 Markdown GTK render order / history repro (investigation)

**Status:** done

See **`.cursor/rules/CODING_STANDARDS.md`** checklist before any future code changes.

---

## Purpose

Long assistant / session content in **`MarkdownGtk.Render`** sometimes appears **out of order** or **mangled at the tail** (fragments from earlier in the document, metrics line not last, text ‚Äúglued‚Äù across logical lines). This plan records **what was tried** in a debugging thread, **GTK errors seen**, and **suggested next steps** ‚Äî without prescribing random renderer edits until the failure mode is isolated.

## Acceptance criteria (for a *future* approved fix)

- A **minimal** repro (fixture + steps) demonstrates wrong order **or** documents that the bug is parser/markdown semantics (e.g. soft breaks).
- Any renderer change is justified by **buffer/mark invariants** (same `Gtk.TextBuffer` for marks, tags, iters) or by **measured** GTK behaviour ‚Äî not by trial-and-error.
- No merge until **ChatView** and **`oc-test-gtkmd`** paths are both considered (single block vs multiple `ui` messages).

## Symptom (issue)

- **Expected:** Visual order matches serialized message order (e.g. last `ui` message is only a short metrics line when the session JSON says so).
- **Observed:** End of view can show a **concatenated / shuffled** mix of link titles, paths, `task://‚Ä¶`, and phrases like `no further tool calls needed` ‚Äî i.e. content that **belongs earlier** in the same markdown stream, not a simple ‚Äúmissing `\n`‚Äù between widgets.
- **Hypothesis (unconfirmed):** `Gtk.TextMark` / `Gtk.TextTag` / `State.add_text` interaction, **or** **`render.current_buffer`** vs table **fake/cell** buffers, **or** incremental parse boundaries ‚Äî needs isolation before coding.

## Errors encountered during attempts (historical)

**At the time of debugging (no longer present after full revert to last commit):**

```text
G_LOG_LEVEL_CRITICAL : Gtk : gtk_text_buffer_apply_tag: assertion 'gtk_text_iter_get_buffer (start) == buffer' failed
```

Meaning: **`gtk_text_buffer_apply_tag`** was invoked with a **start `TextIter`** whose buffer was not the buffer on which **`apply_tag`** was called (or end/start buffer mismatch). This is consistent with mixing **`this.render.current_buffer`** with marks living in **another** `Gtk.TextBuffer` (e.g. table paths in **`libocmarkdowngtk/Table.vala`**).

**Current tree (after revert):** no local patches; **no** assertion reproduced until the same repro steps are run again.

## What was tried (reverted ‚Äî do not reapply blindly)

### 1. `examples/oc-test-gtkmd.vala` ‚Äî history replay alignment

**Idea:** Mirror **`ChatView`**: between consecutive **`ui` / `content-stream`** bodies, call **`end_block()`** + **`start()`** so parser state does not carry across assistant chunks like a single stream.

**Sketch (conceptual):**

```vala
// Pseudocode ‚Äî NOT in tree after revert
if (need_split) { md_renderer.end_block(); md_renderer.start(); }
else if (!main_ready) { md_renderer.start(); }
md_renderer.add(body);
md_renderer.flush();
```

**Result:** Helps **test harness parity** with the app; **does not** by itself fix a bug inside **`State`/`Render`** if the same wrong order appears in **both** places.

---

### 2. `libocmarkdowngtk/Render.vala` ‚Äî trailing newline in `end_block()`

**Idea:** Before dropping **`current_buffer`**, if text exists and the buffer does not end with `\n`, append `\n`.

**Sketch:**

```vala
// Pseudocode ‚Äî NOT in tree after revert
if (current_buffer != null) {
    // get_end_iter, backward_char, if last char != '\n' then insert "\n"
}
```

**Result:** User reported **no visible improvement** for the ordering / mangling issue.

---

### 3. `libocmarkdowngtk/State.vala` ‚Äî `add_text` / `update_ranges_from`

Several variants were attempted:

**A. Tag the inserted slice using explicit iters after `insert` (no `get_iter_at_mark(this.end)` between insert and tag):**

```vala
buf.get_iter_at_mark(out range_start, this.end);
Gtk.TextIter range_end = range_start;
buf.insert(ref range_end, text, -1);
buf.apply_tag(this.style, range_start, range_end);
buf.move_mark(this.end, range_end);
```

**B. Use `this.end.get_buffer()` instead of `this.render.current_buffer`** for insert/tag (table fake/cell vs main).

**C. `insert_with_tags(ref ins, text, -1, this.style)`** to avoid separate **`apply_tag`**.

**D. `update_ranges_from`:** buffer checks (`from_child.end` vs `ancestor.end`), skip sync on mismatch.

**Result:** **`apply_tag` assertion** could still fire; **ordering** issue not clearly resolved. Conclusion: **stop iterating on `State.vala` without a controlled repro** (step-through or buffer dump).

## Ideas for *structured* next steps (no hacking)

### A. Pause after each history item (`oc-test-gtkmd`)

**Idea:** After each JSON message that feeds the renderer (or after each `ui` / `content-stream` body), **wait for Enter on stdin** (optional flag, e.g. `--history-step`).

**Why:** Confirms whether the corruption is **time/event-order** related (idle, resize, nested flush) vs **purely** deterministic from parse + insert sequence.

**Sketch:**

```vala
// Future ‚Äî only with approval
if (opt_history_step) {
    stdout.printf("Press Enter for next message‚Ä¶\n");
    stdin.read_line();
}
```

---

### B. Golden buffer dump

For a **fixed** session tail JSON, dump **`Gtk.TextBuffer` plain text** (and optionally tag names) after each message boundary to a file. Compare to **expected** concatenation of bodies. Locates whether corruption is **buffer content order** vs **Pango/layout**.

---

### C. Single vs multi `TextView`

Verify whether **`ChatView.append_complete_assistant_message`** + **`finalize`** already **end_block** per message. If yes, the bug is **inside one block‚Äôs** parse/render, not missing **`end_block`** between DB messages.

---

### D. Table / fake buffer

Audit **`libocmarkdowngtk/Table.vala`** (`set_renderer_to_fake`, cell `current_buffer`) against **`State`** lifetime: any path where **`current_state`** still points at states whose marks live in **buffer A** while **`render.current_buffer` is B** is a candidate for **`apply_tag`** assertion and visual corruption.

---

### E. Markdown semantics

For ‚Äúline A then line B‚Äù **without** a blank line, CommonMark often treats a single newline as a **soft break** (space / `<br>` behaviour). **`Render.on_softbr`** may map to space ‚Äî that is **not** the same as ‚Äúwrong order‚Äù, but can look like ‚Äúglued‚Äù lines. Separate **content** test from **renderer** test.
