## Purpose

Capture current issues observed when running the skills process (e.g. task creation, refinement, execution). These are testing/UX issues that need to be addressed so the pipeline behaves reliably and doesn’t waste effort or drift from the user’s intent.

## Problem Summary

During testing (e.g. trying to find option args on “the tour”), several pain points emerged: the code search tool cannot target a specific file; the task list keeps retrying searches when nothing is found; the original goal gets altered by each LLM in the chain (“Chinese whispers”); the refinement stage is overloaded with information and it’s unclear what each skill’s refinement responsibility is; and some tasks lack refinement stages, so they don’t receive task list summary or a clear way to update references from prior work.

---

## 1. Code search tool — cannot target a specific file

**Issue:** The code search tool does not appear to support targeting a specific file. When the agent needs to find something in a particular file (e.g. option args in a given tour or module), search runs broadly and may not focus on that file.

**Goal:** Allow or add a way for the code search tool to target a specific file (or narrow scope) so that “search in file X” is possible and the agent doesn’t have to rely on unfocused global search.

**Possible directions:** Add a file/path parameter to the code search tool; or a “scope” (e.g. path prefix, single file) so the tool can restrict results to that file or subtree.

---

## 2. Task list prompts — limit retries when nothing is found

**Issue:** When doing searches (in documentation or in code), if nothing is found, the system keeps trying over and over. This wastes time and tokens.

**Goal:** Instruct the model (via task list / planner prompts) that when searches in documentation or code return nothing, it should **not** keep retrying indefinitely. Limit to a few tries (e.g. 2–3), then either use a fallback strategy (e.g. different query, different source) or record “not found” and move on.

**Scope:** Prompt changes for task creation and/or task list behaviour so the stated policy is “limit search retries when nothing is found.”

---

## 3. Original prompt / goal — “Chinese whispers” (goal drift)

**Issue:** Each LLM in the pipeline seems to modify the original prompt or goal. The intent drifts as it passes through task creation, refinement, and execution (“Chinese whispers”).

**Goal:** The **goal**, once set, should be **fixed**. Nobody (and no LLM step) should change it during task refinement or task iteration. Options to enforce this:

- **Fix the goal at the start:** The original user prompt/goal is captured once and passed through unchanged; all downstream steps receive the same immutable goal text.
- **Do not have the LLM generate the “original prompt”:** If the “original prompt” in the task list is meant to be the user’s goal, it should be **injected** from the actual user input, not generated or paraphrased by the LLM. That way the goal is never rewritten.

**Scope:** Runner and/or prompt design so that (a) the goal is fixed at the beginning, and (b) task refinement (and any other iteration) does **not** modify the goal/original prompt.

---

## 4. Refinement stage — too much information; unclear per-skill refinement task

**Issue:** The refinement stage is getting confused by too much information. It’s not clear what the **refinement** responsibility is for each skill (e.g. analyze code vs others).

**Desired behaviour (to clarify in prompts and possibly Runner):**

- **Analyze code:** The refinement step for “analyze code” should take the **reference file** (and any other references) and **extract and add extra references** as needed. So “analyze code” refinement is clearly: use the referenced file(s), optionally pull in more references, and prepare the precursor so the skill has what it needs.
- **Outputs as references:** If a task uses the **result of a previous task’s output** (e.g. “output of task 3”), once that output is **filled in**, the refinement doesn’t need to **include that output again** as a separate reference blob if it’s already injected. Avoid duplicating or overloading the precursor with the same content in multiple forms.
- **Reference files:** Clarify in the refinement prompt (and possibly in each skill’s definition) what “refinement” means for that skill: e.g. for “analyze code” it’s “take reference file(s), extract/add references”; for others, define similarly so the LLM knows exactly what to do per skill.

**Tasks without refinement stages / empty refinement stages:**

- Some tasks do **not** have refinement stages (or have effectively "empty" refinement). In those cases we still need a way for the pipeline to **update references** from prior work. **Inject a summarized version of the task list so far** into those steps so they can consume and propagate reference information.
- For **empty refinement stages** (or when a skill has no custom refinement): (1) **Give them the task list summary so far** (summarized outcomes of previous tasks). (2) Their job is to **extract any reference information that has previously been gathered** and **add it to the references section**. (3) **Do not add whole files** — focus on **sections of files**, **methods**, and **result summaries** from previous tasks. The refinement output should curate precise references (file + section/method/snippet) rather than dumping full file contents.

**Goal:** Reduce confusion and token bloat in refinement by (1) defining per-skill refinement responsibilities (with “analyze code” explicitly: reference file + extract/add references), and (2) not re-including filled-in outputs redundantly when they’re already in context, and (3) ensuring tasks without refinement still receive task list summary and contribute to references via section/method-level extraction from prior results.
