# 7.14.7 — Task progress tree empty on first send (streaming / new chat)

**Status:** done (archived)

**Pointer:** `.cursor/rules/CODING_STANDARDS.md` — Checklist for all plans

**Layout:** `docs/guide-to-writing-plans.md`

**Related:** `docs/plans/done/7.14-DONE-task-progress-tree-ui.md`, `docs/plans/done/7.14.5-DONE-skill-progress-session-bind.md`

---

## Purpose

- While the Skills agent streams the **first** task-list response on a **new chat** (`EmptySession` → real `Session`), the task progress strip stays empty or shows **stale** rows.
- **Cause:** `OLLMcoder.Task.ProgressView` is wired only from `Skill.Factory.activate` → `set_runner(host.session_agent())`. That path does **not** run when the user sends the first message without changing the agent dropdown, so the view keeps a `TreeListModel` bound to a **previous** session’s `Runner.progress` (or never refreshes for the new `Runner`).
- **Outcome:** After first send, `ProgressView` tracks the **current** session’s `Runner` so `ProgressList` updates (`ProgressRunner`, then `add_pending` after parse) appear during/after that run.

---

## Scope

| In scope | Out of scope |
| -------- | ------------- |
| Ensure **`session.agent`** exists **before** UI factories bind after **`EmptySession` → `Session`** | Incremental markdown parse into tasks **while** streaming (would need streaming `ResultParser` / partial tasks) |
| **`libollmchat`** ordering in **`EmptySession.send`** | Rewriting **`ProgressList`** signalling |
| **`ollmapp`**: one mechanical **`get_active_agent().activate(window)`** on **`session_activated`** (same pattern as **`session_restored`**) | **`OLLMcoder`** type checks or **`is Skill.Factory`** in **`Window`** (**👎** per **`7.14.5`**) |

---

## Discussion

- **ℹ️** `Runner.send_async` begins with `this.progress.add(rp)` — the model for the **active** `Runner` gains a “Task Creation” row immediately; `parse_task_list` + `add_pending` run only after the **full** assistant text is available (not mid-chunk).
- **⚠️** If the UI is bound to the wrong `Runner`, **no** row updates appear for the run the user is watching — this matches “tree not filled by the first task generation” while chat still streams correctly (different code path).
- **ℹ️** `Manager.switch_to_session` already calls `ensure_agent_handler()` **before** `session_activated` — loaded sessions are OK; the gap is **`EmptySession.send`** only.
- **🔶** After this fix, the **detail** rows still appear only after parse completes; the **runner** row may appear as soon as `send_async` starts — that is existing behaviour.

---

## Acceptance criteria

1. **New chat + Skills agent + first user message:** while the model streams the task list, the progress strip is bound to the **new** session’s runner (at least the “Task Creation” row appears for that run when `send_async` runs).
2. **No regression** for **`switch_to_session`** / **`session_restored`** progress binding (`7.14.5`).
3. **`Window`** does not import **`OLLMcoder`**; **`get_active_agent().activate`** is generic.
4. **`Skill.Factory.activate`** stays free of **`session_agent() == null`** as the primary guard — **`ensure_agent_handler`** runs before activate on the fixed path (**⚠️** **`7.14.5`**).

---

## Concrete code proposals

Mandatory pattern: each fenced block has a **`####`** heading whose **first word** is **Keep**, **Remove**, **Replace with**, or **Add**.

---

### 1. `EmptySession.send` — ensure handler before `session_activated`

**Target:** `libollmchat/History/EmptySession.vala` — inside `send`, after building `real_session` and before `manager.session_activated`.

#### Add — call ensure before activating UI session signal

After `real_session.activate();` (or immediately before `session_activated`), insert:

```vala
			real_session.ensure_agent_handler();
```

So `manager.session` points at a `Session` whose **`agent`** is the **`Runner`** that **`send_async`** will use.

---

### 2. `Window` — bind agent chrome on `session_activated`

**Target:** `ollmapp/Window.vala` — **`history_manager.session_activated`** handler (the block that adjusts **`agent_dropdown`** / project).

#### Add — factory activate alongside existing session switch logic

At the **start** of the **`session_activated`** callback body (after null-checks if any), invoke the same pattern as **`session_restored`**:

```vala
				var factory = this.history_manager.get_active_agent();
				factory.activate.begin(this, (obj, res) => {
					factory.activate.end(res);
				});
```

**Note:** This may pair with an existing **`session_restored`** activate on session switch — **`Skill.Factory.activate`** should remain **idempotent** (**`set_runner`** + visibility). If double **`activate`** causes measurable issues, follow-up can narrow to “only when **`session.agent`** identity changed” (**💩** confirm before adding identity tracking).

- Regression: switch between two Skills sessions with same **`agent_name`** — progress matches active session (**`7.14.5`** acceptance).
