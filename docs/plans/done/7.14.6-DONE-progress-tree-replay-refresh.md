# 7.14.6 — Progress tree: replay vs UI refresh (why rows lag, what to fix)

**Status:** done (archived)

**Related:** `docs/plans/7.14.6-progress-tree-click-scroll.md`, `liboccoder/Skill/Factory.vala`, `liboccoder/Skill/Runner.vala` (`on_replay`), `libollmchat/History/Session.vala` (`restore_messages`)

**Implemented:** `liboccoder/Skill/Factory.vala` — `ProgressView.set_runner` and `window` run immediately after widgets are created, **before** `yield initialize_widget()`; one-shot SourceView/project load guarded by private **`done_init`** inside `initialize_widget()`. Proposals **B–E** remain optional follow-ups.

---

## Purpose

- Assume **`ProgressList` is meant to grow and reshape incrementally** during restore whenever `on_replay` calls `add` / `rebuild` / `add_completed` / `add_pending` — that **should** surface in `ProgressView` as **`items_changed`** propagate through `Gtk.TreeListModel`.
- Document **why incremental model updates might still not show incrementally** in the strip (bind/show timing, **`can_replay`**, optional GTK edge cases), not “rows aren’t added until late” as the headline insight.
- Propose **concrete changes** so the column view is attached to **`runner.progress`** early enough that live replay mutations are visible.

---

## How replay and the tree actually connect

### 1. Session restore drives both chat and agent replay

`Session.restore_messages()` (see `libollmchat/History/Session.vala`) does, per message (except the last streaming edge case):

1. `manager.message_added(message, session)` → ChatWidget renders the chat line.
2. `agent.on_replay(message)` → agent-specific side effects.

Between messages it schedules a **100 ms** timeout and **yields**, so the main loop can run other sources (GTK layout, paint, other idle work). So replay is **already** structured to interleave UI work — it is not one giant synchronous loop without yields.

### 2. Incremental tree updates are expected once progress mutates

Whenever **`on_replay`** updates **`Runner.progress`**, **`ProgressList`** emits **`items_changed`**. Over a typical skill transcript that happens **many times** (task list parse → runners + pending details; **`rebuild`** on refinement/exec/post-exec; **`add_completed`** on step completion; etc.). So **after** the first mutation, the reasonable expectation is: **the strip should reflect each subsequent mutation as restore advances**, not jump only at the end.

If that incremental behaviour is **not** what you see, the interesting hypotheses are **not** “rows are sparse vs chat lines” but:

- **Model not bound yet:** **`ProgressView.set_runner(runner)`** ran **after** **`yield initialize_widget()`** on first skill activation — restore may already be applying **`on_replay`** while **`ColumnView`** is still wired to the **placeholder** `ListStore` from **`ProgressView()`**, not **`runner.progress`** (**§5**). The strip can already be **on-screen** (widget is appended; GTK defaults to visible); **binding** is what lags. *(Addressed by binding before `initialize_widget()` — see **Implemented** above.)*
- **Replay skipped:** **`session.can_replay == false`** → **`on_replay`** returns immediately every time → **no** `ProgressList` growth during restore (**§3**).
- **Rare GTK/view gap:** model emits changes but column view doesn’t refresh until a later frame — **instrument first** (**proposal E**, **D**).

### 3. `can_replay` gates **all** of `on_replay`

At the top of `Skill.Runner.on_replay()`:

```vala
if (!this.session.can_replay) {
    return;
}
```

`can_replay` is set when loading JSON (`SessionJson` marks it when any **`agent-stage`** message appears, and it can be stored as **`can-replay`** on disk). **If `can_replay` is false for a loaded session**, **`on_replay` is a no-op for every message** → **`ProgressList` never grows during restore** even though chat replay still runs via `message_added`.

If we ever see “chat restores but tree stays forever empty”, **verify `session.can_replay`** for that session (serialization path, migration, or skill-only sessions).

### 4. When `ProgressList` changes, GTK **should** refresh the column view

`ProgressList` implements `GLib.ListModel` and calls `items_changed()` when rows are added/removed/replaced (`add`, `rebuild`, `add_completed`, `add_pending`, etc.). `Gtk.TreeListModel` listens to the root model and forwards changes; `Gtk.ColumnView` with `Gtk.SingleSelection` should reflect updates **without** manual `queue_draw`.

So “signals not firing” is **unlikely** if:

- `items_changed` is actually emitted (guard clauses like `added == 0 || !call_changed` on `add_pending` mean some calls intentionally emit **no** change), and  
- the widget is bound to **this** model instance (`runner.progress`).

### 5. UI timing: **`set_runner`** ran late on first activation (primary suspect for “not incremental”)

**Historical diagnosis (pre-fix):** `Skill.Factory.activate()` created **`ProgressView`** and **`yield`ed `initialize_widget()`** before **`set_runner`**.

The strip **widget was already in the layout**; the gap was:

- **`ColumnView`** showed the **placeholder** root model until **`set_runner`** replaced **`column_view.model`** with **`Gtk.TreeListModel(runner.progress, …)`**.
- **`session_activated`** vs **`restore_messages`** could **interleave**: **`on_replay`** mutated **`runner.progress`** before **`set_runner`**, so **`items_changed`** fired with **no** bound column view — then **`set_runner`** attached and the UI **caught up**.

**Current behaviour:** **`set_runner`** / **`window`** run before **`yield initialize_widget()`**; **`done_init`** ensures init runs once.

## Proposed solutions (in order of leverage)

### A. Call **`set_runner(runner)`** before **`yield initialize_widget()`** (first activation only)

**Done** — see **Implemented** at top.

### B. Confirm `can_replay` for skill sessions end-to-end

**Goal:** Eliminate silent “replay chat but never replay task state” for skill sessions.

**Sketch:**

- On session load, if `agent_name` is the skill runner and messages contain **`agent-stage`** (or other replay markers), ensure **`can_replay`** is true (derive from messages if missing in older JSON).
- Optionally assert / log in debug when `on_replay` is skipped due to `!can_replay` during `restore_messages`.

### C. Optional UX: “pending” or skeleton row

**Goal:** If the preamble is long, show a single placeholder row (“Restoring task progress…”) driven by a small flag during `restore_messages`, cleared when the first real `ProgressItem` appears — **only if** product wants non-empty strip during replay.

**Risk:** Extra state; keep behind a simple `Session.is_restoring` or reuse existing `ChatWidget.restoring_session`.

### D. Only if a real GTK gap is proven: post-bind refresh

**Goal:** If instrumentation shows `items_changed` firing while the column view stays stale (unexpected), try **`queue_allocate()` / `queue_resize()`** on the `ColumnView` or `ScrolledWindow` after `set_runner`, or re-verify **`Gtk.ColumnView.model`** assignment order.

**Risk:** Treat as last resort; usually masks a bug elsewhere.

### E. Investigation checklist (before coding D)

1. Log **`ProgressList.get_n_items()`** after each `on_replay` that should mutate progress vs what `ColumnView` shows.  
2. Log **`session.can_replay`** at start of restore.  
3. Timestamp **`Factory.activate`** (start / after `initialize_widget` / after `set_runner`) vs **`restore_messages`** progress index.

---

## Acceptance criteria

- After **proposal A**, **`set_runner`** runs before the first **`yield initialize_widget()`**, so **`ProgressView`** observes **`ProgressList`** during overlapping **`restore_messages`** (subject to **`can_replay`**).  
- **First activation:** **`runner.progress`** bound without waiting on **`initialize_widget()`** unless we document a hard dependency.  
- **No silent skip:** skill sessions with replayable transcripts do not leave **`can_replay`** false by accident.  
- **Optional:** debug logging behind **`GLib.debug`** for `can_replay` / correlation of **`set_runner`** vs restore index.

---

## Summary one-liner

Incremental **`ProgressList`** updates during replay **should** show up incrementally in the strip once **`ColumnView`** is bound to **`runner.progress`**; the fix-shaped issue was **`set_runner`** running **after** the first **`yield initialize_widget()`**, so **`items_changed`** could fire on **`ProgressList`** while the UI still showed the **placeholder** model — plus **`can_replay`** false, which suppresses **all** progress replay.
