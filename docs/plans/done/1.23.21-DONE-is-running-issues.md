# Plan 1.23.21: session.is_running flag issues

## Summary

The `session.is_running` flag is currently set to `false` when we **process the final stream chunk** (when `response.done` is true in the stream data), **not** when the async `chat_call.send()` yield returns. This document records where `is_running` is set/cleared, how messages are sent to the LLM (send/yield flow), and the resulting semantic/timing issue. It also notes that **no code sets `is_running` from Idle/timeout**; the only deferred use is ChatWidget **reading** it in an Idle callback when updating the Send/Stop button after session switch.

---

## 1. Where we currently set `is_running`

### 1.1 Property definition

- **File:** `libollmchat/History/SessionBase.vala` (line 43)
- `public bool is_running { get; set; default = false; }`
- Comment: "agent can set this as running when doing tools"

### 1.2 Set to `true`

| Location | File | When |
|----------|------|------|
| Session.send() | `libollmchat/History/Session.vala` ~659 | Before `yield this.agent.send_async(message, cancellable)` for user messages. |
| Agent.send_async() | `libollmchat/Agent/Base.vala` ~360 | Before `yield this.chat_call.send(messages, cancellable)`. |
| Agent.execute_tools() | `libollmchat/Agent/Base.vala` ~275 | When returning tool replies that will continue the conversation (before caller sends them). |
| Runner.send_async() | `liboccoder/Skill/Runner.vala` ~116 | At start of send, before first `yield this.chat_call.send(...)`. |
| Runner.handle_task_list() | `liboccoder/Skill/Runner.vala` ~156 | At start of handle_task_list (so refinement/iteration streams are accepted by UI). |
| Details.refine() | `liboccoder/Task/Details.vala` ~313 | Before `yield this.chat_call.send(messages, null)` for refinement. |

### 1.3 Set to `false`

| Location | File | When |
|----------|------|------|
| Session.finalize_streaming() | `libollmchat/History/Session.vala` ~276 | When `response.done` is true — i.e. when the **stream chunk** containing `"done": true` is processed (inside the stream callback). |
| SessionBase.add_message() | `libollmchat/History/SessionBase.vala` ~360 | When `message.role == "done"` (done message added to session). |

| Runner.handle_task_list() | `liboccoder/Skill/Runner.vala` ~181 | At end of handle_task_list (normal exit). |

### 1.4 Where we do **not** set `is_running`

- **After `Session.send()` yield returns:** There is no code that sets `is_running = false` when `yield this.agent.send_async(...)` returns in `Session.send()`. By then, `finalize_streaming()` has usually already run (see below).
- **On send error:** If `agent.send_async()` or `chat_call.send()` throws (e.g. network error), nothing in Session or Agent clears `is_running`. ChatWidget’s `cleanup_streaming_state()` only calls `chat_input.set_streaming(false)`; it does not set `session.is_running = false`.

## 2. How we send messages to the LLM (send / yield flow)

### 2.1 Entry from UI

1. **ChatWidget** `send_chat_request()` creates a user `Message`, then:
   - `yield this.manager.send(this.manager.session, user_message, cancellable)`
2. **Manager.send()** (`libollmchat/History/Manager.vala` ~341):
   - `yield session.send(user_message, cancellable)`
3. **Session.send()** (`libollmchat/History/Session.vala` ~629):
   - For user messages: ensures agent, sets `this.is_running = true`, then
   - `yield this.agent.send_async(message, cancellable)`
   - (No `is_running = false` when this yield returns.)

### 2.2 Agent sends to Chat

4. **Agent.send_async()** (e.g. `libollmchat/Agent/Base.vala`):
   - Builds `messages` from session, sets `this.session.is_running = true`, `yield this.fill_model()`, then
   - `var response = yield this.chat_call.send(messages, cancellable)`
   - When this yield returns, agent is done with this send (no clearing of `is_running` here).

5. **Chat.send()** (`libollmchat/Call/Chat.vala` ~467):
   - Stores messages, then either
   - `return yield this.execute_streaming()` or
   - `return yield this.execute_non_streaming()`

### 2.3 Streaming path (typical)

6. **Chat.execute_streaming()** (~532):
   - Builds HTTP request, then:
   - `yield this.handle_streaming_response(message, (chunk) => { this.process_streaming_chunk(chunk); })`
   - When this yield returns, the HTTP stream has been fully read and the callback has been invoked for every chunk (including the final one with `done: true`).
   - Then it may `return yield this.toolsReply(response)` if there are tool calls, else returns `response`.

7. **handle_streaming_response()** (`libollmchat/Call/Base.vala` ~219):
   - `input_stream = yield this.connection.soup.send_async(...)` (get response body stream)
   - `yield this.process_json_streaming(input_stream, on_chunk)` (read line-by-line, call `on_chunk` for each JSON line)

8. **process_json_streaming()** (Base.vala ~273):
   - Loop: `line = yield data_input.read_line_async(...)`; when `line != null`, call `process_json_chunk(trimmed, on_chunk)`.
   - So each chunk (including the final one with `"done": true`) is processed **inside** this loop, synchronously in the callback. The **yield** of `handle_streaming_response` only returns when the loop exits (e.g. `line == null` or cancel).

### 2.4 What happens when the “done” chunk arrives

9. For a chunk with `"done": true`:
   - **process_streaming_chunk()** (Chat.vala ~592): `response.addChunk(chunk)` → in **Response.Chat.addChunk()** (`libollmchat/Response/Chat.vala` ~180) `this.done = chunk.get_boolean_member("done")`.
   - Then Chat emits `stream_chunk` and calls `this.agent.handle_stream_chunk(new_text, is_thinking, response)`.
   - **Agent.handle_stream_chunk()** calls **session.handle_stream_chunk()**.
   - **Session.handle_stream_chunk()** (~227): `if (response.done) { this.finalize_streaming(response); }`
   - **Session.finalize_streaming()** (~241): adds end-stream/done/metrics messages, then **`this.is_running = false`**.

All of that runs **synchronously** inside the same stream callback. So **`is_running` is set to false while we are still inside the stream processing loop**, before `process_json_streaming` does the next `read_line_async` and eventually returns, and before `execute_streaming`’s yield returns.

### 2.5 When does the “yield on the chat connection” actually return?

- The yield of `chat_call.send()` in the agent returns when:
  - `execute_streaming()` returns (after `handle_streaming_response` returns and any `toolsReply` completes), or
  - `execute_non_streaming()` returns.
- So the **chat connection yield** returns **after** we have already set `is_running = false` in `finalize_streaming()` (on the last chunk). For the simple no-tool-call case, the order is:
  1. Last chunk with `done: true` is processed → `finalize_streaming()` → `is_running = false`.
  2. Callback returns.
  3. `process_json_streaming` reads again, gets null, breaks, returns.
  4. `handle_streaming_response` returns.
  5. `execute_streaming` returns.
  6. `chat_call.send()` returns.
  7. `agent.send_async()` continues and returns.
  8. `Session.send()` continues after the yield (no code sets `is_running` here).

So we are **not** tying `is_running = false` to “the yield on the chat connection has returned”; we are tying it to “we have seen the stream chunk with `done: true`”.

## 3. Agent behaviour with tools: the tool-call loop

**Yes — we have exactly that loop.** The flow is:

1. **Message is sent, yield**  
   `Session.send()` → `agent.send_async()` → `chat_call.send(messages)` → `execute_streaming()` (or `execute_non_streaming()`). We yield until the HTTP stream (or request) completes.

2. **Response is processed**  
   For streaming: chunks are processed in the callback; when the chunk with `"done": true` is seen, `finalize_streaming()` runs and sets `is_running = false`. Then `handle_streaming_response` returns and `execute_streaming` continues with the full `response` (with `response.done` and `response.message.tool_calls`).

3. **If tools were requested**  
   `execute_streaming()` (and similarly `execute_non_streaming()`) checks `response.done && response.message.tool_calls.size > 0`. If true, it calls **`yield this.toolsReply(response)`** and returns whatever that returns.

4. **Inside `toolsReply`:**  
   - **Run tools and yield:** `var tool_reply_messages = yield this.agent.execute_tools(response.message.tool_calls);`  
     So we run all requested tools; the agent executes each, builds tool-reply messages, sets **`session.is_running = true`** (so the UI shows "running" for the next round), and returns the list of tool messages.  
   - **Send request with results and yield:**  
     `var next_response = yield this.send_append(messages_to_send);`  
     where `messages_to_send` = assistant message (with tool_calls) + tool reply messages.  
     `send_append()` appends to `this.messages` and calls **`yield this.send(this.messages, ...)`**, so we go back through `execute_streaming()` (or non-streaming) and wait for the next LLM response.

5. **Loop back**  
   When `send_append` returns, we have `next_response`. If `next_response.done && next_response.message.tool_calls.size > 0`, we **recursively** `return yield this.toolsReply(next_response)` — so we run tools again, send again, and wait for the next response. We only exit the loop when the last response has no tool calls (or is not done).

So the structure is:

- **Send (user or tool replies) → yield** → response  
- **If response has tool_calls** → run tools (yield) → send tool results (yield) → **same loop** (process next response, possibly more tool_calls).

**Where `is_running` sits in this loop:**

- At the **start** of each "send" (user message or tool continuation), `is_running` is set to `true` (in `Session.send()` or `Agent.send_async()` for the first send; in **`Agent.execute_tools()`** for each tool-continuation send).
- When each **stream** finishes, `finalize_streaming()` sets `is_running = false` (as soon as the "done" chunk is processed).
- So **between** "stream done" and the next `execute_tools()` call we have a window where `is_running` is **false** even though we are still inside the same logical "conversation" (inside `toolsReply` recursion). The UI could briefly show "Send" instead of "Stop" in that gap, and any logic that gates on `is_running` (e.g. accepting stream chunks) would see it false until the next round sets it true again.

## 4. Key issue: stream vs yield semantics

- **Current behaviour:** `is_running` is set to `false` when we **process the final stream chunk** (in `finalize_streaming()`), i.e. when we see `response.done` from the stream.
- **Alternative (what you have in mind):** `is_running` would be set to `false` when the **yield on the chat connection drops** — i.e. when the async send is truly “finished” from the caller’s point of view (e.g. when `chat_call.send()` returns, or when `Session.send()`’s `yield agent.send_async()` returns).

Implications of the current design:

1. **Refinement / iteration (Skill Runner):** The first response’s completion sets `is_running = false` in `finalize_streaming()`. The Runner then sets `is_running = true` again at the start of `handle_task_list()` so that refinement/iteration chunks are accepted by the UI. That works only because the Runner explicitly re-sets the flag; any path that forgets to set it again will see chunks dropped when the UI checks `is_running` (see `docs/plans/skill-runner-refinement-stream-to-ui.md`).
2. **Tool calls:** After the first assistant response (with tool_calls) we set `is_running = false` in `finalize_streaming()`. Then `execute_streaming` returns and we call `toolsReply()` → `execute_tools()` sets `is_running = true` again before sending tool results. So we rely on other code to re-assert “running” after we’ve already cleared it.
3. **Single source of truth:** The “running” state is not derived from one place (e.g. “we are inside a send until the top-level yield returns”) but from a mix of stream-done and various agents setting it true again.
4. **Errors:** If the send throws before or after any chunk, nothing in Session/Agent sets `is_running = false` when the yield returns, so the session can remain “running” after an error.

## 5. Delays / Idle.add / deferred execution — does anything modify `is_running` out of sequence?

**Finding: nothing sets `is_running` from inside Idle.add, timeout, or other deferred callbacks.**

All writes to `is_running` happen on the direct call stack: `Session.send()`, `finalize_streaming()`, `add_message()`, `cancel_current_request()`, `Agent.send_async()` / `execute_tools()`, Runner and Details. None of these are scheduled via `GLib.Idle.add`, `Timeout.add`, or similar.

**One deferred read (can observe stale or changed value):**

- **File:** `libollmchatgtk/ChatWidget.vala` — `on_session_activated()` (lines 275–281)  
  When the manager emits `session_activated(session)`, the handler schedules an **Idle** that runs later:
  ```vala
  GLib.Idle.add(() => {
      this.chat_input.set_streaming(session.is_running);
      return false;
  });
  ```
  So the **value of `session.is_running` is read when the Idle runs**, not when `session_activated` was emitted. By then, another part of the flow may have already set `is_running` to `true` or `false` (e.g. a stream finished and `finalize_streaming()` set it to `false`, or the Runner set it to `true` for refinement). So the Send/Stop button and input streaming state can be updated one main-loop iteration late, and reflect whatever `is_running` is at Idle run time. We do **not** modify `is_running` here — we only read it — but the **timing is deferred**, so the UI can briefly show a state that no longer matches the moment of activation, or can correct itself one tick later.

**Summary:** No delayed **writes** to `is_running`. One delayed **read** in ChatWidget that uses Idle to set the input’s streaming state; that read can see a value that changed after `session_activated` was emitted.

## 6. Possible directions (for later implementation)

- **Option A – Clear on yield return:** Do **not** set `is_running = false` in `finalize_streaming()`. Set it to `false` only when the top-level send is done, e.g. in `Session.send()` when `yield this.agent.send_async(...)` returns (in a `finally` or after the yield), and ensure error paths also clear it. Then “running” means “we have not yet returned from the send”.
- **Option B – Two concepts:** Keep a “stream done” notion for UI (e.g. stop button / streaming indicator) and a separate “send in progress” notion if needed (e.g. to block duplicate send or show “working” until the full send, including tools, is done).
- **Option C – Centralize in Session:** Have only `Session.send()` set `is_running = true` at start and `is_running = false` when its yield returns (and on cancel/error), and remove direct `is_running` writes from Agent, Runner, and Details so the session is the single owner of the flag.

**Adopted:** We implement a variant of Option A: centralize in **agent `send_async`** (not Session). Set `is_running = true` at entry and `is_running = false` in a `finally` so it clears when the send returns (including after the full tool loop). See **Section 7** for the concrete code changes.

## 7. Concrete code changes: centralize is_running in agent send_async (for review)

**Rationale:** The single place that “owns” the “send in progress” lifecycle is the agent’s `send_async`: when it returns (normally or by exception), the send is truly done, including any tool loop. So we set `is_running = true` at entry and `is_running = false` in a `finally` so it always clears on return. All other writes to `is_running` are removed. **Keep** `Session.cancel_current_request()` setting `is_running = false` so the UI updates immediately when the user clicks Stop.

**Why Base.send_async doesn't need to "handle" tools:** The agent only does `yield this.chat_call.send(messages, cancellable)`. That call does **not** return until `Chat.send()` has finished the entire turn. Inside `Chat`, when the first response has `tool_calls`, `execute_streaming()` calls `yield this.toolsReply(response)`, which runs tools, then `yield this.send_append(...)` (another HTTP send), and if the next response also has tool_calls, it recursively calls `toolsReply` again. So `chat_call.send()` only returns when the last response has no (further) tool calls. For the whole of that time, `send_async` is still suspended at the `yield chat_call.send(...)` line — so `is_running` stays true until the entire tool loop is done and `send_async` finally returns and the `finally` runs. No extra handling for tools in `send_async` is needed.

### 7.1 Agent.Base.send_async — set at start, clear in finally

**File:** `libollmchat/Agent/Base.vala`

Set `is_running = true` at the very start; wrap the rest of the method in `try { ... } finally { this.session.is_running = false; }`. Remove the existing `is_running = true` (and debug) that appear before `fill_model()`.

**Replace the method body (from line 335) with:**

```vala
public virtual async void send_async(Message message, GLib.Cancellable? cancellable = null) throws GLib.Error
{
	this.session.is_running = true;
	try {
		// Build full message history from this.session
		var messages = new Gee.ArrayList<Message>();

		// Add the current "user" message to session.messages (after processing)
		this.session.messages.add(message);

		// Filter and add messages from this.session.messages (full conversation history)
		foreach (var msg in this.session.messages) {
			if (msg.role == "user"
				|| msg.role == "assistant" || msg.role == "tool") {
				messages.add(msg);
			}
		}

		yield this.fill_model();

		var response = yield this.chat_call.send(messages, cancellable);
		// Response processed via streaming callbacks / session.handle_stream_chunk
	} finally {
		this.session.is_running = false;
		GLib.debug("Agent.send_async: is_running=false session %s", this.session.fid);
	}
}
```

### 7.8 Details.refine — remove is_running = true, and pass cancellable so Stop works

**Why:** `Details.refine()` currently calls `yield this.chat_call.send(messages, null)`, so the refinement request is not tied to the session’s cancellable. When the user clicks Stop during refinement, the in-flight HTTP request is never cancelled. Thread the cancellable from `Runner.send_async` → `handle_task_list` → `task_list.refine` → `Details.refine`, and pass it to `chat_call.send()`. Then when the user clicks Stop, the same cancellable is cancelled, the send throws `IOError.CANCELLED`, the exception propagates out, and `send_async`’s `finally` clears `is_running`.

**Note:** The line `this.session.is_running = true` was already removed from `Details.refine()` in a prior edit; no further removal needed. We stop and return when cancelled by **checking `cancellable.is_cancelled()`** and returning (no throw, no changes to Call/Base or Call/Chat).

#### 7.8.6 chat_call.send() — no cancellation errors (document and optionally clarify comments)

**Current behaviour:** `chat_call.send()` already does **not** throw when the user cancels. In **Call/Base** we catch `IOError.CANCELLED` in `handle_streaming_response` and `process_json_streaming` and return (or break); in **Call/Chat** `execute_streaming` catches `IOError.CANCELLED` and returns the response (possibly partial). So cancellation is a normal exit: callers get a return value and can check `cancellable.is_cancelled()` if they need to know.

**Optional (no behaviour change):** Clarify in comments that cancellation is not an error — e.g. in Chat.vala and Base.vala catch blocks use "User cancelled — return/break (normal exit, not an error)". Optionally add to the `send()` doc comment: "Does not throw on cancellation; returns the current (possibly partial) response when the operation is cancelled."

### 7.10 OLLMcoder.Agent.send_async — set at start, clear in finally (additional agent)

**File:** `liboccoder/Agent.vala`

**Reason this was missing from the original list:** This agent **fully overrides** `send_async` and does **not** call `base.send_async()`, so it never runs the Base logic. Once we remove `is_running = true` from `Session.send()`, the Code Assistant agent would never set `is_running`, and the Send/Stop button would never show "Stop" when using that agent.

Apply the same pattern as Base and Runner: set `this.session.is_running = true` at the very start of `send_async`, wrap the rest of the method body in `try { ... } finally { this.session.is_running = false; }`.

**Replace the method with:**

```vala
public override async void send_async(OLLMchat.Message message, GLib.Cancellable? cancellable = null) throws GLib.Error
{
	this.session.is_running = true;
	try {
		var messages = new Gee.ArrayList<OLLMchat.Message>();

		string system_content = this.factory.system_message(this);
		if (system_content != "") {
			var system_msg = new OLLMchat.Message("system", system_content);
			this.session.messages.add(system_msg);
			messages.add(system_msg);
		}

		var user_content = this.factory.generate_user_prompt(message.content);
		this.session.messages.add(new OLLMchat.Message("user", user_content));

		foreach (var msg in this.session.messages) {
			if (msg.role == "user"
				|| msg.role == "assistant" || msg.role == "tool") {
				messages.add(msg);
			}
		}

		yield this.fill_model();

		var response = yield this.chat_call.send(messages, cancellable);
	} finally {
		this.session.is_running = false;
		GLib.debug("OLLMcoder.Agent.send_async: is_running=false session %s", this.session.fid);
	}
}
```

### 7.11 How the UI knows when to switch the button (no UI change needed)

The UI **does** use session state, but the button is not bound to `is_running` via a property binding. It is updated by **explicit calls** to `ChatInput.set_streaming(bool)`:

| When | What happens |
|------|----------------|
| **User clicks Send** | `ChatWidget.on_send_clicked()` calls `this.chat_input.set_streaming(true)` (line 467) *before* starting the request → button shows "Stop". |
| **Final response received (no tool calls)** | `on_stream_chunk_handler()` sees `response.done` and `response.message.tool_calls.size == 0`, then calls `this.chat_input.set_streaming(false)` (line 446) → button shows "Send". |
| **Response has tool calls** | Handler does *not* call `set_streaming(false)`; it returns and leaves the button as "Stop" until the next chunk for the final (no-tool) response. |
| **Error / cancel / Stop clicked** | `cleanup_streaming_state()` or `on_stop_clicked` calls `set_streaming(false)` → button shows "Send". |
| **User switches session** | `on_session_activated()` schedules an Idle that calls `set_streaming(session.is_running)` → button reflects the *new* session's running state. |

So the switch to "Send" happens when the **chunk handler** sees the final `response.done` with no tool calls and calls `set_streaming(false)` — it does **not** depend on `session.is_running` having been set to false elsewhere. With the centralized change, we no longer set `is_running = false` in `finalize_streaming()`, but the chunk handler still runs for that last chunk and still calls `set_streaming(false)` explicitly. So the button still flips at the right time and **no UI code change is required**.

`session.is_running` is still used in the UI for: (1) the early-return in `on_stream_chunk_handler` (drop chunks when not running); (2) initial button state when switching sessions (`on_session_activated`); (3) `handle_error` deciding whether to finalize. All of these remain correct when only the agent's `send_async` and `cancel_current_request` set/clear the flag.
