## Purpose

- **🔷** **Future** UX: when the LLM requests **`action: press`**, show a **permission overlay on the WebKit surface** around the press target (instead of, or in addition to, the standard chat permission row).
- **🔷** User chooses **Allow** / **Deny** / **Always allow** / **Always deny** on that overlay before the press proceeds (or is refused).
- **🔷** Overlay blocks clicks in a region around the target so the page cannot steal the decision UI.
- **🔷** **Do not implement this plan now.** Parent **5.0** uses **standard tool permission gating** for `press` first.
- **ℹ️** Keep this doc so the overlay can be picked up later without re-deriving the UX from chat.

---

## Overlay layers

### Blocking region

- **🔷** Overlay a **box around the press target** (a11y extents / `{x,y}` from dump).
- **🔷** That region **captures** pointer events and **blocks** them from reaching the page.
- **🔷** Start with a **visible** blocking box (debug / first UX).
- **💩** Later: make the fill transparent while keeping hit-testing (outline-only) if the visible fill is too noisy.

### Target highlight

- **🔷** Always draw a **red rectangle** around the click target while a press is pending.
- **🔷** The red outline stays even when the permission buttons are hidden.

### Permission buttons

- **🔷** Four actions on the blocking region:
  - **Deny**
  - **Always deny**
  - **Allow**
  - **Always allow**
- **🔷** Layout:
  - **Above** the target: **deny** family in **red** (`Deny`, `Always deny`)
  - **Below** the target: **allow** family in **green** (`Allow`, `Always allow`)
- **🔷** Buttons sit **on** the blocking region (same overlay stack, above the WebView).

---

## Show / hide behaviour

- **🔷** When the LLM first requests the press: **show buttons immediately** (and the red target rect + blocking region).
- **🔷** Buttons stay visible until the user has **entered** the blocking region at least once.
- **🔷** After that first enter: when the pointer **leaves** the blocking region, **hide the buttons**.
- **🔷** While buttons are hidden: keep the **red target rectangle** (and still block clicks in the region).
- **🔷** When the pointer **re-enters** the blocking region: **show the buttons again**.
- **💩** Keyboard / touch parity (focus, tap-outside) — decide at implement time; pointer hover is the locked desktop rule.

---

## Outcomes

- **🔷** **Allow** — run the pending a11y press once; dismiss overlay.
- **🔷** **Deny** — refuse this press; dismiss overlay; tool returns a clear “denied” result (no page activation).
- **🔷** **Always allow** — same as Allow, and **record** the choice for this identity (see below).
- **🔷** **Always deny** — same as Deny, and **record** the choice for this identity.
- **🔷** On a later matching press request: if recorded **always allow** → press without UI; if **always deny** → refuse without UI.
- **💩** Where to store the record (chat session vs app config vs host→path map) — confirm at implement; identity key is locked first.

---

## Remember-key (always allow / deny)

Position is **not** part of the key (window resize moves coordinates a lot).

- **🔷** Key parts:
  - **URL** (page / document the control belongs to)
  - **Button text** (a11y label / name used in the dump)
  - **Sequence among matching items** — press-ref / accessibility order count for that text on the page (not pixel `{x,y}`)
- **🔷** Do **not** key on layout coordinates.
- **⏳** Exact string normalisation (trim, case, role prefix) — lock when wiring storage; do not invent extra heuristics in code until then.
- **💩** Whether “URL” means full URI, origin, or origin+path — confirm at implement.

## Suggested order (when this plan is opened)

1. **⏳** **🔷** Overlay widget on `Browser` / stack host: blocking region + red target rect from a11y extents.
2. **⏳** **🔷** Four buttons + colour / placement (deny above, allow below).
3. **⏳** **🔷** Hover show/hide rules (initial show; hide only after enter-then-leave).
4. **⏳** **🔷** Gate `press` on Allow / recorded always-allow; Deny / always-deny refuse (may replace or sit beside standard `ChatPermission`).
5. **⏳** **🔷** Persist remember-key (URL + label + sequence).
6. **⏳** **💩** Wire into chat globe view once the host is visible (**5.0.4**).

**🚫** No code fences yet — design lock only; add Remove / Replace / Add when implementation starts.
