## Goal

Automated **round-trip** tests for every wire method that has a server handler today, plus explicit **negative** cases for client stubs that lack a daemon relay.

Tests must **not** touch `~/.local/share/ollmchat/` — isolated `data_dir`, socket, and pid under the Meson build tree (same idea as `oc-test-files --test-db`, but for the whole daemon data layout via `ApplicationInterface.data_dir`).

**First pass is shell-only** — pipe NDJSON to the Unix socket, assert on JSON with **`jq`**. No Vala test binary, no `RpcClient`, no `Folder`/`File` `rpc_register()` (typed `result` blobs stay opaque JSON until [`2.10.4.9`](2.10.4.9-v2-wiring-and-cutover.md) wires the client library).

## Harness (proposed)

### Layout

```text
build/test-rpc-<pid>/
  files.sqlite          # daemon DB (ApplicationInterface.data_dir)
  ollmfilesd.sock
  ollmfilesd.pid
  projects/             # fixture tree (git-tracked under tests/rpc-fixtures/)
```

### Components

| Piece | Role |
| ----- | ---- |
| **`tests/test-rpc.sh`** | Meson `test()` entry — spawn daemon, run cases, teardown |
| **`tests/test-rpc-common.sh`** | Shared helpers: `rpc_call`, `wait_for_socket`, `jq` asserts (like `test-common.sh`) |
| **`tests/rpc-fixtures/`** | Minimal project tree copied into temp dir per run |
| **`build/ollmfilesd/ollmfilesd`** | Daemon binary under test |

**Deferred:** `tests/oc-test-rpc.vala` — only worth it once `libocrpc` + V2 types are Meson-wired and we want typed deserialize (tranche 3 / cutover).

**Not in initial pass:** `RpcClient`, `RpcClientBoot`, V2 `ProjectManager`.

### Flow per test run

1. Create `build/test-rpc-$$/` (or fixed `build/test-rpc/` with wipe).
2. Copy `tests/rpc-fixtures/minimal-project/` into fixture path.
3. **`test-rpc.sh`** starts `ollmfilesd --data-dir={test_dir}` (forks to background like production).
4. Poll until `{test_dir}/ollmfilesd.sock` accepts.
5. Each case: build JSON line → **`socat`** (or `nc -U`) → socket → read one response line → **`jq -e`** assertions.
6. Teardown: SIGTERM via **pid file** (not shell `$!` — parent exits after fork), remove temp dir.

### Shell RPC helper (sketch)

```bash
# tests/test-rpc-common.sh
rpc_call() {
  local id="$1" method="$2" params="${3:-{}}"
  printf '%s\n' "{\"jsonrpc\":\"2.0\",\"id\":${id},\"method\":\"${method}\",\"params\":${params}}" \
    | socat - UNIX-CONNECT:"${RPC_SOCKET}"
}

# Example assert
resp=$(rpc_call 1 "Daemon.hello" '{"protocol":1,"client":"test-rpc"}')
echo "$resp" | jq -e '.error == null and .result.server == "ollmfilesd"'
```

Assert on **JSON fields** only (`.result.path`, `.result.id`, `.result | type == "array"`, `.error.code`). No GObject deserialize.

## Fire-and-forget and DB persistence

### In-memory DB, deferred disk write

`ollmfilesd` uses `SQ.Database` — **in-memory SQLite** with backup to `{data_dir}/files.sqlite`. Daemon opens DB with **`autosave: true`**: every 60s while `is_dirty`, calls `backupDB()` (coalesced, up to ~2.5s). **`cleanup()`** calls `backup_real()` (immediate). Most `saveToDB(..., sync: false)` calls set `is_dirty` only.

**Implication:** a successful RPC response does **not** mean `files.sqlite` on disk is updated yet.

### Server replies before async work finishes

Some handlers return `ok` (or a row) while work continues on the main loop:

| Wire method | RPC response | Work after reply |
| ----------- | ------------ | ---------------- |
| `File.write` | immediate `msg: ok` | `buffer.write_real.begin()` — disk + DB |
| `File.delete` | immediate `msg: ok` | `delete_manager.remove.begin()` + `cleanup.begin()` |
| `File.register` | waits | reply in `convert_fake_file_to_real` callback |
| `ProjectManager.activate_project` | waits | reply at end of `activate_project` async (can be slow if scan runs) |
| `ProjectManager.create_project` | sync | in-memory DB updated; disk backup still deferred |
| `ProjectManager.remove_project` | sync | same |

(V2 client also fire-and-forgets some calls — `remove_project`, `activate_project`, `activate_file` — but the shell suite talks to the **daemon** directly, not V2 PM.)

### Preferred validation pattern: hello → command → shutdown → SQLite

For mutations that touch the DB (and for fire-and-forget server methods), do **not** rely on the RPC response alone or a fixed sleep.

**Per-case flow:**

1. Spawn daemon (or reuse one session — see below).
2. `Daemon.hello`
3. Setup calls (`create_project`, `activate_project` with `skip_scan: true`, …)
4. **Mutate** (`File.write`, `remove_project`, …)
5. **`Daemon.shutdown`** — daemon replies `ok`, then quits; **must flush DB to disk** (H2).
6. Assert on **`{data_dir}/files.sqlite`** with `sqlite3` (row present/absent, `delete_id`, path, `id > 0`, …).
7. Assert filesystem where relevant (`test -f`, `cmp`).

```bash
rpc_call 1 "Daemon.hello" '{"protocol":1,"client":"test-rpc"}' >/dev/null
rpc_call 2 "ProjectManager.create_project" "{\"path\":\"$PROJECT_PATH\"}" >/dev/null
# …
rpc_call 9 "Daemon.shutdown" '{}' >/dev/null
wait_for_exit   # pid file gone / poll

sqlite3 "$TEST_DIR/files.sqlite" "SELECT path FROM filebase WHERE delete_id = 0 AND path LIKE '%new.txt'"
```

**Do not** use shell `$!` after spawn — parent exits on fork. Use **pid file** for shutdown wait, or `Daemon.shutdown` RPC.

### When to use RPC reload vs SQLite

| Assert | Use |
| ------ | --- |
| Wire shape, error codes, immediate `result` fields | `jq` on RPC response |
| Row persisted, soft-delete, project list in DB | `sqlite3` after **shutdown** (or after H2 lands: shutdown in-case) |
| File bytes on disk | filesystem after shutdown (or short wait for `write_real` if not shutting down) |

### Suite structure

- **Smoke / negative cases (T0):** single daemon session; no DB assert needed.
- **Persistence cases (T1–T2):** prefer **one daemon per case** with shutdown + SQLite assert — deterministic, no sleep.
- **Optional shared daemon:** faster CI; only safe for read-only or fully synchronous calls; document ordering.

Fire-and-forget RPC handlers (`File.write`, `File.delete`) may still be in flight at shutdown — tests must **`shutdown` after** those complete in the main loop, or assert in-session via a follow-up RPC read.

## Assertions and wire details

- Tranches **0–2:** shell + **`socat`** + **`jq`** + **`sqlite3`**.
- Success: `jq -e '.error == null'`. Failure: log full line; assert `.error.code`, `.error.message`.
- **`result` objects:** assert scalar JSON fields (`.result.path`, `.result.id`, array length). No Vala types.
- **Persistence:** for DB mutations and fire-and-forget methods, end with **`Daemon.shutdown`** then assert `files.sqlite` (requires **H2**).
- **`File.changed.check`**: server returns status in `msg` as int string (current draft) — test documents contract.
- **Scan side effects:** use `skip_scan: true` on activate unless explicitly testing scan (slow, flaky in CI).
- **Filesystem checks:** `test -f`, `cmp`, `sqlite3` — especially after shutdown.
