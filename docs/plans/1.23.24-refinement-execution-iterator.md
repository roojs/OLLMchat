## Step 1: Completed tasks summary for refine stage

**What this step does:** Builds a structured summary of completed tasks (task name/slug, Result summary) and injects it into the refinement prompt so the refiner can fill in References and decide tool calls. In the new design, **reference links live inside the summary** (e.g. within the Result summary or task output); we do **not** output section/reference links separately when building the completed task list.

**Scope:** Runner (build summary from task list + session files); refinement prompt placeholder **`{completed_task_list}`**; format (e.g. markdown list: task name + Result summary, with reference links embedded in the summary content). Do not emit reference/section links as a separate list when constructing the completed-task summary.

**Concrete to-do:**

- Add placeholder **`{completed_task_list}`** to refinement prompt and Runner fill.
- In Runner / task list, build the summary from current task list (and session files when using file-based storage).
- Define and document the summary format (task name, Result summary; reference links are inside the summary, not output separately).

**Concrete code changes (Step 1):**

Use this to fill in References **only when relevant** — do not add irrelevant information to References. 2. **`liboccoder/Task/Details.vala`** — new phase and use in refinement fill

   Add a new markdown phase for “completed task” summary in the refine prompt. In **`MarkdownPhase`** enum add:

   ```vala
   REFINE_COMPLETED
   ```

   In **`to_markdown(MarkdownPhase phase)`**, add `case "References":` (skip when `phase == REFINE_COMPLETED`), extend the Output condition to include `REFINE_COMPLETED`, and return before Tool Calls when `phase == REFINE_COMPLETED`. Caller (List.completed_tasks()) only calls this for completed tasks, so no extra guard at start. Full method:

   ```vala
   public string to_markdown(MarkdownPhase phase)
   {
   	string[] order = {
   		"Name",
   		"What is needed",
   		"Skill",
   		"References",
   		"Expected output",
   		"Output"
   	};
   	var ret = "";
   	for (var i = 0; i < order.length; i++) {
   		var key = order[i];
   		switch (key) {
   			case "References":
   				if (phase == MarkdownPhase.REFINE_COMPLETED) {
   					continue;
   				}
   				break;
   			case "Output":
   				if ((phase != MarkdownPhase.LIST && phase != MarkdownPhase.REFINE_COMPLETED) || !this.exec_done || this.result == "") {
   					continue;
   				}
   				ret += "- **Output** " + this.result.replace("\n", " ") + "\n";
   				continue;
   			default:
   				break;
   		}
   		if (!this.task_data.has_key(key)) {
   			continue;
   		}
   		ret += "- **" + key + "** " + this.task_data.get(key).to_markdown() + "\n";
   	}
   	if (phase == MarkdownPhase.REFINE_COMPLETED) {
   		return ret;
   	}
   	if (this.tools.size == 0) {
   		return ret;
   	}
   	ret += "\n## Tool Calls\n\n";
   	foreach (var tool in this.tools) {
   		var obj = new Json.Object();
   		obj.set_string_member("name", tool.name);
   		if (tool.tool_call != null && tool.tool_call.id != "") {
   			obj.set_string_member("id", tool.tool_call.id);
   		}
   		if (tool.arguments != null) {
   			obj.set_object_member("arguments", tool.arguments);
   		}
   		var node = new Json.Node(Json.NodeType.OBJECT);
   		node.set_object(obj);
   		var gen = new Json.Generator();
   		gen.set_root(node);
   		ret += "```json\n" + gen.to_data(null) + "\n```\n\n";
   	}
   	return ret;
   }
   ```

   ```vala

   {
   	var out = "## Tasks\n\n";
   	var section = 0;
   	foreach (var step in this.steps) {
   		section++;
   		var step_out = "";

   		}
   		out += "### Task section " + section.to_string() + "\n\n" + step_out;
   	}
   	return out;
   }
   ```

## Step 2: Reference resolution (task/session URLs → content)

**What this step does:** Resolves References produced by refinement (task URLs, session paths, file paths, anchors) into concrete content that execution can use as Precursor.

**Scope:** Resolution logic (e.g. in Runner `reference_content()` and/or Details `reference_contents()`). Support: file paths, `#task-name-results`, **task:// URLs**, **session task output paths** (e.g. `history/…/task-name.md`). Prioritize task and session paths so links to prior task outputs work.

**Concrete to-do:**

- Implement or extend resolution for `task://` and session task output paths first.
- Ensure file paths and `#anchor` (e.g. `#task-name-results`) resolve to content.
- Reference resolution must return content that can be concatenated or passed as Precursor to execution.

**Why second:** Execution “on reference documents” (Step 5) requires resolved content. Parsing refinement output (Step 3) can happen in parallel or after, but resolution must exist before we run execution on References.

## Step 3: Refinement output parsing and Runner storage

**What this step does:** Reads from the refinement response the task’s **References** (and Skill call, Tool calls); stores the list of references on the task (e.g. `task.references`); determines whether this task has tool calls so the Runner can branch.

**Scope:** Parser (refinement response → structured output); Runner (store `task.references`, set or detect “has tool calls” per task).

**Concrete to-do:**

- Parser extracts References and Tool calls from refinement response.
- Runner stores the reference list on the task so execution can use it to build Precursor.
- From refinement output (or skill definition), determine “has tool calls” vs “no tool calls” and branch execution flow (Step 4 vs Step 5).

**Why third:** Steps 4 and 5 need to know whether the task has tool calls and what References to resolve. Step 1 ensures the refiner has input to produce References; Step 2 ensures we can resolve them.

## Step 4: Runner — execution when skill has tool calls

**What this step does:** For a task that has tool calls: runs that task’s tool calls only, collects all tool outputs for that task, builds a single Precursor from those outputs, and invokes execution **once** for that task.

**Scope:** Runner only. No change to “we output all tool calls” — just scoped to a single task’s calls and one execution call per task on aggregated tool output.

**Concrete to-do:**

- For the current task, run its tool calls and collect outputs.
- Build Precursor from those outputs (combined for that task).
- Invoke execution once for that task with that Precursor.
- Execution produces Result summary + body sections (see Step 6).

**Why fourth:** Clear branch: “has tool calls” → run tools, then one execution on tool output. No references needed as primary precursor (tool output is the precursor).

---

## Step 5: Runner — execution when skill has no tool calls

**What this step does:** For a task with **no** tool calls: does not run any tools. Resolves the task’s References to content, then runs the executor on that content. **Default:** run execution **once per reference** (each reference’s resolved content as Precursor). **Optional execute-combined:** run execution **once per task** with **all** resolved references (and any tool outputs) combined into a single Precursor.

**Scope:** Runner; optional skill/task flag (e.g. frontrunner value) **execute-combined**. Use case for execute-combined: e.g. code-writing skills that need full context (all references) in one request.

**Concrete to-do:**

- After refinement, if no tool calls: resolve each of `task.references` to content (using Step 2).
- If **execute-combined** is set: concatenate all resolved reference content into one Precursor; invoke execution once for the task.
- If **execute-combined** is not set: for each reference, invoke execution once with that reference’s content as Precursor (run on each reference).
- Execution produces Result summary + body sections (see Step 6). Optionally: map task tool ids or reference ids to TaskResult for per-call/per-reference results; Details.write(suffix) for `slug + suffix + ".md"` (Phase 2 scope).

**Why fifth:** This is the “execution on reference documents” path. Depends on Steps 1–3 (summary, resolution, stored references and “no tool calls” detection).

## Step 6: Task execution template and Precursor contract

**What this step does:** Documents and aligns the execution contract: what Precursor is (tool outputs and/or resolved References), and what the executor must output (Result summary + body sections with descriptive titles and section links; no Output References section).

**Scope:** `task_execution.md`; skill format / task execution template in docs. Optionally: TaskResult or similar for per-tool or per-reference result storage and resolution (e.g. for section links).

**Concrete to-do:**

- Confirm `task_execution.md` describes Precursor as “tool outputs + resolved References” or “resolved References only” when no tools.
- Confirm executor output format: Result summary + body sections with descriptive titles; list sections as links; no Output References section.
- If needed for section links or downstream refs: define TaskResult (or equivalent) and map (e.g. task tool ids ⇒ TaskResult) on task list or task; Details.write(suffix) for per-call/per-reference files.
- **Completed task list (Step 1):** TaskResult (or equivalent) must expose a **summary** (e.g. `TaskResult.summary`) that is used to build the completed task list for the refinement prompt. When TaskResult exists, List.to_markdown(REFINE_COMPLETED) and/or Details.to_markdown(REFINE_COMPLETED) should use this summary instead of `task.result` directly.

**Why sixth:** Execution behaviour (Steps 4 and 5) must match what the prompt and downstream expect. This step locks the contract and any result storage needed for references to individual tool/reference outputs.

## Step 7: Task iterator — run after every step

**What this step does:** Invokes the task list iteration (intermediary analyst) after **each** completed task (each “step”), not only after the full round. Keeps the plan responsive: new or follow-up tasks can be proposed as soon as one task finishes.

**Scope:** Runner (`liboccoder/Skill/Runner.vala`) and any orchestration that decides when to call `run_task_list_iteration()`. Today iteration runs once per round; change to after each task completion, with current task list (including that task’s output) as input.

**Concrete to-do:**

- Call `run_task_list_iteration()` (or equivalent) after each task completion.
- Pass the current task list including the completed task’s output.
- Define “step” clearly (e.g. one task = one step). Consider throttling or summarization to avoid overwhelming the user with iteration prompts.

**Why last:** Iterator timing is independent of refinement/execution branching. It uses the same task list and outputs that Steps 1–6 produce, so implementing it after the execution flow is stable avoids rework.
