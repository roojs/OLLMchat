**Proposed design to apply when implementing Step 7:**

- **Runner**
  - Two lists only: **`completed`** and **`pending`** (no `task_list`). Both non-null, initialised to empty `List` in constructor. **`completed`** = tasks that have gone through execution (unchanged reference data). **`pending`** = tasks to be run (initial plan or new/revised from iteration). Only **pending** is ever run.
  - **Initial parse:** `parse_task_list()` populates **`runner.pending`** only; `completed` stays empty until after first run.
  - **After run:** After the first run (and each run), **completed** contains only the **completed steps** (those that have been run). The **uncompleted steps** remain in **pending**. Then call `run_task_list_iteration()`; iteration parses into `this.pending` (new proposed); `completed` is unchanged.
  - **Iteration loop:** At start of `run_task_list_iteration()` set **`existing_proposed = this.pending`** (local). On parse or validation error: use the **raw LLM response** (last iteration output) as the failed content — set **`last_failed_proposed`** to that text (do **not** use `pending.to_markdown()`; the parsed list may not include all the LLM generated). Restore **`this.pending = existing_proposed`**. In the UI, show **only the error messages** (parser.issues / skill_issues) — do not repeat the full iteration output; it is already in the UI. The raw response is still stored as last_failed_proposed for the **next prompt** (previous_proposed_task_list) so the LLM can see what it proposed and fix it. Parser must **not** clear `runner.pending` on failure so Runner can restore.
  - **Iteration prompt:** `iteration_prompt(previous_proposal_issues, existing_proposed, previous_proposed_md)`. **previous_proposed_md** is a **string** (empty when not retrying, not null). When retrying, it is the **raw LLM response** from the last iteration (the output that failed parse/validation), not our serialized pending. Three placeholders: **{completed_task_list}**, **{outstanding_task_list}** (from existing_proposed), **{previous_proposed_task_list}** (when retrying, that raw response; include section header only when non-empty).
  - **Task reference resolution:** Register **tasks** in the List's slug map (slug → task). We look up the task from the map, get that task's **tool run**, and use the **tool result document** to resolve **only by #fragment** (e.g. `run.document.headings.get(link.hash).to_markdown_with_content()`). We do **not** resolve whole documents (task://slug.md with no hash returns ""). No `find_run_by_slug`, no iterating/concatenating exec_runs; slug → task → task's tool run → run's document headings.

  - **`parse_task_list_iteration()`:** Create a **new List object** and assign to **`runner.pending`** at start, then build into it. On validation/parse failure **do not** clear `runner.pending` (leave the parsed list so Runner can capture and restore).

  - Deal with **names** and **slugs** separately: **names** (e.g. `Gee.HashMap<string,Details> names` — name => task) for ensuring unique names; **slugs** (e.g. `slugs` => task or slug => run/output) for link resolution (`task://slug.md`). - **Reference validation (extra layer)**

  - **Next step to be run:** References in those tasks must refer only to things that **already exist in completed** — i.e. only **tool/output references** (task://slug from a completed task) or task://slug#section, plus files/URLs. Validate that every reference in the next step resolves from **completed** (and file/env) before running it.

So we need validation/update when a step is promoted to "next".

   ```vala
   public string to_markdown(MarkdownPhase phase)
   {
   	string[] order = {
   		"Name",
   		"What is needed",
   		"Skill",
   		"References",
   		"Expected output",
   		"Output"
   	};
   	var ret = "";
   	for (var i = 0; i < order.length; i++) {
   		var key = order[i];
   		switch (key) {
   			case "References":
   				if (phase == MarkdownPhase.REFINE_COMPLETED) {
   					continue;
   				}
   				break;
   			case "Output":
   				if ((phase != MarkdownPhase.LIST && phase != MarkdownPhase.REFINE_COMPLETED) || !this.exec_done || this.result == "") {
   					continue;
   				}
   				ret += "- **Output** " + this.result.replace("\n", " ") + "\n";
   				continue;
   			default:
   				break;
   		}
   		if (!this.task_data.has_key(key)) {
   			continue;
   		}
   		ret += "- **" + key + "** " + this.task_data.get(key).to_markdown() + "\n";
   	}
   	if (phase == MarkdownPhase.REFINE_COMPLETED) {
   		return ret;
   	}
   	if (this.tools.size == 0) {
   		return ret;
   	}
   	ret += "\n## Tool Calls\n\n";
   	foreach (var tool in this.tools) {
   		var obj = new Json.Object();
   		obj.set_string_member("name", tool.name);
   		if (tool.tool_call != null && tool.tool_call.id != "") {
   			obj.set_string_member("id", tool.tool_call.id);
   		}
   		if (tool.arguments != null) {
   			obj.set_object_member("arguments", tool.arguments);
   		}
   		var node = new Json.Node(Json.NodeType.OBJECT);
   		node.set_object(obj);
   		var gen = new Json.Generator();
   		gen.set_root(node);
   		ret += "```json\n" + gen.to_data(null) + "\n```\n\n";
   	}
   	return ret;
   }
   ```

## Step 7: Task iterator — run after every step

**What this step does:** Invokes the task list iteration (intermediary analyst) after **each** completed task (each “step”), not only after the full round. Keeps the plan responsive: new or follow-up tasks can be proposed as soon as one task finishes.

**Scope:** Runner (`liboccoder/Skill/Runner.vala`) and any orchestration that decides when to call `run_task_list_iteration()`. Today iteration runs once per round; change to **after each step**, with current task list (including that step's output) as input. Also: prompt (`resources/skill-prompts/task_list_iteration.md`) and how the Runner applies the iteration result.

**Concrete to-do:**

- Call `run_task_list_iteration()` (or equivalent) after each step.
- Pass the current task list including the completed task’s output.
- Define “step” clearly (e.g. one task = one step). Consider throttling or summarization to avoid overwhelming the user with iteration prompts.

**Prompt (task_creation_initial.md) — initial plan / “Original prompt” guidance (to implement after review):**

- Say that the author should **break down the work into steps**, and that **after each step progress will be reviewed**. So the plan is a suggestion for the full flow that will be revised as the run progresses.
- Structure using **task sections**: e.g. initial research in one section, then analysis, then whatever follows. Create enough sections (and tasks) so that the list can be **reviewed and refined at each point**. Separation into sections is what allows the task list to be refined after each step; do not pack everything into one block.

**Critical change to the refinement process: keep completed tasks, replace uncompleted**

This is a **critical** change to how the task list is updated after iteration. It must be clearly documented and implemented in both the Runner and the iteration prompt. Splitting **completed** vs **pending** tasks explicitly avoids continually resetting the Runner's task list and reduces the risk of losing state.

- **Behaviour:** When the Runner applies the iteration result, it does **not** replace the entire task list with the parser output. Instead:
  1. **Keep** all tasks that are **completed** (have run and have output; `exec_done`).
  2. **Remove** all tasks that are **not yet done** (uncompleted).
  3. **Replace** that uncompleted set with the **tasks returned by the iteration** (the analyst’s output). The analyst outputs only the updated “tasks to be done” (new or revised uncompleted tasks).
- **Result:** The canonical task list = **completed tasks (preserved)** + **tasks to be done (from iteration)**. Completed work is never discarded; only the “what’s left to do” is revised by the intermediary analyst.
- **Why it matters:** The iteration prompt and parser today assume “full task list again” (all tasks plus Goals, Original prompt). The new model is: completed tasks stay; only the uncompleted portion is re‑authored by the analyst. Code (Runner and any parser path for iteration) and the iteration prompt must both reflect this.

**Validation of iteration task steps (what they should and should not contain)**

Iteration output describes **tasks to be done** only. The parser must validate that each parsed task conforms to that. This is **not** already there; it needs to be added.

**Should contain (allowed keywords only):**

- **Name** (optional)
- **What is needed** (required)
- **Skill** (required)
- **References** (optional)
- **Expected output** (required)
- **Requires user approval** (optional) — exact label only when the task modifies code or files

No other field names are allowed. Any other key in a task block (e.g. typos, or extra labels) must be rejected with a clear issue.

**Should NOT contain:**

- **Output** — new tasks must not have results. If the analyst includes an **Output** line, treat it as invalid (iteration tasks are “to be done”; the Runner keeps completed tasks and their Output separately).
- Any key that is not in the allowed list above.

**Name collision (names map):** Pending **names** must not collide with **completed** or with **other tasks in the same proposed list**. List has a **names** map (name => task) for uniqueness. When we register a task: (1) check name not in `runner.completed.names`, (2) check name not already in `this.list.names`. If collision, append to issues; else register `this.list.names.set(name, t)`. Slugs (for link resolution) are populated when tasks complete; see design at top of plan.

Validation runs in `parse_task_list_iteration()` after building each step/task (see concrete code in section 1.1 below). For each task, check `task_data` keys: if **Output** is present, or any key is not in the allowed set, append to `issues` (e.g. “Task list iteration task must not contain Output” / “Task list iteration task has disallowed field: …”) Name-creation step (above) handles collision with completed and proposed list. On parse/validation failure **do not** clear `runner.pending` — Runner restores `this.pending = existing_proposed`.

```vala
	 * Step uses this.list.runner and this.list.names. Slugs for link resolution are
	 * populated when tasks complete (on completed list).

	{
```

```vala
	 * Cached in ''required_key_cache''. Use to validate that required keys are present.
	 *
	 * @return cached list: What is needed, Skill, Expected output

	{

			return required_key_cache;
		}
		string[] required_keys_array = {
			"What is needed",
			"Skill",
			"Expected output"
		};
		required_key_cache = new Gee.ArrayList<string>();
		foreach (string s in required_keys_array) {
			required_key_cache.add(s);
		}
		return required_key_cache;

	 * Validates one task from a task list (initial or iteration): must not contain
	 * Output; only allowed field names; required keys must be present.
	 * Appends to {@link issues} when invalid. Call only after {@link List.fill_names}
	 * has been run (during parsing, before this); this method only reads the task
	 * name for issue messages and does not inject or set names.
	 *

	 *
	 * @param t the task to validate (its task_data keys are checked)

	{
		// Caller (parse path) must have run fill_names() so Name is set and non-empty.
		var label = t.task_data.get("Name").to_markdown().strip();

			this.issues += "\n" + "Task \"" + label + "\" must not contain Output " +
				"(tasks in the list have no results yet).";

		foreach (var req in required_keys()) {
			if (!t.task_data.has_key(req)) {
				this.issues += "\n" + "Task \"" + label + "\" is missing required field: \"" + req + "\". " +

			if (valid_keys().contains(k)) {

			this.issues += "\n" + "Task \"" + label + "\" has disallowed field: \"" + k + "\". " +

				"whether its content belongs in one of the valid fields; then resubmit the task list " +
				"with only the valid fields for each task.";
		}
```

### 8.1 Populating `pending.slugs` (Step.register_slugs)

**Applied.** After `t.validate_references()`, register slug: `var s = t.slug(); if (s != "") this.list.slugs.set(s, t);` so `validate_references()` can resolve task refs within pending.

**Why:** `Details.validate_references()` looks up from both `runner.completed.slugs` and `runner.pending.slugs`; without populating `this.list.slugs`, refs from one pending task to another would not resolve.

**Full method (for reference):**

```vala
	public void register_slugs(int step_index, ResultParser parser)
	{
		var completed_list = this.list.runner.completed;
		foreach (var t in this.children) {
			t.step_index = step_index;
			var name = t.task_data.get("Name").to_markdown().strip();
			if (name != "") {
				if (completed_list.names.has_key(name)) {
					parser.issues += "\n" + "Task \"" + name + "\" has name already used in the " +
						"completed list; use a different name for new tasks.";
				} else if (this.list.names.has_key(name)) {
					parser.issues += "\n" + "Task \"" + name + "\" has name already used in the " +
						"proposed task list; use a different name.";
				} else {
					this.list.names.set(name, t);
				}
			}
			parser.validate_task(t);
			t.validate_references();
			var s = t.slug();
			if (s != "") {
				this.list.slugs.set(s, t);
			}
			if (t.issues != "") {
				parser.issues += "\n" + "Task (References): " + t.issues;
			}
		}
	}
```
