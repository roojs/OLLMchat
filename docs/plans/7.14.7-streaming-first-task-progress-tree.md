# 7.14.7 вЂ” Task progress tree empty on first send (streaming / new chat)

**Status:** proposed

**Pointer:** `.cursor/rules/CODING_STANDARDS.md` вЂ” Checklist for all plans

**Layout:** `docs/guide-to-writing-plans.md`

**Related:** `docs/plans/7.14-task-progress-tree-ui.md`, `docs/plans/done/7.14.5-proposed-skill-progress-session-bind.md`

---

## Purpose

- While the Skills agent streams the **first** task-list response on a **new chat** (`EmptySession` в†’ real `Session`), the task progress strip stays empty or shows **stale** rows.
- **Cause:** `OLLMcoder.Task.ProgressView` is wired only from `Skill.Factory.activate` в†’ `set_runner(host.session_agent())`. That path does **not** run when the user sends the first message without changing the agent dropdown, so the view keeps a `TreeListModel` bound to a **previous** sessionвЂ™s `Runner.progress` (or never refreshes for the new `Runner`).
- **Outcome:** After first send, `ProgressView` tracks the **current** sessionвЂ™s `Runner` so `ProgressList` updates (`ProgressRunner`, then `add_pending` after parse) appear during/after that run.

---

## Scope

| In scope | Out of scope |

| **`libollmchat`** ordering in **`EmptySession.send`** | Rewriting **`ProgressList`** signalling |
| **`ollmapp`**: one mechanical **`get_active_agent().activate(window)`** on **`session_activated`** (same pattern as **`session_restored`**) | **`OLLMcoder`** type checks or **`is Skill.Factory`** in **`Window`** (**рџ‘Ћ** per **`7.14.5`**) |

## Discussion

- **в„№пёЏ** `Runner.send_async` begins with `this.progress.add(rp)` вЂ” the model for the **active** `Runner` gains a вЂњTask CreationвЂќ row immediately; `parse_task_list` + `add_pending` run only after the **full** assistant text is available (not mid-chunk).
- **вљ пёЏ** If the UI is bound to the wrong `Runner`, **no** row updates appear for the run the user is watching вЂ” this matches вЂњtree not filled by the first task generationвЂќ while chat still streams correctly (different code path).
- **в„№пёЏ** `Manager.switch_to_session` already calls `ensure_agent_handler()` **before** `session_activated` вЂ” loaded sessions are OK; the gap is **`EmptySession.send`** only.
- **рџ”¶** After this fix, the **detail** rows still appear only after parse completes; the **runner** row may appear as soon as `send_async` starts вЂ” that is existing behaviour.

---

## Acceptance criteria

1. **New chat + Skills agent + first user message:** while the model streams the task list, the progress strip is bound to the **new** sessionвЂ™s runner (at least the вЂњTask CreationвЂќ row appears for that run when `send_async` runs).
2. **No regression** for **`switch_to_session`** / **`session_restored`** progress binding (`7.14.5`).
3. **`Window`** does not import **`OLLMcoder`**; **`get_active_agent().activate`** is generic.
4. **`Skill.Factory.activate`** stays free of **`session_agent() == null`** as the primary guard вЂ” **`ensure_agent_handler`** runs before activate on the fixed path (**вљ пёЏ** **`7.14.5`**).

## Concrete code proposals

Mandatory pattern: each fenced block has a **`####`** heading whose **first word** is **Keep**, **Remove**, **Replace with**, or **Add**.

---

### 1. `EmptySession.send` вЂ” ensure handler before `session_activated`

**Target:** `libollmchat/History/EmptySession.vala` вЂ” inside `send`, after building `real_session` and before `manager.session_activated`.

#### Add вЂ” call ensure before activating UI session signal

After `real_session.activate();` (or immediately before `session_activated`), insert:

```vala
			real_session.ensure_agent_handler();
```

So `manager.session` points at a `Session` whose **`agent`** is the **`Runner`** that **`send_async`** will use.

---

### 2. `Window` вЂ” bind agent chrome on `session_activated`

**Target:** `ollmapp/Window.vala` вЂ” **`history_manager.session_activated`** handler (the block that adjusts **`agent_dropdown`** / project).

#### Add вЂ” factory activate alongside existing session switch logic

At the **start** of the **`session_activated`** callback body (after null-checks if any), invoke the same pattern as **`session_restored`**:

```vala
				var factory = this.history_manager.get_active_agent();
				factory.activate.begin(this, (obj, res) => {
					factory.activate.end(res);
				});
```

**Note:** This may pair with an existing **`session_restored`** activate on session switch вЂ” **`Skill.Factory.activate`** should remain **idempotent** (**`set_runner`** + visibility). If double **`activate`** causes measurable issues, follow-up can narrow to вЂњonly when **`session.agent`** identity changedвЂќ (**рџ’©** confirm before adding identity tracking).
