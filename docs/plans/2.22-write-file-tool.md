# 2.22 Write File Tool

## Coding standards

Apply [.cursor/rules/CODING_STANDARDS.md](../../.cursor/rules/CODING_STANDARDS.md). Checklist:

- **Nullable types**: Avoid nullable where possible; use default objects/flags.
- **Null checks**: Only where design explicitly requires null.
- **String interpolation**: No `@"..."` except multi-line usage/help/docs.
- **Temporary variables**: No one-use temporaries; no trivial aliases (except 4+ chain).
- **Brace placement**: Line breaks for namespace/class/method; inline for if/for/while. No one-line if-with-body.
- **`this.` prefix**: Use for instance members in Vala.
- **GLib prefix**: Fully-qualified `GLib.*`; no `using` for new code.
- **Property initialization**: Defaults via `get; set; default =` or field defaults.
- **Line length**: Break long method calls and concatenations.
- **StringBuilder**: Avoid; use `string.joinv()` and `+`.
- **String building in loops**: Forbidden; use `string.nfill()`, `replace()`, `string.joinv()`.
- **ArrayList for strings**: Prefer `string[]`; init with `string[] x = {}`.
- **Loops**: Use `continue` to denest; avoid `else` in loops.
- **Character looping**: Avoid; prefer string methods and `GLib.Regex`.
- **File info try/catch**: Only when existence unknown.
- **Try/catch scope**: Minimal; only the call that may throw.
- **Underscore prefix**: No leading `_` on variables/fields/properties.
- **get_* methods**: Prefer properties or verb-less names.

## Overview

New tool **write_file**: one-shot write (no streaming). Same semantics as EditMode (permissions, backups, apply flow) but arguments include **content** and mode (ast_path + location, line numbers, or complete_file). Registered as `write_file`.

## Scope (strict)

- **Permitted**: Only code under **liboctools/WriteFile/** (Tool.vala, Request.vala) plus **registration and build** so the tool compiles: Registry.vala (add WriteFile.Tool to init_config, setup_config_defaults, fill_tools), liboctools/meson.build (add two source files), docs/meson.build (add two source paths).
- **Not permitted**: Any change to code outside WriteFile other than the above. **No modifications to libocfiles (Tree.vala, FileChange.vala) or any other existing library or tool.** Existing code is approved; this plan does not change it. Any change to existing code would require a separate, explicitly approved plan—not hidden here.

## Tool contract

- **Name**: `write_file`
- **No stream**: Single tool call; no `on_stream_chunk` / `on_message_completed`.
- **Arguments**:
  - `file_path` (string, required)
  - `content` (string, required) — text to write
  - `ast_path` (string, optional) — e.g. `Namespace-Class-Method`
  - `location` (string, optional) — required when ast_path set. `""` or `"replace"` (replace target), `"replace-with-comment"`, `"before"`, `"after"`, `"remove"`, `"before-comment"`
  - `start_line` (int, optional) — 1-based inclusive
  - `end_line` (int, optional) — 1-based exclusive
  - `complete_file` (bool, optional, default false) — replace entire file; if true, `content` is full file body
  - `overwrite` (bool, optional, default false) — when `complete_file=true` and file exists

Mode: exactly one of (ast_path set), (start_line/end_line set), or (complete_file true). Same validation as EditMode (e.g. one code block for complete_file).

## Rules (same as EditMode)

- No approval for project files (normalize path, project_manager.get_file_from_active_project / folder_map → clear permission_question, return false from build_perm_question).
- Create backup before change: FileHistory before first apply (create_file_history then apply_change then sync_and_update_metadata); db.backupDB() after sync.

## Existing APIs only (no new library code)

WriteFile uses **only existing, already-approved APIs**. No methods are added to Tree, FileChange, or any other library. Proof:

| What WriteFile needs | Existing API (unchanged) |
|----------------------|--------------------------|
| Normalize path for project | Same logic as EditMode.Request.normalize_file_path (implement that logic inside WriteFile/Request.vala; do not change RequestBase or EditMode). |
| Permission for project files | project_manager.get_file_from_active_project(path), project_manager.active_project.project_files.folder_map.has_key(dir_path) — already exist. |
| Get/create File | project_manager.get_file_from_active_project(norm), OLLMfiles.File.new_fake(project_manager, norm) — exist. |
| Buffer | project_manager.buffer_provider.create_buffer(file), file.buffer.read_async(), file.buffer.write(...), file.buffer.sync_to_file(), file.buffer.get_text(), file.buffer.apply_edit(change) — exist. |
| FileChange (build, resolve, apply) | new OLLMfiles.FileChange(file) { ast_path, operation_type, include_comments, replacement }, change.resolve_ast_path(), change.apply_change(bool) — exist. resolve_ast_path() already sets change.result on failure; we use that string. |
| FileHistory / backup | new OLLMfiles.FileHistory(db, file, change_type, date), yield file_history.commit() — exist. |
| Sync after apply | file.buffer.sync_to_file() or catch NOT_SUPPORTED and buffer.write(buffer.get_text()) — exist. convert_fake_file_to_real, saveToDB, review_files.refresh(), db.backupDB() — exist. |
| Tree / AST path listing | Not used. When AST path is invalid, we return change.result as set by existing FileChange.resolve_ast_path() (e.g. "AST path not found: ..."). No list_ast_paths(); no changes to Tree or FileChange. |

All of the above are calls from WriteFile code into existing types. No new methods on OLLMfiles.Tree, OLLMfiles.FileChange, or elsewhere. Concrete code below is confined to WriteFile/Tool.vala and WriteFile/Request.vala plus the minimal Registry/meson edits already listed.

### 4. create_file_history and sync reuse

- Request holds a `bool history_created = false` (or one-off local in execute_request). No new Stream class; no on_stream_chunk/on_message_completed. Request.execute_request does everything synchronously (async) and returns the result string.

## Validation

**Location (when ast_path set)**: Validate location is one of: `""` (accept empty string for replace), `"replace"`, `"replace-with-comment"`, `"before"`, `"after"`, `"remove"`, `"before-comment"`. Also accept legacy `"with-comment"` as alias for replace-with-comment.

**AST path** = existing `FileChange.resolve_ast_path()` only:

```vala
// Tool.validate() when project_manager != null && has_ast:
var change = new OLLMfiles.FileChange(file) { ast_path = ast_path.strip(), replacement = content };
yield change.resolve_ast_path();
if (change.has_error || (change.result != "" && change.result != "applied")) {
    return change.result != "" ? change.result : "AST path did not resolve";
}
```

Existing: tree_factory(file), tree.parse(), tree.lookup_path(ast_path, ...). Not found → change.result set; we return it.

**Tool.validate**: `async string validate(...)`. Return "" = valid, else error string. (1) Structure → first error; (2) when has_ast → validate location (replace / empty string, replace-with-comment, before, after, remove, before-comment); (3) pm + modify mode → file exists; (4) pm + has_ast → resolve_ast_path block above; (5) return "".


**execute_request**: `var err = yield Tool.validate(this.project_manager, ...); if (err != "") throw or return err;` then apply/sync.

## Summary

- **Scope**: Only liboctools/WriteFile/ (Tool.vala, Request.vala) plus Registry + meson for compile. No changes to Tree, FileChange, or any other existing code.
- Add WriteFile.Tool and WriteFile.Request; register in Registry and meson (minimal edits as listed).
- Request: file_path, content, ast_path, location, start_line, end_line, complete_file, overwrite. build_perm_question / normalize_file_path same logic as EditMode (implemented inside Request.vala). execute_request builds one FileChange, create_file_history → apply_change → sync_and_update_metadata using existing APIs only.
- **Validation**: `Tool.validate(project_manager?, ...)` async returns `string` ("" = valid, else error). Uses existing FileChange.resolve_ast_path() for AST. Skill/refine calls validate before submitting.
