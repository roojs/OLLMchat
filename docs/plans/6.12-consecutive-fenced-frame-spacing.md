# 6.12 Consecutive fenced frame spacing

**Status:** proposed — **do not implement until explicitly approved** (see **`.cursor/rules/plan-implementation-workflow.mdc`**).

**Checklist:** **`.cursor/rules/CODING_STANDARDS.md`** — *Checklist for all plans*.

See **`docs/guide-to-writing-plans.md`**.

---

## Purpose

When **two or more fenced code blocks** are rendered **back-to-back** in the markdown GTK view, vertical spacing can look **too large** (stacked **`margin-bottom`** on **`.oc-frame`**). Adjust spacing so consecutive frames read as a coherent group without double the intended gap.

---

## Scope

| In scope | Out of scope |
| -------- | ------------- |
| **`resources/frame.css`** — `.oc-frame` margins / adjacent-sibling rules | Changing frame colors or themes |
| Optionally **`libocmarkdowngtk/Render.vala`** / **`RenderSourceView.vala`** if a **structural** fix is required (e.g. wrapper spacing) | Parser / markdown grammar |

---

## Current behaviour (baseline)

- **`resources/frame.css`**: **`.oc-frame`** uses **`margin-bottom: 10px`** (and related layout). Each frame contributes bottom margin, so **adjacent** frames can **stack** margins.

## DOM verification (before final CSS)

1. Run the app, render an assistant message with **two fenced code blocks** in a row.
2. Use **GTK Inspector** (or equivalent) to record the **actual** widget hierarchy and CSS node names for each frame.
3. Confirm whether **`.oc-frame`** elements are **direct siblings** in the same parent. If not, adjust selectors (e.g. descendant combinator) or use **§ Complete change checklist** item 4 (Render hook).

---

## Proposed behaviour

- Prefer **CSS**: reduce **combined** vertical gap when a frame immediately follows another frame, e.g. adjacent-sibling selector (exact selector depends on GTK widget/CSS path — **verify in Inspector**).
- Example direction (not final values — tune after review):

```css
/* Example only — verify selectors match rendered DOM */
.oc-frame + .oc-frame {
    margin-top: 0; /* or negative margin-top / reduced margin-bottom on predecessor */
}
```

- If CSS cannot target the structure (frames not direct siblings), add a **minimal** structural hook in **`Render.vala`** / **`RenderSourceView.vala`** (e.g. CSS class **`oc-frame-consecutive`**) — **only** if approved as fallback.

## Concrete code proposals

### 1. `resources/frame.css`

#### Keep

Existing **`.oc-frame`** block (lines ~52–63 in current tree) — baseline for margins.

#### Add

After **`.oc-frame { ... }`**, a new subsection **“Consecutive frames”**, starting with a comment such as:

```css
/* Consecutive code frames: avoid doubled vertical gap (values TBD after review) */
```

Then the **approved** rule set (adjacent sibling and/or last-child adjustments). Examples:

- Adjacent-sibling reduction: **`.oc-frame + .oc-frame`** (selector **must match** Inspector output).
- Optional: **`:first-child`** / **`:last-child`** tweaks if the first/last frame in a message need different spacing.

**Do not** merge without visual approval (values are not fixed in this plan).

---

### 2. `libocmarkdowngtk/Render.vala`

#### Keep (before)

`on_code_block(bool is_start, string lang)` — starts **`RenderSourceView`**, calls **`create_textview()`** (see current implementation ~752–785).

#### Add (only if CSS insufficient)

- **Option A:** Add a **CSS class** on the frame widget when **`is_start`** and the **previous** sibling was also a code frame (requires tracking state on **`Render`** — **only** if approved; may touch `Render` fields).
- **Option B:** Set **`margin_top`** / **`margin_bottom`** on the GTK widget from Vala (last resort — prefer CSS).

Document the chosen option in the PR; **do not** implement both without approval.

---

### 3. `libocmarkdowngtk/RenderSourceView.vala`

#### Add / Replace with

Only if **`Render`** delegates frame chrome here and CSS must target an inner node — **spike first**, then update this plan with a **§ Replace with** hunk.

## Acceptance criteria

1. Two **consecutive** fenced code frames in one assistant message: **no** excessive double gap (compared to single frame before paragraph text).
2. **Single** frame before normal text: unchanged or improved — **no** regression.
3. **Thinking** vs **code** frames: if both use `.oc-frame`, verify **both** theme variants.
4. **GTK 4** — stylesheet loads as today (no broken theme); **`frame.css`** still loads via **`gresources.xml`** and **`ChatView`** (see checklist).
5. **Inspector** notes attached to PR or plan archive (selectors used).
