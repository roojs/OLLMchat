# 7.14.10 Task progress tree — tool row tooltip (argument summary)

**Status:** proposed

**Pointer — `.cursor/rules/CODING_STANDARDS.md` (`## Checklist for all plans`):** nullable design, avoid defensive null branches unless justified; **⚠️** this plan **explicitly** authorizes **new API** only where marked (**tooltip/describe** surface) — otherwise follow **no new methods**.

**See also:** `docs/guide-to-writing-plans.md` · [`7.14.9`](done/7.14.9-DONE-task-tool-display-name.md) (`Task.Tool.title` = `BaseTool.title`)

## Purpose

- Task progress **tool** rows (`liboccoder/Task/ProgressView.vala`) show **`ProgressItem.title`** — the tool’s friendly name from the registry (**7.14.9**). That label is often too short to see **what** the model asked for (**query**, **paths**, filters).
- **⚠️** Add a **tooltip** on the **title column** for **tool** rows so hovering shows an **argument / intent summary**.
- **⚠️** The ideal tooltip text **matches** the body the user sees when the tool runs in chat (e.g. codebase search **Query:** / **Language:** / … — see **`libocvector/Tool/RequestCodebaseSearch.vala`** lines 191–206).

## Feasibility (investigation summary)

### ✅ GTK

- **`Gtk.Label`** inside **`Gtk.TreeExpander`** in **`ProgressView`** title factory: call **`Gtk.Widget.set_tooltip_text()`** (plain text, newlines OK) or **`set_tooltip_markup()`** if we ever need lightweight markup.
- Bind **once** with **`title`** or add **`notify::tooltip`** if the string is computed from mutable fields — **`Task.Tool.arguments`** / **`tool_call`** are stable after parse for a given row refresh (recycled list items must **unbind/clear** tooltip when row is reused for non-tool **`ProgressItem`** rows).

### ℹ️ Two different “tool display” surfaces today

| Surface | Where | What the user sees for “what was called” |
| ------- | ----- | ---------------------------------------- |
| **Chat UI** (`ui` messages) | Each **`Request.*.execute_request()`** (and similar) builds a **local** `request_message` / fenced body, then **`agent.add_message(...)`**. Example: **`RequestCodebaseSearch`** builds **Query:/Language:/…** then sends **`Message.fenced(..., request_message)`**. |
| **Skill executor prompt** | **`Task.Tool.tool_call_details()`** | **`### Tool call <name>`** + **JSON** from **`Json.gobject_to_data(tool_call)`** + optional **result** — **not** the same prose as chat. |

- **⚠️** “Match execution display” means matching the **chat / Request** prose, **not** the **`tool_call_details()`** JSON block.
- **⏳** There is **no** shared **`RequestBase.describe_for_ui()`** today; each tool that shows a nice block **inlines** string building next to **`add_message`**.

### ✅ Reproducing chat-style content (options)

| Approach | Parity with chat | Effort | Notes |
| -------- | ---------------- | ------ | ----- |
| **A — Pretty-print `arguments` JSON** | **Low** — keys/values only, label style may differ (**`query`** vs **Query:**) | **Small** | Always have **`Task.Tool.arguments`** or **`tool_call.function.arguments`**. Could title-case **`snake_case`** keys for readability. **Does not** match **`RequestCodebaseSearch`** labels without more work. |
| **B — Deserialize + shared “UI body” helper** | **High** — can be **identical** if the **same function** builds the string used in **`add_message` and tooltip** | **Medium / large** | **`BaseTool.execute`** already **`deserialize(parameters_node)`** into **`RequestBase`**. Add a path that builds the **same** multi-line string **without** **`execute()`** (no side effects): e.g. **`RequestBase` virtual `ui_argument_summary()`** with **default** = simple key/value listing; **`RequestCodebaseSearch`** overrides to the **exact** **Query:/…** format (extract from **`execute_request`** into one method called from both places). **Wrapped tools** need a parallel on **`WrapInterface`** or dedicated branch. |
| **C — Store summary on run** | **High** after first run only | **Small hook, wrong for tree before run** | Caching **`request_message`** on **`Task.Tool`** when **`run()`** executes **does not** help **refinement / proposed** rows or **hover-before-run**. |

- **💩** Optional: tooltip also shows **fence title** (e.g. **“Code search: …”**) — confirm with user; plan default is **body only**, like the **inner** part of the chat frame.

### 🔶 Risks

- **⏳** **`deserialize`** can **fail** on bad JSON; tooltip should **fall back** (e.g. compact JSON string or **“Invalid arguments”**) — **minimal** branch, not a blanket null-guard policy change.
- **⏳** Very long queries / paths: GTK tooltips may need **truncation** or rely on native wrap — verify on **Gtk 4** with real strings.

## Scope

| In scope | Out of scope |
| -------- | ------------ |
| Tooltip on **tool** rows in **`ProgressView`** title column | Tooltips on **`Details`** / **`ProgressRunner`** rows (unless we add a trivial **empty** default) |
| Investigation captured here + **one** chosen parity approach (**A** fast vs **B** exact) | Changing **executor** **`tool_call_details()`** JSON |
| Clearing tooltip when list item **unbind** / non-tool row | Markdown / Pango inside tooltip (unless we pick **markup** later) |

## Acceptance criteria

- **⚠️** Hovering a **tool** row in the task progress **title** column shows a **non-empty** tooltip when **`arguments`** exist (or shows a short **“No parameters”** / tool name-only line when object is empty — behaviour **explicitly** chosen during implementation).
- Tooltip text **either** matches **Approach B** prose for **at least** **`codebase_search`** (and any other tools we migrate in the same change), **or** matches **Approach A** with documented **known** deltas from chat.
- **Row recycling:** switching a **list item** from a **tool** to a **non-tool** row does **not** leave a stale tooltip.
- No new **defensive** null chains beyond the **deserialize / missing registry** cases this feature intrinsically needs.

## Concrete code proposals

### 1) `ProgressView.vala` — tooltip on title **`Gtk.Label`**

- **`setup`:** unchanged except ensure **`label`** has **`overflow = Gtk.Overflow.VISIBLE`** only if clipping affects tooltip hit-testing (often unnecessary).
- **`bind`:** After **`pi.bind_property("title", …)`**, **`if (pi is OLLMcoder.Task.Tool)`** → **`tool_label.set_tooltip_text(...)`** from **`Task.Tool`** property (see §2). **`else`** → **`set_tooltip_text(null)`**.
- **`unbind`:** **`set_tooltip_text(null)`** so recycled rows do not keep tooltips.

### 2) `Task.Tool` — owned string for tooltip (**Approach A** sketch)

- Add **`public string argument_tooltip { owned get { … } }`** (name as decided at implement time).
- **Minimal** implementation: iterate **`this.arguments`** members (or **`tool_call.function.arguments`**), build **`Title Case: value\n`** lines. Empty → **`""`** and **`ProgressView`** skips tooltip or shows **`title`** only — **pick one** in implementation and keep consistent.

### 2b) **`RequestBase` + tools** — (**Approach B** sketch, **⚠️** explicit **new methods** authorized here only)

- **⚠️** Add **`public virtual string ui_argument_summary()`** on **`RequestBase`** with **default** implementation (**simple key/value** from deserialized properties **or** JSON fallback).
- For **`RequestCodebaseSearch`:** **move** lines 191–202 into **`ui_argument_summary()`** (or a **private** builder called from **`execute_request`** and **`ui_argument_summary`**). **Duplicate** prevention is the **whole point**.
- **`BaseTool`:** add **`public string tooltip_for_arguments(Json.Object args)`** that **`deserialize`**s → **`request.ui_argument_summary()`** with **`request.agent`** unset / dummy — **only if** summaries never touch **`agent`** (they must not call **`add_message`**). **Verify** each migrated **`Request`**.
- **`Task.Tool.argument_tooltip`** getter: resolve **`BaseTool`** via **`session.manager.tools.get(key)`**, call **`tooltip_for_arguments(this.arguments)`**, catch deserialize failure → fallback string.

### 3) Tests / manual verification

- Run skill with **`codebase_search`** in task list; hover tool row — tooltip matches chat **Query:/…** block (**Approach B**) or is clearly acceptable (**Approach A**).

## Notes

- **ℹ️** User example **“writes to proposed list file”** may refer to a **different** tool or a **natural-language** summary; **tooltips** here are **parameter-centric**, not LLM-authored task prose.
- **👎** Do not use **`tool_call_details()`** JSON as the tooltip **unless** we explicitly choose parity with the **executor** rather than **chat**.
