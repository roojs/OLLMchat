# 7.14.13 Chat ↔ rendered widget span (RenderBox)

Status: proposed

Checklist: `.cursor/rules/CODING_STANDARDS.md` — Checklist for all plans.

Workflow: `docs/guide-to-writing-plans.md` — implement only approved hunks.

## Separation (non-negotiable)

- 🔷 The Gtk.Box subclass is a Gtk/layout concern only. It does not know `OLLMchat.Message`, session, or progress types.
- 🔷 Anything that says “this span belongs to that message” lives outside — e.g. `ChatView` / `ChatWidget` correlates spans or IDs when it drives rendering.
- 🔷 The box holds a query centre: widget IDs implied by overriding Gtk.Box append (no append_register). One method `mark()` records the start of a range; the end is always derivable from the list length — no long span_first / span_last names.

## Purpose

- 🔷 Replace fragile `Message.idx` ↔ tail of `idx_widgets` / `current_textview` with `RenderBox` (ordered `by_id`, `mark()` for range start).
- 🔷 The container is not a “message object”; it tracks append order, `mark()`, and length-derived end.
- 🚫 No GObject signals on `MarkdownGtk.Render` for this; append + `mark` only.
- ℹ️ Related: `docs/plans/7.14.6-progress-chat-scroll-issues.md`.

## Scope

| In scope | Out of scope |
| --- | --- |
| 🔷 Remove `ChatView.idx_widgets` and all tail-dedupe index return paths; use `RenderBox` for id truth | 🚫 `Message` type inside `RenderBox` |
| 🔷 `RenderBox` + `mark` + storing ids on `Message` from `ChatView` at the right instants | 🚫 Rewriting markdown parser |
| 🔷 One small **debug** append helper (label row) for tracing layout / ids on the page | 🚫 Shipping debug UI to end users by default (flag or compile-time if needed) |

## Acceptance criteria

- 🔷 `RenderBox` has no `libollmchat` message imports; public `by_id`; override append; `mark()`; no `append_register`, no `widget_for_id`.
- 🔷 `ChatView` no longer maintains `idx_widgets` or equivalent parallel tail list; scroll and `Message.idx` (or `idx_first` / `idx_last`) come from the active `RenderBox` and the mark/store contract.
- 🔷 Documented mark/store steps: who calls `mark()`, when `Message` fields are written, and how `scroll_to_idx` resolves a widget.
- 🔷 Debug helper exists and uses normal append so the label receives an id in `by_id` (usable for “what index am I looking at?” during development).
- 🔷 Replay documented.

## Current behaviour

- ℹ️ `ChatView` owns `Gee.ArrayList<Gtk.Widget> idx_widgets` (public field today).
- ℹ️ On streaming done (`append_assistant_chunk`), complete assistant (`append_complete_assistant_message`), and tool (`append_tool_message`), code may append `renderer.current_textview` to `idx_widgets` when it differs from the list tail, then returns `idx_widgets.size - 1` as the index passed back as `Message.idx`.
- ℹ️ `scroll_to_idx` walks `idx_widgets` by integer index.
- ℹ️ One flat `text_view_box` feeds one `MarkdownGtk.Render`.

## Proposed behaviour

- 🔷 Each logical render episode that needs stable ids uses a `RenderBox` instance as the render target (`renderer.box` points at it; box is a child of `text_view_box` or sits in the same vertical stack per agreed hierarchy).
- 🔷 `mark()` is called from `ChatView` (or a thin wrapper it owns) at the start of a span the outer layer wants to correlate (e.g. before body for one assistant message). After appends, `first_id` and `last_id` on that `RenderBox` define the range; `ChatView` copies those values onto the relevant `Message` (or only `idx` as last id — migration in hunks) before the next `mark()` on the same box.
- 🔷 `Message.idx` for progress/scroll: prefer `last_id` (or `idx_last`) as the “scroll to this message” target unless product wants “top of message” = `first_id` — pick one in implementation and document.
- 🔷 **Debug cheat sheet:** a single method (e.g. on `ChatView` or `RenderBox`) that appends one full-width `Gtk.Label` (e.g. `halign` start, `hexpand` true, small monospace or `add_css_class` for a debug stripe) with caller-supplied text, using the same append path as everything else so the label gets the next id in `by_id`. Use for “page / phase / message boundary” labelling while debugging mapping. Optional: guard with `--debug` or a `const bool` so production builds stay clean if you want that.

## Mark and store (contract)

- 🔷 `mark()` sets `RenderBox.first_id` to the current `by_id.size` (start of the next run of appends).
- 🔷 After the renderer (and any tool/debug appends) has finished for that episode, `ChatView` reads `first_id` and `last_id` and assigns them to the `Message` (or session mirror) that this episode belongs to. **Store happens in `ChatView` (or `ChatWidget` if you keep all `Message` touch there) — not inside `RenderBox`.**
- 🔷 `ChatWidget` calls into `ChatView` at: stream start / per message completion / `message_created` complete paths / `tool_message` so that `mark()` runs at the right boundary and store runs before the next `mark()` for the same `RenderBox` instance.
- ℹ️ If one `RenderBox` is reused across episodes, order must be: store previous span → `mark()` for next span. If each episode gets a **new** `RenderBox`, `mark()` is typically once per box at creation.

### 1. `libollmchatgtk/RenderBox.vala`

#### Add — new file + meson entry.

#### Add — shape (adjust names in PR).

```vala
namespace OLLMchatGtk
{
	/** Vertical Gtk.Box: append registers widgets in order; mark() sets first_id; use public by_id[index]. No Message type. */
	public class RenderBox : Gtk.Box
	{
		public Gee.ArrayList<Gtk.Widget> by_id { get; private set; default = new Gee.ArrayList<Gtk.Widget>(); }

		public uint first_id { get; private set; default = 0; }

		public uint last_id {
			get {
				uint n = this.by_id.size;
				return n > 0 ? n - 1 : 0;
			}
		}

		public RenderBox()
		{
			Object(orientation: Gtk.Orientation.VERTICAL, spacing: 0);
		}

		public override void append(Gtk.Widget child)
		{
			base.append(child);
			this.by_id.add(child);
		}

		public void mark()
		{
			this.first_id = (uint) this.by_id.size;
		}
	}
}
```

Verify `Gtk.Box.append` overridable in Vala/GTK; if not, use the GObject-approved replacement (still no `append_register`).

### 3. `libollmchatgtk/ChatView.vala` — remove `idx_widgets`, use `RenderBox`

#### Remove

- Public field `idx_widgets` and every use: growth in `append_assistant_chunk` (done path), `append_complete_assistant_message`, `append_tool_message`.
- Return values that were `this.idx_widgets.size - 1` as the sole index source; replace with logic derived from the **active** `RenderBox` for that path (e.g. `last_id` after the append chain, or `by_id.size - 1` on the box that received the widget).
- `clear()`: `idx_widgets.clear()` — remove; if `RenderBox` instances are destroyed with the view, no separate clear; if a long-lived list of boxes is kept, clear that instead.
- `scroll_to_idx` and related highlight: **Replace** index walk from `idx_widgets` with resolution via the agreed map — e.g. `Message` stores `idx` as an id into the **current** session’s `RenderBox.by_id`, or `ChatView` keeps a single active `RenderBox` for the tail and indexes into `by_id`. Exact resolution rule in hunks (must be one mechanical story, not two).

#### Add

- Active `RenderBox?` (or list / stack) and rules for which box `scroll_to_idx` uses.
- **Mark / store:** methods or inline flow that call `mark()` on the new box or at span start, and after render assign `message.idx` (and optional `idx_first` / `idx_last`) from `first_id` / `last_id` on the correct `RenderBox`.
- **Debug cheat sheet** — e.g. `append_debug_label(string line)` on `ChatView` (or static helper): creates `Gtk.Label` with `hexpand = true`, `xalign` 0, optional `add_css_class("oc-debug-chat-row")`, `append`s to the **current** `RenderBox` (or the same `text_view_box` + `RenderBox` policy as other single-row UI). Document in CSS plan or `style.css` if a class is added. One call = one id in `by_id` for tracing.
