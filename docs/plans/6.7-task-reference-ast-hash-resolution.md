# 6.7 Task references: resolve `file#…` AST hashes like `read_file`

**Status:** Not started

**Related:** [1.23.31 — reference file anchor validation](1.23.31-reference-file-anchor-ast.md) (validation of anchors in `validate_references()`; separate concern). This plan is **injection / resolution** when building reference blocks for refinement and execution.

---

## Purpose

Task **References** can use `file:///…/path#fragment` where `fragment` is an **AST path** (same string family as vector metadata and tools: e.g. `OLLMcoder.Task-List-write`), not only `#L10-L20` line ranges.

**Today:** `Details.link_content()` treats only `^[Ll](\d+)-[Ll](\d+)$` as a line range. Any other non-empty hash still calls `header_file()` with **no** range; on **execution** (and non–refinement-preview paths) that loads the **entire file** via `OLLMfiles.File.contents(-1, -1)`.

**Expected:** Non–`Lx-Ly` fragments should be resolved the same way as the **`read_file`** tool with `ast_path`: **Tree-sitter** parse + `Tree.lookup_path()` → `comment_start` … `end` line range, matching **`OLLMtools.ReadFile.Request.resolve_ast_path()`** (see `liboctools/ReadFile/Request.vala`).

SQLite vector rows also store `ast_path`, but **live** resolution in the app already goes through **`ProjectManager.tree_factory(file)`** + **`yield tree.parse()`** + **`lookup_path`** — not a DB lookup for this use case.

---

## Current behaviour (reference)

| File | Role |
|------|------|
| `liboccoder/Task/Details.vala` | `link_content()` — regex only for `#L…-L…`; else full file when no range |
| `liboctools/ReadFile/Request.vala` | `resolve_ast_path()` — tree parse + `lookup_path`; sets `start_line` / `end_line` from comment + node end |
| `libocfiles/Tree.vala` | `lookup_path()` — exact match, then suffix match |
| `liboccoder/Task/Tool.vala` | Execution: `link_content(..., EXECUTION)` after `load_links` |
| `liboccoder/Skill/Runner.vala` | `load_file()` / `load_links()` — buffer read only; no hash→range |

Refinement calls `reference_contents(REFINEMENT)` from `refinement_prompt()` synchronously; execution builds reference blocks from `link_content` in async `Tool.run()` but the resolver is still sync-only for file ranges.

---

## Proposed behaviour

1. **Shared resolution API** (name indicative; adjust to match codebase style):
   - On **`OLLMcoder.Skill.Runner`** (or, if preferred, a small helper next to `Tree` used by both ReadFile and task refs): an **`async`** method that, given **`OLLMfiles.File`** and **`hash`** string:
     - If hash matches **`#Lx-Ly`** (existing regex): return **success** with **1-based inclusive** start/end for `header_file` / `File.contents` range mode (same as today).
     - Else if hash is non-empty: treat as **`ast_path`** — require `file` in active project (`id > 0`); **`tree_factory` → `yield parse()` → `lookup_path(hash, …)`**; on success return **comment_start** and **end** line the same way **`ReadFile.Request.resolve_ast_path()`** assigns to `start_line` / `end_line`.
     - Else (empty hash): **no** range (keep current head / full-file semantics by phase).
     - On AST lookup **failure**: define a single policy (see **Open decisions**).

2. **Slice semantics:** The text shown in the reference block for an AST-resolved range should match what **`read_file`** would return for that `file_path` + `ast_path` (same `buffer.get_text` / line conventions as **`ReadFile.Request.execute_request()`** after AST resolution — avoid a second off-by-one drift versus `Details.header_file` → `File.contents` if they differ; **pick one authoritative path**, ideally by calling the same helper or **`file.buffer.get_text`** with the same indices as ReadFile).

3. **Async plumbing:** `link_content` cannot perform **`yield tree.parse()`** while remaining synchronous. Introduce **`async` `link_content_file_reference`** (or **`reference_contents_async`**) used from:
   - **`Tool.run()`** — already async; **`yield`** resolution.
   - **`Details.refine()`** — already async; **`yield`** before building `refinement_prompt` (today **`yield load_links(this.references)`**; ensure buffers exist, then resolve AST hashes — consider whether **`shared_references`** need the same **`load_links`** pass).

4. **`refinement_prompt()`:** Takes the precomputed **`task_reference_contents`** string (or call **`yield reference_contents_async(REFINEMENT)`** inside an async factory) so the refine loop does not call synchronous **`reference_contents()`** when AST hashes need resolution.

5. **Callers outside the main refine/exec path:** Update **`examples/oc-test-skill-agent.vala`** (and any other **`refinement_prompt()`** call sites) to supply reference contents or use an async path.

6. **Optional DRY:** Factor **`ReadFile.Request.resolve_ast_path()`** internals to call the new shared helper so behaviour stays single-sourced (only if it does not balloon scope).

---

## Open decisions

- **AST resolution failure** (typo, file not in project, tree parse error): fallback to **full file** with a visible one-line note, **empty** block with error line, or **fail** validation — pick one; default recommendation: **warning log + full-file fallback** for exec (parity with “best effort” previews) unless product prefers strict.
- **`shared_references`** in refine: confirm whether **`load_links(shared_references)`** is required before resolving (execution **`Tool.run`** already merges exam + references for load).

---

## Scope

| In scope | Out of scope |
|----------|----------------|
| `liboccoder/Skill/Runner.vala` — async hash → line range (L-regex + AST) | Changing vector search or SQLite schema |
| `liboccoder/Task/Details.vala` — async reference assembly; `refinement_prompt` signature / refine loop | **1.23.31** validation-only work (can follow after resolution exists) |
| `liboccoder/Task/Tool.vala` — `yield` async link resolution for execution | Rewriting `read_file` UX or tool JSON |
| Optional: thin refactor so **ReadFile** shares the resolver | Broader markdown anchor types (GFM heading slug in `.md` only) unless trivial reuse of same tree |

---

## Acceptance criteria

- A task reference like `file:///…/List.vala#OLLMcoder.Task-List-write` (or project-relative equivalent with the same hash) injects **only the resolved symbol range** (including doc comment start when that is how **`read_file`** behaves), not the whole file, in **execution** prompts when the file is in the active project and the AST path resolves.
- `#L10-L20` references continue to work unchanged.
- Refinement prompts that include file references with AST hashes resolve the same way (after **`yield`**), not the full file.
- No regression for references with **no** hash or **only** `L` line-range hashes.
- **`oc-test-skill-agent`** (or documented equivalent) builds a refinement prompt without compile errors.
