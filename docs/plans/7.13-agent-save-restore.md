## Purpose

Restore **skill-runner** task state (`Runner.pending` / `completed`, refinement/exec progress, etc.) when the user opens a **saved history session**, **without calling the model** — **only if** that session includes **message markers** (**`agent-stage`** / **`agent-issues`**). The mechanism is **replay**: drive the existing **`Runner`** + **`ResultParser`** code paths with **`ReplayChat`**, which feeds the **same assistant content** stored in **`session.messages`** (`content-stream` / `content-non-stream` / `assistant`). **Implementation order:** emit markers on **live** runs first; **then** gate restore on their presence (see §Phased implementation).

**Structured stage metadata** via dedicated **message roles** (`agent-stage`, `agent-issues` — see §Stage messages) records **when a stage starts** (**`stage_id`**) and **issues after that stage’s run** (empty = none), so replay can detect **parser/library drift** and support **Continue** and future **restore points** without scraping **`ui`** prose.

---

## Chosen approach (B only)

| Principle | Detail |
| --------- | ------ |
| **Source of truth** | Persisted **`session.messages`** only. **Do not rely on `task_dir()`** for restore logic — it may be removed later; any on-disk task files are **not** part of this plan. |
| **No LLM on restore** | **`ReplayChat`** implements **`send()`** by returning the next stored assistant body; **`Runner.in_replay`** already calls **`report_replay_outcome()`** after parses (to be made **non-fatal** and **flag-driven**). |
| **Drift handling** | Compare **replay** **`ResultParser.issues`** (and executor validation) to **`agent-issues`**: **blank** means no issues on the live run; non-blank is the recorded issues string. On mismatch: **stop**, surface UI, **Continue** can re-send the **previous** system/user payload as a **fresh** attempt (no stale retry budget). |
| **No `ui` scraping for outcomes** | **Do not** infer “had issues” by scanning **`ui`** fenced text. Replace **`ReplayChat`’s substring `run_issues` heuristics** with **`agent-stage` / `agent-issues`** (see §Stage messages). |
| **Restore is marker-gated** | Task-state **restore / replay runs only** if the loaded session contains the **markers** (**`agent-stage` / `agent-issues`** in expected shape — e.g. at least one **`agent-stage`** after markers ship). Sessions **without** markers keep **chat transcript only**; no silent **`Runner.replay()`**. |

## `ui-waiting` and persistence

- **`ui-waiting`** is a **transient** UI affordance (“waiting for … to reply”). It is **not** stored in session JSON: **`Session.serialize_property`** for **`messages`** **skips** **`role == "ui-waiting"`** (`libollmchat/History/Session.vala`).
- Some **`ui-waiting`** instances are emitted only via **`manager.message_added`** (never appended to **`session.messages`**).
- **Replay / restore must not depend on `ui-waiting`** appearing in the saved file; ordering is defined by **content messages** + **`agent-stage` / `agent-issues`**, not by waiting placeholders.

## Stage messages (`agent-stage` / `agent-issues`)

Two **message types** (roles), **plain-text bodies** — no JSON in **`content`**.

| Role | **`Message.content` (body)** |
| ---- | ----------------------------- |
| **`agent-stage`** | **`stage_id`** — stable string id for **entering** this stage (align with **`replay_step`** / internal phase names: `task_list_parse`, `task_list_iteration`, `refinement`, `exec`, `exec_validate`, `post_exec`, …). Emitted **at stage start** (before **`send`** or before synchronous work). |
| **`agent-issues`** | **Issues text** **after** that stage’s run — from **`ResultParser.issues`** (or executor validation). **Blank / empty means no issues** — same as **`issues == ""`**. Emitted **after** parse/validation for the **`agent-stage`** that opened the same step. |

**Success:** **Yes — still emit `agent-issues`.** When a stage finishes **without** parser/validation errors, append **`agent-issues`** with **`content == ""`** (empty string). That records an explicit **successful** outcome for replay (drift check: replay should also yield empty issues). **Do not** skip **`agent-issues`** on success — the **`agent-stage`** / **`agent-issues`** pair would be incomplete and “no message” would be ambiguous vs “crashed before outcome.”

**Ordering / pairing (start → run → outcome)**

- **`agent-stage`**: emit **when the stage starts** — immediately **before** **`chat_call.send(...)`** for that model round (after **`system` / `user`** for the request are built, but those messages go only into the **`send()`** array, not into marker content). If a substage has **no** model call (e.g. synchronous **validate** only), emit **`agent-stage`** immediately **before** that work begins (see executor **validate** path below).
- **`agent-issues`**: emit **after** that stage’s **run** finishes — immediately **after** **`parse_*` / `extract_*` / `exec_post_extract` / validation**, with the issues string (**empty** = success for that boundary).

Transcript shape: **`agent-stage`** → … **`system` / `user`** (and later **`content-stream`**, **`ui`**, etc.) … → **`agent-issues`**. Replay matches **replay** parser output to the **`agent-issues`** that **follows** the **`agent-stage`** with the same **`stage_id`** (ordering in **`session.messages`**). **Blank vs non-blank** / text mismatch → drift / **`Continue`** path.

**Why two types?** Separates **intent to enter** a stage from **outcome**; if a run **crashes** mid-stage, the last **`agent-stage`** may have **no** following **`agent-issues`** yet — useful for future restore-point logic.

**Avoid array index as the primary key** — use **transcript order** between **`agent-stage`** and the next **`agent-issues`** for the same logical step. Optional later: a **correlation id**.

**`ReplayChat`** — derive **expected “had issues”** from **`agent-issues`** (non-blank vs blank), not from scanning **`ui`** for `"had issues"`.

**Why not parse `ui` strings for replay?** Fragile; we already know **`issues`** when **`add_message`** runs — persist it in **`agent-issues`** instead.

## Masking: LLM request vs session file vs UI

Messages are used in **three** places; new stage metadata must stay **out of model input** the same way other session-only rows do.

| Path | Behaviour today (relevant bits) |
| ---- | -------------------------------- |
| **Per-request LLM `Chat.send()`** | Skill runner builds a **small `Gee.ArrayList<Message>`** (e.g. **system + user**) per round — **not** the full **`session.messages`** list. **`agent-stage` / `agent-issues`** must **never** be appended to that array. |
| **Session JSON on disk** | **`Session.write()`** serializes **`messages`** with per-message history info; **`ui-waiting`** is **omitted**. **`agent-stage`** and **`agent-issues`** **are** persisted (unless we explicitly skip a role — prefer **persist** so replay has recorded **`stage_id`** and issues after reload). |
| **UI** | Roles like **`project`** are hidden from chat; **`agent-stage`** / **`agent-issues`** should be **`is_hidden`** / non-visible (or optional debug-only visibility), same family as metadata — **not** shown as normal chat unless we opt in. |

**Implementation rule:** add **`agent-stage`** and **`agent-issues`** to **`Message`**’s **`role` switch** (flags for hidden / not sent to API). Follow the same pattern as **`project`**: stored for replay, **not** merged into **`Chat`** request payloads. Audit any code path that builds API history from **full** **`session.messages`** (if present) and **exclude** these roles there too.

## Current assets (what we already have)

### 1. `ReplayChat` (`libollmchat/Call/ReplayChat.vala`)

- Walks **`replay_messages` in order**; each **`send()`** returns the **next** message whose role is **`content-stream`**, **`content-non-stream`**, or **`assistant`**, with non-empty **`content`**.
- Tracks **`last_return_ix`** (array index of the message last fed to the runner).
- **`run_issues`** today: **heuristic** — substring checks on following **`ui` / `ui-warning`**. **To be replaced** by **`agent-stage` / `agent-issues`** for new sessions.
- **`report_replay_outcome(replay_issues)`** — must stop using **`Process.exit(1)`** in app flows; compare replay **`issues`** string (blank vs non-blank and/or text) to **`agent-issues`**, not reconstructed **`ui`** text.

### 2. `Runner.replay()` (`liboccoder/Skill/Runner.vala`)

- Sets **`in_replay`**, swaps in **`ReplayChat`**, calls **`send_async(...)`**.
- **`replay_step(step, content)`** — good **human** hook; **`agent-stage`** / **`agent-issues`** carry the **machine** contract for drift detection ( **`agent-issues`** body mirrors **`issues == ""`** vs not).

### 3. Persisted transcript

- **`session.messages`** holds **`system`**, **`user`**, **`user-sent`**, **`content-stream`**, **`ui`**, **`project`**, etc. **`ui-waiting`** is **not** in the JSON file (see above).
- **Runner** still emits fenced **`ui`** for warnings; those remain for **humans** — **replay decisions** should use **`agent-stage` / `agent-issues`**, not regex over **`ui`** strings.

### 4. `task_dir()` (explicitly out of scope for restore)

- May be removed or repurposed later. **No dependency** for replay correctness. A **rendered task visualization** from files is **not** in scope for **7.13**.

## Feasibility assessment

| Question | Verdict |
| -------- | ------- |
| Rebuild **`pending` / `completed`** without the model? | **Yes**, via **`ReplayChat`** + parsers. |
| **Index ↔ stage** reliability? | **Weak** if tied only to array position; use **`agent-stage` / `agent-issues`** pairs in transcript order (§Stage messages). |
| Parser drift? | **Yes** — compare replay **`issues`** to **`agent-issues`** + **Continue**. |
| **`Runner.replay()` on loaded GTK session?** | **Not safe as-is** — **duplicate `session.messages`** / **`message_added`**. Needs **replay-suppressed mutation** or **state-only** replay path — **Phase 2** (with **`can_replay`** gate; not deferred). |

---

## Mismatch and Continue (replace `exit(1)`)

1. **Stop** replay; show **`stage_id`** + **expected vs actual** issues ( **`agent-issues`** vs current parser output).
2. **Continue:** one **live** model call with the **same system/user** as that stage’s request (**first attempt** for recovery), **`in_replay = false`**.

## Phased implementation

### Phase 1 — **Message markers in the live flow (prerequisite)** — **DONE**

- **`Message` roles** **`agent-stage`**, **`agent-issues`**; **`Message`** flags (**`is_hidden`**, **`is_ui_visible` = false**, same masking as **`project`** — stored, not shown, not merged into per-request **`Chat.send()`** arrays).
- At each replay-relevant boundary: **`agent-stage`** at **stage start** (before **`send`** or before synchronous work), **`agent-issues`** **after** parse/validation — via **`add_message`** (**no** shared helper).
- **Deliverable:** live runs **persist markers** in **`session.messages`** / JSON. **No restore / replay wiring** in this phase.

#### Phase 1 — Scope

| In scope | Out of scope |
| -------- | ------------ |
| **`Message`** role docs + **`role` switch** flags | **`ReplayChat`**, GTK restore, **`in_replay` suppress** |
| Inline **`add_message`** pairs in **`Runner`**, **`Details`**, **`Tool`** | Changing **`task_dir()`** writes |
| **`agent-stage`** / **`agent-issues`** persisted by existing **`Session.write()`** (not **`ui-waiting`**) | Migration of old sessions |

#### Phase 1 — `stage_id` strings (`agent-stage` body)

Use stable ids; pair **`agent-stage`** → … work … → **`agent-issues`** by **order** in **`session.messages`**.

| Location | **`agent-stage`** (emit when) | **`agent-issues`** (emit when) | `stage_id` / body |
| -------- | ----------------------------- | -------------------------------- | ----------------- |
| **`Runner.send_async`** — task creation try | **Before** **`chat_call.send`** for that try | **After** **`parse_task_list()`** | `task_list_parse` / **`parser.issues`** |
| **`Runner.run_task_list_iteration`** | **Before** **`chat_call.send`** | **After** **`parse_task_list_iteration()`** | `task_list_iteration` / **`parser.issues`** |
| **`Details.refine`** — each outer attempt | **Before** **`chat_call.send`** for that attempt | **After** **`extract_refinement`** | `refinement` / **`result_parser.issues`** |
| **`Details.run_post_exec`** — each **`for`** try | **Before** **`chat_call.send`** | **After** **`exec_post_extract`** | `post_exec` / **`parser.issues`** |
| **`Tool.run`** — executor **try_count** round with model | **Before** **`chat_call.send`** | **After** **`exec_extract`**: **`agent-issues`** with **`parser.issues`** if parse failed; if parse **ok**, run **validate** — if validate fails, emit **`agent-stage`** `exec_validate` **before** **`validate`**, then **`agent-issues`** with **`validate_issues`**; if both ok and writes succeed, **`agent-issues`** `""` **once** before **`return`** (single outcome message for that try — see note) | `exec` / `exec_validate` as needed |

**Executor (implemented):** **`agent-stage` `exec`** before **`send`**; **`agent-issues`** after **`exec_extract`** on parse failure; on parse success **`agent-issues`** `""`, then **`agent-stage` `exec_validate`**, **`validate`** loop, **`agent-issues`** with validate outcome; **`send`** errors in **`catch`** get **`agent-issues`** with the error message. Markers use **`Tool.add_message`** (same session as **`Runner`**).

### Phase 2 — **Restore gate, replay, and no `session.messages` pollution**

Restore must not **dirty** the saved transcript: replay must **not** duplicate persisted rows or spam **`message_added`** for UI already on disk. **Anti-pollution** is **in the same phase** as **`can_replay`** and the first safe **`Runner.replay()`** path (old “Phase 3” merged here).

**Comments:** When implementing, **preserve existing comments** in touched files unless factually wrong (then fix wording).

- **`can_replay`** on **`Session`/`SessionBase`**, persisted as **`can-replay`** in session JSON (**`Session.serialize_property`**). **`SessionPlaceholder.load()`**: **`real_session.can_replay = json_session.can_replay`**. Derive from **`SessionJson`** top-level JSON and/or **`messages`** loop: **`if (!this.can_replay && msg.role == "agent-stage") { this.can_replay = true; }`** (same in **`add_message`**). **Remove** **`this.messages.clear()`** in **`SessionJson`** deserialize for **`messages`** (fresh object — nothing to clear).
- **Gate:** **`Runner.replay()`** returns immediately if **`!this.session.can_replay`**.
- **No pollution:** While **`in_replay`**, suppress duplicate **`session.messages`** / spurious **`message_added`** / unnecessary **`save_async`** for replay replay of on-disk content.

#### Phase 2 — Proposed code (reference)

**`SessionJson`** — add **`can_replay`**; in **`messages`** loop after **`add(msg)`**:

```vala
if (!this.can_replay && msg.role == "agent-stage") {
	this.can_replay = true;
}
```

**`Session.serialize_property`** — add **`case "can-replay":`** with **`project-path`** → **`default_serialize_property`**.

**`SessionBase.add_message`** — after **`messages.add`**, **`if (!this.can_replay && message.role == "agent-stage") { this.can_replay = true; }`** (keep existing comments in method).

**`Runner.replay()`** — at start: **`if (!this.session.can_replay) return;`** (keep existing doc comment on **`replay`**).

See **`libollmchat/History/SessionJson.vala`**, **`SessionPlaceholder.vala`**, **`SessionBase.vala`**, **`Session.vala`**, **`liboccoder/Skill/Runner.vala`**.

### Phase 3 — **`ReplayChat` + drift handling**

- **`ReplayChat`:** read **expected issues** from **`agent-issues`** (and **`stage_id`** from **`agent-stage`**); drop substring **`run_issues`** for **marker-backed** sessions.
- **`report_replay_outcome`:** return recoverable errors, **no `exit`** in app.

### Phase 4 — **GTK wiring**

- **`session_restored`**, skill-runner, **only if `session.can_replay`** (or agreed marker predicate) → async replay; distinct **“restoring tasks”** vs live streaming.

### Phase 5 — **Continue from mismatch**

- Wire **Continue** to re-issue **last system/user** for that stage.

### Sessions without markers (legacy)

- **No task restore** until the user runs a **new** flow that emits markers (or a future optional migration tool adds markers — out of core scope). **No** reliance on **`ui`** substring replay for production restore.

## Retry semantics (restore)

- **Live:** caps at **5** tries in places — **Continue** after drift is a **fresh** attempt for that recovery send.
- **Replay:** one **`ReplayChat.send()`** per logical model round in the stored ordering.

## Acceptance criteria (draft)

1. **Phase 1 (done):** Live skill-runner runs emit **`agent-stage`** at **stage start** and **`agent-issues`** **after** parse/validation; bodies are plain **`stage_id`** and issues (**empty = none**); roles **masked** from LLM and **persisted** in session JSON.
2. **Restore gate:** **`session.can_replay`** false until set from JSON, transcript repair on load, or **`add_message`**; no replay when false.
3. **Phase 2:** No duplicate **`session.messages`** / spurious **`message_added`** during replay (merged with gate); no **`Process.exit`** from replay mismatch once **Phase 3** drift handling ships.
4. Marker-backed replay uses **`agent-issues`** (and **`agent-stage`**) for drift checks, not **`ui`** substring matching.
