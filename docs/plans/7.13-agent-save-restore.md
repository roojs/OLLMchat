## Purpose

**Goal:** Restore **task flow** to where the user can **Continue** or **inject more prompt** later ŌĆö **not** to archive every intermediate string for its own sake.

Restore **skill-runner** task state (`Runner.pending` / `completed`, refinement/exec progress, etc.) when the user opens a **saved history session**, **without calling the model** ŌĆö **only if** that session includes **message markers** (`**agent-stage`** / `**agent-issues**`). The mechanism is **replay** (and/or lighter **hydration**): drive or refill paths so **`ReplayChat`** can supply stored **model** content from `**session.messages`** (`content-stream` / `content-non-stream` / `assistant`). **Implementation order:** **Phase 1** markers ŌåÆ **Phase 2** gate (`**can_replay`**) ŌåÆ later phases ŌĆö see ┬¦Phased implementation.

**Note:** **Processed tool results** are **not** reused by the system after that step; if something is needed again, the user **re-runs** the flow. So **persisting raw tool output** (`**tool-output`**) is **not** a prerequisite for this goal ŌĆö see ┬¦Phase 1.5 (optional).

**Structured stage metadata** via dedicated **message roles** (`agent-stage`, `agent-issues` ŌĆö see ┬¦Stage messages) records **when a stage starts** (`**stage_id`**) and **issues after that stageŌĆÖs run** (empty = none), so replay can detect **parser/library drift** and support **Continue** and future **restore points** without scraping `**ui`** prose.

## `Process.exit` and replay (hard boundary)

TodayŌĆÖs **`exit(1)`** calls exist so **automated / CLI** replay can fail fast with a non-zero status. **Session restore in the desktop app is not allowed to do that.**

| Context | `**Process.exit`** |
| ------- | ------------------- |
| **CLI test harness** (e.g. **`examples/oc-test-skill-agent.vala`**, **`run_replay()`**) | May keep **`exit`** for scripts and CI if desired ŌĆö **not** the product UI. |
| **GTK / normal session lifecycle** | **Forbidden** on replay drift, empty replay, **`ReplayChat.send`** exhaustion, or **`Runner.replay`** catch ŌĆö propagate **`Error`**, signals, or UI state; user keeps a usable session. |

Implementation splits **`ReplayChat` / `Runner`** behaviour by call site or introduces a **non-fatal** replay mode ŌĆö details in **Phase 5** (verbatim hunks TBD); the **Phase 2** **`Runner.replay`** proposal below still shows legacy **`exit`** in the **catch** only to match the tree today; **replace** that path before shipping restore to GTK.

## `ui-waiting` and persistence

- `**ui-waiting**` is a **transient** UI affordance (ŌĆ£waiting for ŌĆ” to replyŌĆØ). It is **not** stored in session JSON: `**Session.serialize_property`** for `**messages**` **skips** `**role == "ui-waiting"`** (`libollmchat/History/Session.vala`).
- Some `**ui-waiting**` instances are emitted only via `**manager.message_added**` (never appended to `**session.messages**`).
- **Replay / restore must not depend on `ui-waiting`** appearing in the saved file; ordering is defined by **content messages** + `**agent-stage` / `agent-issues`**, not by waiting placeholders.

## Stage messages (`agent-stage` / `agent-issues`)

Two **message types** (roles), **plain-text bodies** ŌĆö no JSON in `**content`**.


| Role               | `**Message.content` (body)**                                                                                                                                                                                                                                                                     |
| ------------------ | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| `**agent-stage`**  | `**stage_id**` ŌĆö stable string id for **entering** this stage (align with `**replay_step`** / internal phase names: `task_list_parse`, `task_list_iteration`, `refinement`, `exec`, `exec_validate`, `post_exec`, ŌĆ”). Emitted **at stage start** (before `**send`** or before synchronous work). |
| `**agent-issues**` | **Issues text** **after** that stageŌĆÖs run ŌĆö from `**ResultParser.issues`** (or executor validation). **Blank / empty means no issues** ŌĆö same as `**issues == ""`**. Emitted **after** parse/validation for the `**agent-stage`** that opened the same step.                                    |


**Success:** **Yes ŌĆö still emit `agent-issues`.** When a stage finishes **without** parser/validation errors, append `**agent-issues`** with `**content == ""**` (empty string). That records an explicit **successful** outcome for replay (drift check: replay should also yield empty issues). **Do not** skip `**agent-issues`** on success ŌĆö the `**agent-stage**` / `**agent-issues**` pair would be incomplete and ŌĆ£no messageŌĆØ would be ambiguous vs ŌĆ£crashed before outcome.ŌĆØ

**Ordering / pairing (start ŌåÆ run ŌåÆ outcome)**

- `**agent-stage`**: emit **when the stage starts** ŌĆö immediately **before** `**chat_call.send(...)`** for that model round (after `**system` / `user**` for the request are built, but those messages go only into the `**send()**` array, not into marker content). If a substage has **no** model call (e.g. synchronous **validate** only), emit `**agent-stage`** immediately **before** that work begins (see executor **validate** path below).
- `**agent-issues`**: emit **after** that stageŌĆÖs **run** finishes ŌĆö immediately **after** `**parse_*` / `extract_*` / `exec_post_extract` / validation**, with the issues string (**empty** = success for that boundary).

Transcript shape: `**agent-stage`** ŌåÆ ŌĆ” `**system` / `user**` (and later `**content-stream**`, `**ui**`, etc.) ŌĆ” ŌåÆ `**agent-issues**`. Replay matches **replay** parser output to the `**agent-issues`** that **follows** the `**agent-stage`** with the same `**stage_id**` (ordering in `**session.messages**`). **Blank vs non-blank** / text mismatch ŌåÆ drift / `**Continue`** path.

**Why two types?** Separates **intent to enter** a stage from **outcome**; if a run **crashes** mid-stage, the last `**agent-stage`** may have **no** following `**agent-issues`** yet ŌĆö useful for future restore-point logic.

**Avoid array index as the primary key** ŌĆö use **transcript order** between `**agent-stage`** and the next `**agent-issues**` for the same logical step. Optional later: a **correlation id**.

`**ReplayChat`** ŌĆö derive **expected ŌĆ£had issuesŌĆØ** from `**agent-issues`** (non-blank vs blank), not from scanning `**ui**` for `"had issues"`.

**Why not parse `ui` strings for replay?** Fragile; we already know `**issues`** when `**add_message**` runs ŌĆö persist it in `**agent-issues**` instead.

## Masking: LLM request vs session file vs UI

Messages are used in **three** places; new stage metadata must stay **out of model input** the same way other session-only rows do.


| Path                              | Behaviour today (relevant bits)                                                                                                                                                                                                                                                                   |
| --------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **Per-request LLM `Chat.send()`** | Skill runner builds a **small `Gee.ArrayList<Message>`** (e.g. **system + user**) per round ŌĆö **not** the full `**session.messages`** list. `**agent-stage`** / `**agent-issues`** (and **`tool-output`** if ever added) must **never** be appended to that array.                                                                          |
| **Session JSON on disk**          | `**Session.write()`** serializes `**messages**` with per-message history info; `**ui-waiting**` is **omitted**. **`agent-stage`** and **`agent-issues`** are persisted; **`tool-output`** only if we add that role (optional ŌĆö ┬¦Phase 1.5). |
| **UI**                            | Roles like `**project`** are hidden from chat; `**agent-stage`** / `**agent-issues`** (and optional **`tool-output`**) ŌĆö `**is_hidden**` / non-visible, same family as metadata.                       |


**Implementation rule:** add `**agent-stage`** and `**agent-issues`** to `**Message**`ŌĆÖs `**role` switch** (done in **Phase 1**). **`tool-output`** only if we implement optional persistence (┬¦Phase 1.5). Stored for replay, **not** merged into `**Chat`** request payloads. Audit any code path that builds API history from **full** `**session.messages`** (if present) and **exclude** these roles there too.

## Mismatch and Continue (app must not use `Process.exit`)

1. **Stop** replay **without** exiting the process; show `**stage_id`** + **expected vs actual** issues (`**agent-issues`** vs current parser output).
2. **Continue:** one **live** model call with the **same system/user** as that stageŌĆÖs request (**first attempt** for recovery), **`in_replay = false`**.
3. **`Process.exit(1)`** remains acceptable **only** in the **CLI replay** harness, not in GTK session restore (**┬¦`Process.exit` and replay**).

### Phase 1.5 ŌĆö **`tool-output` (optional / deferred)**

**Not required** for the **product goal** in ┬¦Purpose (**Continue** / **inject prompt**): after a tool step, **processed** results are **not** read back by the system later; if something is needed again, the user **re-runs** the flow. Persisting raw tool strings is **not** needed for that.

**When `tool-output` might matter:** only if we implement **full `Tool.run` replay** without **`tool_impl.execute`** and still need **`executor_input`**-shaped text for that path ŌĆö **edge case**. Otherwise prefer **lighter hydration** (**markers**, task list rebuild, **skip** re-parse ŌĆö **Phase 4**) without a **`tool-output`** role.

**If implemented later:** **`tool-output`** = session-only role (mask like **`agent-*`**); **`Message`** switch + **`add_message`** after **`execute`**; hunks per **`docs/guide-to-writing-plans.md`**.

### Phase 2 ŌĆö **Restore gate and first safe `Runner.replay()` wiring (`can_replay`)**

- **Gate:** Run task replay / state rebuild **only** when the session is marker-capable (e.g. `**can_replay**`). If **false**, behaviour stays **transcript-only**; **do not** call `**Runner.replay()`** by mistake.
- **Deliverable:** **`can_replay`**, JSON **`can-replay`**, **`Runner.replay()`** early exit + **`finally`** clearing **`in_replay`** ŌĆö see **`## Concrete code proposals ŌĆö Phase 2`**.

### Phase 3 ŌĆö **Transcript anti-pollution during replay (session integrity)**

This is the material that used to sit ŌĆ£nextŌĆØ after the gate in older outlines: **ensure restore/replay does not corrupt `session.messages` or the on-disk JSON**.

**Why it looked optional:** Until **GTK** calls **`Runner.replay()`** on a **loaded** session, the bad behaviour (duplicate rows, double **`message_added`**, extra **`save_async`**) mostly shows up in tests or manual replay ŌĆö not in normal ŌĆ£open historyŌĆØ flows. That does **not** make it irrelevant; it is **blocking** for real restore UX.

**What goes wrong today (representative paths):**

- **`SessionBase.add_message`** always **`messages.add`** + **`manager.message_added`** when active ŌĆö replay **re-enters** the same paths and can **append duplicates** of content already in the loaded transcript.
- **`Runner.send_async`** / **`Tool.run`** (e.g. **`Tool.run`** lines that **`session.messages.add(system/user)`** before **`chat_call.send`**) add **new** rows during replay while **`ReplayChat`** is only supposed to substitute **model** output ŌĆö the transcript can **grow** and **dirty** the session file.
- **`ui-waiting`** and other ephemera may be re-emitted.

**Deliverable:** With **`in_replay`** (and/or a dedicated flag), **suppress** duplicate appends and spurious UI/save for content **already satisfied** by the loaded history ŌĆö see **`## Concrete code proposals ŌĆö Phase 3`** for **session vs UI pollution**, **file anchors**, and **where verbatim `Remove` / `Replace` hunks will go** (fences **TBD** until the approach is fixed; must follow **`docs/guide-to-writing-plans.md`**). Possible directions: skip **`session.messages.add`** for replay-fed assistant/system/user copies; coalesce **`add_message`**; defer **`write()`** during replay ŌĆö **user approval** on the chosen approach if trade-offs are large.

**Relation to Phase 2:** Phase 2 **does not** implement anti-pollution; it only adds **`can_replay`** and a safe **entry** to replay. Phase 3 is **required** before treating restore as shippable.

### Session pollution (persisted transcript grows or duplicates)

| Symptom | Typical cause |
| ------- | ------------- |
| **Duplicate rows** in **`session.messages`** after replay | **`SessionBase.add_message`** / direct **`messages.add`** replaying paths that **already** serialized those rows at live run. |
| **Dirty JSON on disk** right after open + replay | **`Session.write()`** / **`save_async`** runs while replay **appends** ephemera or duplicates. |
| **Double history** | **`Runner.send_async`**, **`Tool.run`**, **`Details`** add **system/user** (and similar) to **`session.messages`** **again** during replay while **`ReplayChat`** only substitutes **model** content ŌĆö transcript **length** diverges from the loaded file. |

| Symptom | Typical cause |
| ------- | ------------- |
| **Duplicate bubbles** or duplicate **waiting** rows | **`manager.message_added`** (or equivalent) fired **again** for content **already** in the loaded UI model. |
| **Spurious ŌĆ£restoringŌĆØ noise** | **`ui-waiting`** / **ephemeral** frames re-emitted during replay when they should be **suppressed** or **not** re-appended. |

## Phase 4 ŌĆö **Skip `tool_impl.execute` whenever `in_replay`** (roadmap)

Replay drives the **same** **`Runner`** / **`Details`** / **`Tool`** code paths with **`ReplayChat`** so the **LLM** is not called: **`ReplayChat.send()`** returns the **next** stored **`content-stream`** / **`content-non-stream`** / **`assistant`** body from **`session.messages`** (`libollmchat/Call/ReplayChat.vala`).

**`in_replay` is sufficient:** whenever it is **true**, **`Tool.run`** must **not** call **`tool_impl.execute`**. Re-running tools during replay would be **wrong** for restore (side effects, filesystem, unsafe). There is **no** separate **`replay_reexecute_tools`** (or similar) flag ŌĆö we are **not** supporting ŌĆ£re-execute tools inside replayŌĆØ in this plan.

**CLI / test harness (`examples/oc-test-skill-agent.vala`):** exercising real **`execute`** paths against a recorded session is **deferred**. A plausible later design is **replay stored messages up to a chosen marker / boundary**, then **leave replay** (**`in_replay = false`**, restore normal **`Chat`**) and **continue the run live** so tools run in the real environment ŌĆö not a second flag on **`Runner`**.

**Current code** (`**liboccoder/Task/Tool.vala`**, **`Tool.run`**) calls **`execute`** whenever **`tool_call != null`** with **no** guard. **Deliverable:** **`if (this.tool_call != null && !this.parent.runner.in_replay) { ŌĆ” execute ŌĆ” }`** ŌĆö no **`else`** (initial **`tool_output`** and default **`tool_run_result`** are already empty). **GTK** does **not** need **`tool-output`** for the **continue / inject-prompt** goal; only if we replay **`Tool.run`** without **`execute`** and still need **`executor_input`**-shaped text does **Phase 1.5** **`tool-output`** become relevant.
