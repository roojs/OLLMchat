## Purpose

1. Refine the **project contents** sent to the project-summary LLM: show a **nested tree** (filesystem hierarchy) and **file type labels** (code language, documentation category, or “file”) so the model sees structure and role of each item.
2. Document and support **testing** the project summary path via `oc-vector-index --project-summary`.

## Background

- **Project summary** is generated in `libocvector/Indexing/ProjectAnalysis.vala` using `analysis-prompt-project.txt`. The prompt receives `{project_path}` and `{folders_and_files_list}`.
- Currently `folders_and_files_list` is a **flat** list: all folders then all files, each line like `  - (folder) name` or `  - (file) name: description`.
- `oc-vector-index` normally: (1) scans the filesystem, (2) runs the indexer (tree-sitter, embeddings, per-file/folder analysis). The project summary is produced as part of full indexing; there was no way to run *only* the project summary step for quick iteration or debugging.

## Design: nested tree and file types

- **Single entry point**: Always get the project root folder and call **`root_folder.to_summary(keymap, "")`**. No branch: no "flat" fallback, no recurse flag. Folder always iterates its children and passes `indent + "  "`; each child formats itself.
- **Line format and type**: Each item implements **`to_summary(keymap, indent)`**. Folder: one line then append each child's `to_summary(keymap, indent + "  ")`. File: one line with type (category / language / "file"). Indent is passed in, not computed.
- **Aliases**: `folder.children` can contain **FileAlias** (symlinks). FileAlias must implement `to_summary`: when `points_to != null` delegate to `points_to.to_summary(keymap, indent)` so the tree shows the resolved target; when null, one line e.g. `- (alias) name`.

## How to test `--project-summary`

- **Prereqs**: A project folder already in the files DB (e.g. created with `oc-vector-index --create-project /path/to/repo` or by the app). Some existing vector metadata helps (from a prior full index) so the project summary has descriptions to work with; otherwise the list will show names with empty descriptions.
- **Run**:
  ```bash
  ./build/oc-vector-index --project-summary /path/to/your/project
  ```
- **With debug** (see exact prompt and LLM response):
  ```bash
  ./build/oc-vector-index --debug --project-summary /path/to/your/project
  ```
- **Behaviour**: The tool (1) resolves the folder, (2) runs the **filesystem scan** (read_dir, update project_files), (3) **does not** run the indexer (no tree-sitter, no embeddings), (4) loads existing file/folder metadata from the DB, (5) builds the project-summary prompt, (6) calls the analysis LLM, (7) saves the project summary to the DB. So you can iterate on the prompt or the list format without re-indexing the whole codebase.

### 1. FileBase / Folder / File / FileAlias: to_summary(keymap, indent)

**FileBase** (`libocfiles/FileBase.vala`): add abstract method. Caller passes **indent** (string); Folder passes `indent + "  "` to children.

```vala
/**
 * Returns one or more lines for the project summary list.
 * @param indent Leading indent for this line; Folder passes indent + "  " to children.
 */
public abstract string to_summary(Gee.HashMap<int, OLLMfiles.SQT.VectorMetadata> keymap, string indent);
```

**Folder** (`libocfiles/Folder.vala`): override; output this line then iterate children (always).

```vala
public override string to_summary(Gee.HashMap<int, OLLMfiles.SQT.VectorMetadata> keymap, string indent)
{
	var line = indent + "- (folder) " + GLib.Path.get_basename(this.path);
	if (keymap.has_key((int)this.id)) {
		var vm = keymap.get((int)this.id);
		if (vm.description != "") {
			line += ": " + vm.description;
		}
	}
	var child_indent = indent + "  ";
	foreach (var child in this.children.items) {
		line += "\n" + child.to_summary(keymap, child_indent);
	}
	return line;
}
```

**File** (`libocfiles/File.vala`): override; one line with indent. Type = keymap category (if set and not "other") else `this.language` else `"file"`.

```vala
public override string to_summary(Gee.HashMap<int, OLLMfiles.SQT.VectorMetadata> keymap, string indent)
{
	var type = "file";
	var description = "";
	if (keymap.has_key((int)this.id)) {
		var vm = keymap.get((int)this.id);
		if (vm.category != "" && vm.category != "other") {
			type = vm.category;
		}
		description = vm.description != "" ? ": " + vm.description : "";
	}
	if (type == "file" && this.language != "") {
		type = this.language;
	}
	return indent + "- (" + type + ") " + GLib.Path.get_basename(this.path) + description;
}
```

**FileAlias** (`libocfiles/FileAlias.vala`): override; when `points_to != null` delegate to target so the tree shows resolved content; otherwise one line for the alias.

```vala
public override string to_summary(Gee.HashMap<int, OLLMfiles.SQT.VectorMetadata> keymap, string indent)
{
	if (this.points_to != null) {
		return this.points_to.to_summary(keymap, indent);
	}
	return indent + "- (alias) " + GLib.Path.get_basename(this.path);
}
```
