## Purpose

1. Refine the **project contents** sent to the project-summary LLM: show a **nested tree** (filesystem hierarchy) and **file type labels** (code language, documentation category, or “file”) so the model sees structure and role of each item.
2. Document and support **testing** the project summary path via `oc-vector-index --project-summary`.

## Background

- **Project summary** is generated in `libocvector/Indexing/ProjectAnalysis.vala` using `analysis-prompt-project.txt`. The prompt receives `{project_path}` and `{folders_and_files_list}`.
- Currently `folders_and_files_list` is a **flat** list: all folders then all files, each line like `  - (folder) name` or `  - (file) name: description`.
- `oc-vector-index` normally: (1) scans the filesystem, (2) runs the indexer (tree-sitter, embeddings, per-file/folder analysis). The project summary is produced as part of full indexing; there was no way to run *only* the project summary step for quick iteration or debugging.

## Design: nested tree and file types

- **Tree source**: Use `OLLMfiles.Folder.children` (e.g. `root_folder.children.items`). Recursively walk; for each child append `indent + child.to_summary(keymap)`; recurse into folders. Indent by depth (e.g. two spaces per level).
- **Line format and type**: Each item (File or Folder) formats itself via **`to_summary(keymap)`** on `OLLMfiles.FileBase` (overridden in Folder and File). Folder returns `- (folder) name` [+ `: description`]; File returns `- (type) name` [+ `: description`] where type = keymap category (if set and not `"other"`) else `file.language` else `"file"`.
- **Fallback**: If the hierarchy is not available (e.g. `children` empty), keep the existing flat list behaviour using the same `to_summary(keymap)` for each folder and file (with a single leading indent).

## How to test `--project-summary`

- **Prereqs**: A project folder already in the files DB (e.g. created with `oc-vector-index --create-project /path/to/repo` or by the app). Some existing vector metadata helps (from a prior full index) so the project summary has descriptions to work with; otherwise the list will show names with empty descriptions.
- **Run**:
  ```bash
  ./build/oc-vector-index --project-summary /path/to/your/project
  ```
- **With debug** (see exact prompt and LLM response):
  ```bash
  ./build/oc-vector-index --debug --project-summary /path/to/your/project
  ```
- **Behaviour**: The tool (1) resolves the folder, (2) runs the **filesystem scan** (read_dir, update project_files), (3) **does not** run the indexer (no tree-sitter, no embeddings), (4) loads existing file/folder metadata from the DB, (5) builds the project-summary prompt, (6) calls the analysis LLM, (7) saves the project summary to the DB. So you can iterate on the prompt or the list format without re-indexing the whole codebase.

### 1. FileBase / Folder / File: to_summary(keymap)

**FileBase** (`libocfiles/FileBase.vala`): add abstract method so each item can format itself for the project summary list.

```vala
/**
 * Returns one line for the project summary list (no leading indent).
 * Format: "- (folder) name" or "- (type) name" with optional ": description" from keymap.
 */
public abstract string to_summary(Gee.HashMap<int, OLLMfiles.SQT.VectorMetadata> keymap);
```

**Folder** (`libocfiles/Folder.vala`): override.

```vala
public override string to_summary(Gee.HashMap<int, OLLMfiles.SQT.VectorMetadata> keymap)
{
	var line = "- (folder) " + GLib.Path.get_basename(this.path);
	if (keymap.has_key((int)this.id)) {
		var vm = keymap.get((int)this.id);
		if (vm.description != "") {
			line += ": " + vm.description;
		}
	}
	return line;
}
```

**File** (`libocfiles/File.vala`): override; type = keymap category (if set and not "other") else `this.language` else `"file"`.

```vala
public override string to_summary(Gee.HashMap<int, OLLMfiles.SQT.VectorMetadata> keymap)
{
	var type = "file";
	if (keymap.has_key((int)this.id)) {
		var vm = keymap.get((int)this.id);
		if (vm.category != "" && vm.category != "other") {
			type = vm.category;
		}
	}
	if (type == "file" && this.language != "") {
		type = this.language;
	}
	var line = "- (" + type + ") " + GLib.Path.get_basename(this.path);
	if (keymap.has_key((int)this.id)) {
		var vm = keymap.get((int)this.id);
		if (vm.description != "") {
			line += ": " + vm.description;
		}
	}
	return line;
}
```

### 2. Recursive tree builder (ProjectAnalysis)

Uses `child.to_summary(keymap)`; caller prepends indent. Call with `folder = this.root_folder`, `depth = 0`.

```vala
/**
 * Returns a nested tree of folder/file lines for the project summary.
 * Indent is two spaces per depth; each line is indent + child.to_summary(keymap).
 */
private string[] folder_tree_lines(
	OLLMfiles.Folder folder,
	int depth,
	Gee.HashMap<int, OLLMfiles.SQT.VectorMetadata> keymap)
{
	var indent = string.nfill(depth * 2, ' ');
	string[] lines = {};
	foreach (var child in folder.children.items) {
		if (child is OLLMfiles.Folder) {
			var sub = (OLLMfiles.Folder) child;
			lines += indent + sub.to_summary(keymap);
			lines += this.folder_tree_lines(sub, depth + 1, keymap);
			continue;
		}
		lines += indent + child.to_summary(keymap);
	}
	return lines;
}
```

### 3. Method that builds the project contents list (tree or flat) (ProjectAnalysis)

Insert after `folder_tree_lines()` and before `analyze()`. When root has children, use tree; else flat list using each folder’s and file’s `to_summary(keymap)` with a single leading indent.

**Intended tree format** (two spaces per level; folders and files under each folder):

```text
- (folder) lib
  - (folder) libocvector
    - (vala) Database.vala
    - (vala) Indexer.vala
  - (vala) main.vala
- (folder) docs
  - (documentation) README.md
- (plan) ROADMAP.md
```

```vala
/**
 * Builds the folders-and-files list for the project summary prompt.
 * When root_folder has children: nested tree via folder_tree_lines().
 * When children is empty: flat list, each line "  " + item.to_summary(keymap).
 */
private string build_folders_and_files_list(Gee.HashMap<int, OLLMfiles.SQT.VectorMetadata> keymap)
{
	if (this.root_folder.children.items.size > 0) {
		return string.joinv("\n", this.folder_tree_lines(this.root_folder, 0, keymap)).strip();
	}
	string[] folder_lines = {};
	foreach (var folder in this.root_folder.project_files.folder_map.values) {
		folder_lines += "  " + folder.to_summary(keymap);
	}
	string[] file_lines = {};
	foreach (var pf in this.root_folder.project_files) {
		file_lines += "  " + pf.file.to_summary(keymap);
	}
	return (string.joinv("\n", folder_lines) + "\n" + string.joinv("\n", file_lines)).strip();
}
```
