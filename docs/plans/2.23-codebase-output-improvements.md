# 2.23 Codebase search output improvements (markdown-friendly)

Implement per **.cursor/rules/CODING_STANDARDS.md** with one exception: this code is a **strong case for variable interpolation and multi-line strings** (and for locals `file` / `metadata`). Repeated `result.file()` / `result.metadata` and long `append_printf` argument lists hurt readability; building the result block with locals and `@"..."` / `@"""` (per standard’s exception for multi-line/readability) is preferred here.

## Goal

Make codebase search tool output markdown-friendly so it renders cleanly in chat UIs and in LLM context:
- Use bullet format for key/value: ` - **KEY** value` (no result numbering like "1.", "2.").
- Keep code in ``` fenced blocks; use **language** in the info string (e.g. ```vala), not line:file path.

## Current output format (exact)

**What is actually output** (from `libocvector/Tool/RequestCodebaseSearch.vala` `format_results()`):

- Intro line: `Found %d result(s) for "%s":`
- For each result:
  - One header line: `%d. %s (%s) - %s:%d-%d` → e.g. `1. format_results (function) - libocvector/Tool/RequestCodebaseSearch.vala:384-454`
  - If non-empty: `Description: %s`
  - If non-empty: `ast-path: %s`
  - Fenced code block: ` ```%d:%d:%s\n%s\n``` ` (currently uses citation in info string; no language tag)
  - If snippet truncated: `... (%d more lines)\n`

**What is available but not output:**  
Each `SearchResult` has a `distance` property (float, FAISS L2 distance; lower is better). It is **not** currently written into the formatted string. So distance is available for a future **Distance** line if desired.

Example of current output:

```
Found 2 result(s) for "format results":

1. format_results (function) - libocvector/Tool/RequestCodebaseSearch.vala:384-454
Description: Format search results for LLM consumption.
ast-path: OLLMvector.Tool-RequestCodebaseSearch-format_results
```384:454:libocvector/Tool/RequestCodebaseSearch.vala
private async string format_results(
    ...
)
```

... (20 more lines)

2. ...
```

**Issues:**
- Labels like `Description:` and `ast-path:` don’t render as bold in markdown; they look like plain text.
- The first line mixes element name, type, path, and line range in one long line; no clear key/value structure.
- Consistency: some keys use "Description", another uses "ast-path" (lowercase with hyphen).

## What is ast_path (from codebase)

**Definition:** `VectorMetadata.ast_path` (libocvector/VectorMetadata.vala): "AST path representing the hierarchical location of the element. Format: namespace-class-method or namespace-function etc. (using '-' separator). Empty string for file elements (files don't have AST paths)."

**How it is built:** `TreeBase.ast_path(node, code_content)` (libocfiles/TreeBase.vala): traverses up the AST from the element node; collects namespace (dots within namespace), then class/struct/interface hierarchy, then element name; joins all with **hyphen** `-`. Example from docblock: `OLLMvector.Indexing-OuterClass-InnerClass-methodName`. So the real format is **hyphen-separated** (e.g. `OLLMvector.Tool-RequestCodebaseSearch-format_results`), not "File > Class > method".

**Used for:** Caching/keying in VectorBuilder (ast_path or fallback `element_type:element_name`), filtering in VectorMetadata query by ast_path, and navigation/link fragment.

## Proposed output format

- **Key/value:** Bullet format: ` - **KEY** value` (one key per line; no result numbering).
- **Code:** Fenced block info string = **language** (e.g. ```vala, ```python) from `file.language`, not line:end:path. Citation (path and lines) is already in **Location** and **Link**.
- **Result header:** Each result is a block of bullet lines; no "1.", "2." — bullets only.

Example of one result in the new format:

```
Found 2 result(s) for "format results":

 - **Element** format_results (function)
 - **Location** libocvector/Tool/RequestCodebaseSearch.vala:384-454
 - **Link** [format_results](libocvector/Tool/RequestCodebaseSearch.vala#OLLMvector.Tool-RequestCodebaseSearch-format_results)
 - **Distance** 0.2145
 - **Description** Format search results for LLM consumption.
 - **AST path** OLLMvector.Tool-RequestCodebaseSearch-format_results

```vala
private async string format_results(
    ...
)
```

... (20 more lines)

 - **Element** another_name (class)
 - **Location** ...
```

Rules:
- Format: every line in the result block is a bullet: ` - **KEY** value`. No numbering of results.
- **Element** — element name and type: `name (type)`.
- **Location** — file path and line range: `path:start-end`.
- **Link** — markdown link for navigation: `[title](/path/to/file#ast_path)` with title = e.g. method/element name (`metadata.element_name`). Fragment = `metadata.ast_path` (hyphen-separated, e.g. `OLLMvector.Tool-RequestCodebaseSearch-format_results`); URL-encode fragment if needed for `#` in links.
- **Distance** — **Distance** value (float, L2; lower = better). From `result.distance`; include so relevance is visible in the new format.
- **Description** — always output (required). One line: ` - **Description** value` (value may be empty).
- **AST path** — always output (required). One line: ` - **AST path** value` (value may be empty).
- Code block: opening fence = ```{language} (e.g. ```vala) from `file.language`; fallback to plain ``` if language empty. Citation (path, lines) is in **Location**/**Link**, not in the fence.
- **Fence collision:** If the snippet contains ``` (three backticks), the default fence would break the block. Test `snippet.contains("```")` and use an alternative fence when true (e.g. four backticks ```` or more) so the snippet is never wrapped in a fence that appears inside it.
- **Truncation:** When snippet is truncated (> 50 lines), add a line *inside* the code block (not markdown after it): e.g. `// content truncated - original code was N lines`. So the fenced body is `snippet + "\n" + that comment` when truncated.

- **File:** `libocvector/Tool/RequestCodebaseSearch.vala`
- **Method:** `format_results()`

**Replace with:** local `file` and string interpolation so the output format is obvious. CODING_STANDARDS.md allows multi-line/interpolation for this case. `metadata` inlined (`result.metadata`). AST path is hyphen-separated and does not contain `#`, so no fragment escaping.

```vala

				GLib.debug(

**Notes:**
- **Link** fragment: `result.metadata.ast_path` is hyphen-separated; AST path does not contain `#`, so no escaping needed.
- **Distance:** always output (e.g. `%.4f`).
- **Language:** per CODING_STANDARDS.md avoid null checks; assume `File.language` is non-null (default empty string). Use `result.file().language != ""` only.
- **Gee.ArrayList:** use `.get(i)` not `[i]` (CODING_STANDARDS.md).

- **Changes summary:** (1) **Element** / **Location** lines; (2) **Link** with path and ast_path; (3) **Distance**; (4) **Description** and **AST path** always output (required); (5) code fence with language from `file.language` or plain ```; (6) if snippet contains ``` use alternative fence (e.g. ````); (7) when truncated, add comment inside the block: `// content truncated - original code was N lines`.

## Summary

| Aspect        | Current                         | Proposed                    |
|---------------|----------------------------------|-----------------------------|
| Key/value     | `Description: ...`, `ast-path: ...` | **Description** ..., **AST path** ... |
| Result header | `1. name (type) - path:start-end` | Bullets: ` - **Element**`, ` - **Location**`, etc. (no result numbering) |
| Link          | (not present)                    | **Link** [title](path#ast_path), title = e.g. method name |
| Distance      | Not output (available on `SearchResult.distance`) | **Distance** value (L2, lower = better) |
| Code          | ```startLine:endLine:path in fence (no language) | ```{language} from `file.language`; if snippet contains ``` use longer fence (e.g. ````) |
| Description / AST path | Description/ast-path only if non-empty | Always output (required) |
