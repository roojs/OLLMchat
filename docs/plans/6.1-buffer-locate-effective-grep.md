## Purpose

Define how OLLMchat finds **all** occurrences of a search string (or multiline pattern) in file buffers, with two backends:

1. **DummyFileBuffer** (non-GTK): **`locate` must not load the whole file into memory** ‚Äî use **grep-style chunked sequential I/O** with a **carry / overlap** window so multiline needles cannot be split across reads (Part D). **Bare minimum** elsewhere: no dedicated unit tests, no new callers, trimmed semantics ignored (same as literal match). Optional later: faster substring algorithms (e.g. Horspool) ‚Äî not required to ship chunked `locate`.
2. **GtkSourceFileBuffer** (GTK): use **GtkSourceView‚Äôs search API** (`GtkSourceSearchSettings` + `GtkSourceSearchContext`) on the live `GtkSource.Buffer`.

Requirements beyond classic grep:

- **Multiline search** ‚Äî the needle may span `\n`.
- **Trimmed multiline search** ‚Äî conceptually: compare a **per-line trimmed** view of the buffer to a **per-line trimmed** view of the needle (each line stripped of leading/trailing whitespace), and report matches in terms of real buffer coordinates / matched text.
- **Return every match**, not just the first or ‚Äúcurrent‚Äù match.
- **Result shape (proposed):** a map from **start line (0-based, consistent with `FileBuffer`)** to the **exact matched substring** as it appears in the buffer (including original whitespace and newlines). The matched string is the raw span so callers can highlight or replace accurately.

**Explicit scope decision (current):** For **DummyFileBuffer**, **do not implement trimmed multiline semantics** for now. If `match_trimmed` is requested, treat it as **plain full-text / substring matching** (same as non-trimmed ‚Äî document on the API). Trimmed matching matters on **GtkSourceFileBuffer** / project UI paths. Chunked `locate` is still **required** for dummy so tools never need a full-file string just to search.

## Part A ‚Äî What we can learn from grep (and related tools)

### A.1 GNU grep‚Äôs performance model (high level)

From GNU‚Äôs own documentation ([Performance (GNU Grep 3.12)](https://www.gnu.org/software/grep/manual/html_node/Performance.html)) and common descriptions of GNU grep:

- **Large read buffers:** read the file in big chunks rather than line-by-line I/O.
- **Avoid per-byte work when possible:** for fixed strings, implementations often use fast substring algorithms (historically associated with **Boyer‚ÄìMoore‚ÄìHorspool**-style skipping) so many bytes are skipped without full comparisons.
- **Line reporting without scanning every newline first:** a useful mental model is ‚Äúfind a match in the byte stream, **then** find surrounding newlines to assign a line number,‚Äù rather than splitting the whole file into lines up front.

### A.2 Functional lessons relevant to our design


| Idea                | Takeaway                                                                                                                                                                     |
| ------------------- | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Chunked input       | **DummyFileBuffer `locate`:** large reads via `FileInputStream` (or async read), **not** `get_contents()` / loading the whole file for the search path.                      |
| Cross-chunk matches | **DummyFileBuffer:** keep a **carry** tail between reads (length ‚â• needle length minus one, or full needle) so a multiline match cannot straddle a chunk boundary undetected. |
| Line index          | **Running line count** while scanning, or derive line from offset + newline count in the window ‚Äî same spirit as grep (line number = newlines before match start).          |
| Trimmed multiline   | **Not** how grep works; on GtkSource use constructed regex (Part E) or fallback scan. Dummy ignores trimmed semantics (Part D).                                                |


### A.3 What we are *not* copying

- Default grep is **line-oriented** for simple patterns; we **explicitly** want multiline literal (and optionally regex on SourceView).
- We need **all** occurrences; grep prints lines or counts ‚Äî we need a structured result for programmatic use.

## Part B ‚Äî GtkSourceView search API (gtksourceview 5)

Official overview: [GtkSource.SearchContext](https://gnome.pages.gitlab.gnome.org/gtksourceview/gtksourceview5/class.SearchContext.html), [GtkSource.SearchSettings](https://gnome.pages.gitlab.gnome.org/gtksourceview/gtksourceview5/class.SearchSettings.html).

### B.1 `GtkSourceSearchSettings`

- `**search-text`** ‚Äî string or pattern (if regex enabled).
- `**case-sensitive`**
- `**regex-enabled**` ‚Äî use `search-text` as a regular expression.
- `**wrap-around**` ‚Äî for UI-style ‚Äúfind next‚Äù; for ‚Äúall matches‚Äù we will likely **disable wrap** and iterate from buffer start, or use forward from start until end.
- `**at-word-boundaries`**
- `**visible-only`** (since 5.12) ‚Äî exclude invisible text; may matter for folded regions if used.

### B.2 `GtkSourceSearchContext`

- Construct with `(buffer, settings)`.
- `**get_occurrences_count()**` ‚Äî total matches; docs note count may be **-1** until the buffer is fully scanned.
- `**get_occurrence_position(index, out start, out end)`** ‚Äî random access to match *n* by index (useful to build full result list without re-searching).
- `**forward` / `forward_async*`* ‚Äî from a `GtkTextIter`, find next match; `**backward`** similarly.
- `**replace` / `replace_all**` ‚Äî optional future UI; not required for read-only locate.
- **Highlighting** ‚Äî `set_highlight(false)` if we do not want side effects on the widget (if the buffer is attached to a visible view, consider whether highlighting is desired; for headless/tool use, disable).

### B.3 Multiline behaviour

The SearchContext documentation explicitly notes that **regex searches can match multiple lines**. So for **GtkSourceFileBuffer**:

- **Multiline literal** might be implemented as **regex with the pattern escaped**, plus flags that allow `.` to span lines if needed, or a constructed multiline-safe pattern ‚Äî **verify** against the exact regex engine (PCRE via GLib/GTK) and GtkSource‚Äôs restrictions (`get_regex_error()` when invalid).
- **Trimmed multiline** is not a SearchSettings flag, but it **can still be a single GtkSource regex search**: split the needle on `\n`, **trim each line** to get the literal core, **regex-escape** each core, then wrap every line with **prefix/suffix** ‚Äúgutter‚Äù for indentation, and join the line patterns with a newline matcher (e.g. `\R` or `\n` depending on engine and platform). Conceptually each line becomes `\h*escaped_core\h*` (or `\s*` if vertical whitespace should count as trim ‚Äî match whatever `string.strip()` means in Vala for the product). That is **mechanical**, not an unmaintainable hand-written pattern.
  - **Empty lines** after trim: use a line pattern that allows ‚Äúonly whitespace‚Äù on that line (e.g. `\h*` with a line break), not `escaped_core`.
  - **Strict line alignment** (whole logical lines only): consider `(?m)^‚Ä¶$` per segment or equivalent so a match cannot start mid-line unless that is explicitly desired ‚Äî product decision.
  - **Fallback:** if GtkSource rejects the built pattern or edge cases misbehave, fall back to a **custom iterator scan** (Part E.2).

### B.4 Collecting *all* matches

Recommended approach (conceptual):

1. Configure settings (case, regex, wrap off).
2. Create `GtkSourceSearchContext`.
3. Wait until occurrences count is known (property or scan completion ‚Äî check docs / test).
4. For `i` in `0 .. count-1`, call `get_occurrence_position(i, ‚Ä¶)`, extract slice with `buffer.get_text(start, end, false)`, map `start` to **0-based line** via `start.get_line()`.

Edge cases: empty search text, invalid regex, buffer mutations during search (freeze or copy strategy if needed).

## Part C ‚Äî Result type: `Gee.HashMap<int, string>`

**Design:** `Gee.HashMap<int, string>` ‚Äî **key:** 0-based **start line** of the match (same convention as `FileBuffer`); **value:** the **exact** matched substring as it appears in the buffer (may contain `\n`). Tools that expose 1-based line numbers convert at the boundary.

If two matches share the same start line, a plain hash map keeps **one** entry; that situation is **out of scope** for current use cases ‚Äî no alternate container type is required.

## Part D ‚Äî DummyFileBuffer: chunked sequential `locate`

### D.1 Scope (this plan)

- **Memory:** `locate` must **not** read the entire file into one string for searching (no `get_contents()` / no ‚Äúload all then `index_of`‚Äù as the primary path). Use **chunked reads** like grep-style tools.
- **Process load:** **bare minimum** ‚Äî no dedicated unit tests, no new callers; implementation can stay straightforward (e.g. `index_of` in a sliding window) without Boyer‚ÄìMoore unless someone optimizes later.
- **`match_trimmed`:** **ignored for semantics** ‚Äî same as plain literal substring match; document on the method.
- **Multiline needle:** must work across chunk boundaries via **carry / overlap** (D.2).
- **Return:** fill `Gee.HashMap<int, string>`; **overlapping** occurrences are **not** a product concern ‚Äî simplest forward scan is fine.

### D.2 Chunked search strategy (required)

1. Open the file for sequential read (`FileInputStream` or async equivalent).
2. Maintain a **search window:** **carry** from the end of the previous iteration plus the newly read chunk. **Carry size:** at least **`needle.length - 1`** characters (UTF-8 **character** length for index math in `string`), or the **full needle** length for simplicity ‚Äî so any match spanning a chunk boundary appears entirely inside the combined window at least once.
3. **Global state:** running **line number** (or file offset + newline count) so each match can be mapped to **0-based start line** = number of `\n` before the match‚Äôs first byte/character, consistent with `FileBuffer`.
4. In the window, find matches with repeated **`index_of`** (or equivalent) on the **literal** needle; only **commit** matches that end before the ‚Äúunsafe‚Äù tail (the region that might be incomplete until the next read). Advance: drop consumed prefix, keep carry tail, read next chunk, repeat until EOF.
5. **Matched text:** slice the exact span from the window / accumulated context so the map value matches what would appear in the file (including `\n` and indentation).
6. **UTF-8:** treat file as UTF-8 text. When mixing **byte** positions from raw reads with **character** indices in Vala `string`, use the **built-in `string` methods** that wrap the GLib UTF-8 helpers ([valadoc: `string`](https://valadoc.org/glib-2.0/string.html)): **`utf8_offset (long offset)`** ‚Üí `g_utf8_offset_to_pointer` (returns `unowned string` at that character offset), and **`pointer_to_offset (string pos)`** ‚Üí `g_utf8_pointer_to_offset` (character offset of a slice / position inside the string). **Do not** assume one byte == one character; no custom `[CCode]` needed for these two.

### D.3 Algorithm class

- First implementation: **repeated `index_of`** on the window is enough.
- **Optional later:** Horspool / Boyer‚ÄìMoore‚Äìstyle skipping inside the window if profiling shows need ‚Äî not part of the minimum deliverable.

### D.4 `lines` cache

- **`locate` should not** populate `this.lines` from a full-file read just to search. Either leave `lines` untouched by `locate`, or document interaction if a partial cache is updated ‚Äî avoid surprising `get_line()` after `locate()`.

## Part E ‚Äî GtkSourceFileBuffer: SearchContext design

### E.1 Normal and multiline literal

- Escape needle for regex **or** use literal search if GtkSource supports non-regex multiline literal reliably ‚Äî **verify in implementation**; fallback: regex with `\Q‚Ä¶\E` or GLib‚Äôs regex escaping.
- Configure `GtkSourceSearchSettings`, attach `GtkSourceSearchContext` to `this`.

### E.2 Trimmed multiline (full support here)

When `match_trimmed` is true on **GtkSourceFileBuffer**, **prefer regex via SearchContext**:

1. Split needle on `\n`; for each segment, `strip()` to get the core (may be empty).
2. Regex-escape each non-empty core (GLib / GLib.Regex quote, or manual escape of metacharacters).
3. Build one pattern: for each line `i`, segment pattern = `\h*quoted_core\h*` (adjust `\h` vs `\s` to mirror trim semantics); for empty core after strip, use a ‚Äúwhitespace-only line‚Äù subpattern before the line terminator.
4. Join segments with `\R` (or `\n`) so the match spans multiple buffer lines.
5. Set `regex-enabled`, run `GtkSourceSearchContext`, collect spans; **matched text** comes from `buffer.get_text(start, end, ‚Ä¶)` so callers get the **real** indentation and newlines, not the trimmed view.

**Fallback if needed:** custom walk with buffer iterators (same as earlier ‚ÄúOption 1‚Äù) when the composed regex is invalid, too slow, or semantics differ (e.g. strict `^`/`$` per line). **Avoid** ‚Äúnormalized copy‚Äù unless a clear offset map exists ‚Äî easy to get wrong.

### E.3 Occurrence collection

Use `get_occurrences_count()` + `get_occurrence_position()` **or** repeated `forward_async` from start; prefer the API that minimizes duplicate scans and matches GtkSource‚Äôs caching behaviour.

## Part F ‚Äî `FileBuffer` interface

Add one abstract method only:

```text
locate (string needle, bool match_trimmed, bool case_sensitive) ‚Üí Gee.HashMap<int, string>
```

- `**needle**` ‚Äî **literal** search text (may contain `\n` for multiline), not a user regex pattern.
- `**match_trimmed**` ‚Äî full semantics on `GtkSourceFileBuffer`; on `DummyFileBuffer`, same as plain full-text match (Part D.1).
- `**case_sensitive**` ‚Äî no separate ‚Äúoptions‚Äù type and no overloads; one signature on the interface.

**Regex in the public API:** **not** in this plan. `locate` is literal-only. GtkSource may still use regex **internally** to implement multiline literal / trimmed construction (Parts E.1‚ÄìE.2). A **separate** `FileBuffer` method for user-supplied regex could be added **later** if needed ‚Äî out of scope now.

**Return:** `Gee.HashMap<int, string>` as in Part C (0-based start line ‚Üí matched text).

Implementations:

- `DummyFileBuffer` ‚Äî **chunked sequential** `locate` (Part D); literal needle only; no whole-file load for search.
- `GtkSourceFileBuffer` ‚Äî `GtkSourceSearchContext` inside `locate` (trimmed branch per Part E).

Concurrency: **no** `locate_async` (or other second entry point) on `FileBuffer` ‚Äî if a caller must not block the UI, it runs `locate` from a worker thread or task itself.

### I.2 `DummyFileBuffer` ‚Äî chunked `locate`

Replace the broken WIP `locate` in [libocfiles/DummyFileBuffer.vala](../libocfiles/DummyFileBuffer.vala). **`match_trimmed` is ignored** (same as non-trimmed). Uses a sliding window and overlap of `needle.length` characters (simple and safe for multiline needles). Empty needle after strip returns an empty map.

```vala
		/**
		 * Append to result every match of needle in the current window whose end lies
		 * on or before the safe search limit (full window at EOF; otherwise exclude the
		 * carry tail so a match spanning the next chunk is not committed early).
		 *
		 * @return true if the caller should slide the window (drop prefix, advance
		 * line_base): i.e. not at EOF and window is longer than the needle.
		 */
		private bool locate_in_window(
			string window,
			string needle,
			long line_base,
			bool eof,
			Gee.HashMap<int, string> result
		)
		{
			var search_limit = eof
				? window.length
				: (window.length > needle.length ? window.length - needle.length : 0);
			var pos = 0;
			var prev = 0;
			long nl_before_match = 0;
			while (true) {
				var found = window.index_of(needle, pos);
				if (found < 0) {
					break;
				}
				if (found + (int) needle.length > search_limit) {
					break;
				}
				nl_before_match += found > prev
					? window.substring(prev, found - prev).split("\n").length - 1
					: 0;
				result.set(
					(int) (line_base + nl_before_match),
					window.substring(found, needle.length)
				);
				prev = found;
				pos = found + (int) needle.length;
			}
			return !eof && window.length > needle.length;
		}

		public Gee.HashMap<int, string> locate(string needle, bool match_trimmed, bool case_sensitive)
		{
			var result = new Gee.HashMap<int, string>();
			if (needle.strip() == "") {
				return result;
			}
			if (!case_sensitive) {
				GLib.debug("DummyFileBuffer.locate: case_insensitive chunked search not implemented yet");
				return result;
			}
			/* Trimmed multiline not implemented on dummy; same as literal search. */
			var path_file = GLib.File.new_for_path(this.file.path);
			GLib.FileInputStream stream;
			try {
				if (!path_file.query_exists()) {
					return result;
				}
				stream = path_file.read(null);
			} catch (GLib.Error e) {
				GLib.debug("DummyFileBuffer.locate: %s", e.message);
				return result;
			}
			var window = "";
			long line_base = 0;
			if (needle.length < 1) {
				return result;
			}
			var eof = false;
			var chunk_buf = new uint8[65536];
			while (!eof) {
				ssize_t n_read;
				try {
					n_read = stream.read(chunk_buf);
				} catch (GLib.Error e) {
					GLib.debug("DummyFileBuffer.locate: %s", e.message);
					break; /* continuation: fall through to return partial result */
				}
				if (n_read < 0) {
					break; /* same: best-effort map so far */
				}
				eof = eof || n_read == 0;
				window = n_read > 0 ? window + (string) chunk_buf[0:n_read] : window;
				if (this.locate_in_window(window, needle, line_base, eof, result)) {
					var keep_from = window.length - needle.length;
					line_base += keep_from > 0
						? window.substring(0, keep_from).split("\n").length - 1
						: 0;
					window = window.substring(keep_from);
				}
			}
			return result;
		}
```

**Review notes:**

- **Try/catch scope:** Per [.cursor/rules/CODING_STANDARDS.md](../../.cursor/rules/CODING_STANDARDS.md), one small `try` for **`query_exists`** + **`path_file.read`** (both can throw on open), and a `try` **inside the read loop** only around **`stream.read`**; search / `result.set` stay outside `try`.
- **Inner search loop:** **`locate_in_window`** computes **`search_limit`** from **`eof`** and **`needle.length`**, appends matches, then returns **`true`** when the caller should slide the carry (`!eof && window.length > needle.length`).
- **Newlines:** Inside the helper, `substring(prev, found - prev).split("\\n").length - 1`; sliding prefix uses the same pattern. Window is bounded (overlap + chunk). Helper **accumulates** between successive match starts, not from 0 each time.
- **`window = window + chunk`:** Each append **copies the whole window**, so over a huge file this is **O(file_size¬≤)** in the worst case. Acceptable for the first land if windows stay bounded by overlap + chunk; if profiling shows cost, switch to **`GLib.StringBuilder`** (or a ring of chunks) for the sliding buffer instead of repeated `string` concatenation.
- **Case sensitivity:** Chunked dummy path is **case-sensitive only** until a correct insensitive strategy is added (e.g. `GLib.Regex` with `CASELESS` over the window, or byte-oriented scan). Returning an empty map when `case_sensitive == false` is a deliberate stub so reviewers do not ship wrong `index_of`/casefold index bugs.
- **Chunk as string:** `(string) chunk_buf[0:n_read]` assumes valid UTF-8; invalid sequences may need a carry for split code units (future hardening; see Part D.2).
- **`int` casts:** `result.set((int) abs_line, ‚Ä¶)` assumes line count fits `int`; matches `HashMap<int, string>`.
- **GtkSource `while (forward)`:** No single API returns all slices at once; repeated `forward` matches how [SourceView.vala](../liboccoder/SourceView.vala) steps matches ‚Äî that loop is appropriate.

### I.3 `GtkSourceFileBuffer` ‚Äî `locate` via `GtkSource.SearchContext`

Add in [liboccoder/GtkSourceFileBuffer.vala](../liboccoder/GtkSourceFileBuffer.vala) (same class as other `FileBuffer` methods). Collect matches with repeated `forward()` from increasing buffer offsets (same idea as [liboccoder/SourceView.vala](../liboccoder/SourceView.vala)); disable highlight.

```vala
		/**
		 * Build a regex pattern for trimmed multiline literal search (Part E.2).
		 * Each logical line: optional horizontal space, escaped trimmed core, optional space;
		 * lines joined by "\\R".
		 */
		private string build_trimmed_multiline_pattern(string needle)
		{
			var lines = needle.split("\n");
			var line_pats = new string[lines.length];
			for (var i = 0; i < lines.length; i++) {
				var core = lines[i].strip();
				line_pats[i] = core == ""
					? "\\h*"
					: "\\h*" + GLib.Regex.escape_string(core) + "\\h*";
			}
			return string.joinv("\\R", line_pats);
		}

		public Gee.HashMap<int, string> locate(string needle, bool match_trimmed, bool case_sensitive)
		{
			var result = new Gee.HashMap<int, string>();
			if (needle.strip() == "") {
				return result;
			}
			var settings = new GtkSource.SearchSettings();
			settings.case_sensitive = case_sensitive;
			settings.wrap_around = false;
			settings.at_word_boundaries = false;
			settings.regex_enabled = match_trimmed || needle.index_of("\n") >= 0;
			var search_text = match_trimmed
				? this.build_trimmed_multiline_pattern(needle)
				: (needle.index_of("\n") >= 0 ? GLib.Regex.escape_string(needle) : needle);
			settings.set_search_text(search_text);
			var ctx = new GtkSource.SearchContext(this, settings);
			ctx.set_highlight(false);
			var scan_pos = 0;
			while (true) {
				Gtk.TextIter from_iter;
				this.get_iter_at_offset(out from_iter, scan_pos);
				Gtk.TextIter m_start;
				Gtk.TextIter m_end;
				bool wrapped = false;
				if (!ctx.forward(from_iter, out m_start, out m_end, out wrapped)) {
					break;
				}
				result.set(
					m_start.get_line(),
					this.get_text(m_start, m_end, false)
				);
				scan_pos = m_end.get_offset();
				if (scan_pos <= from_iter.get_offset()) {
					break;
				}
			}
			return result;
		}
```

- **Invalid regex:** If `set_search_text` / SearchContext rejects the composed pattern, add a `try/catch` or check `ctx.get_regex_error()` (if exposed) and return an empty map or log ‚Äî not shown above to keep the snippet short.
- **`get_occurrences_count`:** Optional alternative: once count is stable, loop `get_occurrence_position(i, ‚Ä¶)` if your VAPI exposes it ‚Äî may avoid repeated `forward` scans; verify against installed gtksourceview VAPI.

### I.5 Negative cases and continuation (dummy `locate`)

**Negative inputs / outcomes** (expect **empty** `Gee.HashMap`, no throw from `locate`):

| Case | Expected |
|------|----------|
| `needle` empty or only whitespace (`strip() == ""`) | Empty map |
| `case_sensitive == false` (dummy only, until implemented) | Empty map + debug line |
| File path does not exist (`query_exists` false) | Empty map |
| `query_exists` or `read` throws on open | Empty map + debug |
| `needle.length < 1` (defensive; empty needle already returned) | Empty map |

**Continuation on mid-stream failure** (expect **partial** map ‚Äî matches from chunks already processed):

| Case | Behaviour |
|------|-----------|
| `stream.read` throws | Log, **break** read loop, **return `result`** with whatever matches were already recorded |
| `n_read < 0` | Same: **break**, **return partial `result`** |

So a ‚Äúnegative‚Äù I/O test after matches were found earlier in the file should assert **non-empty** map containing only the safe prefix of matches, not a throw and not a full scan.
