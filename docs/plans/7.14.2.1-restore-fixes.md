## Purpose

- **Replay:** **`Runner.on_replay`** must drive **`this.progress`** at the same logical points as live **`send_async`** / **`handle_task_list`** / **`List.run_step`*** (**§ Proposed behaviour (mapping)**).
- **UI:** **`7.14.2`** attached a **flat** **`Gtk.ColumnView`** to **`ProgressList`**. **`ProgressItem.children`** (**`Details`** → **`ToolList`**) exists in data (**`7.14.1.3`**) but was **never** connected — users see **top-level rows only**, no **expand** for tools. **`docs/plans/7.14-task-progress-tree-ui.md`** expects a hierarchy; implement **`Gtk.TreeListModel`** + **`Gtk.TreeExpander`** (**§ TreeListModel + TreeExpander**).
- **Titles:** **`Details.title`** currently returns **`slug()`** — wrong for **UI**. Task **creation / validation** (**`fill_names`**, **`validate_task`**, etc.) already guarantees **`task_data["name"]`** is present and **non-empty** for tasks that enter the run — **`title`** should **only** expose that human string (**`to_markdown().strip()`**), with **no** empty checks, **no** **`slug()`** fallback, and **no** **`has_key`** guards (**§ Title column**). If **`Details`** appears without a valid **name**, that is a bug in the **parse/validation** path, not something **`title`** defensively papers over.

---

## Diagnosis (what was missing vs red herrings)

| Observation | Conclusion |
|-------------|------------|
| Rows appear during restore / replay | **`ProgressList`** + **`on_replay`** **`progress.*`** calls are largely doing their job — the **model is built**. |
| Nested **tools** invisible | **Not** a replay bug first — **`ColumnView`** is bound to a **flat** **`GListModel`**. **`Tool`** rows live under **`Details.children`**; without **`TreeListModel`**, they **never** appear. **Missing: tree model + expander.** |
| **`rebuild()`** “breaks bindings” | **Red herring** for this issue. **`rebuild()`** clears the pending tail and re-appends the same **`Details`** references from **`runner.pending`** — **`Gtk`** column factories re-bind **`ListItem`** rows; **`ProgressItem`** instances are typically **unchanged**. Do not chase “binding breakage” here unless a **separate** bug is reproduced with a minimal case. |
| **Titles** look like **links** / slugs | **`Details.title`** is **`return this.slug();`**. Replace with **`task_data["name"]`** plain text only — **non-empty** is guaranteed by task **creation/validation** before tasks hit **`progress`**. |

## TreeListModel + TreeExpander — notes

**Goal:** Wrap **`Runner.progress`** in **`Gtk.TreeListModel`**; **`passthrough = false`** → **`ListItem.item`** is **`Gtk.TreeListRow`**; **`TreeExpander`** on the **Title** column only; **Stage** column binds **`status_str`** via **`row.get_item()`**. **Child model:** **`ProgressItem.children`** when **`get_n_items() > 0`**, else **`null`**.

**GTK4:** [Gtk.TreeListModel](https://docs.gtk.org/gtk4/class.TreeListModel.html), [Gtk.TreeExpander](https://docs.gtk.org/gtk4/class.TreeExpander.html), [Gtk.TreeListRow](https://docs.gtk.org/gtk4/class.TreeListRow.html).

**Verbatim hunks:** **§ Concrete code proposals — `ProgressView` + `Details.title`** (ordered **Keep** / **Remove** / **Replace with**). Do not introduce **`private static`** tree factory methods; duplicate the **`Gtk.TreeListModelCreateModelFunc`** **inline** in **`ProgressView()`** and **`set_runner`**.

## Title column — `liboccoder/Task/Details.vala` (**display** vs **slug**)

**Problem:** **`title`** returns **`slug()`** — kebab-case for **`task://`**, not the UI label.

**Contract:** Task **creation** and **validation** (**`List.fill_names()`**, **`ResultParser.validate_task`**, etc.) ensure **`task_data["name"]`** exists and is **non-empty** for any **`Details`** that participates in the normal task list. **`ProgressItem.title`** must **not** re-check emptiness, **not** fall back to **`slug()`**, and **not** use **`"Task"`** placeholders — that would imply **`Details`** can appear in **`progress`** without a valid **name**, which contradicts that contract (fix **parse/validation** instead).

**`Tool.title`:** unchanged unless product wants pretty-print (**`7.14.3`**).

## Concrete code proposals — **`ProgressView`** + **`Details.title`**

**Layout:** `docs/guide-to-writing-plans.md` — **Ordered chunk format for large methods**; **each** **`####`** has its **own** fence. **No** private tree helpers — inline **`Gtk.TreeListModelCreateModelFunc`** in **`ProgressView()`** and **`set_runner`** (duplicate body). Anchors match **`liboccoder/Task/ProgressView.vala`** in the tree (**`set_min_content_height`** may be **`96`** with a testing comment — preserve or set to **`48`** per **`7.14.4`**).

**No impossible-case guards (`.cursor/rules/CODING_STANDARDS.md` — defensive code):** **`TreeListModelCreateModelFunc`** receives **`item`** from **our** **`ListModel`**s — every row is **`ProgressItem`** (**`ProgressRunner`**, **`Details`**, **`Tool`**). Use **`var pi = (ProgressItem) item;`** — **do not** use **`as`** + **`if (pi == null)`**. **`if (pi.children.get_n_items() == 0) return null;`** stays — that is **GTK**’s “no child **`ListModel`**”, not a type check.

**`bind_property` only:** Factories call **`ProgressItem.bind_property(…, label, …)`** in **`bind`** with **`SYNC_CREATE`** — **no** **`set_data`**, **no** stashing **`GLib.Binding`**, **no** **`Binding.unbind()`** in **`unbind`**. Same pragmatic choice as **`HistoryBrowser`** / **`Approvals`**: list recycling could theoretically leave stale bindings; for this strip it is acceptable. **Title** column **`unbind`** is only for **`Gtk.TreeExpander.set_list_row(null)`** (row lifecycle), not for clearing bindings.
