# 7.14.8 — Multi-tool tasks: progress tree nesting review

**Status:** proposed

**Pointer:** `.cursor/rules/CODING_STANDARDS.md` — Checklist for all plans

**Layout:** `docs/guide-to-writing-plans.md`

**Related:** `docs/plans/7.14-task-progress-tree-ui.md`, `docs/plans/done/7.14.1-DONE-phase1-core.md`, `docs/plans/7.14.7-streaming-first-task-progress-tree.md`

---

## Purpose

- **Symptom:** A task that should run **multiple** tools (multiple execution rows under one `Details`) does **not** appear as **nested** items under the task row in the progress strip — the user expects a tree (`Details` → `Tool` × N) per **`docs/plans/7.14-task-progress-tree-ui.md`**.
- **Outcome:** **`ProgressList.rebuild()`** on the **live** path immediately after **`Details.build_run_queue()`**, matching **replay’s** **`build_run_queue` → `progress.rebuild()`** ordering (**Fix 2**). **`ProgressView` / `TreeListModel` callback changes** (**former Fix 1**) are **out of scope** — rejected.

**⚠️** **Do not add new methods** — only extend existing **`run_child`** with **`runner.progress.rebuild()`** (no wrappers on **`List`**, **`Runner`**, **`ProgressList`**, etc.).

---

## Scope

| In scope | Out of scope |
| -------- | ------------- |
| **`List.run_child`** → **`runner.progress.rebuild()`** after **`build_run_queue()`** | **`ProgressView`** **`Gtk.TreeListModel`** create-func changes |
| One **inline** call — **no** new methods | **Private helpers**, **`ProgressList`** API changes beyond existing **`rebuild()`** |
| **`ProgressList.rebuild()`** behaviour (`clear_pending` + `add_pending`) | Streaming **partial** tool rows before refinement (**7.14.7** optional follow-up) |
| Manual verification (multi-tool, single-tool, exam-only) | Changing **skill markdown** for tool calls |

---

## Intended behaviour (normative summary)

- **`ProgressList`** root rows include **`ProgressRunner`** slices and **`Details`** rows (`liboccoder/Task/ProgressList.vala` — `add_pending` / `add`).
- **`Details`** owns **`children`** = **`ToolList`** — one **`Tool`** per execution pass (`Details.build_run_queue`, `Details.run_exec`).
- **`ProgressView`** wraps the root model in **`Gtk.TreeListModel`** (`liboccoder/Task/ProgressView.vala`).
- **Multiple tools:** Refinement fills **`proposed_tools`**; **`build_run_queue`** copies into **`children`** (`Details.vala`).

---

## Why live differs from replay — **confirmed**

**✅** **`Skill.Runner`** calls **`this.progress.rebuild()`** only in **`on_replay`** today (exec ~704, refinement ~636, post-exec ~675). **Live** execution never called **`progress.rebuild()`** after **`build_run_queue()`**.

**ℹ️** **`ProgressList.rebuild()`** is **not** “restore-only” — it means **“resync the pending segment”**: **`clear_pending`** (drop non-completed tail rows) + **`add_pending`** (re-append **`Details`** from **`runner.pending`**) + **`items_changed`** when the tail changed (`ProgressList.vala`). Replay uses it often because replay walks messages incrementally; **live** can call the **same** method whenever the **same** resync is needed after **`ToolList`** mutates.

**✅** Replay exec path: **`build_run_queue()`** → … → **`progress.rebuild()`**, which **re-splices** pending **`Details`** on **`ProgressList`** **after** **`ToolList`** is populated — **live** did **not** resync that way.

**🔶** **Fix 2** — call **`runner.progress.rebuild()`** right after **`t.build_run_queue()`** in **`List.run_child`** — gives **live** the same **list-model refresh** replay already gets; it is **not** loading history or replaying transcripts.

**👎** **`rebuild()`** does **not** mean “restore session” by itself — it only refreshes **`ProgressList`** rows from current **`runner.pending`**.

---

## Fallback — data never produces multiple `Tool` rows

**⏳** If **Fix 2** passes manual tests but tool **count** is still wrong:

- Multiple fences under **`## Tool Calls`** (`ResultParser.extract_refinement`).
- **`Details.build_run_queue`** branches (**`proposed_tools`** / **`exam_references`**).
- Distinct **`Tool`** ids (**`tool-0`**, **`tool-1`**, …).

---

## Acceptance criteria

1. **Live run:** After refinement proposes **N ≥ 2** tool calls, once execution starts (**after `build_run_queue`**), the progress strip shows **`Details`** **expandable** with **N** child **`Tool`** rows.
2. **Single-tool** / exam-only / lone **`exec`** rows still behave correctly.
3. **Replay** unchanged or improved — no regression (**`7.14.5`**).

## Concrete code proposals

### Fix 2 — **`run_child`**: **`progress.rebuild()`** after **`build_run_queue()`**

**Target:** `liboccoder/Task/List.vala` — **`run_child`** only (**⚠️** **no new methods**).

#### Keep — surrounding logic

- **`wait_refined`**, **`run_exec`**, **`write`** — unchanged except inserting one call as below.

---

#### Add — **`rebuild()`** immediately after **`build_run_queue()`**

**Insert** between **`t.build_run_queue();`** and **`yield t.run_exec();`**:

```vala
		this.runner.progress.rebuild();
```

**Rationale:** **`ProgressList.rebuild()`** (`liboccoder/Task/ProgressList.vala`) **`clear_pending(false)`** + **`add_pending(false)`** + **`items_changed`** — same pattern replay relies on after **`ToolList`** is filled.

---

#### Replace with — full **`run_child`** (optional paste)

```vala
	private async void run_child(Details t) throws GLib.Error
	{
		yield t.wait_refined();
		t.build_run_queue();
		this.runner.progress.rebuild();
		yield t.run_exec();
		t.write();
	}
```

### Fix 3 — **`set_autoexpand`** on **`TreeListModel`** (**💩** optional UX)

## Verification

- **Manual:** Skills agent → **two** fenced tool calls under **`## Tool Calls`** → refine → run → task row shows **two** nested tools.
- **Regression:** Single-tool task; **`ProgressRunner`** rows; session **replay** still loads progress strip correctly.

---

## Risks

- **🔶** **`rebuild()`** per **`run_child`** — extra **`items_changed`** on **`ProgressList`**; watch flicker or performance on steps with many tasks (**💩** measure before optimizing).
- **ℹ️** **`bind_property`** / **`TreeListRow`** lifecycle — unchanged code path but more list churn; watch duplicate bindings if GTK recycles widgets oddly (existing **`unbind`** on title factory).
