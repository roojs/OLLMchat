# 2.31. Chatter agent — summarized history and recall links

Status: ⏳ proposed

ℹ️ Plan checklist: `docs/guide-to-writing-plans.md` — Checklist for plans.
Vala code must follow `docs/coding-standards.md`. Plan layout and
implementation workflow: `docs/guide-to-writing-plans.md`.

## Purpose

- 🔷 ⏳ Add a new **Chatter** agent in `libollmchat` with summarized
  conversation history and hash-style recall links.
  - Do **not** extend or complicate **Just Ask** — it stays a simple
    pass-through agent.
- 🔷 ⏳ After each completed Chatter turn, run a background summarization
  request.
  - The user-visible assistant reply finishes normally first.
  - The UI is free to react to completion before the summary call runs.
- 🔷 ⏳ Store an under-250-word concise summary in the session transcript.
  - The summary is updated after each completed Chatter turn.
  - The latest summary becomes the compact history sent to the model on later
    turns.
- 🔷 ⏳ Use markdown links for hash-style references inside the summary.
  - User messages: `[#user-N](#user-N)`.
  - Assistant responses: `[#agent-N](#agent-N)`.
  - Model thinking (thinking-capable models): `[#think-N](#think-N)`.
  - Tool calls or tool results: `[#tool-N](#tool-N)` when available.
  - Numbers are stable message/session references, not per-role sequential
    counters.
- 🔷 ⏳ On follow-up turns, build the outbound message list from the latest
  `summary` onward (see **Main request message assembly**).
  - The system message includes the active summary and `session_fetch` tool
    instructions.
  - Messages before the active summary are not sent.
- 🔷 ⏳ Provide a **`session_fetch`** tool that recalls a referenced user message,
  thinking trace, assistant reply, or tool event by markdown hash link.
- 🔷 ⏳ Display the latest summary in the chat UI as a collapsed summary-style
  output.
  - While the summary call runs, show the standard dotted waiting message with
    text like `summarizing conversation`.
  - Stream summary text into a collapsed summary frame as chunks arrive.

## Architecture

- 🔷 ⏳ New namespace **`OLLMchat.Chatter`** in `libollmchat/`.
  - Follows the same factory/agent split as Just Ask and Code Assistant.
- 🔷 ⏳ **`OLLMchat.Chatter.Factory`** — extends `OLLMchat.Agent.Factory`.
  - `name = "chatter"`, `title = "Chatter"`.
  - Creates a `Chatter.Agent` per session.
- 🔷 ⏳ **`OLLMchat.Chatter.Agent`** — extends `OLLMchat.Agent.Base`.
  - Owns main-request assembly (summary-reset message scan).
  - Triggers background summarization after assistant completion.
- 🔷 ⏳ **`OLLMchat.Chatter.Summarizer`** — extends `Object` (same pattern as
  `History.TitleGenerator`).
  - Runs the separate streaming LLM call that produces/updates the summary.
  - Validates markdown links before persisting.
  - Does not live on `Session` — called by `Chatter.Agent` only.
- 🚫 Do not add Chatter-specific logic to `History.Session.send()`.
  - Session stays generic; the agent owns request and post-turn behaviour.
- 🚫 Do not modify `OLLMchat.Agent.JustAsk` or `JustAskFactory`.

## Current behaviour

- ℹ️ `History.Session.send()` stores a user turn as `user-sent` plus a visible
  `ui` frame, then calls `agent.send_async()` with a `user` message.
- ℹ️ `Agent.Base.send_async()` appends the current `user` message to
  `session.messages`, then sends every prior API-compatible `user`,
  `assistant`, and `tool` message to the model.
- ℹ️ Just Ask uses `Agent.Base` with no overrides — full transcript every turn.
- ℹ️ Session JSON persists top-level metadata and the full `messages` array.
  There is no summary message role yet.

## Session transcript — messages and tool recording (current)

ℹ️ This section documents **today's** persistence in `session.messages` so
Chatter summarizer input and `#user-N` / `#think-N` / `#agent-N` / `#tool-N`
links match reality. Code refs: `History/Session.vala`, `Agent/Base.vala`,
`Call/ChatBase.vala`, `Message.vala`.

### Roles in the transcript

| Role | Persisted in JSON | Sent to LLM API | Typical use |
| ---- | ----------------- | --------------- | ----------- |
| `user-sent` | yes | no | Raw user text for history; **use for `#user-N`** |
| `user` | yes (added in `send_async`) | yes | API user message for the request |
| `ui` | yes | no | Fenced display block ("You said", metrics footer, …) |
| `content-stream` | yes | no | Visible assistant reply text (streamed) |
| `think-stream` | yes | no | Visible thinking stream (thinking models) |
| `end-stream` | yes | no | Stream end marker |
| `assistant` | yes | yes | API assistant message; **`tool_calls`** array when model invokes tools |
| `tool` | yes | yes | Tool result: `tool_call_id`, `name`, `content` |
| `ui-waiting` | no (transient signal only) | no | Dotted waiting label |
| `done` | no | no | Runtime completion signal for tools |

Tool status lines (`Executing tool: …`, errors) go through
`handle_tool_message` → `Manager.tool_message` → UI only — **not** appended to
`session.messages`.

### Typical turn without tools

1. `user-sent` — user text (history source).
2. `ui` — fenced "You said" block.
3. `user` — added when `Agent.send_async()` runs (API).
4. `think-stream` — model reasoning (thinking-capable models); text in
   `content`; may precede `content-stream`.
5. `content-stream` — assistant reply as it streams.
6. `end-stream` — stream finished.
7. `ui` — timing / model metrics line.

### Typical turn with tool calls

After the model returns tool calls (`Call/ChatBase.toolsReply`):

1. Same user messages as above.
2. `think-stream` — optional reasoning before visible reply or tool use.
3. `content-stream` — may hold brief assistant text before/during tools (can be
   empty on tool-only replies).
4. `end-stream` — end of that stream segment.
5. **`assistant`** — added via `agent.add_message(response.message)` with
   `Message.tool_calls` (`Response.ToolCall`: `id`, `function.name`,
   `function.arguments` as JSON). **One message per model response that
   requested tools** — multiple calls share one `assistant` row.
6. **`tool`** — one per executed call via `Message.tool_reply(tool_call_id,
   name, content)` (or `tool_call_fail` / `tool_call_invalid` on error).
   Added via `agent.add_message(reply_msg)`.
7. Further `content-stream` / `think-stream` / `assistant` / `tool` cycles if
   the model continues after tool results (multi-step tool loop).
8. Final `ui` metrics when the outer response completes.

### Hash reference mapping (Chatter plan)

- **`#user-N`** → `session.messages[N]` where role is `user-sent` (preferred)
  or API `user`.
- **`#think-N`** → model reasoning for that turn — `think-stream` at index `N`
  (`content` holds the thinking text). Omit from `{turn_references}` when the
  turn had no thinking.
- **`#agent-N`** → visible assistant reply text — usually `content-stream` at
  index `N` (concatenate consecutive `content-stream` segments in the same turn
  if needed).
- **`#tool-N`** → either:
  - `assistant` at `N` with non-empty `tool_calls` (tool **call** — name +
    arguments), or
  - `tool` at `N` (tool **result** — `name` + `content`).

`N` is always the message index in `session.messages` at append time.

### Summarizer `{turn_references}` layout

🔷 ⏳ `Summarizer` builds the **new exchange** for the completed turn using
section headings that include the hash link for that row:

```markdown
### User ([#user-12](#user-12))

{content from user-sent at index 12}

### Thinking ([#think-14](#think-14))

{content from think-stream at index 14 — omit section if no thinking}

### Assistant ([#agent-16](#agent-16))

{content from content-stream at index 16 — omit section if empty}

### Tool call ([#tool-18](#tool-18))

{tool name + serialized arguments from assistant.tool_calls at index 18}

### Tool result ([#tool-19](#tool-19))

{content from tool message at index 19}

… repeat Tool call / Tool result pairs for each call in the turn …
```

- Omit **Thinking** when the turn had no `think-stream` message.
- Omit **Assistant** if the turn was tool-only with no `content-stream` text.
- Omit **Tool call** / **Tool result** sections when the turn had no tools.
- **Allowed references** lists every `[#user-N](#user-N)`, `[#think-N](#think-N)`,
  `[#agent-N](#agent-N)`, and `[#tool-N](#tool-N)` used in those headings plus
  links carried forward from the previous summary.
- 🔷 In the summary text, cite **`#think-N`** only when the thinking content is
  **particularly valuable** (e.g. non-obvious reasoning, key decision chain,
  error diagnosis) — not for routine or empty reasoning.
- 🔷 When a think reference is warranted and the trace is long, use the link —
  do not paste full thinking inline.

## Proposed behaviour

- 🔷 ⏳ First Chatter turn:
  - Send the latest user request with full history (same as Just Ask — no summary
    yet).
  - After the assistant response is complete, `Chatter.Agent` starts
    `Summarizer` in the background.
  - Disable thinking for the summarizer call.
  - The summarizer receives the completed turn as `{turn_references}` sections
    (User, Thinking, Assistant, Tool call, Tool result — see **Session
    transcript**).
  - The summarizer returns a concise summary under 250 words.
  - Persist the summary as a transcript message with role **`summary`**.
  - Show that message in the UI as the collapsed summary output.
- 🔷 ⏳ Later Chatter turns:
  - `Chatter.Agent.send_async()` builds the API message list by scanning
    `session.messages` in order (see **Main request message assembly**).
  - After the latest `summary`, only messages since that summary (plus system
    built from it) go to the model — not the full prior transcript.
  - After completion, `Summarizer` receives the previous summary plus the new
    `{turn_references}` block for that turn (including tool call/result sections
    when tools ran).
  - Append the new summary as the latest `summary` transcript message.
- 🔷 ⏳ Summary-call UI reuses the standard session-message pattern.
  - Add a `ui-waiting` message with label `summarizing conversation`.
  - Stream returned text into the visible `summary` message.
  - Render it collapsed with `markdown.oc-frame-summary.collapsed`.
- 🔷 ⏳ Summary references stay useful to the model.
  - Cite important prior turns with markdown links.
  - The summarizer may drop older references as the conversation grows.
  - The summary should not become a catalogue of historical links.
  - The summary should not quote long source text or tool output.
  - If exact details are needed, the model calls the **`session_fetch`** tool.

## Reference model

- 🔷 ⏳ Use markdown links whose `href` is a local hash anchor.
  - `[#user-N](#user-N)` → persisted `user-sent` or API `user` message.
  - `[#think-N](#think-N)` → persisted `think-stream` message (`content`).
  - `[#agent-N](#agent-N)` → persisted assistant-visible reply (`content-stream`).
  - `[#tool-N](#tool-N)` → persisted `assistant` tool-call or `tool` reply.
- 🔷 ⏳ `N` is the message's index in `session.messages` (0-based array
  position).
  - References are assigned from append order only.
  - There is no renumbering step and no separate persisted ref field on
    `Message`.
  - The **`session_fetch`** tool resolves `#user-N`, `#think-N`, `#agent-N`, and
    `#tool-N` by looking up `session.messages[N]`.

## Summary validation

- 🔷 ⏳ The summarizer input provides a closed set of allowed markdown links.
  - Output may refer only to those links.
  - Links not in the input reference set are invalid.
  - Allowed set = links in the previous summary plus links for the latest user,
    thinking, agent, and tool messages.
- 🔷 ⏳ Validate the returned summary before storing it.
  - Parse the summary text with `Markdown.Document.Render.parse()` (same as
    skills code in `WriteExec` / `Action.Base`).
  - Collect links from `render.document.links` (`Markdown.Document.Format`).
  - Confirm every link is a local hash reference in the allowed set
    (`path == ""`, `hash` is `user-N`, `think-N`, `agent-N`, or `tool-N`).
  - Reject links with a non-empty `path`, `http`/`https`/`task` schemes, or
    hashes not in the allowed set.
  - Bare `#user-N` text outside a markdown link is not in `document.links` —
    treat as valid only when wrapped as `[…](#user-N)`.
- 🔷 ⏳ If validation fails, retry the summarizer once with the validation
  issue.
  - If it still fails, store no new summary and keep the previous summary as
    active history context.
- ℹ️ Reference implementation: `liboccoder/Action/WriteExec.vala` —
  `sum_render.parse(…)` then `validate_all(sum_render.document.links)`.
- ℹ️ Section link extraction: `Block.links()` on parsed heading nodes (as in
  `Task.Details` for References / Shared references).

## Summary prompt contract

- 🔷 ⏳ The summarizer prompt (`chatter_summary.md`) should say:
  - Role: update a **running summary** — concise, actionable, markdown-friendly.
  - Input: previous summary, new exchange (user, thinking, assistant, tool
    refs), allowed link set.
  - Output: **only** the updated summary text (no preamble, no fenced code
    block around it).
  - Preserve decisions, constraints, unresolved questions, tool calls and key
    results, errors that block next steps, IDs/counts/filters the user cares
    about.
  - Replace long assistant replies, thinking traces, JSON blobs, or tool output
    with markdown hash links; the main agent uses **`session_fetch`** to retrieve
    full text.
  - Cite **`#think-N`** in the summary **only when thinking was particularly
    valuable** — skip think links for routine or discardable reasoning even if
    `{turn_references}` included a Thinking section.
  - Use **only** links from **Allowed references** — `[#user-N](#user-N)`,
    `[#think-N](#think-N)`, `[#agent-N](#agent-N)`, `[#tool-N](#tool-N)`; never
    bare `#user-N`.
  - Merge related facts across turns; drop older detail when over 250 words.
  - Omit greetings, thanks, and conversational fluff.
  - If the new exchange adds nothing material, return the previous summary
    unchanged.
  - If input is malformed or empty, return the previous summary unchanged.
- 🔷 ⏳ Load prompts via **`OLLMchat.Prompt.Template`** in `libollmchat`
  (`libollmchat/Prompt/Template.vala`).
  - Add **`OLLMchat.Prompt.ChatTemplate`** — extends `Template`, sets
    `source = "resource:///"`, `base_dir = "chat-prompts"` (same wrapper pattern
    as `OLLMcoder.Skill.PromptTemplate` and plan **2.30**).
  - Template files live in `resources/chat-prompts/`; register prefix
    `/chat-prompts` in `resources/gresources.xml`.
  - Each file splits on the first **`---`**: system message above, user
    template below (`load()` requirement).
  - Fill placeholders with **`system_fill()`** / **`fill()`** — `{key}` → value
    (optional `{key/start}` … `{key/end}` sections from skills templates).
  - **`Summarizer`** uses `chatter_summary.md`: instructions in system half;
    user half filled with `{previous_summary}`, `{turn_references}`,
    `{allowed_references}`.
  - **`{turn_references}`** — see **Summarizer `{turn_references}` layout**
    (`### User`, `### Thinking`, `### Assistant`, `### Tool call`,
    `### Tool result`; omit empty sections).
  - **`Chatter.Agent`** main chat:
    - `chatter_initial.md` — `system_fill("environment", …)` when no summary.
    - `chatter_followup.md` — `system_fill("conversation_summary", …,
      "environment", …)` when `active_summary` is set.
- ℹ️ Reference: `OLLMcoder.Skill.PromptTemplate.template()` +
  `system_fill()` / `fill()` in `liboccoder/Skill/Runner.vala`.
- 🔷 ⏳ Run the summarizer with tools disabled and thinking disabled.
- 🔷 ⏳ Run the summarizer with streaming enabled.
  - Stream text into the collapsed `summary` message as it arrives.

  - Clear or replace the waiting indicator when summary content starts.

## Main request message assembly

- 🔷 ⏳ `Chatter.Agent` builds the outbound message array by looping
  `session.messages` in order:
  - Start with an empty working array and `active_summary = null`.
  - For each message:
    - **`summary`**: clear the working array; set `active_summary` to this
      message; do not add the summary to the working array.
    - **`user` / `assistant` / `tool`**: append to the working array.
    - Skip UI-only and non-API roles (`ui`, `ui-waiting`, `content-stream`,
      etc.) — same filtering as `Agent.Base` today.
  - Each `summary` in the transcript resets the working array; a later summary
    replaces an earlier one the same way.
  - When the loop finishes:
    - If `active_summary` is set, prepend a **system** message from
      `chatter_followup.md` (`system_fill("conversation_summary",
      active_summary.content)` plus `{environment}`).
    - If `active_summary` is null, prepend **system** from `chatter_initial.md`
      (`system_fill("environment", …)` only — first turn and any turn before
      the first `summary` exists).
  - Send `system` (if any) plus the working array to the model.
- 🔷 ⏳ When no `summary` message exists yet, the loop never clears — working
  array holds the full API-compatible history (same as Just Ask).
  - First turn has no summary.
  - Sessions restored from older JSON behave the same until a turn completes
    and writes the first `summary`.

## Main chat system prompts

🔷 ⏳ Chatter uses **two** `ChatTemplate` resources for the **main** chat (not
the summarizer):

| Template | When used |
| -------- | --------- |
| `chatter_initial.md` | No `summary` in transcript yet — first message and full-history turns until the first summary is written. |
| `chatter_followup.md` | `active_summary` set after the summary-reset scan — summarized follow-up turns. |

Both load via `OLLMchat.Prompt.ChatTemplate` (`---` split: system above, user
below). `Chatter.Factory` or `Chatter.Agent` fills placeholders at send time.

### Runtime environment block

🔷 ⏳ Fill `{environment}` on every main-chat system prompt (both templates).
  - Reuse lightweight context similar to Code Assistant / skill prompts — not
    the full Code Assistant rule stack.
  - **Include when available:** current date (`YYYY-MM-DD`), OS / pretty name
    (`Factory.get_os_version()` or `GLib.Environment.get_os_info`), shell
    (`$SHELL` if set).
  - **Include when set:** `session.project_path` as workspace path.
  - 💩 Format as a short bullet list (same spirit as `Skill.Runner.env()`):
    `Date`, `OS`, `Shell`, `Workspace`.
  - 🚫 Do not instruct the model to prefer shell/command output over tools or
    internal reasoning — keep environment factual only.

### `chatter_initial.md` — first / full-history turns

🔷 ⏳ System prompt should say:
  - You are a helpful assistant (Chatter agent); help the user with their request.
  - Tools are available when this session supports tool calling — use them when
    they help answer the question or complete the task.
  - Brief `{environment}` block (date, OS, shell, workspace as available).
  - 🚫 No summarized-history or `session_fetch` instructions on this template
    (no summary exists yet).

### `chatter_followup.md` — summarized follow-up turns

🔷 ⏳ System prompt should say:
  - You are continuing an earlier conversation.
  - The section below is a **concise summary** of what happened before — not
    the full transcript.
  - **Use the summary when** the user's latest message continues that thread;
    **ignore it when** the question is unrelated or starts a new topic.
  - Markdown links in the summary (`[#user-N](#user-N)`, `[#think-N](#think-N)`,
    `[#agent-N](#agent-N)`, `[#tool-N](#tool-N)`) point at stored messages.
  - Call **`session_fetch`** with a link when you need exact wording, thinking,
    assistant text, or tool output — the summary is intentionally incomplete.
  - `{conversation_summary}` — active summary message content.
  - `{environment}` — same runtime block as the initial template.

## Main request system message (assembly summary)

- 🔷 ⏳ **Initial template** when no `active_summary` after the scan.
- 🔷 ⏳ **Follow-up template** when `active_summary` is set:
  - Continuing-conversation framing and relevance guidance (use vs ignore summary).
  - Active `summary` content in `{conversation_summary}`.
  - `session_fetch` instructions for hash links.
- 🔷 ⏳ Messages after the active summary in the working array are sent as-is
  (typically the latest `user` message on a follow-up turn; may include
  assistant/tool messages from the current in-flight turn when applicable).

## Session fetch tool

- 🔷 ⏳ Add tool **`session_fetch`** (`libollmchat/Tool/SessionFetch.vala`).
  - Accepts one or more markdown hash references.
  - Examples: `[#user-1](#user-1)`, `[#think-2](#think-2)`,
    `[#agent-3](#agent-3)`, `[#tool-6](#tool-6)`.
  - For `#think-N`, return `think-stream` message `content`.
  - Returns exact stored message excerpts for those references.
- 🔷 ⏳ Works against the current session only.
- 🔷 ⏳ Register on **`History.Manager.tools`** for all sessions and agents —
  not Chatter-only; no harm if other agents ignore it.

## Session persistence and UI

- 🔷 ⏳ Persist each generated summary in the session transcript.
  - New message role: **`summary`** (not `conversation-summary`).
  - Store in the standard session JSON `messages` array.
  - Latest `summary` message is the active summary.
- 🔷 ⏳ Display the latest summary as collapsed summary output in the chat UI.
- 🔷 ⏳ Do not store the active summary as top-level session metadata.
  - Restore finds the latest summary by scanning `messages`.
- 🔷 ⏳ Mark **`summary`** as UI-visible and API-hidden.
  - Visible through the collapsed summary UI.
  - Excluded from the generic API-compatible history filter in
    `Chatter.Agent` / `Agent.Base` message assembly.
  - Streaming summary chunks update this message, not `content-stream`.
- 🔷 ⏳ **`History.Session` needs no Chatter-specific show/hide logic.**
  - `Message.role = "summary"` sets `is_ui_visible` in `Message.vala`.
  - `ChatWidget` renders `summary` like other UI-visible roles.
  - Session JSON load/save is unchanged apart from the new role.

### 4. `resources/chat-prompts/chatter_summary.md` — summarizer prompt

**Why:** Running-summary instructions, DO/DON'T, hash-link rules, 250-word limit,
and worked example — adapted for Chatter + `session_fetch`.

**Depends on:** Phase 1.

#### Add

```markdown
You are a conversation summariser for the OLLMchat **Chatter** agent.

Your job is to **update a running summary** of the conversation, keeping it
**concise**, **actionable**, and **markdown-friendly**.

The next main assistant turn receives **only this summary** (plus messages
after the latest summary boundary), not the full transcript. The main agent can
call **`session_fetch`** with your markdown links to retrieve exact stored
messages when it needs full wording or tool output.

---

### Output rules

1. **Produce only the updated summary** — no preamble ("Here is your summary"),
   no markdown code fence around the whole output.

2. **Keep under 250 words.** Trim the oldest, least relevant sentence first if
   you need space. Do not become a catalogue of hash links.

3. **Preserve essential state:**
   - User intent, decisions, constraints, and unresolved questions.
   - Tool calls and **primary outcomes** (what was invoked, key result in one
     line — not raw dumps).
   - Errors or blockers that affect the next turn.
   - IDs, counts, paths, filter values, and names the user may refer to again.

4. **Use markdown hash links for retrievable detail:**
   - When you would paste a long assistant reply, **thinking trace**, JSON blob,
     file contents, or a long list, use a link from **Allowed references**
     instead, e.g. `[reasoning](#think-11)`, `[full reply](#agent-12)`, or
     `[tool output](#tool-14)`.
   - **Only** use links listed under **Allowed references** below.
   - Write `[#user-3](#user-3)`, not bare `#user-3`.
   - Do not invent link targets.
   - **Thinking links are optional in the output.** Include `[#think-N](#think-N)`
     only when the thinking content is **particularly valuable** — e.g. explains
     a non-obvious decision, resolves ambiguity, or records error diagnosis.
     Do not cite thinking for routine filler or when the assistant reply alone
     is enough.
   - When a think reference **is** warranted and the trace is long, link it in
     one short phrase — do not paste the full thinking block.

5. **Merge across turns** — combine related facts (e.g. two tool calls about
   the same file) into one line instead of repeating.

6. **Consistent tense** — past for completed actions, present for current state.

7. **No new material** — if the new exchange is fluff or adds nothing material,
   return **CURRENT SUMMARY** unchanged.

8. **Malformed input** — if **NEW EXCHANGE** is empty or unusable, return
   **CURRENT SUMMARY** unchanged.

---

### Do

- **Do** state what the user wanted and what was decided or left open.
- **Do** note tool usage in shorthand: `read_file foo.vala → [outline](#tool-8)`.
- **Do** link thinking **only when it is particularly valuable**, e.g. `chose X
  after [reasoning](#think-11)`.
- **Do** replace large payloads with hash links the main agent can fetch.
- **Do** drop older references when the summary gets crowded.
- **Do** omit greetings, thanks, politeness, and repeated clarifications.

### Don't

- **Don't** cite `#think-N` for every turn that had thinking — only when the
  reasoning is worth recalling later.
- **Don't** output raw JSON, stack traces, long thinking blocks, or long quoted
  text — link instead.
- **Don't** repeat the same fact every update — summarise once.
- **Don't** use links not in **Allowed references**.
- **Don't** guess details that were not in the exchange.
- **Don't** wrap the summary in a code block.

---

### Example

**CURRENT SUMMARY:**
User asked how sessions are saved. Assistant explained JSON under `history/`.

**NEW EXCHANGE:**
User: Show me the exact Session.vala save method.
Assistant: [long multi-paragraph explanation with code]

**Good output:**
User asked how sessions are saved (JSON under `history/`). User then asked for
the exact `Session.vala` save method; see [save method explanation](#agent-22).

**Bad output:** pasting the full assistant paragraphs or code inline.

---

### Current summary

{previous_summary}

### New exchange

{turn_references}

(Sections: `### User ([#user-N](#user-N))`, `### Thinking ([#think-N](#think-N))`,
`### Assistant ([#agent-N](#agent-N))`, `### Tool call ([#tool-N](#tool-N))`,
`### Tool result ([#tool-N](#tool-N))` — built by `Summarizer`; omit sections
with no content.)

### Allowed references

Use **only** these markdown links when citing stored messages or tools:

{allowed_references}
```
