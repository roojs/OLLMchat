# 7.14.12 — Progress strip: no new rows at post-exec (duplicate `Details`)

Status: proposed

Pointer: `.cursor/rules/CODING_STANDARDS.md` (checklist for all plans), `docs/guide-to-writing-plans.md`

## Purpose

- 🔷 Product rule for the strip when synthesis runs after multiple tools:
  - `POST_EXEC` means `Details.run_post_exec()` (LLM combines tool outputs).
  - That phase must not behave like a new task line in the strip.
  - Same task row as execution; not an extra `ProgressItem` for the same work unit.
- 🔷 Refinement: same idea if refine completion is ever wired like “another row”. Secondary until post-exec story is settled.

## Scope

| In scope | Out of scope |
| -------- | ------------ |
| Any `runner.progress.*` / `ProgressList` mutation tied to `run_post_exec` or POST_EXEC replay | `msg_idx` / Idx column |
| `Runner.on_replay` POST_EXEC branch (`progress.rebuild`, etc.) | Gtk.TreeListModel internals |
| `List.run_child` / `run_exec` ordering vs `rebuild()` | Chat transcript layout |

## Acceptance criteria

- 🔷 After multi-tool `run_post_exec`, top-level `ProgressList` row count for that task must not grow because of post-exec alone.
  - Same `Details*` must not appear twice in `rows`.
- 🔷 During POST_EXEC, one logical row per task: bindings update, no second insertion for the same slug.
- 🔷 Document allowed sync points (e.g. `build_run_queue` then `rebuild` before `run_exec`) vs what post-exec must not do.

## Code review — what touches `ProgressList`

- ℹ️ `Details.run_post_exec` (`liboccoder/Task/Details.vala`)
  - No `runner.progress` or `ProgressList` calls.
  - Only `GLib.debug` text contains the word “progress”; does not mutate the strip.
  - Live post-exec does not append by itself. Duplicates timed with post-exec are indirect (replay, `add_completed`, tree vs flat list, UI).
- ℹ️ Live `List.run_child`
  - Calls `this.runner.progress.rebuild()` once after `build_run_queue()` and before `yield t.run_exec()`.
  - `run_exec()` runs tools then `run_post_exec()` when `tools().size > 1`.
  - No further `progress.*` inside `Details.run_exec` or `run_post_exec`.
- ℹ️ `ProgressList` mutators only: `add`, `add_pending`, `clear_pending`, `rebuild`, `add_completed`, `clear`.
- ℹ️ `runner.progress` call sites (repo grep)
  - Runner: task creation / iteration (`add`, `clear_pending`, `add_pending`).
  - Runner `on_replay`: `rebuild` on REFINEMENT, POST_EXEC, EXECUTION `content-stream`; `add_completed` where used.
  - `List.run_step` / `run_step_until_approval`: `add_completed`.
  - `List.run_child`: `rebuild`.
- 🔷 Conclusion: “extra append during live post-exec via rebuild inside `run_post_exec`” does not match current code. Live `rebuild` already ran before exec; post-exec does not call `rebuild` again.

### If duplicates are real (ideas, not exhaustive)

- 🔷 Tree vs flat list
  - TreeListModel shows `Details` plus `Details.children` (tool rows).
  - Many tools can look like duplicate tasks while `ProgressList` has one top-level `Details`.
  - Compare `ProgressList.rows` to column-view depth in a debugger.
- 🔷 Replay
  - `on_replay` POST_EXEC / REFINEMENT / EXECUTION each call `progress.rebuild()`.
  - Check whether `clear_pending` + `add_pending` ever attach the same `Details*` twice for one pending graph (`items_changed`, ordering, duplicate refs in `step.children`).
- 🔷 Step completion
  - `add_completed` appends `step.children` then `add_pending(false)`.
  - Overlap of the same `Details` between completed and pending graphs could duplicate pointers in `rows`.
- 🔷 `clear_pending`
  - Walks from the tail removing until `status == COMPLETED`.
  - Interacts with `ProgressRunner` rows and `Details` in POST_EXEC; worth stepping through on a bad session.

## Investigation

- ℹ️ Log `ProgressList.rows` (pointer or `slug()`) before and after `run_post_exec` on multi-tool path.
  - Expect unchanged top-level count if the bug is not live post-exec appending.
- ℹ️ `grep progress.` helps replay and list completion paths, not `run_post_exec` body.

## Proposed fixes (direction)

Revise after measurements.

1. 🔷 If `rows` has duplicates
   - Separate same `Details*` twice vs two `Details` with same title.
   - Fix graph, replay, or `add_completed`, not debug strings in `run_post_exec`.
2. 🔷 Replay REFINEMENT / POST_EXEC / EXECUTION
   - Decide if `rebuild()` every `content-stream` is required or should be coalesced / replaced with notifies on existing rows.
   - Only if duplicates tie to `items_changed` here.
3. 🔷 If only the UI doubles
   - `ProgressView` / expander / factory; leave `ProgressList` alone.
4. 🚫 Dedupe with `rows.contains`
   - Last resort after the above.
