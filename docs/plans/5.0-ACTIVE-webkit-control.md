## Purpose

- **🔷** New library **`libocwebkit`** — browser host **and** the OLLMchat **browser tool** (hand in hand). Not born inside chat UI chrome; chat registers/uses the library.
- **🔷** Library depends on **OLLMchat tool interfaces** (`BaseTool`, request types, etc.) so the tool is a first-class citizen of the lib.
- **🔷** Early work: **`oc-test-webkit`** under **`libocwebkit/examples/`** (Phase **2.05** Snappr host → **2.1** a11y → Phase **3** recipes) before globe UI.
- **🔷** Tool help is **action-based** (overview + per-action topic pages); browsing is driven by the **platform accessibility tree**, **not** DOM/JS page scrapes (and **not** Snappr SERP JS).
- **🚫** Do **not** implement dump / fill / press with injected page JavaScript or DOM walks (`document.body`, `querySelector`, ARIA-attribute scrape). **No** `evaluate_javascript` in `Browser` for those paths.
- **🔷** Each **chat instance** will own one browser session — **via Phase 6 generic UI affordance**, not a hardcoded `BrowserStack` field on chat. Until then `Tool.stack` defaults to its own stack.
- **🔷** Tool wire name: **`browser`** (title **Browser**).
- **🔷** Actions: **`help`**, **`fetch`**, **`search`** (Google only for now), **`press`**, **`download`** (URL), **`whereami`** (“where am I?”).
- **🔷** **`fill` is an argument**, not an action — pass a fill **object** (press-ref id → text) on a call (typically with **`action: press`**) to type into fields then activate.
- **🔷** Tool replies return page content (default a11y markdown with inline press refs + References) so the model can continue with **`press`** (+ optional **`fill`**).
- **🔷** UI: **globe toggle** chat ↔ browser on **desktop** so the user can see the live page when needed; default stays on chat.
- **🔷** Port Snappr **target-site Cloudflare detection + UI handoff**; ignore Snappr login / PMAN auth.
- **🔷** **Platform for this plan’s spike:** **Linux** (WebKitGTK) only. **Windows** a11y → **[`5.0.1`](5.0.1-windows-webkit-accessibility.md)**. **Android** → **[`5.0.2`](5.0.2-android-webkit-control.md)**.
- **✔️** Phase 2 scaffold. **✔️** Phase **2.05** Snappr host. **✅** Phase **2.1** a11y. **✅** Phase **3** recipes. **Next:** Phase **4** CF → Phase **5** Registry (no chat stack yet).

## Actions (v1)

### `help` (action)

- **🔷** **`action: "help"`** — not a boolean-only Hub-style `help: true` as the primary API (may still accept `help: true` as alias for overview — **💩**).
- **🔷** **Overview:** `{"action": "help"}` — short index: what the tool is, default `format=a11y` dump shape, list of actions, “call help for an action name before using it”, prefer `press` over inventing URLs.
- **🔷** **Per-action page:** `{"action": "help", "topic": "<action>"}` (or equivalent arg name) — e.g. `topic: "press"`, `topic: "search"`. Also allow **`topic: "fill"`** and **`topic: "format"`** even though those are arguments, not actions.
- **🔷** Must re-call overview help when that text is missing from context (same rule as Hub). Must call the **topic** help when about to use an action whose page is not in context.
- **🔷** Overview and topic pages are **for the model** — omit engine internals (settle timers, Cloudflare machinery). Light note that the chat keeps one browser session is OK if useful; no CF essay.

### `fetch`

- **🔷** Load a URL in the chat’s WebKit instance.
- **🔷** Wait for load settle (see **Settle**).
- **🔷** Return page content in the requested **output format** (default: accessibility markdown).

### `search`

- **🔷** Args: **`query`** — search terms (not a URL). Google only in v1; no `engine` in schema.
- **🔷** Use **accessibility** to fill/submit when needed — **not** Snappr’s `search-google.js` / DOM scrape.
- **ℹ️** Under the hood related to **`action: press`** + **`fill`** on homepage controls; spike may open a Google results URL built from `query`.
- **💩** Optional **`engine`** later — omit from schema for now.
- **🔷** After settle, return in the requested **output format** (default: a11y markdown).

### `press` (action)

- **🔷** **`action: "press"`** — activate a press-ref from the last a11y dump.
- **🔷** Which control: argument **`press`** (integer) — the ref from content markers / **References**.
- **🔷** Optional argument **`fill`** may be present on the same call (see below).
- **🔷** Order when both: apply **`fill`** first, then press, then settle, then return **`format`**.
- **🔷** Help recommends **`press`** over **`fetch`** when following an on-page result.
- **ℹ️** Press ids come from the a11y tree even if a later reply used `html` / `markdown` format.

### `fill` (argument only — not an action)

- **🔷** **`fill` is never an `action` value.** It is a tool **argument**: a JSON **object** (associative map) of press-ref id → text.
- **🔷** Shape: `{ "1": "typed text", "3": "other field" }` — keys are press-ref ids (strings in JSON); one text per ref.
- **🔷** Used with **`action: "press"`** so one call can type into fields and then activate (e.g. submit).
- **💩** Whether `fill` is allowed on other actions (e.g. dump-only after fill without press) — optional later; v1 = with **`press`**.
- **🔷** No `{n, value}` object array — avoids awkward/`value` naming and duplicate refs.
- **🔷** Apply all fills via accessibility (not JS form scrape).

### `download` (action)

- **🔷** **`action: "download"`** with a **URL** — download that resource through the browser session (cookies / auth of this chat’s WebKit).
- **💩** Save path / progress / approval UX — follow existing Hub download patterns where it fits; spike or later wiring.
- **ℹ️** Distinct from `fetch` (which returns page content in `format`).

### `whereami` (action) — “where am I?”

- **🔷** One-word action **`whereami`** — same idea as Unix **`whoami`**, for the browser: **where am I?**
- **🔷** Return the current page in the requested **output format** — no navigation.
- **🔷** Use after challenges, or when the model needs a fresh dump before choosing `press`.
- **🚫** Not a spike CLI flag (no `--whereami`); tool action + optional script line only.

## Output formats

Shared option on page-bearing actions (`fetch` / `search` / `press` / current-page) and on `oc-test-webkit` (`--format`).

- **🔷** **`a11y`** (default) — markdown projection of the **platform accessibility tree** (see contract below). Primary format for browse / `press` (+ `fill`).
- **🚫** **`a11y` must not** be produced by DOM traversal / `querySelector` / ARIA-attribute scraping in page JS.
- **🔷** **`html`** — page HTML via **JavaScript** `outerHTML` after settle.
- **🔷** **`markdown`** — HTML→Markdown via **`Markdown.HtmlParser`** (same as `web_fetch`).
- **🚫** No `--settle` / `settle_ms` — fixed quiet only (see Snappr host alignment).

## Accessibility markdown contract (`format=a11y`)

Two parts: **content first** (screen-reader-ish flow + inline press pointers), then a **reference index**.

```markdown
# Page
- URL: …
- Title: …

## Content
### Web results
[index.htm – webkitgtk-6.0 Valadoc https://valadoc.org › webkitgtk-6.0](^press:25){160,189}
[How to create a multi-process…](^press:28){160,240}

## References
(^press:25): [index.htm – webkitgtk-6.0 Valadoc…](https://valadoc.org/…)
(^press:28): [How to create…](https://stackoverflow.com/…)
```

- **🔷** **Two-pass layout** — walk registers {@link A11yNode}s with `{x,y}`; emit sorts by ''y'' then ''x''. Same ''y'' shares a Content line (space-separated); new ''y'' starts a new line. Headings always get a **blank line** before `###`.
- **🔷** Before the tree walk, **scroll bottom → top once** so below-fold accessible names fill in — no per-step / per-control jumping.
- **🔷** Label vs value: only strip a trailing Text value from the accessible name for **editable** roles (searchbox/entry/…). Links like “Page 2” keep the full name.
- **🔷** Named **leaf** nodes (no children) that are not pressable still emit as plain text — containers are skipped to avoid repeating child labels.
- **🔷** **Location** — after each press marker, append **`{x,y}`** window coordinates from the a11y Component extents when available (squiggly braces). Model can use that as layout hint; do not invent grids.
- **🔷** **Labels** — collapse internal newlines/whitespace to single spaces so markdown links stay on one line.
- **🔷** Real URLs live in **References** as markdown `[text](url)` only — no extra `[link]` role tag.
- **🔷** Prefer real labels — never emit bare role names like `button` as the only label.
- **🔷** **References** use `(^press:N): [role] label` for non-links; `(^press:N): [label](url)` for links.
- **🔷** `press` takes the same id as the marker (integer).
- **💩** Marker form locked to `[label](^press:N){x,y}` / `(^press:N):`.
- **⏳** Spike: tune Google SERP noise (chrome, duplicate title nodes) from evidence.
- **⏳** Windows WebView2 a11y spike → **5.0.1**. Android → **5.0.2**.
- **🚫** Do **not** port Snappr `resources/search-google.js` SERP extraction.
- **🚫** Do **not** lead with a standalone “Interactive 1…N” catalogue ahead of the content.

## Ownership and cookies

- **🔷** **One browser per chat instance** (not shared across chats).

### Overview draft (`help.md`)

```
BROWSER TOOL
---
General-purpose web browser for THIS chat session. Search the web, open pages,
fill forms, follow links via press-refs from the page dump, download files.

NOT a one-shot HTTP API. Prefer the accessibility dump + press/fill over
shell/curl browsing.

MUST — HELP WHEN MISSING
  If this overview is NOT already in your current context (including after a
  conversation summary), call {"action": "help"} first.
  Before using an action whose topic page is not in context, call
  {"action": "help", "topic": "<action>"} (e.g. topic "press", "search").
  A vague recollection is not enough.

ACTIONS (call help with topic for details)
  help      — this index, or topic page when "topic" is set
  fetch     — open a URL
  search    — Google search (v1)
  press     — activate a press-ref; optional fill object on the same call
  download  — download a URL through this browser session
  whereami  — where am I? current page (no navigation)

DEFAULT PAGE DUMP (format "a11y")
  # Page → URL / Title
  ## Content — structured text; pressables as [label](^press:N)
  ## References — (^press:N): role, label; links as [text](url)

  Prefer press on a ref from the last dump over fetch with a hand-copied URL
  when the control is already listed. fill is an ARGUMENT on press, never an
  action. For format "html" / "markdown", call {"action": "help", "topic": "format"}.

TYPICAL FLOW
  help → search or fetch → read Content + References →
  press (+ fill) → read dump → repeat → download if needed

OTHER TOOLS
  google_search / web_fetch may still be registered. Prefer this tool for
  multi-step browsing and press/fill. Do not use them to replace press/fill
  on this tool’s dumps.
```

### Topic drafts (summaries — full pages in resource files)

**`topic: fetch`**
- Args: `url` (required), `format` (optional, default a11y)
- When: known URL not already a press-ref
- Example: `{"action": "fetch", "url": "https://example.com/docs"}`
- Returns page dump; then use press/fill from References

**`topic: search`**
- Args: `query` (required search terms), `format`
- Opens Google via accessibility (fill + submit under the hood) when needed
- Then press a result ref — do not invent SERP scraping
- Example: `{"action": "search", "query": "Vala WebKitGTK"}`

**`topic: press`** (include **fill**)
- Args: `press` (required press id), `fill` (optional object), `format`
- `fill`: `{"1": "…", "3": "…"}` — NOT an action; only with press
- Order: fill entries → press → return dump
- Prefer over fetch when the link/control is in the last References
- Examples: press only; fill+press for forms/search boxes

**`topic: download`**
- Args: `url` (required)
- Saves through this chat’s browser session (distinct from fetch content)
- Prefer over run_command/wget for that URL
- Example: `{"action": "download", "url": "https://example.com/a.pdf"}`

**`topic: whereami`**
- “Where am I?” — current page, no navigation
- Args: `format` optional
- Use when unsure of the current page or after the UI was shown for a challenge
- Example: `{"action": "whereami"}`

**`topic: format`** (optional separate page)
- `a11y` (default) — Content + References; needed for press/fill
- `html` — page HTML; `markdown` — HTML→MD
- Press/fill need a prior a11y dump with References

**🚫 Model help must not document:** settle millisecond defaults, Cloudflare detection headers, cookie harvest APIs, Snappr internals.
