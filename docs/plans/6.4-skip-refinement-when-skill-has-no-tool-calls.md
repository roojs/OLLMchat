# 6.4 Skip refinement when skill has no tool calls — **do not implement**

**Conclusion:** We must **not** skip refinement for skills that have no `tools` in their YAML header. Refinement is still required to set up **References** (prior task output link and links extracted from that output's Detail). Skipping would break tasks that take their input from a previous task's result.

**Context:** Refinement is per-step (refine step 1 → execute step 1, then refine step 2 → execute step 2). This plan considered skipping refinement when the skill has no tools; that optimisation is rejected for the reasons below.

---

## 1. What the skill definitions say about Refinement

Skills that take input from a **previous task's result** (e.g. analyze_docs, analyze_code, research_pages) have no tool calls but their **Refinement** section still has work to do.

### analyze_docs.md — #### Refinement

- No tool calls. **Ensure the task's References include the prior analyze_docsbase task output** (refiner adds a link to that task's results so Precursor contains Result summary + Analyze docsbase results and any resolved reference content).

### analyze_code.md — #### Refinement

- No tool calls. **Ensure the task's References include the prior analyze_codebase task output** (refiner adds a link to that task's results so Precursor contains Result summary + Analyze codebase results and any resolved reference content).

### research_pages.md — #### Refinement

- No tool calls. **Ensure the task's References include the prior research_topic task output** (refiner adds a link to that task's results so Precursor contains Result summary + Detail).

So even when there are no tool calls, the refiner must **add the link to the prior task's output** to this task's References so the Runner can inject that output into the precursor when the executor runs.

---

## 2. What the refinement prompt says about references

From **task_refinement.md**:

### References from prior task output (Detail links)

When this task references a **prior task's output** (e.g. `[Research 1 Results](#research-1-results)`), the task reference contents include that output (Result summary + Detail). The **Detail** section often contains markdown links (URLs, file paths, or file sections with AST references) to sources. **Extract those links from the Detail** and **add them to this task's References** in your refined output. Use the same link format: `[Title](url)`, `[Title](/absolute/path)`, or `[Title](/path/to/file#ast_path)` for a code symbol. The runner will then inject both the prior task's output and the resolved content of those links into the precursor, so the executor receives the Detail together with the content of the links mentioned in it.

### Output format — ## Task

- **References** … When this task references a prior task's output, **include in References any links extracted from that output's Detail section** (see "References from prior task output" above).

So refinement is responsible for:

1. **Tool calls** (when the skill has tools) — ## Tool Calls section.
2. **References** — ensuring the task's References include (a) the link to the prior task's output so the precursor gets it, and (b) links extracted from that output's Detail so the executor gets the full context.

If we skip refinement for “no tools” skills, we never run step 2. Tasks that depend on a previous task's result would not get the prior output or the extracted links in their References, and the executor would not receive that content.

---

## 3. Rejected: skip refinement when `tools` is empty

- **Idea:** When the skill has no `tools` in its YAML header (or `tools` empty), skip the refine stage and mark the task as refined, to save an LLM round-trip.
- **Why not:** Refinement is still required for reference setup. Skills like analyze_docs, analyze_code, research_pages have no tools but depend on the refiner to add the prior-task output link and (per the prompt) to extract links from the prior task's Detail into this task's References. Skipping refinement would break that flow.
- **Action:** Do **not** skip refinement. Keep running refinement for every task.

---

## 4. Implemented: hide tool-call sections in refine prompt when skill has no tools

- **Idea:** Use the fact that the skill has no `tools` in its YAML header to **hide** the tool-call-related parts of the refinement template from the LLM. Refinement still runs (so references are set up), but the prompt is shorter and the model is not asked to produce ## Tool Calls when the skill does not use any.
- **Implementation:** In `task_refinement.md`, wrap the tool-call-only content in `{toolcall1/start}` … `{toolcall1/end}`, `{toolcall2/start}` … `{toolcall2/end}`, etc. (four blocks: derive paragraph, Tools definition bullet, “How to run tools” section, Output/Example for ## Tool Calls). In `Details.refinement_prompt()`, pass `toolcall1` … `toolcall4` with value `"DEFAULT"` when `definition.has_refinement_tool_calls()` is true (keep sections), or `""` when false (remove those sections). `Definition.has_refinement_tool_calls()` returns true when `header.has_key("tools") && header.get("tools").strip() != ""`.
- **Result:** Skills with no tools still get refinement (for references), but the refiner no longer sees instructions about producing ## Tool Calls, which reduces confusion and prompt length.

---

## 5. Future: split refine vs execution system/prompts

- **Observation:** Having both refine and execution in one flow can confuse the LLM and inflate prompt length. The skill document is used for both stages.
- **Idea:** Use a **split system** for “refine stage” vs “execution stage”: separate system (and possibly user) content for each. Refine prompt would contain only what the refiner needs (task + references + optional tool-calls); execution prompt would contain only what the executor needs (precursor, skill execution instructions). That would reduce the length of each prompt and make the model’s role clearer in each step.
- **Status:** Not implemented; captured here for a future plan.
