## ℹ️ Overview

Phase 2 integration of OpenAI-compatible **generate** for **all libraries** that use it. Parent plan: **1.24** (OpenAI API chat and embed compatibility). Ollama has `/api/generate`; OpenAI has `/v1/completions` (different shape, no conversation). This plan covers every generate usage across the codebase.

## 🔷 Scope

**Method:** Generate only.

**Libraries:** libollmchat (Client) — currently the only consumer. `Client.generate()` uses `Call.Generate` (Ollama `/api/generate`), returns `Response.Generate`.

**Refer to plan 1.24 for:** rationale, API differences.

### ℹ️ Naming: Generate = Completions (not Chat Completions)

| Our name      | Ollama endpoint   | OpenAI endpoint           | Class              |
|---------------|--------------------|---------------------------|--------------------|
| **Generate**  | `/api/generate`    | **`/v1/completions`**     | **`Call.Completions`**  |
| Chat          | `/api/chat`       | `/v1/chat/completions`    | `Call.ChatCompletions`  |

So we add a class **`Call.Completions`** for the **text completions** API (`/v1/completions`): single prompt in, completion text out. That is **not** the same as `Call.ChatCompletions`, which is for the **chat** API (`/v1/chat/completions`): messages in, assistant message out.

## ℹ️ Request and response differences (generate): Ollama vs OpenAI

### ℹ️ Request

| Aspect | Ollama classic (`/api/generate`) | OpenAI-style (`/v1/completions`) |
|--------|----------------------------------|----------------------------------|
| Endpoint | `POST /api/generate` | `POST /v1/completions` |
| Input | `prompt` (string), optional `system`, `suffix` | `prompt` (string or array of strings/tokens) |
| Model | `model` | `model` |
| Generation | Nested `options` (e.g. `num_predict`, `temperature`) | Flat: `max_tokens`, `temperature`, `top_p`, **`n`** (see below), `stop`, etc. |
| Extra (Ollama) | `images[]`, `format`, `stream`, `think`, `raw`, `keep_alive` | — |
| Extra (OpenAI) | — | `suffix`, `best_of`, `logprobs`, `echo`, `presence_penalty`, `frequency_penalty`, `logit_bias`, `user` |

**Ollama request body example:**

```json
{
  "model": "llama3.2",
  "prompt": "Complete this: The capital of France is",
  "system": "You are helpful.",
  "stream": false,
  "options": { "num_predict": 64, "temperature": 0.7 }
}
```

**OpenAI-style request body example:**

```json
{
  "model": "llama3.2",
  "prompt": "Complete this: The capital of France is",
  "max_tokens": 64,
  "temperature": 0.7,
  "stop": ["\n"],
  "stream": false
}
```

### ℹ️ Response

| Aspect | Ollama classic | OpenAI-style |
|--------|----------------|--------------|
| Generated text | `response` (string, accumulated if streaming) | `choices[0].text` (or `choices[].text` when `n` > 1) |
| Per-choice index | none | `choices[].index` |
| Finish | `done`, `done_reason` | `choices[].finish_reason` |
| Token counts | `prompt_eval_count`, `eval_count` | `usage.prompt_tokens`, `usage.completion_tokens`, `usage.total_tokens` |
| Timing | `total_duration`, `load_duration`, `eval_duration` | none |
| Thinking | `thinking` (Ollama extended) | none |
| Top-level | `model`, `response`, `done`, timing, eval counts | `id`, `object`, `created`, `model`, `choices`, `usage` |

**Ollama response (non-streaming) example:**

```json
{
  "model": "llama3.2",
  "created_at": "2025-01-01T00:00:00.000Z",
  "response": " Paris.",
  "done": true,
  "done_reason": "stop",
  "total_duration": 12345678,
  "prompt_eval_count": 12,
  "eval_count": 3
}
```

**OpenAI-style response example:**

```json
{
  "id": "cmpl-xxx",
  "object": "text_completion",
  "created": 1704067200,
  "model": "llama3.2",
  "choices": [
    { "text": " Paris.", "index": 0, "finish_reason": "stop", "logprobs": null }
  ],
  "usage": { "prompt_tokens": 12, "completion_tokens": 3, "total_tokens": 15 }
}
```

**Why `choices` is an array:** The request supports **`n`** (number of completions to generate). With `n: 1` (default) you get one choice; with `n: 3` the server returns three different sampled completions for the same prompt, so `choices` has three elements. We only need a single completion for our generate flow, so we use `choices[0]` and do not add `n` to the request (default is 1). If we ever need multiple alternatives, we would add `n` to `Call.Completions` and expose multiple results (e.g. from `Response.Generate` or a list).

## ℹ️ Options, required prompt, and how Completions differs from Generate

### ℹ️ Options in Completions

- **Generate (Ollama)** sends a nested **`options`** object; the API expects e.g. `options: { num_predict, temperature, top_p, stop, ... }`. So `Call.Generate` has `Call.Options options` and serializes it as a single nested object (with hyphen→underscore keys).
- **Completions (OpenAI)** has no nested options; the API expects **flat** top-level params: `max_tokens`, `temperature`, `top_p`, `stop`, `presence_penalty`, `frequency_penalty`, etc.

We still want **options support on Completions**: use the same **`Call.Options`** type, defaulting to a standard options instance. If the caller sets **`options`** (e.g. from a model or config), those values are used as fallbacks when building the request. **On serialization:** for each param that exists both as a top-level Completions property and in Options, **if the top-level property is set, send that; otherwise if options has a value, send from options** (with name mapping for the API). We do **not** send a key `"options"` in the JSON (OpenAI has no such key); we only emit flat top-level fields.

**Mapping Options → Completions request:**

| Call.Options (options.) | Completions top-level / API key | Note |
|------------------------|----------------------------------|------|
| `num_predict`          | `max_tokens`                    | Use top-level if max_tokens >= 0, else options.num_predict |
| `temperature`          | `temperature`                   | Use top-level if >= 0, else options.temperature |
| `top_p`                | `top_p`                         | Use top-level if >= 0, else options.top_p |
| `stop`                 | `stop`                          | Use top-level if non-empty, else options.stop |
| `presence_penalty`     | `presence_penalty`              | Completions can add top-level or use only from options |
| `frequency_penalty`    | `frequency_penalty`             | Completions can add top-level or use only from options |

So: add **`public Call.Options options { get; set; default = new Call.Options(); }`** to Completions. In `serialize_property`, for `max_tokens` / `temperature` / `top_p` / `stop` (and any penalty): if the corresponding top-level property is set, serialize it; else if `options` has the corresponding value (`has_values()` and the specific field), serialize that value with the **API key name** (e.g. `max_tokens` from `options.num_predict`). Do **not** serialize the property name `"options"` (return null for that case).

### ℹ️ Required prompt

- **OpenAI completions:** **`prompt`** is required. If we send a request without `prompt`, the API will error.
- **Generate (Ollama):** In our code we **omit** `prompt` when it is empty (`serialize_property` returns null). So we don’t validate prompt non-empty; the server may return an error if it’s missing.

**Completions:** We should **require prompt** before sending: in `exec_completions()` (or in `serialize_property` when building the body), if `prompt == ""` then throw `OllmError.INVALID_ARGUMENT("Prompt is required for completions")`. That matches the API and avoids sending an invalid request. **Generate** keeps current behaviour: omit prompt if empty (caller’s responsibility / server-side error if required).

### ℹ️ How this differs from Generate

| Aspect | Generate (Ollama) | Completions (OpenAI) |
|--------|-------------------|----------------------|
| **Options** | Single nested `options` object; serialize `options` as JSON object when `options.has_values()`. | No `options` key. Flat params only; `options` is a fallback: for each param, use top-level if set, else value from `options` (with mapping). |
| **Prompt** | Optional in serialization (omit if empty). | Required: validate non-empty and throw before send if empty. |
| **Param names** | Options keys serialized with hyphen→underscore (e.g. `num_predict`). | Flat keys: `max_tokens`, `temperature`, `top_p`, `stop`, etc. Options’ `num_predict` → request `max_tokens`. |
| **Endpoint** | `POST /api/generate` | `POST /v1/completions` |

## ✅ Approach: reuse existing types, flatten v1 into Response.Generate (implemented)

- **Reuse the Ollama response object:** We keep a single response type, **`Response.Generate`**. The v1/completions response is parsed and mapped into it (same pattern as embed: one `Response.Embed`, v1 response mapped in via `read_data` / `read_usage`). No new response type; no nested `Usage` object.
- **Call namespace:** Add **`Call.Completions`** for the OpenAI **completions** endpoint (`POST /v1/completions`), i.e. the generate equivalent — not chat. Same namespace as `Call.Generate`; callers use `Call.Completions` when using the v1 completions API.
- **Client:** Add **`completions(model, prompt, options?, cancellable?)`** — same shape as **`generate()`** (model, prompt, options, cancellable) but **no system** (v1/completions has no system prompt). Builds `Call.Completions` inside and returns `Response.Generate`.
- **Response shape — flatten usage and choices (like Embed):** Do **not** add a real `Usage` object to `Response.Generate`. Mirror **Embed**: add flat properties for v1 usage. **No extra methods on Response.Generate** — all mapping from v1 JSON into `Response.Generate` is done in the **JSON parsing code** (in `Call.Completions` when it handles the response), not on the response class.
  - **Embed pattern:** `Response.Embed` has `prompt_tokens`, `total_tokens` (flat); the call’s parse path fills them from the v1 `usage` object. No `Response.Usage` on Embed.
  - **Generate — extend with choices:** Add **`choices`** to **`Response.Generate`** as **`Gee.ArrayList<string>`** (list of completion text strings). In **Call.Completions**’ parsing code: read v1 `choices[].text` into `choices`, set **`response`** = first choice (kludge so existing code that reads `response` still works). Ollama path sets only `response` (no choices). So: **`response`** = first string; **`choices`** = full list when we have multiple (v1 with `n` ≥ 1).
  - **Generate — usage:** Reuse existing **`prompt_eval_count`** and **`eval_count`** (they are token counts). When deserializing v1 `usage`, set `prompt_eval_count` from `usage.prompt_tokens` and `eval_count` from `usage.completion_tokens`. No new `prompt_tokens` / `completion_tokens` / `total_tokens` properties.

Full contents:

```vala

namespace OLLMchat.Call
{
	/**
	 * OpenAI-compatible completions API call (prompt → text).
	 * Uses v1/completions; same URL pattern as Call.Models.
	 * Request: model, prompt (required), optional max_tokens, temperature, top_p, n, stop, presence_penalty, frequency_penalty.
	 * Options: same Call.Options as Generate; used as fallback when top-level param not set (options.num_predict → max_tokens, etc.).
	 * Returns Response.Generate; v1 JSON is parsed via deserialize_property.
	 */
	public class Completions : Base
	{
		public string model { get; set; }
		public string prompt { get; set; default = ""; }
		public bool stream { get; set; default = false; }
		public int max_tokens { get; set; default = -1; }
		public double temperature { get; set; default = -1.0; }
		public double top_p { get; set; default = -1.0; }
		public int n { get; set; default = 1; }
		public string stop { get; set; default = ""; }
		public double presence_penalty { get; set; default = -1.0; }
		public double frequency_penalty { get; set; default = -1.0; }
		public Call.Options options { get; set; default = new Call.Options(); }

		public Completions(Settings.Connection connection, string model)
		{
			base(connection);
			if (model == "") {
				throw new OllmError.INVALID_ARGUMENT("Model is required");
			}
			this.model = model;
			this.is_openai = true;
			this.url_endpoint = "v1/completions";
			this.http_method = "POST";
		}

		public override Json.Node serialize_property(string property_name, Value value, ParamSpec pspec)
		{
			switch (property_name) {
				case "options":
					// Do not serialize "options" as a key; we flatten options into top-level params below.
					return null;
				case "prompt":
					return default_serialize_property(property_name, value, pspec);
				case "max_tokens": {
					var v = this.max_tokens >= 0 ? this.max_tokens : this.options.num_predict;
					if (v < 0) {
						return null;
					}
					var node = new Json.Node(Json.NodeType.VALUE);
					node.set_int(v);
					return node;
				}
				case "temperature": {
					var v = this.temperature >= 0 ? this.temperature : this.options.temperature;
					if (v < 0) {
						return null;
					}
					var node = new Json.Node(Json.NodeType.VALUE);
					node.set_double(v);
					return node;
				}
				case "top_p": {
					var v = this.top_p >= 0 ? this.top_p : this.options.top_p;
					if (v < 0) {
						return null;
					}
					var node = new Json.Node(Json.NodeType.VALUE);
					node.set_double(v);
					return node;
				}
				case "stop": {
					var s = this.stop != "" ? this.stop : this.options.stop;
					if (s == "") {
						return null;
					}
					var node = new Json.Node(Json.NodeType.VALUE);
					node.set_string(s);
					return node;
				}
				case "presence_penalty": {
					var v = this.presence_penalty >= 0 ? this.presence_penalty : this.options.presence_penalty;
					if (v < 0) {
						return null;
					}
					var node = new Json.Node(Json.NodeType.VALUE);
					node.set_double(v);
					return node;
				}
				case "frequency_penalty": {
					var v = this.frequency_penalty >= 0 ? this.frequency_penalty : this.options.frequency_penalty;
					if (v < 0) {
						return null;
					}
					var node = new Json.Node(Json.NodeType.VALUE);
					node.set_double(v);
					return node;
				}
				case "n":
					if (this.n <= 1) {
						return null;
					}
					return default_serialize_property(property_name, value, pspec);
				default:
					return base.serialize_property(property_name, value, pspec);
			}
		}

		/**
		 * Execute request; uses Json deserialize (gobject_from_data). Response.Generate
		 * deserialize_property handles v1 shape (choices[], usage) and maps into flat properties.
		 * Prompt is required for the API; we throw if empty.
		 */
		public async Response.Generate exec_completions() throws Error
		{
			if (this.prompt == "") {
				throw new OllmError.INVALID_ARGUMENT("Prompt is required for completions");
			}
			var bytes = yield this.send_request(true);
			var root = this.parse_response(bytes);
			if (root.get_node_type() != Json.NodeType.OBJECT) {
				throw new OllmError.FAILED("Invalid JSON response");
			}
			var generator = new Json.Generator();
			generator.set_root(root);
			var json_str = generator.to_data(null);
			var gen = Json.gobject_from_data(
				typeof(Response.Generate), json_str, -1) as Response.Generate;
			if (gen == null) {
				throw new OllmError.FAILED("Failed to deserialize completions response");
			}
			return gen;
		}
	}
}
```

**2c. `deserialize_property`** — Add an override so v1 JSON is parsed via the normal Json deserialize path. Handle v1 keys `choices`, `usage`, and `created`; delegate the rest to `default_deserialize_property`. This is how parsing works: the call uses `Json.gobject_from_data(typeof(Response.Generate), json_str, -1)` and the framework calls `deserialize_property` for each JSON key.

```vala
		public override bool deserialize_property(
			string property_name, out Value value, ParamSpec pspec, Json.Node property_node)
		{
			switch (property_name) {
				case "choices": {
					// v1: array of { "text", "index", "finish_reason" } — we only use text; ignore finish_reason for now
					this.choices.clear();
					var array = property_node.get_array();
					for (var i = 0; i < array.get_length(); i++) {
						var choice_obj = array.get_object_element(i);
						if (choice_obj.has_member("text")) {
							var t = choice_obj.get_string_member("text");
							this.choices.add(t != null ? t : "");
						}
					}
					this.response = this.choices.size > 0 ? this.choices[0] : "";
					value = Value(typeof(Gee.ArrayList));
					value.set_object(this.choices);
					return true;
				}
				case "usage": {
					// v1: reuse prompt_eval_count / eval_count (same as tokens)
					var usage = property_node.get_object();
					if (usage.has_member("prompt_tokens")) {
						this.prompt_eval_count = (int)usage.get_int_member("prompt_tokens");
					}
					if (usage.has_member("completion_tokens")) {
						this.eval_count = (int)usage.get_int_member("completion_tokens");
					}
					value = Value(typeof(int));
					value.set_int(0);
					return true;
				}
				case "created": {
					// v1: unix timestamp (int) → created_at string
					this.created_at = property_node.get_int().to_string();
					value = Value(typeof(string));
					value.set_string(this.created_at);
					return true;
				}
				default:
					return default_deserialize_property(property_name, out value, pspec, property_node);
			}
		}
```

### ✅ 3. Changes to `libollmchat/Client.vala`

Add the following method after `generate(...)` (before the closing `}` of the class). Signature matches `generate()` (model, prompt, options, cancellable) but **no system** — v1/completions has no system prompt.

```vala
		/**
		 * Runs a completions call (OpenAI v1/completions) and returns the result as Response.Generate.
		 * Same argument shape as generate() but without system prompt (API has no system).
		 *
		 * @param model The model name to use
		 * @param prompt The text prompt (required)
		 * @param options Optional Call.Options; used as fallback for max_tokens, temperature, etc.
		 * @param cancellable Optional cancellable
		 * @return Response.Generate with choices list and response = first choice; usage flattened.
		 */
		public async Response.Generate completions(
			string model,
			string prompt,
			Call.Options? options = null,
			GLib.Cancellable? cancellable = null
		) throws Error
		{
			if (model == "") {
				throw new OllmError.INVALID_ARGUMENT("Client.completions() requires a model parameter.");
			}
			var call = new Call.Completions(this.connection, model) {
				cancellable = cancellable,
				prompt = prompt,
				options = options == null ? new Call.Options() : options
			};
			return yield call.exec_completions();
		}
```

## ℹ️ Notes

- Generate is a small surface; only libollmchat uses it. `Client.generate(model, prompt, system, options, cancellable)` uses `Call.Generate` (Ollama); new path is `Client.completions(model, prompt, options, cancellable)` (no system) → builds `Call.Completions`, returns `Response.Generate`.
- Reusing `Response.Generate` and flattening usage/choices keeps the same contract as embed and avoids new types.
