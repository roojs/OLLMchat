## Goal

[`2.10.4.5`](2.10.4.5-DONE-folder.md) and sibling plans **remove** daemon-only code from client types. That leaves:

1. **Caller reshape** — UI and tools still read `project_files`, `child_map`, `folder_map` on shipping paths.
2. **Folder fallout** — client `Folder` no longer has `project_files`; V2 code still references it in places.
3. **Meson cutover** — swap `libocfiles/*.vala` → `libocfiles/V2/*.vala` in the app build.

This plan owns **wiring those together** — decide first, then implement. **No ad-hoc stubs** in V2 PM (e.g. reusing `file_cache` as a project index).

### T8 — `notify_config_updated()` (detail)

**Problem:** Config persistence is scattered — settings UI knows to call `config.changed()`, but there is no standard method on `ApplicationInterface` for “config was saved / reloaded; react.”

**Proposed (option A — minimal):**

```vala
// ApplicationInterface — default implementation
public void notify_config_updated() {
    this.config.changed();
}
```

**Call sites after cutover / tidy-up:**

| Caller | When |
| ------ | ---- |
| `ollmapp/SettingsDialog/MainDialog` | Replace direct `this.app.config.changed()` with `this.app.notify_config_updated()` |
| `ollmfilesd/Application` | After any future config reload path (e.g. SIGHUP or RPC) — same method as GUI app |
| Tests / CLI | Same entry point if they mutate and save config |

**Listeners:** unchanged for option A — still `config.changed.connect(...)`. Option B adds `app.config_updated` for components that should not depend on `Config2` directly.

**Out of scope for T8:** daemon RPC to push config to `ollmfilesd`; file-indexer config keys — those stay on `Config2` / tool registries until a separate plan says otherwise.
