## What is going on (plain English)

Today the chat app keeps a **full in-memory picture of each project’s files** inside the GUI process. Folders, files, maps of paths to file objects — all of that lives in `libocfiles` code linked directly into `ollmapp`.

The long-term design moves that work into a **background daemon** called `ollmfilesd`. The daemon owns scanning disks, building the project tree, and serving file operations. The app becomes a **thin client**: it asks the daemon over RPC instead of walking directories itself.

Much of the new code already exists:

- `libocfiles/V2/` — rewritten client-side types that talk RPC instead of doing disk I/O.
- `ollmfilesd/` — the daemon process with its own copies of server-side types.
- `libocfiles/RpcClient.vala` — shared transport to talk to the daemon.

But the app **still builds and runs the old shipping sources**. The V2 tree is in the repo; it is not what Meson compiles for `ollmapp` yet.

Earlier plans (especially folder and file) **removed daemon responsibilities from the client types**. That was correct, but it left two problems this plan owns:

1. **The app still calls the old APIs.** Many screens and tools still reach for `project_files`, `child_map`, and `folder_map` as if the tree were local.
2. **V2 code still mentions removed fields.** For example, V2 ProjectManager still references `Folder.project_files`, which the client Folder no longer has.

This plan is the **glue work**: decide how to bridge each gap, update callers, flip the build to V2, and prove the app works with the daemon.

## What we need to do (order of work)

Work in four phases. Do not skip the decision phase and jump straight to coding.

### Phase A — Decide the open questions

Every item in **Open decisions** below must have a chosen option before implementation starts on that item. Document the choice in this plan (or a linked note). Do not patch V2 with “probably right” behaviour.

### Phase B — Reshape callers

Change UI modules and tools so they **fetch data from the daemon** (or from a small local cache of objects the user has actually touched) instead of reading a populated `project_files` tree on the active project.

See **Parts of the app that must change** below.

### Phase C — Meson cutover and compile fixes

Point the app build at `libocfiles/V2/*.vala` (plus shared RPC sources as agreed in the layout plan). Fix whatever fails to compile after Phase B decisions — especially fallout in V2 ProjectManager and shared types like ProjectList.

Ensure `ollmfilesd` is running (spawned or already on disk) via `RpcClientBoot` or equivalent.

Do **not** edit the shipping `libocfiles/Folder.vala` (and siblings) in place; V2 replaces them.

### Phase D — End-to-end test

Run through the scenarios in **How we know it works** and fix regressions before marking this plan done.

## Open decisions

Rule for all items: **write down the gap, pick an option, then implement.** Do not conflate `file_cache` (files the user has opened or touched) with `project_files` (the full project membership index the daemon now owns).

### V2 ProjectManager and shared client code

**Removed `Folder.project_files`, but V2 ProjectManager still uses it**

- Where: `libocfiles/V2/ProjectManager.vala`
- Today in shipping: callers use `child_map`, `update_from`, `get_active_file` on a local tree.
- Pick one:
  - Temporarily add an empty `project_files` shell on client Folder until callers are reshaped.
  - Edit V2 ProjectManager now (ties to the next three items).
  - Defer PM edits until Meson cutover when caller reshape lands.

**Looking up a file by path (`get_file_from_active_project`)**

- Where: V2 ProjectManager.
- Today: `project_files.child_map.get(path)`.
- Direction: per-path **RPC** (file register / lookup on the wire). Not `file_cache`.

**After registering a new file (`convert_fake_file_to_real`)**

- Where: V2 ProjectManager.
- Today: updates the local list and emits `new_file_added`.
- Direction: drop the client-side list update; refresh the file dropdown via RPC or an event from [`2.10.4.6`](done/2.10.4.6-DONE-file.md). Keep `file_cache.set` and buffer handling only if still needed for open editors.

**Restoring which file was open (`restore_active_state`)**

- Where: V2 ProjectManager.
- Today: `project_files.get_active_file()`.
- Direction: handled in [`2.10.4.7`](2.10.4.7-active-state-client-only.md) — store paths in agent config / window state, not in a daemon-owned active-file pointer on the client tree.

**Finding a folder anywhere in a project (`ProjectList.get_folder_in_any_project`)**

- Where: `ProjectList.vala` (shared between old and new).
- Today: walks `p.project_files.folder_map`.
- Direction: RPC `get_folder_at_path` at cutover. Shipping code can stay until then.

**Empty `children` stub on client Folder**

- Where: `libocfiles/V2/Folder.vala`.
- Today: empty `FolderFiles` for type compatibility.
- Pick one: keep for compat, or remove if nothing in the tree UI still needs it.

**What `file_cache` on V2 ProjectManager is for**

- Document clearly: path → materialized `FileBase` for **objects the client has touched**, not project membership, not session restore.

**Config change notification (`notify_config_updated`)**

- Problem: after settings save, callers invoke `app.config.changed()` directly. There is no shared “config was saved or reloaded” entry point on `ApplicationInterface`, so `ollmapp` and `ollmfilesd` do not follow the same pattern.
- Pick one:
  - **A (minimal):** add `notify_config_updated()` on `ApplicationInterface`; default body calls `this.config.changed()`. Settings dialog and future daemon reload paths call that instead of touching `Config2` directly.
  - **B:** same as A, plus a `config_updated` signal on the interface for listeners that should not depend on `Config2`.
  - **C:** leave as-is; document that apps must call `config.changed()` after persisting (no interface method).
- Call sites to update if A or B: `ollmapp/SettingsDialog/MainDialog.vala`, any future `ollmfilesd` reload path, tests/CLI that save config.
- Out of scope: pushing config over RPC to the daemon; indexer config keys stay on `Config2` until a separate plan says otherwise.

**Suggested order for the client-side items**

1. Compile strategy for the `project_files` removal (first bullet above).
2. Active-file restore direction (align with 2.10.4.7 before reimplementing restore).
3. Path lookup, fake-to-real conversion, and folder lookup — together with caller reshape.
4. `children` stub and `file_cache` documentation — small cleanup.
5. Config notification — can land before or with Meson cutover.

## Parts of the app that must change

The daemon holds the authoritative project graph. Client code must stop assuming `active_project.project_files` is filled in locally.

**File dropdown**

- Today: `set_item_model(project.project_files)`.
- Need: the widget owns its own `ListModel`; populate it via RPC when opened or refreshed (exact wire TBD — see [`2.10.4.1`](2.10.4.1-ollmfilesd-rpc-api.md)).

**Approvals**

- Today: reads `project_files.review_files`.
- Need: RPC or query for pending approvals.

**Tools (ReadFile, WriteFile, EditMode, link handling)**

- Today: `get_file_from_active_project` → `child_map`; also uses `folder_map`.
- Need: per-path RPC to resolve files.

**AgentFactory recent list**

- Today: `get_recent_list()` from local state.
- Need: RPC (deferred in 2.10.4.1 wire doc).

**Codebase search**

- Today: `project_files.get_ids()`.
- Need: daemon search RPC.

**SourceView**

- Today: file dropdown plus `get_active_file()` on project switch.
- Need: dropdown changes above; active file handling moves to 2.10.4.7.

**Project description**

- Client RPC side is done; daemon relay still pending.

Incremental UI updates should use events from 2.10.4.6, not the cross-window reload machinery in 2.10.4.8 (that plan is for after cutover).

## Meson cutover (what “flip the build” means)

Per the layout plan:

- Change app `libocfiles` source lists from `libocfiles/*.vala` to `libocfiles/V2/*.vala`, plus shared `Rpc/`, `RpcClient`, and related files as already agreed.
- Resolve compile fallout in V2 ProjectManager and shared types using the decisions above.
- Run or spawn `ollmfilesd` on the RPC socket path the client expects.

Until this step, it is acceptable that V2 does not compile in the main app target — gaps are tracked in **Open decisions**, not patched silently.

## How we know it works

1. Open a project → `activate_project` RPC → daemon scans disk; client `Folder` does not call `read_dir` locally.
2. File dropdown lists files from RPC, not from an empty local `project_files`.
3. Tools resolve paths via RPC without a local `child_map`.
4. Save and reload the active file through `File.read` / `File.write` RPC.
5. Session restore brings back project and file paths (full active-file story when 2.10.4.7 is ready).
