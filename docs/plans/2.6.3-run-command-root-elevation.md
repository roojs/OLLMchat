# 2.6.3 Run Command Root Elevation

**Status:** proposed

**Checklist:** [docs/guide-to-writing-plans.md](guide-to-writing-plans.md) — Checklist for plans. Proposed Vala must follow [docs/coding-standards.md](coding-standards.md).

**Parent:** [2.6 DONE run terminal command tool](done/2.6-DONE-run-terminal-command-tool.md) · related: [2.22.1.4 DONE seccomp write paths](done/2.22.1.4-DONE-seccomp-write-paths-and-followups.md)

## Purpose

- **🔷** Give the agent a **declared, explicit** way to run a shell command with **root privileges** when the user truly needs it (e.g. `apt install`, system config), instead of embedding `sudo` in the command string (which fails or is unsafe in the sandboxed `run_command` path).
- **🔷** **Every** root run must pass through the existing **ChatPermission** flow first.
  - No cache bypass.
  - No **Allow Always** shortcut for this case (same pattern as `network` and non-default `allow_write`).
- **🔷** After the user approves in ChatPermission, **elevate via a system GUI password dialog** (PolicyKit / `pkexec`), not by running `sudo` inside our subprocess with no UI.
- **⏳** Implementation backlog — see **Phase tasks** below.
- **ℹ️** Current tool: `liboctools/RunCommand/` (`Tool.vala`, `Request.vala`). Permission UI: `libollmchatgtk/ChatPermission.vala`, `libollmchatgtk/Tools/Permission.vala`.

## Current behaviour

- **ℹ️** `run_command` runs as the normal user.
  - With bubblewrap: sandboxed via `OLLMfiles.Sandbox.Bubble` (`libocfiles/Sandbox/Bubble.vala`).
  - Without bubblewrap: `GLib.Subprocess` via `/bin/sh -c` (`Request.execute_with_subprocess()`).
- **ℹ️** Permission prompts today cover **network**, **extra `allow_write` roots**, and **every command when bwrap is unavailable**.
  - Default sandboxed runs with normal writes skip execute permission (`build_perm_question()` returns false when `Bubble.can_wrap()` and default `allow_write`).
- **ℹ️** There is **no** root-elevation path.
  - Agent commands like `sudo apt-get install …` run inside `sh -c` as the user.
  - `sudo` typically fails (no TTY, no password prompt wired to our UI).
- **ℹ️** GTK app uses `OLLMchatGtk.Tools.Permission` → `ChatPermission.request()` (`ollmapp/Window.vala` wires `history_manager.permission_provider`).

## Proposed behaviour

### Tool parameter

- **🔷** Add boolean tool argument **`run_as_root`** (default **`false`**).
  - Document in `Tool.description` and `Tool.parameter_description` alongside `network` and `allow_write`.
  - Tell the model: set **`run_as_root: true`** when the command **requires root**.
  - Tell the model: **do not** prefix the command with `sudo`, `pkexec`, `su`, etc. — the tool performs elevation when this flag is set.

### Permission (mandatory, no bypass)

- **🔷** When **`run_as_root`** is **`true`**:
  - Always call `build_perm_question()` → **`true`** (permission required) **before** execution.
  - Set **`one_time_only = true`** (hide **Allow Always** / **Deny Always** — same as `network`).
  - Set **`permission_target_path`** to a **unique key** each invocation (e.g. `run_as_root#` + `GLib.get_real_time()`), so stored **Always** permissions never match.
  - Set **`permission_operation`** to **`EXECUTE`**.
  - **`permission_question`** tone: **scary and explicit** — user must understand real risk before clicking Allow.
    - Lead with danger, not politeness.
    - State the command runs **as root** on the **host**, outside the normal sandbox.
    - Warn this **may damage your system** if the command is wrong or malicious.
    - Tell the user to **only proceed if they understand the command** — e.g. they may **not be able to log in tomorrow** if it breaks auth, disk, or boot.
    - Note a **system password dialog** appears after Allow.
    - Include command preview (first line, same as other prompts).
  - **🔷** Verbatim copy for `permission_question` (adjust only line breaks; keep wording):
    - Title line: `Confirm — Run as ROOT (high risk)`
    - Body: `This command will run with root privileges on your system, outside the normal sandbox.`
    - Warning: `WARNING: This may damage your system. Only allow this if you understand exactly what the command does. If you get it wrong, you may not be able to log in tomorrow (or worse).`
    - Password note: `After you click Allow, you will be asked for your password in a system dialog.`
    - Command line: `Command: ` + `<cmd_preview>` (first line of command, same as other prompts)

  - **🔷** `⏳` Use **destructive styling** on the ChatPermission **Allow** button for root prompts (same `destructive-action` / warning feel as deny buttons, or a dedicated root-warning CSS class on the frame) so the row reads as high-risk at a glance.
- **🔷** If **`run_as_root`** is **`false`** but the command text appears to invoke elevation (`sudo`, `pkexec`, `su`, etc.):
  - **💩** `⏳` Return **`ERROR:`** with guidance to use **`run_as_root: true`** instead — optional guard; confirm before build.

### Execution path when `run_as_root` is true

- **🔷** **Do not use bubblewrap** for these runs.
  - Root + user-namespace sandbox is the wrong model.
  - No overlay scan / approvals flow for root runs (host changes are outside our bwrap overlay).
- **🔷** After ChatPermission **Allow** (once or session — but **once** is the only durable option exposed in UI when `one_time_only`):
  - Spawn elevation helper on Unix via **`pkexec`** (PolicyKit).
    - PolicyKit shows the native GTK password dialog the user described (modern replacement for deprecated `gksu` / `gnome-sudo`).
  - Wrap the user command in a shell one-liner that sets working directory when needed:
    - e.g. `pkexec env … sh -c 'cd <work_dir> && <command>'` with correct shell quoting (`GLib.Shell.quote` on inner parts).
  - Capture stdout/stderr the same way as `execute_with_subprocess()` (pipe read, truncate, UI fenced message).
- **🔷** **`network`** and **`allow_write`** semantics when **`run_as_root`**:
  - **💩** `⏳` Treat root runs as **full host access** — ignore sandbox `network` / `allow_write` restrictions for the elevated child (root can do anything). Document that in the tool description. Confirm before build.
- **ℹ️** Order in `Request.execute()`:
  - Validate parameters (including **`run_as_root`** mutual exclusivity / parse order — see [2.22.1.4](done/2.22.1.4-DONE-seccomp-write-paths-and-followups.md) “validate before permission”).

### Platform scope

- **🔷** **Linux (GTK app, non-Flatpak or Flatpak with working `pkexec`)** — primary target for v1.
- **💩** `⏳` **Flatpak** — document behaviour when `pkexec` is unavailable or blocked; return clear **`ERROR:`** (do not hang). Optional follow-up: Flatpak portal / host command exposure.
- **💩** `⏳` **Windows** — UAC elevation (`runas` / ShellExecute elevate) as a separate phase; v1 may return **`ERROR: run_as_root is not supported on Windows`**.
- **💩** `⏳` **CLI** (`ollmchat-cli`) — no GTK PolicyKit UI; v1 deny **`run_as_root`** with **`ERROR:`** unless a future CLI elevation path exists.

## Security and UX notes

- **🔷** Two-step consent is intentional:
  1. ChatPermission — user reads the command and clicks **Allow**.
  2. PolicyKit dialog — user enters password (or biometrics per OS policy).
- **🔷** Cancelling the PolicyKit dialog → treat as failure; return **`ERROR:`** to the model (include stderr if PolicyKit prints any).
- **🔷** ChatPermission row for root must look **high-risk** (scary copy above + destructive/warning button styling).
- **🚫** Do not persist “always allow root commands” via permission store for this feature.
- **🚫** Do not run elevated commands inside bubblewrap.
- **🚫** Do not pipe the user’s password through our app (no `sudo` with password on stdin).
