# 7.14.11 — `tool-call-summary` message role + replay tooltip hydration

**Status:** proposed

**Pointer — `.cursor/rules/CODING_STANDARDS.md` (`## Checklist for all plans`),** **`docs/guide-to-writing-plans.md`**

## Checklist — plans (**`CODING_STANDARDS.md`**)

This plan aims to satisfy the **plan checklist** (verify before marking **done**):

- **Nullable types / null checks / defensive code**: Prefer a single optional **`string`** default **`""`** on **`Task.Tool`** for replay tooltip text; no extra null guards beyond existing **`tool_request`** checks.
- **`@"` interpolation**, one-use temps, **`get_*`**, long method names — avoid per standards.
- Brace style, **`this.`**, **`GLib.`** qualification, line length — match project rules.
- **`StringBuilder`** / **`Gee.ArrayList<string>`** for string accumulation — avoid unless justified.
- **New methods**: default to extending **`tooltip_text`** and **`Runner.on_replay`** **`switch`** arms only; **⚠️** user asked for persisted transcript metadata — a **single new string property** on **`Task.Tool`** is explicitly in scope (**§ Concrete code proposals**).
- **Single canonical proposals**: mechanical **Remove** / **Replace with** / **Add** fences below — no duplicate full-file copies.
- Every **`#### Add`** documents **Where** + **What** (placement + purpose), not only a fence (**`docs/guide-to-writing-plans.md`**).

---

## Purpose

- **Problem:** Tooltip text on task progress tool rows comes from **`RequestBase.to_summary()`** via **`tool_request`**, which is only set after **`BaseTool.execute()`**. Replay skips **`execute`**, so **`tool_request`** stays **`null`** and tooltips are empty on restore (**ℹ️** discussion in **[ prior chat ]** — same analysis).
- **Outcome:** Emit **`tool-call-summary`** next to each **`Request`**’s existing **`ui`** **`to_summary()`** line so the transcript carries tooltip metadata **without** a second **`add_message`** path in **`Task.Tool.run()`**. On **`Runner.on_replay`**, hydrate **`Task.Tool.replay_tooltip`** from those messages.

---

## Scope

| In scope | Out of scope |
| -------- | ------------ |
| New **`Message.role`** **`tool-call-summary`** (metadata: hidden from chat UI, not sent to LLMs) | Changing **`ReplayChat.send()`** skipping logic beyond verifying **`tool-call-summary`** stays skipped (same as **`agent-stage`**) |
| **`liboctools` / `libocvector` `Request`**: one **`add_message ("tool-call-summary", …)`** at the **same call site** as the existing **`ui`** fence that already shows **`to_summary()`** (or the primary “command running” line for **`run_command`**) — **not** in **`Task.Tool.run()`** | **`tool_reply`** / generic **`Agent.Base.execute_tools`** path unless trivial symmetry is desired later |
| **`Runner.on_replay`**: **`EXECUTION`** phase handles **`tool-call-summary`** **before** **`content-stream`** feeds **`exec_extract`** | Rewriting **`to_summary()`** bodies |
| **`Task.Tool`**: **`replay_tooltip`** + **`tooltip_text`** branch only (**no** extra **`add_message`** in **`run()`**) | Persisting **`tool-call-summary`** for sessions without **`can_replay`** markers |

---

## Acceptance criteria

- **Live:** Where a **`Request`** already **`add_message`**’s **`ui`** content built from **`to_summary()`** (same string the user sees in the fenced block), **`add_message`** a **`tool-call-summary`** whose **`content`** is **that same string** when non-empty (**⚠️** distinct role for restore). **`Task.Tool.run()`** does **not** emit **`tool-call-summary`**.
- **Transcript order:** **`tool-call-summary`** is appended **immediately after** that existing **`ui`** line inside **`execute_request`** / **`execute`** — still **before** **`Task.Tool.run()`** reaches “Reviewing Tool Output …” / **`agent-stage`** **`exec`**, because **`yield tool_impl.execute`** finishes first.
- **Restore:** **`Runner.on_replay`** in **`PhaseEnum.EXECUTION`** assigns **`replay_tooltip`** on the current **`Tool`** row and calls **`ex_run.notify_property ("tooltip_text")`** so **`ProgressView`** updates (see **§ 4**).
- **Chat UI:** **`tool-call-summary`** messages are **not** rendered as bubbles (**`is_ui_visible`** false), same class as **`agent-stage`** / **`agent-issues`**.
- **Backward compatibility:** Sessions without **`tool-call-summary`** replay unchanged; tooltips empty on replay for those rows (current behaviour).
- **ReplayChat:** **`send()`** still advances only **`content-stream`** / **`content-non-stream`** / **`assistant`**; **`tool-call-summary`** never consumes a **`send()`** slot.

---

## Current behaviour (**brief**)

- **`Task.Tool.run()`** sets **`tool_request`** only inside **`!in_replay`** (**ℹ️** **`liboccoder/Task/Tool.vala`**).
- **`tooltip_text`** uses **`tool_request?.to_summary()`** only.
- **`ReplayChat`** ignores non-content roles when satisfying **`send()`**.

---

## Proposed behaviour (**brief**)

- Each **`Request`** implementation emits **`tool-call-summary`** next to its existing **`to_summary()`** **`ui`** output (reuse the same **`string`** variable where the fence already uses one).
- **`Task.Tool`** keeps **`replay_tooltip`**; **`tooltip_text`** prefers **`replay_tooltip`**, else **`tool_request?.to_summary()`**. **`run()`** unchanged aside from **`tooltip_text`** branch.
- **`on_replay`** hydrates **`replay_tooltip`** from **`tool-call-summary`** messages (**§ 4**).

---

## Live transcript fragment (**contract**)

Per tool run (skill **`execute`** path):

1. Existing **`ui`** fence(s) from the **`Request`** (e.g. “File Read Requested”, “Running command …”) — unchanged body.
2. **`tool-call-summary`** — **same** text as **`to_summary()`** passed to that fence (**or** **`run_command`**’s **`to_summary()`** beside “Running command …”).
3. Later: **`Task.Tool`** executor **`ui`** / **`agent-stage`** **`exec`** / **`content-stream`**, …

**⚠️** **`restore_messages`** still sees **`tool-call-summary`** **before** “Reviewing Tool Output …” because it is emitted during **`BaseTool.execute`** / **`execute_request`**, which completes before **`Tool.run()`** builds the executor prompt loop.

## Risks / notes

- **✅** Transcript ordering is the correlation key — **`tool-call-summary`** must stay **inside** **`BaseTool.execute`** / **`Request.execute_request`**, **before** **`Tool.run()`**’s executor **`ui`** / **`agent-stage`** **`exec`**, or **`replay_tool_pos`** will misalign with **`on_replay`**.
- **ℹ️** If a **`Request`** gains a **second** early-return path that **`add_message`**’s errors **without** emitting **`tool-call-summary`**, replay tooltips for failed runs may stay empty — acceptable if those sessions are not **`can_replay`**-complete.
- **👎** Do not use **`tool-call-summary`** for the LLM’s **`## Result summary`** markdown — that remains inside **`content-stream`**; this role is **only** for **`RequestBase.to_summary()`** tooltip text.
