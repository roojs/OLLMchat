**Status:** proposed

**Pointer — `.cursor/rules/CODING_STANDARDS.md` (`## Checklist for all plans`),** **`docs/guide-to-writing-plans.md`**

## Purpose

- **Problem:** Tooltip text on task progress tool rows comes from **`RequestBase.to_summary()`** via **`tool_request`**, which is only set after **`BaseTool.execute()`**. Replay skips **`execute`**, so **`tool_request`** stays **`null`** and tooltips are empty on restore (**ℹ️** discussion in **[ prior chat ]** — same analysis).
- **Outcome:** Persist the same summary string the UI would show live using a dedicated transcript role **`tool-call-summary`**. On **`Runner.on_replay`**, hydrate the corresponding **`Task.Tool`** row so **`tooltip_text`** matches the saved session without re-running tools.

---

## Scope

| In scope | Out of scope |
| -------- | ------------ |
| New **`Message.role`** **`tool-call-summary`** (metadata: hidden from chat UI, not sent to LLMs) | Changing **`ReplayChat.send()`** skipping logic beyond verifying **`tool-call-summary`** stays skipped (same as **`agent-stage`**) |
| **`Task.Tool`**: emit one message **after** successful **`execute`** when **`to_summary()`** is non-empty | **`tool_reply`** / generic **`Agent.Base.execute_tools`** path unless trivial symmetry is desired later |
| **`Runner.on_replay`**: **`EXECUTION`** phase handles **`tool-call-summary`** **before** **`content-stream`** feeds **`exec_extract`** | **`RequestBase`** / **`to_summary()`** implementations |
| **`Task.Tool.tooltip_text`**: prefer replay string when set | Persisting **`tool-call-summary`** for sessions without **`can_replay`** markers |

---

## Acceptance criteria

- **Live:** After **`yield tool_impl.execute(...)`**, when **`in_replay`** is false and **`tool_request.to_summary()`** is non-empty, **`session`** gains a **`tool-call-summary`** message whose **`content`** equals that summary **byte-for-byte** (**⚠️** user requirement: distinct message type carrying summary for restore).
- **Transcript order:** **`tool-call-summary`** appears **after** tool execution completes for that run and **before** the **`ui`** “Reviewing Tool Output …” frame (**before** **`agent-stage`** **`exec`**) for the same executor attempt — so restore walks summaries in lockstep with exec replay.
- **Restore:** **`Runner.on_replay`** in **`PhaseEnum.EXECUTION`** applies each **`tool-call-summary`** to **`pending.steps[replay_step_pos].children[replay_details_pos].tools().get_at(replay_tool_pos)`** via the new stored string; **`progress.rebuild()`** not required solely for tooltip if **`notify`** on **`tooltip_text`** fires (see proposals).
- **Chat UI:** **`tool-call-summary`** messages are **not** rendered as bubbles (**`is_ui_visible`** false), same class as **`agent-stage`** / **`agent-issues`**.
- **Backward compatibility:** Sessions without **`tool-call-summary`** replay unchanged; tooltips empty on replay for those rows (current behaviour).
- **ReplayChat:** **`send()`** still advances only **`content-stream`** / **`content-non-stream`** / **`assistant`**; **`tool-call-summary`** never consumes a **`send()`** slot.

---

## Current behaviour (**brief**)

- **`Task.Tool.run()`** sets **`tool_request`** only inside **`!in_replay`** (**ℹ️** **`liboccoder/Task/Tool.vala`**).
- **`tooltip_text`** uses **`tool_request?.to_summary()`** only.
- **`ReplayChat`** ignores non-content roles when satisfying **`send()`**.

---

## Proposed behaviour (**brief**)

- One **`tool-call-summary`** row per completed **`execute`** with non-empty summary.
- **`Task.Tool`** keeps **`replay_tooltip_summary`** (**`string`**, default **`""`**); **`tooltip_text`** returns **`replay_tooltip_summary`** when non-empty, else live **`to_summary()`** path.
- **`on_replay`** assigns **`replay_tooltip_summary`** when **`replay_phase == EXECUTION`** and **`role == "tool-call-summary"`**, using the same **`replay_*`** cursor as **`content-stream`** **`exec_extract`** for the current tool run.

---

## Live transcript fragment (**contract**)

After implementation, **one tool execution** with a non-empty **`to_summary()`** must persist messages in this **relative order** (names only):

1. **`tool-call-summary`** — **`content`** = **`tool_request.to_summary()`**
2. *(reference preload / resolver — no new messages)*
3. *(first executor attempt)* **`ui`** (Reviewing Tool Output …), **`ui-waiting`**, **`agent-stage`** **`exec`**, **`content-stream`** (executor markdown), **`agent-issues`**, …

**⚠️** **`tool-call-summary`** must appear **before** the **`ui`** “Reviewing Tool Output” message for that **`run()`**, so **`restore_messages`** invokes **`on_replay(tool-call-summary)`** before **`on_replay(agent-stage exec)`** for the same logical exec.

## Risks / notes

- **✅** Transcript ordering is the correlation key — **⚠️** implementers must not emit **`tool-call-summary`** after **`agent-stage`** **`exec`** or **`content-stream`**, or **`replay_tool_pos`** will misalign.
- **ℹ️** **`restore_messages`** calls **`on_replay`** for every message including **`tool-call-summary`** — ensure **`EXECUTION`** phase is already active when the first summary for a step arrives (same order as live **`add_message`** sequence relative to **`agent-stage`** transitions). If a gap appears in testing, document the exact live **`add_message`** order in **`Tool.run()`** as the contract.
- **👎** Do not use **`tool-call-summary`** for the LLM’s **`## Result summary`** markdown — that remains inside **`content-stream`**; this role is **only** for **`RequestBase.to_summary()`** tooltip text.
