**Status:** PROPOSAL — implementation plan (no coding in this doc).

## Premise: one daemon, many lightweight clients

- **Daemon:** one `OLLMfiles.ProjectManager` — SQLite, FAISS queues, filesystem scan, canonical `projects` / caches.
- **Each UI window:** its own **`V2/ProjectManager`** + its own **`RpcClient`** (separate socket sessions are fine).
- Old design shared one fat in-process PM; V2 PM is thin — holds `active_project` / `active_file`, Gtk signals, and **`RpcClient`** only.
- App constructs **`new ProjectManager()`** per window (when cutover); each instance bootstraps RPC on construct.

---

## Phase 1 — Bootstrapping (`RpcClient` only)

**Goal:** Client can always reach a running `ollmfilesd` before any `ProjectManager.*` call.

Today `RpcClient.connect()` only connects to the socket; it does **not** start the daemon.

**Add to `RpcClient` (not V2 PM yet):**

1. **`ensure_daemon()`** (or inside `connect()`):
   - If socket accepts connection → proceed.
   - Else if production → assume **systemd** user service will provide socket *(document path / unit name when known)*.
   - Else **dev / fallback:** spawn `ollmfilesd` (foreground flag for tests, detached for app) — wait until socket exists (poll / short timeout).
2. **`connect()`** sequence: `ensure_daemon` → Unix connect → NDJSON streams → **`Daemon.hello`** (already there).
3. Optional: `client_name` includes window id later; v1 `"ollmchat"` OK.

**Out of scope for phase 1:** `ProjectManager` RPC methods; V2 file edits.

**Test:** no `ollmfilesd` running → `RpcClient.connect()` leaves daemon up and hello succeeds.

## Phase 2 — Client `libocfiles/V2/ProjectManager.vala`

**Goal:** Same public API surface as today where callers care, but **no direct filesystem / DB / scan** — only RPC + local UI state.

### Constructor (client bootstrap)

```text
V2/ProjectManager()
  → rpc = new RpcClient ()
  → yield rpc.connect()     // ensure_daemon + hello inside
  → (no SQ.Database — db stays null on client)
```

- Remove or no-op server-only construction: `DeleteManager` wired to local DB, `db` usage in ctor.
- **`delete_manager`**, **`file_cache`**, **`tree_cache`**, **`scanning`:** drop or keep empty stubs until `File` / `Folder` plans; do not mirror full daemon graphs.

### What stays on the client (no RPC)

| Piece | Why |
| ----- | --- |
| `active_project`, `active_file` | Per-window UI state |
| Gtk signals | Re-emit after RPC success |
| `buffer_provider`, `git_provider` | Local / no-op bases for now |
| `disable_initial_scan` | Passed as `params.skip_scan` on activate |
| `tree_factory` | Local AST on buffer content *(until `File.parse.ast`)* |
| `notify_project_changed` | May become “emit after RPC” only; daemon fires `event.*` later |

### What becomes `rpc.call("ProjectManager.{method}", …)`

Replace body of any method that touches **disk, SQLite, `saveToDB`, `read_dir`, `DeleteManager` on server data**, or builds canonical project tree:

| V2 method | RPC | Strip from V2 body |
| --------- | --- | ------------------ |
| `load_projects_from_db` | `ProjectManager.load_projects_from_db` *(wire name = Vala name)* | DB query, `projects.append` from SQL — **hydrate thin list from `result`** |
| `create_project` | `ProjectManager.create_project` | local `Folder` create + `saveToDB` |
| `remove_project` | `ProjectManager.remove_project` | local `saveToDB` |
| `activate_project` | `ProjectManager.activate_project` | `read_dir`, `saveToDB` on activate |
| `get_folder_at_path` | `ProjectManager.get_folder_at_path` or defer | DB / cache walk |
| `restore_active_state` | orchestrate several RPCs | direct DB / activate |
| `activate_file` | → `File.activate` in [2.10.4.6](2.10.4.6-file.md) | `saveToDB` |
| `on_file_metadata_change` | → `File.metadata.set` | `saveToDB` |
| `on_file_contents_change` | → `File.write` | disk write |
| `convert_fake_file_to_real` | → `File.register` | Gio file create + DB |
| `check_active_file_changed` | → `File.changed.check` | disk stat |
| `write_buffer_to_disk` | → `File.write` | disk |
| `reload_file_from_disk` | → `File.read` | disk |

**Phase 2 tranche (PM-only):** implement RPC relay for **`load_projects_from_db`**, **`create_project`**, **`remove_project`**, **`activate_project`** first — enough for settings + window open.

**Client `projects` list:** after `load_projects_from_db` RPC, rebuild **`ProjectList`** from returned `filebase` rows (paths + ids), not from SQL.

## Phase 3 — Server `ollmfilesd/ProjectManager.vala` + dispatch

**Goal:** Incoming `ProjectManager.{method}` on the wire runs the **real** method on the daemon’s `ProjectManager` instance.

### Daemon process layout

```text
main
  → open DB, ProjectMigrate.migrate_all
  → project_manager = new OLLMfiles.ProjectManager(db)
  → yield project_manager.load_projects_from_db()
  → listen = new Listen(project_manager: ...)
  → listen.start()
```

One **`ProjectManager`** for all connected clients (shared DB). Per-window state (`active_project` on server) may need **`client_id` later** — v1 can use last-writer or single-client assumption for testing.

### `Listen` wiring

- Property **`ProjectManager`** on `Listen` (name matches wire object — dispatch uses `registry.ProjectManager`).
- Value: the server `OLLMfiles.ProjectManager` instance (relay target).

**Relay pattern** (same idea as `Rpc.Daemon` today):

- For each supported wire method, handler receives **`Rpc.Request`**:
  - Read **`request.param`**
  - Call **`this.project_manager.{method}(...)`** (real Vala)
  - **`request.session.reply(request, result)`** or **`reply_error`**
- Handlers can live:
  - in **`construct`** of a small `Rpc.ProjectManagerRelay` wrapper, **or**
  - as **`rpc_{method}`** signals on a GObject that holds `unowned ProjectManager manager`

**Lightweight vs heavy:**

| Style | Example |
| ----- | ------- |
| Trivial in relay ctor | parse params, one line call, serialize result |
| Heavy | `activate_project` — already async on server; relay `yield`s and returns revision / flags |

Do **not** duplicate business logic in relay — only param mapping + result JSON.

### Phase 3 tranche (match phase 2 client)

Wire handlers for:

- `ProjectManager.load_projects_from_db`
- `ProjectManager.create_project`
- `ProjectManager.remove_project`
- `ProjectManager.activate_project`

### Phase 3 does not edit (yet)

- `libocfiles/V2/ProjectManager.vala` beyond what phase 2 needs
- Vector, `Folder`, `File` server relays ([2.10.4.5](2.10.4.5-folder.md), [2.10.4.6](2.10.4.6-file.md))

## Open questions (decide before code)

1. **`ensure_daemon` spawn command** — path to `ollmfilesd` binary, args (`--foreground` for tests), pid file behaviour.
2. **Server `active_project`** with multiple windows — last activate wins vs per-session id.
3. **Exact wire method names** — use full Vala names (`load_projects_from_db` not `list`) — **recommended** per 2.10.4.1.
4. **`Listen.ProjectManager` property** vs lowercase `project_manager` — wire requires **`ProjectManager`** prefix; property name on `Listen` must match dispatch (`ProjectManager`).
