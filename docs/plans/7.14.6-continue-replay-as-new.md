## User-specified requirements (authoritative)

- 🔷 Bridge **progress**, **chat**, and later **rendered output** with **message indexes**.
- 🔷 Each **message** gets a runtime **`idx`** in order; **omit from serialization**; **rebuild on import** when messages are loaded.
- 🔷 Each **`ProgressItem`** gets **`msg_idx`**: **last** chat message index tied to that item’s step (updates as more messages belong to that step); **continue-from-here** continues **after** that message.
- 🔷 When creating/updating **`ProgressItem`** from **streaming** or **replay**, set **`msg_idx`** to that **last related message**’s index.
- 🔷 **Phase A:** connect **`ProgressItem`** ↔ chat messages. **Phase B:** connect chat messages ↔ **rendered output** (same index story).
- 🔷 In the **Gtk chatview widget**, keep **`idx_to_widget`**: a **hash map** from **`Message.idx`** to the **widget** for that rendered row (replaces **`message_widgets`** — **one** structure).
- 🔷 Expose **`scroll_to_idx()`** on that chatview so the UI can **jump** to the message for a given **`idx`** (continue-from, navigation, progress tree).

---

## Design and backlog (implementation)

- ℹ️ Authoritative requirements are in **User-specified requirements** above — do not contradict them.

### Phase A — `ProgressItem` ↔ chat messages

- ⏳ **Messages:** add runtime **`idx`** (`0..n-1`); exclude from save/export; reassign on import.
- ⏳ **`ProgressItem`:** add **`msg_idx`**; **last** related message index; update on stream/replay.
- ⏳ Align **cut / fork** with **`7.13`** when wiring continue/replay-as-new.
- ⏳ **Streaming:** set/update **`msg_idx`** when **`ProgressItem`** is created or updated.
- ⏳ **Replay:** same when progress is synthesized from replay.

### Phase B — chat messages ↔ rendered output

- ⏳ In **`ChatView`**, **`idx_to_widget`**(**`int`** → **`Gtk.Widget`**); keep it consistent when rows are created, torn down, or **`clear()`** runs.
- ⏳ Add **`scroll_to_idx()`** on that widget: given **`idx`**, scroll/jump so that row is visible (used with **`msg_idx`** from progress / navigation).
- ⏳ Wire progress-tree actions and any **continue-from** entry points to **`scroll_to_idx()`** where needed.
- ⏳ **Single contract:** one order index for messages, **`msg_idx`**, and rendered rows; **`idx`** stays ephemeral on disk.

## Required changes (summary)

- ⏳ **Messages:** runtime **`idx`**; omit from serialization; reassign on import.
- ⏳ **`ProgressItem`:** **`msg_idx`** = **last** message index for that item; mutable as the step grows.
- ⏳ **Streaming / replay:** set **`msg_idx`** via **`Runner`** + **`ProgressList`** insert paths (`§12`–`§13`).
- ⏳ **Phase B:** **`ChatView.idx_to_widget`**; **`scroll_to_idx()`**; scroll / highlight / continue target uses **`idx` / `msg_idx`**.
- ⏳ **Continue / replay-as-new:** cut point from **`msg_idx`** + **7.13** — **after** the tied message for the selected node.
