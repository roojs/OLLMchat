## Issue summary (**⚠️** user-reported)

- **ℹ️** Thinking blocks are **normally streamed** (`Render.is_streaming` true at fence open when live).

### Symptom 1 — Body stays collapsed until the fence completes

- **⚠️** While streaming, the thinking body **appears collapsed** (not expanded) **until** the stream **completes** — expected: body **expanded** while the assistant is still streaming thinking content.
- **⚠️** **Desired refinement:** In **streaming** mode, when the fence **ends**, **delay slightly** before applying the automatic collapse (`collapse_on_done` path), so the user can see the final state briefly before the fold (exact ms TBD in implementation).

### Symptom 2 — Chevron shows “expanded” (up arrow) when the body is collapsed

- **⚠️** The header control shows the **up** / **collapse** affordance (`go-up-symbolic`, “Collapse”) **as if** the body were expanded, **while** the body is **actually hidden** / collapsed.

## What the code actually does (**ℹ️** tree — read this before fixing)

### Where the thinking fence comes from

- **ℹ️** `libollmchatgtk/ChatView.vala` — `start_block_direct` opens the thinking code fence with a **fixed** info string (language + frame theme classes):

```450:452:libollmchatgtk/ChatView.vala
						this.renderer.on_code_block(true, 
							"markdown.oc-frame-info.thinking.collapsed-on-done Thinking...");
						this.thinking_frame = this.renderer.childview;
```

- **ℹ️** `libocmarkdowngtk/Render.vala` — `on_code_block(true, lang)` allocates **`RenderSourceView(this, lang, this.is_streaming)`**. Feed mode is whatever **`renderer.is_streaming`** is **at fence open** (set by **`ChatView`** before **`add()`** for that assistant round — see 7.14.2.3).

### How `RenderSourceView` parses that fence

- **ℹ️** After splitting on `"."`, the **frame theme classes** are the segments after the first token (`markdown` → language `markdown`; rest → `oc-frame-info`, `thinking`, `collapsed-on-done`).
- **ℹ️** **`has_collapsed_style`** is **`"collapsed" in frame_theme_classes`** — an **exact token** match. The token **`collapsed-on-done`** does **not** satisfy that (it is a different class name). So for the standard thinking fence, **`has_collapsed_style` is false** and **`collapse_on_done` is true**.

### The ctor gate: when `body_widgets()` runs vs lazy

- **ℹ️** This condition decides whether the embed + revealer are built **in the ctor**:

```264:267:libocmarkdowngtk/RenderSourceView.vala
			if (is_streaming || (!has_collapsed_style && !this.collapse_on_done)) {
				this.body_widgets(is_streaming || !this.collapse_on_done);
				this.body_revealer.reveal_child = !has_collapsed_style;
			}
```

| `is_streaming` | `has_collapsed_style` | `collapse_on_done` | Enter `if`? | `body_widgets` arg | `reveal_child` after |
|----------------|----------------------|--------------------|---------------|----------------------|------------------------|
| **true** | false | **true** (thinking) | **yes** (`is_streaming`) | `true` | **true** (`!has_collapsed_style`) |
| **false** | false | **true** (thinking, bulk) | **no** | — | — (no `body_revealer` yet) |
| **false** | **true** | false | **no** | — | — |

- **ℹ️** **Live streaming thinking** (first row): ctor **does** call **`body_widgets(true)`** and sets **`reveal_child = true`**. On paper the body should **not** stay hidden for the whole fence for that reason alone.
- **ℹ️** **Bulk / history replay** of the same fence shape (second row): ctor **does not** call **`body_widgets()`**. Then **`has_widgets`** stays false, **`has_content`** stays false, and **`add_code_text` only does `code_text +=`** until something else builds embeds — so there is **no GtkSource / nested markdown visible** during replay until **`body_widgets` / `fill_widgets`** run (e.g. user expands). That can look like “nothing until the end” depending on when **`end_code_block`** fires relative to UI.

### Default `Gtk.Revealer` and `add_code_text` side effects

- **ℹ️** **`body_widgets()`** creates **`body_revealer`** with **`reveal_child = false`**, then appends it; the ctor **immediately** sets **`reveal_child = !has_collapsed_style`** when the gate above ran (line 266). So the revealer is created hidden for one instant, then flipped to shown for streaming thinking.

```401:407:libocmarkdowngtk/RenderSourceView.vala
			this.body_revealer = new Gtk.Revealer() {
				reveal_child = false,  // callers set true when the body should paint (eager path, expand click)
				hexpand = true,
				vexpand = false,
				transition_type = Gtk.RevealerTransitionType.SLIDE_DOWN,
				transition_duration = 250
			};
```

- **ℹ️** **`add_code_text`** always inserts into **`source_buffer`** and nested markdown **before** the early return:

```599:616:libocmarkdowngtk/RenderSourceView.vala
		public void add_code_text(string text)
		{
			if (!this.has_content) {
				this.code_text += text;
				return;
			}

			Gtk.TextIter end_iter;
			this.source_buffer.get_end_iter(out end_iter);
			this.source_buffer.insert(ref end_iter, text, -1);

			if (this.code_language == "markdown") {
				this.nested_markdown_render.add(text);
			}

			if (this.body_revealer != null && !this.body_revealer.reveal_child) {
				return;
			}
```

- **ℹ️** If **`reveal_child`** is **false**, chunks still **land in buffers**, but **scroll**, **`code_block_content_updated`**, and **newline resize** are **skipped**. The user can see an **empty or static** body until something reveals the revealer and/or triggers layout — consistent with “looks collapsed / stuck” even though data is present.

### Chevron initial state (Symptom 2 — confirmed in tree)

- **ℹ️** The header button is **not** initialized from **`body_revealer.reveal_child`** (the revealer may not exist yet). It uses **`collapse_on_done` only**:

```122:131:libocmarkdowngtk/RenderSourceView.vala
			// When .collapsed: button before title toggles body (list-add = expand, go-up = collapse)
			this.collapse_toggle_button = new Gtk.Button() {
				icon_name = this.collapse_on_done ? "go-up-symbolic" : "go-next-symbolic",
				tooltip_text = this.collapse_on_done ? "Collapse" : "Expand",
				hexpand = false,
				margin_end = 4,
				can_focus = false,
				focus_on_click = false,
				visible = has_collapsed_style || this.collapse_on_done
			};
```

- **ℹ️** For **thinking**, **`collapse_on_done`** is **true** → **go-up-symbolic** / “Collapse” **always** at construction time.
- **ℹ️** On the **lazy** path (**bulk** row in the table), **`body_revealer` is null** and the body is **not** shown, but the chevron still reads **“Collapse”** / **up** — **wrong affordance** (Symptom 2).
- **ℹ️** On the **streaming** path, **`reveal_child`** becomes **true**, so “up = expanded” matches **if** you ignore the initial **250 ms** slide animation and any frame where the revealer was still false.

### Auto-collapse at fence end

- **ℹ️** **`end_code_block`** collapses after **`GLib.Timeout.add(350, …)`** when **`collapse_on_done`**:

```694:702:libocmarkdowngtk/RenderSourceView.vala
			if (this.collapse_on_done) {
				GLib.Timeout.add(350, () => {
					this.body_revealer.reveal_child = false;
					this.collapse_toggle_button.icon_name = "go-next-symbolic";
					this.collapse_toggle_button.tooltip_text = "Expand";
					this.view_source_toggle.visible = false;
					this.copy_button.visible = false;
					return false;
				});
			}
```

- **ℹ️** There is **no** branch for “this fence was live-streamed vs bulk”; delay is **fixed 350 ms**.

### Hypotheses to validate in a debugger (**💩** — not proven here)

- **💩** **Wrong `is_streaming` at fence open** — If **`renderer.is_streaming`** is **false** when **`on_code_block(true, …)`** runs for thinking, the ctor takes the **lazy** path. Then **`add_code_text`** never hits GtkSource (`has_content` false) until policy changes — matches “no visible streaming body” and wrong chevron. **Verify:** breakpoint at **`RenderSourceView` ctor** and log **`is_streaming`** for a live thinking session.
- **💩** **`reveal_child` false during an active stream** — If anything sets **`reveal_child`** back to **false** before **`end_code_block`** (or a race leaves it false), **`add_code_text`** skips parent scroll/resize (lines 614–616) while still appending text — body may **look** frozen/collapsed. **Verify:** log **`body_revealer.reveal_child`** on each **`add_code_text`** chunk.
- **💩** **Perception vs collapse** — **Nested markdown** (`rendered` page) may stay **visually empty** for the first ticks even when **`reveal_child`** is true; user may read that as “collapsed until done.” **Verify:** distinguish **revealer** state from **nested Render** output.

## Proposed fix (design only — **⏳** not implemented here)

### A. Sync chevron to real body visibility

- **ℹ️** **Grounded in:** `collapse_toggle_button` ctor uses **`collapse_on_done` only** (see citations above); lazy path has **no** revealer but still **go-up**.
- **💩** After **`body_widgets()`** exists and whenever **`body_revealer.reveal_child`** changes (ctor tail, **`collapse_toggle_button.clicked`**, **`end_code_block`** `collapse_on_done` timeout), set:
  - **`reveal_child` true** → icon **`go-up-symbolic`**, tooltip **“Collapse”** (or existing copy).
  - **`reveal_child` false** → icon **`go-next-symbolic`**, tooltip **“Expand”** (or existing copy).
- **💩** **Do not** initialize the chevron from **`collapse_on_done` alone**; derive from **revealer state** (and **lazy**: if **`!has_widgets`**, chevron should match “body not yet shown” — **go-next**, not go-up).
- **👎** Do not add a **third** parallel boolean for “chevron state” — sync from **`body_revealer.reveal_child`** (+ **`has_widgets`** where needed).

### B. Streaming: keep body expanded during the fence

- **ℹ️** **Grounded in:** ctor table — **streaming + thinking** should already call **`body_widgets`** and set **`reveal_child` true**. If the bug persists, priority is **verify `is_streaming` at fence open** and **`reveal_child` during `add_code_text`**, not only re-read the ctor gate.
- **💩** For **`collapse_on_done`** + **streaming** (`ctor` **`is_streaming`** true): ensure **`body_revealer.reveal_child = true`** once the embed exists **before** first visible chunk policy (align with 7.14.2.3 matrix: stream into widgets while fence open).
- **💩** Audit **`ChatView` / `Render`** assignment order so **`renderer.is_streaming`** cannot be **false** when the thinking fence opens on a **live** stream (hypothesis: lazy path mistaken for streaming).

### C. Optional delay before auto-collapse at fence end (streaming)

- **ℹ️** **Grounded in:** fixed **`350`** ms in `end_code_block` (see citation above); no `is_streaming` branch today.
- **💩** In **`end_code_block`**, where **`collapse_on_done`** schedules collapse (existing **`GLib.Timeout.add`**, ~350 ms today), use a **slightly longer** delay when the fence was **streamed** (ctor saw **`is_streaming` true**), *or* add a **small extra** delay only in that case — **exact values** to tune (e.g. +150–300 ms) during implementation.
- **👎** Do not add arbitrary **`get_allocated_width`** / allocation guards (7.14.2.3 **👎** still applies).

## Acceptance criteria (**⏳** for a future fix PR)

- **⚠️** Live **streaming** thinking fence: body **visible** while chunks arrive; chevron matches (**up** only when revealed).
- **⚠️** After stream ends: auto-collapse still runs for **`collapse_on_done`**, with **noticeable short delay** before fold in streaming mode (per Symptom 1).
- **⚠️** **Bulk / history** paths: chevron does not show **collapse** when the body has never been revealed (lazy embed).
- **⚠️** No new **`RenderSourceView`** state flags beyond what 7.14.2.3 allows (`has_widgets`, `has_content`); ctor-local **`is_streaming`** remains non-persistent unless a stored “was streaming fence” bit is explicitly approved (prefer **timeout parameterization** or **Render**-scoped hint over new RSV fields).
