### Phase 5: Background Pull with Progress
5. **Implement `ModelPullManager` class** ‚úÖ **COMPLETED** (`ollmchat/Settings/ModelPullManager.vala`)
   - ‚úÖ Manages background thread for `ollama pull` operations (single thread handles all pulls concurrently)
   - ‚úÖ Uses `OLLMchat.Client` to call `/api/pull` endpoint with streaming
   - ‚úÖ Parses progress from JSON chunks using `Response.Pull` typed objects (status, completed, total, etc.)
   - ‚úÖ Uses `GLib.MainContext.invoke()` to dispatch signals to main thread for UI updates
   - ‚úÖ **Rate limits UI updates**: Throttle progress updates to at most once every 2 seconds per model
     - ‚úÖ Track last update time per model
     - ‚úÖ Only emit progress signal if 2+ seconds have passed since last update, or if status changed
     - ‚úÖ Always emit final status updates (complete/error) immediately
   - ‚úÖ Tracks multiple concurrent pulls (single background thread with MainLoop handles all)
   - ‚úÖ Writes to `data_dir/loading.json` when starting/finishing pulls
   - ‚úÖ Format: JSON array `[{"model_name": "...", "status": "pulling", "completed": 0, "total": 0, "started": "timestamp", ...}]`
   - ‚úÖ **Error handling with retry logic**:
     - ‚úÖ On error, retry up to 5 times with exponential backoff (10s, 30s, 1m, 2m, 5m)
     - ‚úÖ Status "pending-retry" while waiting to retry (just progress indicator update, no notification)
     - ‚úÖ Status "failed" after all retries exhausted (triggers `model_failed` signal for notification)
     - ‚úÖ Reset retry count if data is received during retry (means retry is working)
     - ‚úÖ Remove from cache and file when status is "failed" (same as "complete")
     - ‚úÖ Track retry count in `LoadingStatus` object
   - ‚úÖ Split into three classes for clarity:
     - ‚úÖ `LoadingStatus` - Status data class with JSON serialization
     - ‚úÖ `ModelPullManager` - Main thread logic, UI interaction, persistence
     - ‚úÖ `ModelPullManagerThread` - Background thread logic, network operations

## Technical Details

### Ollama Pull API
- Endpoint: `POST /api/pull`
- Request body: `{"name": "model_name", "stream": true}`
- Response: Streaming JSON chunks with status updates:
  - `{"status": "pulling manifest", "digest": "sha256:...", ...}`
  - `{"status": "pulling <digest>", "digest": "sha256:...", "Progress": "completed/total (percentage%)", ...}`
  - Status values observed: "pulling manifest", "pulling <short_hash>" (e.g., "pulling 415f8f959d80")
  - Progress format: `"Progress": "430243/300796576 (0.1%)"` - shows completed bytes, total bytes, and percentage
  - Digest: Full sha256 hash (e.g., "sha256:415f8f959d807bd4d4da891f01225d7b330416947fb011a8473080ae4fd07885")
  - Final chunk: `{"status": "success", ...}` (or error status)

### Background Thread Pattern
- ‚úÖ Use single `Thread<bool>` with `MainLoop` and `MainContext` for all pull operations
- ‚úÖ Background thread creates its own `MainContext`, pushes it as thread default, runs `MainLoop`
- ‚úÖ Use `MainContext.invoke()` to dispatch signals to main thread for UI updates
- ‚úÖ Track active pulls in `ModelPullManager` to prevent duplicate pulls
- ‚úÖ Thread safety: Pass primitive data between threads, avoid sharing objects directly

### Progress Calculation
- ‚úÖ Parse `completed` and `total` from `Response.Pull` typed objects (available during pull operations)
- ‚úÖ Calculate percentage: `(completed / total) * 100` (stored in `LoadingStatus.progress` as calculated property)
- ‚úÖ Progress updates occur during "pulling" status (not just "downloading")
- ‚úÖ Store `completed` and `total` as `int64` in `LoadingStatus` for persistence
- ‚úÖ Progress is calculated on-demand from `completed`/`total` fields
- ‚è≥ Update banner text: `"Loading {model_name} {progress}% complete"` (Phase 5.6)
