## `ProgressItem.children` vs `ToolList` (normative)

- The interface **`ProgressItem`** declares **`GLib.ListModel children`** — that is the GObject contract for the progress UI.
- **`ToolList`** already implements **`GLib.ListModel`** (items are **`Tool`**, which implements **`ProgressItem`**). It also implements **`Gee.Iterable<Tool>`** / **`Gee.Traversable<Tool>`** — prefer **`foreach (var t in …)`** over indexed **`for`** when walking **`proposed_tools`** or **`this.tools()`** (see **`liboccoder/Task/ToolList.vala`**).
- **Property type:** **`public GLib.ListModel children { get; default = new ToolList(); }`** — Vala **does not** allow **`public ToolList children`** on **`Details`** when implementing **`ProgressItem`** (getter types must match; **incompatible get accessor type** from **`valac`**). The default instance **is** a **`ToolList`**; use **`(ToolList) this.children`** (or a small **`tools()`** returning that cast) anywhere you need **`ToolList.append`**, **`foreach`**, etc.
- **Docblock (required when implementing):** Included in the **4b — fields** block below **`children`**.

## Purpose

- **`proposed_tools`** (**`ToolList`**) — refined proposals; **`children`** (**`GLib.ListModel`**, default **`ToolList`**) — execution queue (**`build_run_queue`** → **`run_exec`**); use **`(ToolList) this.children`** (or **`tools()`**) for **`ToolList`** operations.
- Rename **`build_exec_runs`** → **`build_run_queue`**; use **`this.tools()`** for the execution queue (was **`exec_runs`**); **`ProgressItem.children`** stays **`GLib.ListModel`** for the UI contract.
- Port **`ResultParser`**, **`to_markdown`**, **`run_exec`**, **`foreach`** sites to **`ToolList`** API.
- **`Details` implements `ProgressItem`** — **`status`**, **`title`**, **`status_str`**, **`children`** is **`GLib.ListModel`** (default **`ToolList`**); **`tools()`** returns the same instance for **`ToolList`** APIs.

## `ResultParser` boundary (normative)

**Execution queue (`Details.tools()` / underlying **`children`**):** **`ResultParser` must not mutate it** — no **`clear()`**, no **`append()`**, no replacing the queue. Owning when the queue is emptied or rebuilt is **`Details`** only (e.g. **`build_run_queue()`**, and the legacy synthetic path below).

**Rationale:** Parsing produces structured output and **`issues`**; committing that into the task graph (especially clearing runs) is a **post-approval** step. **`ResultParser`** does not touch the execution **`ToolList`**; **callers** (e.g. **`Details.refine()`** flow, or **`oc-test-skill-agent`**) assign **`tools()`** / **`exec_done`** inline — **no new helper methods** for that.

**Legacy `extract_exec`:** Change to **return **`Tool?`** only** (same name, smaller diff): no **`task.exec_runs`/`tools()` clear**, no **`add`**, no **`exec_done = true`** inside the parser. Callers that need the old behaviour do **`detail.tools().clear(); detail.tools().append(ex); detail.exec_done = true;`** (or equivalent) **next to** the call site.

**Refinement (`proposed_tools`):** This plan still shows **`extract_refinement`** appending to **`proposed_tools`** (parallel to today’s **`task.tools.add`**). That is **not** the execution queue. A stricter follow-up would move that mutation into **`Details.refine()`** after **`issues == ""`** without adding a named helper — optional.

## Status updates (normative — coordinate with **`7.14.1.2`** / **`7.14.1.1`**)

| When | Set **`Details.status`** |
|------|---------------------------|
| Not refining / not executing yet | **`NONE`** |
| **`refine()`** — first attempt, outer loop body (before a failed parse) | **`REFINEMENT`** |
| **`refine()`** — parse failed, will run another refinement attempt (`i < 4`) | **`REFINEMENT_RETRY`** — set **in the same branch** as the existing “retrying” UI message, **before** the next outer iteration. **`to_human()`** is **Refinement Retrying** (not generic **Retrying**): the row **title** is the **task name**, unlike the task-list **`ProgressRunner`** row. |
| Refinement parse succeeded (`issues == ""`), before **`run_exec`** / resume | **`REFINED`** — same as **`NONE`** in the status column (**`PhaseEnum.to_human`** returns empty; not bold) |
| **`run_exec`** active | **`TOOLS_RUNNING`** |
| Task row done | **`COMPLETED`** |
| Failure (refinement / exec / post-exec exhausted) | **`ERROR`** |

Wire in **`refine()`**, **`run_exec()`**, and completion paths — grep **`status_value`** / **`this.status`**.

**`PhaseEnum.REFINEMENT_RETRY`:** add enum value + **`to_human()`** — see **`### liboccoder/Task/PhaseEnum.vala`** in **Files** below (same PR).
