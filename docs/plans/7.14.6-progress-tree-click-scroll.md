# 7.14.6 — Progress tree navigation (pointer + keyboard) → chat `scroll_to_idx`

**Status:** proposed

**Parent:** `docs/plans/7.14.6-continue-replay-as-new.md`

**Checklist:** `.cursor/rules/CODING_STANDARDS.md` — **Checklist for all plans**.

**Plan layout / workflow:** `docs/guide-to-writing-plans.md` — **Required shape**, **Discussion style (emoji prefixes)**, **Concrete code proposals**.

---

## Purpose

- **🔷** User **chooses** a **`ProgressView`** row with **predictable** selection/highlight (**pointer** and **keyboard**).
- **🔷** **Keyboard** this phase: **Arrow Up** / **Arrow Down** only (**Home** / **End** / **PgUp** / **PgDn** **deferred**).
- **🔷** Chat transcript **scrolls** so the tied message is on-screen.
- **🔷** Indices follow **`7.14.6`**: **`ProgressItem.msg_idx`** ↔ **`Message.idx`** ↔ **`ChatView.scroll_to_idx`**.

---

## Scope

- **In scope**
  - **🔷** **Owned** row interaction: **avoid** **`Gtk.ColumnView.activate`** as the primary signal.
  - **🔷** Use **explicit pointer gesture** and **keyboard** (**Arrow Up** / **Arrow Down** only in this phase) so **we** drive **`Gtk.SingleSelection`** and visual **highlight**.
  - **🔷** **`Gtk.ColumnView`** + **`Gtk.SingleSelection`**: resolve **`ProgressItem`** for the **user-intended** row.
  - **🔷** Call **`scroll_to_idx(msg_idx)`** when **`msg_idx >= 0`**.
  - **🔷** Host bridge **without** **`liboccoder` → libollmchatgtk`**.
  - **🔷** **`ChatUserInterface`** adds only a **GObject-safe** hook (**no GTK types** in **`libollmchat`**).
- **Out of scope**
  - **🚫** **`replay_as_new`**, session fork, transcript cut (**parent plan**).
  - **🚫** Changing **`ProgressList`** **`msg_idx`** rules beyond **reading** **`msg_idx`** on the row.
  - **🚫** Relying on **`activate`** firing unpredictably (**Discussion**).
  - **🚫** **CSS** overrides for **`.oc-task-progress`** / **:selected** (**not** planned unless a bug appears).
  - **🚫** **Home**, **End**, **PgUp**, **PgDn** tree navigation (**defer** — **Up**/**Down** only **for now**).

## Discussion

- **🔷** Do **not** use **ColumnView.activate** as the main path.
  - It **fires unpredictably** (focus, keyboard, delegates).
  - **Scroll-to-chat** becomes unreliable (**unwanted** activations).
- **🔷** **Own** what counts as a row pick and what looks selected (**approved** approach below).
  - **🔷** **`Gtk.GestureClick`** on **`column_view`** (**released** / **pressed**, **`Gtk.PropagationPhase`** per spike).
  - **🔷** Map coordinates → row / tree position → **`ProgressItem`** (**`Gtk.TreeListRow`** / list position — match **GTK 4** hit-testing).
  - **🔷** **`Gtk.SingleSelection`**: set **`selection.selected`** (**position**) from **our** handlers (**pointer** and **keyboard**), not from stray **activate**.
  - **🔷** Highlight follows **our** policy, not stray **activate** bursts.
  - **🔷** **Spike:** avoid **widget-driven** **activation** (**GTK** **ColumnView** / **list** defaults — props, controllers, propagation) so **`activate`** is **not** used for selection or side effects.
- **🚫** **No** **CSS** work on **`.oc-task-progress`** or **:selected** overrides **for now** (not expected to be needed).
- **ℹ️** **`/home/alan/gitlive/roobuilder`** has **working** row detection (**GestureClick** → row widget → object); **`set_data`** links **widgets** to domain objects (**e.g.** `WindowLeftTree.vala` around **tree-row** / **`JsRender.Node`**).
  - **🔷** Borrow **`roobuilder`** patterns **only** if **`ColumnView`** hit-testing needs them (**may not** need **`set_data`** when **`TreeListRow`** + model position suffice).
- **🔷** **Keyboard** follows **`Gtk.TreeListModel`** **flattened** **`SingleSelection`** indices (**§** **Concrete**): **Down** — **`children`** **non-empty** → **`expanded`** **+** **`pos+1`** **when** **`< n`**; **else** **`pos+1`**. **Up** — **depth** **>0** → **`get_parent()`**; **else** **`pos-1`**. **ℹ️** **Spike** **only** if **visible** **order** **or** **collapse** **differs** **from** **this** **two-level** **strip**.
  - **🔷** **`Gtk.EventControllerKey`** on **`column_view`** for **`Gdk.Key.Up`** / **`Gdk.Key.Down`** only.
  - **🔷** **Disable** **GTK** **default** keyboard handling for those keys on **`column_view`** / **list** (**GTK 4** spike: **`propagate`**, controller phase, widget flags) so **built-in** **list** movement does **not** run **in parallel** with **our** **`selection.selected`** updates + scroll.
  - **🔷** After defaults are **off**, handle **Up**/**Down** in **our** controller and return **`Gdk.EVENT_STOP`** when consumed (**verify** **no** double step).
  - **🚫** **Home**, **End**, **PgUp**, **PgDn** — **not** in this phase (**Discussion** — **Out of scope**).
  - **🔷** **`column_view`** must have **focus** for **Up**/**Down** (**grab_focus** when **strip** clicked — **spike** hook; **tab** order must reach **`column_view`**).
  - **🔷** **`scroll_to_message`** runs after **`select_item`** from **pointer** and **keyboard** through **one** shared behaviour — **typically** **`connect`** **anonymous** handlers / lambdas (**standard** **code** design — **no** duplicated **`msg_idx`** logic).
- **🚫** **single_click_activate = true** + **activate → scroll** is rejected as the **primary** design.
  - Keep **single_click_activate = false** unless a spike proves **activate** is unused for navigation.
  - **⏳** **Review:** **`Gtk.ListItem.activatable = false`** in **both** **`SignalListItemFactory.setup`** handlers (**title** + **stage**) — **`single_click_activate = false`** alone still allows **double-click** / **Enter** to emit **`Gtk.ColumnView::activate`**; **`activatable = false`** would disable that (**not** **landed** in tree until reviewed — **§3**).
- **🔷** **`selection.changed`** must **not** drive **`scroll_to_message`** (**set_runner**, model churn, programmatic **`selection.selected`** changes).
  - **🔷** **Ignore:** do **not** **`connect`** **`selection.changed`** to transcript scroll (**preferred**).
  - **🔷** **Or block:** if **`selection.changed`** is **`connect`**ed for **other** reasons, **that** handler **must not** call **`scroll_to_message`**.
- **ℹ️** **`ProgressItem.msg_idx`** lives on **`Details`**, **`Tool`**, **`ProgressRunner`** (**`liboccoder/Task/*.vala`**).
  - Cast row to **`ProgressItem`** and read **`msg_idx`**.
- **⏳** **`ChatView.scroll_to_idx`** needs **`idx_to_widget`** filled (**`7.14.6`** §14, **`idx_to_widget.set(m.idx, w)`** when **`m.idx >= 0`**).
  - Land that **before** this subplan **or** extend **`scroll_to_idx`** elsewhere.
  - **Do not** assume **`grab_focus`** alone is enough for visibility.
- **🚫** No cast from **`ChatUserInterface`** to **`OllmchatWindow`** inside **`liboccoder`** (**`7.14.2`**).

## Acceptance criteria

- **🔷** A **deliberate user action** (**click** or **Arrow Up**/**Down** with owned selection policy) scrolls/focuses chat for **`msg_idx`** when **`msg_idx >= 0`**.
- **🔷** **Arrow Down** on a row whose **`ProgressItem`** **has children** selects the **first child** (**expand** if needed).
- **🔷** **Arrow Up** **from** **depth** **0**: **previous** **flattened** row (**`pos - 1`**); **from** **a** **tool** **row**: **parent**. **ℹ️** **Same** **as** **“last** **tool** **under** **previous** **Details”** **when** **children** **are** **present** **in** **the** **flat** **`TreeListModel`** (**`autoexpand`** **true** **in** **`set_runner`**); **spike** **if** **collapse** **differs**.
- **🔷** Spurious **ColumnView.activate** does **not** scroll the transcript.
  - Document in **§3** fences: **`activate`** **not** **`connect`**ed for scroll **vs** **`connect`** **no-op** for scroll.
  - **ℹ️** Optional **`ListItem.activatable = false`** (**§3**, **under review**) suppresses **GTK**-driven **`activate`** on **double-click** / **Enter** if needed.
- **🔷** **`msg_idx < 0`**: no crash and no scroll.
  - **ℹ️** Any alternative (**e.g.** scroll to bottom) needs **explicit** approval in plan fences.

```vala
			this.click_gesture = new Gtk.GestureClick();
			this.column_view.add_controller(this.click_gesture);
			this.click_gesture.released.connect((n_press, x, y) => {
				this.column_view.grab_focus();
				var picked = this.column_view.pick((float) x, (float) y, Gtk.PickFlags.DEFAULT);
				if (picked == null) {
					return;
				}
				Gtk.Widget? w = picked;
				while (w != null) {
					if (w is Gtk.ListItem) {
						var row = (Gtk.TreeListRow) ((Gtk.ListItem) w).item;
						this.select_row(row.get_position());
						break;
					}
					w = w.get_parent();
				}
			});

			this.key_controller = new Gtk.EventControllerKey();
			this.key_controller.propagation_phase = Gtk.PropagationPhase.CAPTURE;
			this.column_view.add_controller(this.key_controller);
			this.key_controller.key_pressed.connect((keyval, keycode, state) => {
				var up = keyval == Gdk.Key.Up || keyval == Gdk.Key.KP_Up;
				var down = keyval == Gdk.Key.Down || keyval == Gdk.Key.KP_Down;
				if (!up && !down) {
					return false;
				}
				var m = this.progress_selection.model;
				var n = m.get_n_items();
				var pos = this.progress_selection.selected;
				if (up) {
					if (pos == Gtk.INVALID_LIST_POSITION || pos == 0) {
						return true;
					}
					var row = (Gtk.TreeListRow) m.get_item(pos);
					if (row.get_depth() > 0) {
						this.select_row(row.get_parent().get_position());
					} else {
						this.select_row(pos - 1);
					}
					return true;
				}
				if (pos == Gtk.INVALID_LIST_POSITION) {
					this.select_row(0);
					return true;
				}
				var row = (Gtk.TreeListRow) m.get_item(pos);
				var pi = (ProgressItem) row.get_item();
				if (pi.children.get_n_items() > 0) {
					row.expanded = true;
				}
				if (pos + 1 < n) {
					this.select_row(pos + 1);
				}
				return true;
			});
```

### 4. `liboccoder/Skill/Factory.vala` — **`progress_view.window`** **assignment**

#### Context (**where** **`host`** **comes from**)

- **ℹ️** **`Factory.activate(GLib.Object window)`** is the **Skills** agent **entry** when the session’s agent UI is shown. The framework passes **`window`**: the app’s main chat shell, which **implements** **`OLLMchat.ChatUserInterface`**.
- **ℹ️** At the **top** of **`activate`**, the code already does **`var host = (OLLMchat.ChatUserInterface) window;`** — that **`host`** is the **same** object used for **`host.above_input_widget()`**, **`host.tab_view()`**, **`host.session_agent()`**, etc. It is **not** a new global; it is the **per-activation** reference to the **window** that owns the chat pane and can forward **`scroll_to_message`** ( **`Window`** in **`ollmapp`** implements **`ChatUserInterface.scroll_to_message`** per **§2**).
- **ℹ️** **`var runner = (OLLMcoder.Skill.Runner) host.session_agent();`** is already there; **`set_runner(runner)`** runs in that **same** method. **`this.progress_view.window = host`** stores the **`ChatUserInterface`** on **`ProgressView`** so **`select_row`** can call **`this.window.scroll_to_message(msg_idx)`** — **`liboccoder`** stays on the **`ChatUserInterface`** API (**no** **GTK** in **`libollmchat`**).
- **🚫** Do **not** add a **`host`** field on **`Factory`**; use the **local** **`host`** in **`activate`** only (or **set** **`progress_view.window = null`** in **`deactivate`** if you need to avoid a stale pointer — **spike**).
