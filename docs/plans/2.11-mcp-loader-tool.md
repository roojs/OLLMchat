# 2.11. MCP Loader Tool

## Overview

Implementation of MCP (Model Context Protocol) loader and client so OLLMchat can start or connect to MCP servers, discover their tools, and expose those tools to the chat agent as first-class `Tool.BaseTool` instances.

## Folder and namespace

- **Folder:** `libocmcp/` (same pattern as `liboctools/`, `libocvector/`).
- **Namespace:** `OLLMmcp.*` (e.g. `OLLMmcp.Registry`, `OLLMmcp.Client`, `OLLMmcp.ToolWrapper`).
- **Front-end:** App uses `OLLMmcp.Registry` like tools and vector: add `mcp_registry` on Application (and in ollmchat-cli), call `init_config()` and `setup_config_defaults(config)` at startup, and in Window call `app.mcp_registry.fill_tools(this.history_manager, this.project_manager)` after `tools_registry.fill_tools()` and `vector_registry.fill_tools()`.

## What does what (and what extends our tool classes)

- **Config**  
  - One MCP server entry from mcp.json (id, enabled, transport, command/args/url, network).  
  - Data only.

- **Registry**  
  - App entry point. Loads mcp.json, **is the client factory** (creates Client from Config: switch on transport → Stdio.Client or Http.Client), calls Loader in `fill_tools()`.  
  - No Stdio.Factory / Http.Factory; Registry does it.

- **Client** (abstract)  
  - One connection to one MCP server: connect, disconnect, list_tools, call_tool.  
  - **Stdio.Client** — extends Client. Subprocess, JSON-RPC over stdin/stdout.
  - **Http.Client** — extends Client. JSON-RPC over HTTP to url.

- **ToolFactory**  
  - From Client.list_tools(): MCP tool metadata (name, description, input_schema). `create_wrapper(client, server_id)` → BaseTool.  
  - Tool factories are the only factories; they create wrappers that call the client to do the work.

- **ToolWrapper**  
  - **Extends `OLLMchat.Tool.BaseTool`.** One per MCP tool. `execute(parameters)` → `client.call_tool(tool_name, parameters)`. Name = `mcp:{server_id}:{tool_name}`.

- **Loader**  
  - Given configs: for each enabled config, Registry creates Client, Loader gets tool factories from client, creates ToolWrappers via tool factories, registers with Manager.

## First test MCP: Chrome (command-line)

A practical first MCP to support is the **Chrome MCP** (or similar browser/IDE MCP) that runs as a **command-line (stdio)** server. That drives the need for stdio transport, bwrap sandboxing, and optional network access (e.g. `network: true` in config) so the server can talk to the browser.

---

## Requirements

### 1. Loader: Start vs Connect

MCP supports two transports; the loader must support both:

| Transport | Behaviour | Config |
|-----------|------------|--------|
| **stdio** | Client **starts** the MCP server as a subprocess (e.g. Chrome MCP, or `npx -y @modelcontextprotocol/server-filesystem`). Communication is over stdin/stdout (JSON-RPC, newline-delimited). **Runs under bwrap** with minimal file access by default (see §1b). | Command + args + optional env + optional `network` |
| **HTTP/SSE** | Client **connects** to an already-running server at a URL. No process lifecycle. | Base URL |

- **stdio**: spawn process (via bwrap when available), keep it alive for the session, pipe stdin/stdout for JSON-RPC.
- **HTTP**: no spawn; connect to `url`, use HTTP POST for requests (and optional SSE for streaming).

So we need a single “loader” that can either **start** (stdio) or **connect** (HTTP) depending on config.

### 1b. Sandboxing (stdio servers)

- **Command-line (stdio) MCP servers should run under bwrap** when available, with **minimal file access** by default (e.g. read-only root, no writable binds except what's needed for the subprocess, or a minimal tmp).
- **Network:** Blocked by default (e.g. bwrap `--unshare-net`). Servers that need network (e.g. Chrome MCP talking to a browser) enable it via config: **`network: true`** in the server entry in `mcp.json`. Same idea as RunCommand's network option; for MCP we make it a config-only opt-in per server (user edits mcp.json), with optional permission prompt later if desired.
- Config can include other sandbox knobs later (e.g. allow list of bind mounts) if needed.

---

### 2. Configuration

**Dedicated MCP config file:** `~/.config/ollmchat/mcp.json` — standalone file; keeps MCP server list separate from main app config.

**File format: just an array** — the JSON file is a single array of server entries (no wrapper object). Each element is one MCP server (one “service”).

**Config class = single service:** Have a single **`OLLMmcp.Config`** class that represents one MCP server entry (id, enabled, transport, command/args/env or url, network, etc.). The file deserializes to an array of `Config` instances.

**Flow:** Registry loads mcp.json → array of Config. For each enabled Config, Registry creates the Client (switch on transport: Stdio.Client or Http.Client). Loader gets tool factories from client (list_tools), each tool factory creates a ToolWrapper that calls the client; register those. No separate client factories — Registry is the client factory.

**Proposed schema for `mcp.json` (root is array):**

```json
[
  {
    "id": "chrome",
    "enabled": true,
    "transport": "stdio",
    "command": "npx",
    "args": ["-y", "@modelcontextprotocol/server-chrome"],
    "env": {},
    "network": true
  },
  {
    "id": "filesystem",
    "enabled": true,
    "transport": "stdio",
    "command": "npx",
    "args": ["-y", "@modelcontextprotocol/server-filesystem", "/tmp/allowed"],
    "env": {}
  },
  {
    "id": "mysql",
    "enabled": true,
    "transport": "http",
    "url": "http://127.0.0.1:3000"
  }
]
```

**Per-entry (`Config` class) fields:**
- **id**: unique key for this server (used for tool name prefix).
- **enabled**: if false, skip this server (no client, no tools).
- **transport**: `"stdio"` | `"http"`.
- **stdio**: `command`, `args` (array), optional `env` (map). Optional **`network`** (boolean, default false): when true, allow network in bwrap (omit `--unshare-net`).
- **http**: `url` (required).

Optional later: per-server timeouts, auth headers for HTTP, additional sandbox options. Alternative: later add an `mcp_servers` section to `config.2.json` or a dedicated MCP settings page.

### 3. Ability to Fetch Tools

- After **starting** (stdio) or **connecting** (HTTP), the client must call the MCP **tools/list** (or equivalent) capability to get the list of tools.
- Each tool has: `name`, `description`, `inputSchema` (JSON Schema for parameters).
- We need to map these into our tool model:
  - `name` → tool name. To avoid clashes across servers, use a prefixed name: `mcp:{server_id}:{tool_name}` (e.g. `mcp:mysql:execute_sql`).
  - `description` → tool description.
  - `inputSchema` → our `parameter_description` / `Function.parameters`. We already have `ParamParser` and a convention for `@param name type Description`; we need a small adapter from JSON Schema to that or to `Function.parameters` directly.

So the “loader” is responsible for: connect/start → **fetch tools** → **register** one `BaseTool` per MCP tool (see below).

---

### 4. Ability to Call Tools

- When the agent selects a tool (e.g. `mcp:mysql:execute_sql`), we need to **call** the MCP server’s **tools/call** with that tool’s name and the provided arguments.
- Flow:
  1. Agent receives a tool call for `mcp:mysql:execute_sql` with arguments.
  2. We resolve `server_id` + `tool_name` from the registered tool (or from the prefixed name).
  3. We send a JSON-RPC request to the correct MCP server (stdio or HTTP): `tools/call` with `name`, `arguments`.
  4. We return the result (and/or error) as the tool result string to the agent.

So each MCP tool is exposed as a `Tool.BaseTool` subclass (or a single generic “MCP tool” that dispatches by name): `execute(Json.Object parameters)` runs the MCP `tools/call` and returns the string result.

### 5. Fit into Existing Infrastructure

| Piece | How MCP fits |
|-------|----------------|
| **Config** | Read `~/.config/ollmchat/mcp.json` on startup (or when opening MCP settings). Optional: register an “mcp_loader” entry in `Config2.tools` later for enable/disable and per-server config. |
| **Tool registration** | Same as other tools: `History.Manager.register_tool(tool)`. Each MCP tool is a `Tool.BaseTool` instance (e.g. `MCPToolWrapper`) with `name`, `description`, `parameter_description`/`function.parameters`, and `execute()` calling the MCP client. |
| **Registry** | `OLLMmcp.Registry` loads mcp.json → array of Config. **Registry is the client factory**: for each enabled Config it creates the Client (switch on transport → Stdio.Client or Http.Client). Then Loader gets tool factories from client (list_tools), creates ToolWrappers (tool factories call the client to do the work), registers them. No Stdio.Factory / Http.Factory. |
| **Lifecycle** | **stdio:** Start subprocess when loading MCP servers (e.g. when Window initializes tools); stop when app exits or when disabling MCP. **HTTP:** No process to manage; just connect/disconnect. |
| **BaseTool** | Each MCP tool is a thin wrapper: `name` = `mcp:{server_id}:{tool_name}`, `description` and params from the server; `execute(parameters)` → MCP client `tools/call` → return result. |
| **Config2.tools** | Optional: one entry per server (e.g. `mcp_filesystem`, `mcp_mysql`) with `active` and maybe server-level options; or one global `mcp` config that lists enabled servers. Tool-level enable/disable can be done by not registering a tool when the server is disabled. |

**Suggested flow:**

1. **Application startup:** No MCP yet; main config and tool config types load as today.
2. **Window / tool init:** After `tools_registry.fill_tools()` and `vector_registry.fill_tools()`, call `mcp_registry.fill_tools()` (same signature as the others). Registry reads `mcp.json`, for each enabled server starts or connects, fetches tools, registers one `OLLMmcp.ToolWrapper` per tool via `manager.register_tool()`. Same `manager.tools` and `chat.tools` as today.
3. **Agent tool call:** Unchanged: agent gets tool call by name, looks up in `chat.tools`, calls `tool.run()` → `execute(parameters)`. For `OLLMmcp.ToolWrapper`, `execute()` delegates to MCP client.
4. **Shutdown:** For stdio servers, terminate subprocesses; for HTTP, close connections.

**Naming:** Use `mcp:{server_id}:{tool_name}` so that (a) tools are unique across servers, (b) we can route to the right server and tool in `execute()` without extra maps.

## Transport: Registry is the client factory

- **Client** (abstract): `connect()`, `disconnect()`, `list_tools()`, `call_tool()`. Concrete **Stdio.Client** and **Http.Client** in their own directories.
- **No ClientFactory / Stdio.Factory / Http.Factory.** Registry creates the Client: switch on `config.transport` → `new Stdio.Client(config)` or `new Http.Client(config)`. Add a branch when adding a transport.
- **Tool factories** (from `Client.list_tools()`): each creates a ToolWrapper; the wrapper calls the client to do the call. The only "factories" are the tool factories — they create the BaseTool wrappers that delegate to the client.

### Summary

| Type | Role |
|------|------|
| `OLLMmcp.Client` | Base: connect, disconnect, list_tools, call_tool. |
| `OLLMmcp.Stdio.Client` | Extends Client; stdio subprocess, JSON-RPC. |
| `OLLMmcp.Http.Client` | Extends Client; HTTP JSON-RPC to url. |
| Registry | Loads config; **creates Client** (switch on transport); drives Loader. |
| ToolFactory | From list_tools; creates ToolWrapper (which calls client). |
| ToolWrapper | Extends BaseTool; execute() → client.call_tool(). |

## Implementation Details

### Features

- MCP server configuration (read from `~/.config/ollmchat/mcp.json`).
- **Client** (abstract) + **Stdio.Client**, **Http.Client**. Registry creates Client (switch on transport); no client factory classes.
- MCP protocol: initialize, `tools/list`, `tools/call`, JSON-RPC message handling behind `OLLMmcp.Client` API.
- Tool wrapper: one `Tool.BaseTool` per MCP tool; name prefix `mcp:{server_id}:{tool_name}`; `execute()` calls the server’s `Client.call_tool()`.
- Map MCP tool schema (name, description, inputSchema) to our `Function` / parameter_description.
- Lifecycle: start/connect when filling tools; stop stdio processes when app exits.

### Files (by phase: 2.11.1 → 2.11.2 → 2.11.3; meson in 2.11.4)

**Root (libocmcp/)** – namespace **OLLMmcp**:

- `libocmcp/Config.vala` – One MCP server (single service). Load `mcp.json` → array of Config.
- `libocmcp/Client.vala` – Abstract `OLLMmcp.Client`: connect, disconnect, list_tools, call_tool.
- `libocmcp/ToolFactory.vala` – MCP tool metadata; `create_wrapper(client, server_id)` → BaseTool.
- `libocmcp/ToolWrapper.vala` – Extends `Tool.BaseTool`; execute() → client.call_tool().
- `libocmcp/Loader.vala` – For each enabled Config, gets Client (from Registry), list_tools → tool factories, creates ToolWrappers, registers with Manager.
- `libocmcp/Registry.vala` – Loads config; **creates Client** (switch on config.transport → Stdio.Client or Http.Client); `init_config()`, `setup_config_defaults()`, `fill_tools()` (uses Loader). No ClientFactory / Stdio.Factory / Http.Factory.

**Stdio (libocmcp/Stdio/):**

- `libocmcp/Stdio/Client.vala` – Stdio.Client extends Client; subprocess, JSON-RPC over stdin/stdout.

**Http (libocmcp/Http/):**

- `libocmcp/Http/Client.vala` – Http.Client extends Client; HTTP JSON-RPC to url.

Optional: a “meta” tool (e.g. in libocmcp or as a wrapped tool) that lists or manages MCP servers (e.g. “list_mcp_tools”); not required for the core “load and call” behaviour.
