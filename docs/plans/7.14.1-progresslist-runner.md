# 7.14.1 — `ProgressList` + `Runner.progress` + adapters (phase 1)

**Status:** proposed

**Parent:** `docs/plans/7.14-task-progress-tree-ui.md`

**Checklist:** `.cursor/rules/CODING_STANDARDS.md` — **Checklist for all plans**.

---

## Purpose

- Introduce **`ProgressList`** and wire **`Runner.progress`** (ctor takes **`Runner`**).
- **`Runner`** calls **`this.progress.replacePending()`**, **`this.progress.addComplete()`**, and related ops at real transitions (**`handle_task_list`**, iteration, **`run_child`**, **`Tool.run`**, etc.).
- **Adapters:** **`Runner`** is **not** one row shape — use **adapter objects** (owned by **`ProgressList`**) for **task-list create**, **task-list iterate**, so the model stays uniform. Same for **`Details`** / **`Tool`** if needed.
- Sketch **`ProgressItem`**: bindable-oriented surface (see parent **Stage column** + **`7.14.3`**).

**No** polished GTK yet beyond what **`7.14.2`** adds.

---

## Scope

| In scope | Out of scope |
|----------|----------------|
| **`ProgressList`** API, **`Runner`** hooks, brutal refill after iterate | **`ProgressView`** shell (**`7.14.2`**) |
| Adapter types for **`Runner`**-backed rows | **`to_human()`** polish (**`7.14.3`**) |

---

## Acceptance criteria

1. **`Runner`** exposes **`this.progress`**; **`ProgressList`** constructed with **`Runner`** reference.
2. **`replacePending()`** / **`addComplete()`** (or equivalent names) called from **`Runner`** at agreed boundaries; after **task-list iteration**, stale rows removed and model refilled.
3. **Adapter** pattern documented in code: create/iterate rows do not poke **`Runner`** from renderers directly.
