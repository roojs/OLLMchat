# 7.14.1 — `ProgressList` + `Runner.progress` + `ToolList` (phase 1)

**Status:** proposed

**Parent:** `docs/plans/7.14-task-progress-tree-ui.md`

**Checklist:** `.cursor/rules/CODING_STANDARDS.md` — **Checklist for all plans**.

**Layout:** `docs/guide-to-writing-plans.md`

---

## Purpose

- Introduce **`ProgressList`** (**`Runner`** holds **`this.progress`**; **`ProgressList`** holds **`weak Runner`** to avoid **`Runner` ↔ `ProgressList` ref cycle). **`ProgressList`** **`implements GLib.ListModel`** — same pattern as **`OLLMfiles.ProjectList`** (`libocfiles/ProjectList.vala`): private **`Gee.ArrayList`** backing store, **`get_item_type`** / **`get_n_items`** / **`get_item`**, and **`items_changed(position, removed, added)`** on mutations so **`7.14.2`** can wire **`Gtk.ColumnView`** (or **`Gtk.ListView`**) with **`Gtk.SingleSelection(this.progress)`**, like **`HistoryBrowser`** (`libollmchatgtk/HistoryBrowser.vala`) chains models + selection.
- **No trivial one-liner wrappers** on **`ProgressList`** — callers invoke **`clear_pending()`**, **`add_pending()`**, **`add_completed()`**, or **`rebuild()`** directly where appropriate.
- **Incremental updates (default):** **`completed`**-backed rows are **append-only** and **not** removed when **`pending`** changes. **`clear_pending()`** drops only the **pending** segment; **`add_pending()`** appends **`Details`** only for the current **`runner.pending`**. **`add_completed(Step)`** inserts **`Details`** for **one** step **before** the pending segment when that step becomes **completed**. **Do not** append **`Tool`** as top-level **`ListModel`** rows — **tools live under **`Details`** via **`ToolList`**. Mutations emit **`items_changed`** (see **`ProjectList`**); **do not** full **`rebuild()`** on every transition.
- **`rebuild()`** — optional **full** resync when incremental state is wrong or unknown (e.g. **`7.14.5`** restore / replay); callers use it **only** when necessary, not as the default path.
- **`ProgressList`** (as **`ListModel`**) reports **`get_item_type()`** = **`ProgressItem`**; **row instances** in phase **1** are **`Details`** only (completed segment + pending segment). **No** **`create_row`** / **`iterate_row`** / **`ProgressItemRunner`**. **Task-list create** / **iterate** phase (try loop, awaiting LLM, “Creating” / “Retrying”) lives on **`Runner`** (or **`7.14.2`**), not as fake **`ListModel`** rows.
- **`Details`** **`implements ProgressItem`** for the **step/detail** row. **`Tool`** **`implements ProgressItem`** for **child** rows when **`7.14.2`** binds **`ToolList`** — **not** duplicated as top-level **`ProgressList`** rows.
- **`ToolList`** (new): **`GLib.Object`** that **owns** a **`Gee.ArrayList<Tool>`** internally and **centralizes** mutations / notifications for tool runs (same *role* as a list store: one place to manage the collection). **Replaces** raw **`Gee.ArrayList<Tool>`** fields on **`Details`** (**`tools`**, **`exec_runs`**) so tool list lifecycle is explicit. **No Gtk** in **`liboccoder`** — this is a model-side manager; **`7.14.2`** may connect it to **`GListModel`** / **`Gtk`** widgets if needed.
- **`ProgressItem.kind`** → **`TaskKind`** (**`TASK_LIST_CREATE`**, **`TASK_LIST_ITERATION`**, **`DETAIL`**, **`TOOL`**) — **one enum** for progress UI; no separate **`RowKind`**. **`TASK_LIST_ITERATION`** aligns by name with **`PhaseEnum.TASK_LIST_ITERATION`**. **`PhaseEnum`** remains for **markdown / `ValidateLink` / wire ids**; **`TaskKind.from_phase(PhaseEnum)`** bridges where a row tracks a phase (**7.14.3** may unify **`to_human()`** inputs).
- Phase **1** may use **stub** strings for **`ProgressItem`** **`stage`** / **`title`** on **`Details`** / **`Tool`** until **`7.14.3`** (**`to_human()`**, real labels).

**No** GTK in **`liboccoder`** — model only (**`7.14.2`** adds **`ProgressView`**).

**Naming:** do **not** prefix fields or locals with **`_`** — use plain names (e.g. **`rows`**, **`pending_begin`**) per project style.

## Acceptance criteria

1. **`Runner.progress`** is non-null after construction; **`ProgressList`** uses **`weak Runner`**.
2. When **`runner.pending`** is replaced or resynced, **`Runner`** calls **`clear_pending()`** then **`add_pending()`** (not a full list rebuild) so stale pending rows drop without touching completed-backed rows.
3. When a **step** becomes **completed**, **`Runner`** calls **`add_completed(step)`** once — **no** rescan of the entire **`completed`** list each time unless **`rebuild()`** is used deliberately.
4. **`ProgressList`** **`ListModel`**: **`get_item_type()`** = **`ProgressItem`**; phase **1** **instances** are **`Details`** only. **`Tool`** rows appear only via **`Details`’** **`ToolList`** (not top-level model rows).
5. **`ToolList`** replaces **`Details`’** raw **`ArrayList<Tool>`** fields; **`build_exec_runs()`** / **`run_exec()`** go through **`ToolList`**’s API.

### 2. `liboccoder/Task/ToolList.vala` — new

**Role:** Replace **`Details`’** raw **`Gee.ArrayList<Tool>`** fields (**`tools`**, **`exec_runs`**) with **one managed type** per field (or share one **`ToolList`** type for both — two instances on **`Details`**). **`ToolList`** is **`GLib.Object`**, owns **`Gee.ArrayList<Tool>`** internally, and exposes **clear / add / get / size** (and whatever **`build_exec_runs()`** / **`run_exec()`** need) so mutations go through one place. **Emit **`notify`** or a custom signal when the backing list changes** so **`7.14.2`** can bind without scanning **`Details`** ad hoc.

This is **not** **`Gtk.ListStore`** (**no Gtk** in **`liboccoder`**); it is the **model-side** “store” pattern — **Object + ArrayList** — and may be wrapped or bridged to **`GListModel`** in GTK code later if useful.

#### Add

```vala
/*
 * Copyright (C) 2026 Alan Knowles <alan@roojs.com>
 */

namespace OLLMcoder.Task
{

/**
 * Manages a list of Tool rows for a Details (parsed tools vs exec runs).
 * Backing store is internal; callers use methods that keep notifications consistent.
 */
public class ToolList : GLib.Object
{
	private Gee.ArrayList<Tool> rows = new Gee.ArrayList<Tool>();

	public uint size {
		get { return this.rows.size; }
	}

	public Tool get_at(int index) {
		return this.rows.get(index);
	}

	public void clear()
	{
		if (this.rows.size == 0) {
			return;
		}
		this.rows.clear();
		this.notify_property("size");
	}

	public void add(Tool t)
	{
		this.rows.add(t);
		this.notify_property("size");
	}

	// Add iterators, add_all, etc. as needed when porting Details off raw ArrayList.
}

}
```

*Exact API must satisfy all current **`this.tools` / `this.exec_runs`** call sites in **`Details.vala`** — grep before merge.*

### 3. `liboccoder/Task/ProgressList.vala` — new

**Pattern:** Match **`OLLMfiles.ProjectList`** — **`public class ProgressList : GLib.Object, GLib.ListModel`**, private **`Gee.ArrayList<ProgressItem>`**, **`get_item_type()`** → **`typeof(ProgressItem)`** (not **`typeof(Details)`**). **`items_changed`** after every mutation. **`Gtk.ColumnView`** / **`Gtk.ListView`** in **`7.14.2`** take **`Gtk.SingleSelection(this)`** (or **`NoSelection`** if no selection) — same idea as **`HistoryBrowser`** list stack.

#### Add

```vala
/*
 * Copyright (C) 2026 Alan Knowles <alan@roojs.com>
 * SPDX-License-Identifier: LGPL-3.0-or-later
 */

namespace OLLMcoder.Task
{

/**
 * ListModel of **`ProgressItem`** rows (completed segment, then pending). Phase **1** only adds **`Details`**
 * instances; **`get_item_type()`** still reports **`ProgressItem`** so factories bind to the interface.
 * Task-list create / iterate UI state is **not** here — **Runner** / **7.14.2**.
 */
public class ProgressList : GLib.Object, GLib.ListModel
{
	public weak OLLMcoder.Skill.Runner runner { get; private set; }

	private Gee.ArrayList<ProgressItem> rows = new Gee.ArrayList<ProgressItem>();

	/** Index of first pending-backed row; equals rows.size when there is no pending segment. */
	private int pending_begin = 0;

	public ProgressList(OLLMcoder.Skill.Runner r)
	{
		Object();
		this.runner = r;
	}

	public Type get_item_type()
	{
		return typeof(ProgressItem);
	}

	public uint get_n_items()
	{
		return (uint) this.rows.size;
	}

	public Object? get_item(uint position)
	{
		if (position >= this.rows.size) {
			return null;
		}
		return this.rows[(int) position];
	}

	/**
	 * Full resync — rare (restore/replay). Emit **items_changed** like **ProjectList.append**:
	 * after clear: **items_changed(0, old_n, 0)**; after repopulate: **items_changed(0, 0, new_n)**.
	 */
	public void rebuild()
	{
		uint old_n = (uint) this.rows.size;
		this.rows.clear();
		if (old_n > 0) {
			this.items_changed(0, old_n, 0);
		}
		this.pending_begin = 0;
		var r = this.runner;
		if (r == null) {
			return;
		}
		int added = 0;
		foreach (var step in r.completed.steps) {
			foreach (var d in step.children) {
				this.rows.add(d);
				added++;
			}
		}
		this.pending_begin = this.rows.size;
		foreach (var step in r.pending.steps) {
			foreach (var d in step.children) {
				this.rows.add(d);
				added++;
			}
		}
		if (added > 0) {
			this.items_changed(0, 0, (uint) added);
		}
	}

	/** Truncate pending tail; emit **items_changed(pending_begin, removed, 0)**. */
	public void clear_pending()
	{
		int n = this.rows.size - this.pending_begin;
		if (n <= 0) {
			return;
		}
		for (int i = 0; i < n; i++) {
			this.rows.remove_at(this.rows.size - 1);
		}
		this.items_changed((uint) this.pending_begin, (uint) n, 0);
	}

	public void add_pending()
	{
		var r = this.runner;
		if (r == null) {
			return;
		}
		uint pos = (uint) this.rows.size;
		int added = 0;
		foreach (var step in r.pending.steps) {
			foreach (var d in step.children) {
				this.rows.add(d);
				added++;
			}
		}
		if (added > 0) {
			this.items_changed(pos, 0, (uint) added);
		}
	}

	public void add_completed(OLLMcoder.Task.Step step)
	{
		uint pos = (uint) this.pending_begin;
		int n = 0;
		foreach (var d in step.children) {
			this.rows.insert(this.pending_begin, d);
			this.pending_begin++;
			n++;
		}
		if (n > 0) {
			this.items_changed(pos, 0, (uint) n);
		}
	}

	public void clear()
	{
		uint old_n = (uint) this.rows.size;
		this.rows.clear();
		this.pending_begin = 0;
		if (old_n > 0) {
			this.items_changed(0, old_n, 0);
		}
	}
}

}
```

* **`rebuild()`** uses two **`items_changed`** emissions (remove all, then add all) — valid for **`GLib.ListModel`**; **`ProjectList`**-style single append would use one emission per row if you prefer. **`Details`** implements **`ProgressItem`**; **`get_item`** returns **`Object?`** (cast to **`Details`** when needed).

## Implementation notes (read before merge)

1. **`pending_begin`** invariant: index of the first **pending**-backed **`Details`** row; **`clear_pending()`** truncates **`rows`** at that index; **`add_completed`** inserts before it and advances it; **`add_pending()`** appends pending steps after it. Every mutation emits **`items_changed`** (see **`libocfiles/ProjectList.vala`**). **Do not** prefix fields with **`_`** — use names like **`rows`**, **`pending_begin`**.
2. **First sync** when **`completed`** / **`pending`** already contain steps (e.g. first **`handle_task_list`**): either **`rebuild()`** once or loop **`add_completed`** for existing completed steps then **`add_pending()`** — do not rely on incremental hooks alone without an initial baseline.
3. **`on_replay`** / session load: **`progress.rebuild()`** or **`progress.clear()`** then incremental repopulate — **`7.14.5`**.
4. **`Gee.ArrayList.insert`:** verify API on the project’s **Gee** version; when chaining **`clear_pending`** + **`add_pending`**, emit **`items_changed`** for each logical step (or merge if GTK accepts a single batched update).
5. **`ToolList`:** grep **`Details`** for **`tools`** / **`exec_runs`**; extend **`ToolList`** until **`build_exec_runs()`** and **`run_exec()`** behave identically; **`ResultParser`** and any other mutators must use **`ToolList`** too.
6. Run **`meson compile`** after reordering **`meson.build`** — **`Tool`** → **`ToolList`** → **`Details`** → **`ProgressList`** as needed.
