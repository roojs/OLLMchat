## AST on wire (daemon — not V2 client)

**Decision:** `OLLMfiles.Tree` (tree-sitter path → line range for **one file**) is **backend-only**. V2 `ProjectManager` has **no** `tree_factory` / `tree_cache`.

Today (shipping, in-process) callers use `project_manager.tree_factory(file)` then `tree.parse()` / `lookup_path()`:

| Caller | Use |
| ------ | --- |
| `liboctools/ReadFile/Request.vala` | `read_file` tool + `ast_path` → line range |
| `libocfiles/FileChange.vala` | Agent edits anchored to AST path |
| `liboccoder/Task/ResolveLink.vala` | Task refs `[[file#ast]]` |

**Not used by editor UI** (`SourceView`, `Window`) — no frontend direct access.

**Target after cutover:**

- Daemon keeps `tree_factory` + cache on server `ProjectManager` (or per-file cache on `File`).
- Wire: e.g. **`File.ast.lookup`** — `params.path`, `params.ast_path` → result `{ start_line, end_line, comment_start }` (exact shape TBD).
- **`File.read`** with `ast_path` may fold into lookup + read slice on server, or client calls lookup then read by lines.
- Tools / task code call **RPC** and consume **results** — they do not hold or parse `Tree` locally.
- Vector `Indexer` already uses `new Tree(file)` on daemon side — same process.

**Shipping `libocfiles/ProjectManager.tree_factory`:** keep until tools migrate to RPC; remove from client when cutover complete.
