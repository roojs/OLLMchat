## Purpose

- **🔷** Confirm Ubuntu **mingw / sqgipkg** Windows cross-build is **not** viable once the browser tool links **`webview2-gtk`** (WebView2 / COM / `WebView2Loader`).
- **🔷** Split release CI: keep **Linux / Debian / AppImage** on Ubuntu; add a **separate Windows job** that builds on a **GitHub Windows runner** (MSYS2 UCRT64).
- **🔷** Mirror patterns that already work:
  - **webview2-gtk** ships its own NSIS + pacman assets from **`windows-latest`** + **`msys2/setup-msys2`**
  - **Snappr2** builds the **same library + app** on a real Windows host (SSH/rsync for day-to-day; same MSYS2 UCRT64 stack)
- **⏳** **🔷** Ship Windows installer / app as **`OLLMchat`** — not `remote-only` in the filename (see **Windows naming** below).
- **⏳** **🔷** Remove `sqgipkg --target win-nsis` from the Ubuntu release job once the Windows job is green.
- **💩** Optional later: Linux-side `agent-remote-build.sh` (Snappr-shaped) for local Windows QA — not required for CI if GitHub `windows-latest` is the release path.

### Windows naming

- **🔷** There is **no** second Windows SKU (no local-GGUF / libllama Windows build today).
- **🔷** `remote-only` on Linux means “no local inference engine”; Windows is that product by default — do **not** put `remote-only` in Windows artifact names.
- **🔷** Installed / portable app binary: **`OLLMchat.exe`**.
- **🔷** Installer filename is **versioned** (Snappr pattern): **`OLLMchat-<version>-Setup.exe`**, with `<version>` from `meson.build` `project(… version: '…')` (same source Snappr reads in `build-windows-installer.sh`).
- **🔷** Pass that version into NSIS as **`PRODUCT_VERSION`** (Snappr: `-DPRODUCT_VERSION=…` + `OUTFILE=…/OLLMchat-${VERSION}-Setup.exe`).
- **🔷** Drop today’s unversioned `OLLMchat-remote-only-Setup.exe` name when cutting over to the native Windows job.
- **ℹ️** Snappr also refuses to overwrite an existing same-version setup exe (forces a meson version bump) — optional but nice for release hygiene.
- **💩** If local GGUF is ever ported to Windows, **then** consider a rename / second SKU — not now.

## Why cross-compile breaks

- **🔷** OLLMchat Windows release today: Ubuntu **`ubuntu-24.04`** → `mingw-w64` + **`sqgipkg --target win-nsis`** ([`release.yml`](../../.github/workflows/release.yml), [`creating-releases.md`](../creating-releases.md)).
- **🔷** Browser tool Windows backend is **`webview2-gtk`** (`#if WINDOWS` in `libocwebkit`), not WebKitGTK.
- **ℹ️** Sibling library states plainly: **no Linux cross-compile path** for webview2-gtk itself (`docs/build-this-library.md`).
- **🔷** WebView2 needs native Windows headers / SDK vendoring (`vendor-webview2-sdk.sh`), `WebView2Loader.dll`, and a Win32 HWND parent — that stack is what Snappr and the library build on **MSYS2 UCRT64 on Windows**.
- **💩** Wine smoke of an old GTK-only cross bundle may still work for non-browser paths; it is **not** a release gate for WebView2.

## Reference builds (what to copy)

### webview2-gtk — GitHub Windows release (canonical CI shape)

- **ℹ️** Workflow: `webview2-gtk/.github/workflows/release.yml`
  - `runs-on: windows-latest`
  - `defaults.run.shell: msys2 {0}`
  - `msys2/setup-msys2@v2` with **`msystem: UCRT64`**
  - pacman toolchain: gcc, vala, meson, ninja, pkg-config, gtk4, libgee, libsoup3, curl, nsis
  - Build NSIS installer + pacman `.pkg.tar.zst`
  - Upload artifact; on `v*` tags attach to GitHub Release
- **ℹ️** Consumer template: `scripts/sample-github-build-windows.yml`
  - Same runner / MSYS2 setup
  - Build **webview2gtk** into a prefix (or clone repo)
  - Set `PKG_CONFIG_PATH`
  - Meson-build the app
  - Package with `copy-exe-runtime-dlls` / `package-windows` (exe + GTK DLLs + `WebView2Loader.dll`)
- **ℹ️** End users still need **WebView2 Runtime** (Evergreen) on the PC — document in release notes / installer text like Snappr.

### Snappr2 — remote Windows machine (canonical product build)

- **ℹ️** Day-to-day: Linux → SSH host `snappr-win` → rsync app + webview2-gtk → MSYS2 UCRT64 bash
- **ℹ️** Order: build/install **webview2-gtk** to a staging prefix → `meson setup` snappr with `-Dwebview2gtk_prefix=…` → `package-snappr.sh` → optional `build-windows-installer.sh` (NSIS)
- **🔷** OLLMchat CI should follow the **same compile/link/package order**, but on **GitHub `windows-latest`** instead of a private SSH box (unless we later add a self-hosted runner).

## Target CI shape

### Job split

- **🔷** **`build` (Ubuntu)** — keep:
  - Debian packages
  - AppImages (x86_64 + aarch64)
  - Android parallel job unchanged
  - **Remove** Windows NSIS step and Windows meson cache paths once Windows job owns the installer
- **🔷** **`build-windows` (new)** — `windows-latest` + MSYS2 UCRT64:
  - Checkout OLLMchat
  - Obtain **webview2-gtk** (submodule, `actions/checkout` of sibling repo at pinned ref, or `pacman -U` published package from library releases)
  - Vendor SDK + build/install library **or** install pacman package
  - `meson setup` / `meson compile` OLLMchat with **`-Dlocal_gguf=disabled`** (no Windows local inference yet — build flag only, not a product name)
  - Package portable tree (`OLLMchat.exe` + DLLs) + NSIS → **`OLLMchat-<version>-Setup.exe`** (version from `meson.build`)
  - Upload artifact; on tag attach to the **same** GitHub Release as Ubuntu assets

### Trigger / publish

- **🔷** **Phase 1:** `workflow_dispatch` only (no tag trigger) — test via Actions UI or `gh workflow run`.
- **⏳** **🔷** Later: same triggers as release (`v*` tags + dispatch); publish installer to the GitHub Release.
- **💩** Prefer one workflow file with two jobs (Ubuntu + Windows) so tag publish can wait on both before changelog finalize — or keep finalize on Ubuntu success and let Windows upload with `--clobber` (Android already non-blocking; decide when wiring tags)
- **🔷** Changelog finalize on `main` must not run until **Windows installer is published** if Windows is a release-blocking asset (preferred).

## Meson / tree work (product must build natively)

### Platform deps (Snappr / consumer-meson pattern)

- **⏳** **🔷** Root / `libocwebkit` Meson: on `host_machine.system() == 'windows'`
  - `dependency('webview2gtk-1')` instead of `webkitgtk-6.0`
  - Vala `-D WINDOWS` (and keep Linux `-D LINUX` where used)
- **⏳** **🔷** Today `libocwebkit/meson.build` still lists `webkitgtk-6.0` unconditionally — that must become platform-conditional before native Windows CI can compile the browser tool.
- **ℹ️** Precedent: Snappr2 `meson.build` `if win` / `else`; webview2-gtk `examples/consumer-meson.build`
- **💩** Optional meson option `webview2gtk_prefix` (Snappr) for CI staging prefix vs pacman install

### Packaging scripts

- **⏳** **🔷** Add Windows-native package path (do **not** rely on Ubuntu `sqgipkg --target win-nsis` for release):
  - Copy or adapt **`copy-exe-runtime-dlls.sh`** from webview2-gtk / Snappr
  - Portable dir (Snappr `package-snappr.sh` shape) including `WebView2Loader.dll` + GTK DLLs + existing launcher env (`FONTCONFIG_*`, emoji font)
  - NSIS installer script (Snappr `build-windows-installer.sh` / library `build-installer.sh` as references) — **versioned outfile** `OLLMchat-${VERSION}-Setup.exe` + `PRODUCT_VERSION`
- **💩** Keep Linux `scripts/build-windows-dir.sh` / sqgipkg cross path only as **legacy / optional** until deleted — not for tagged releases after cutover
