# 5.5 Wiki maker

**Status:** proposed

**Checklist:** `.cursor/rules/CODING_STANDARDS.md` — *Checklist for all plans* (verify before implementation).

**Plan layout / workflows:** `docs/guide-to-writing-plans.md`

## Purpose

- **⚠️** Build a **persistent, LLM-maintained wiki** derived from the codebase (and optionally docs), analogous to how **semantic search** indexes code for retrieval—but here the artifact is **structured markdown** with links, not only vectors.
- **⚠️** Expose wiki maintenance and reading as **skills** (create/update wiki; read/navigate wiki) so the general skill/orchestration design applies.
- **⚠️** Make the wiki the **first port of call** when starting coding: a sub-agent (or refinement loop) **reads from the top**, follows links, and **ships distilled findings** to the **task queue** as reference content—complementing ad-hoc semantic search.
- **⚠️** **Wiki location:** all generated and hand-curated wiki pages live under **`docs/wiki/`** (same tree as the rest of project docs). Avoid hidden dot-directory layouts (e.g. `.ollm/wiki`); they are hard to browse and easy to misconfigure.
- **ℹ️** Prior art: Karpathy’s **LLM Wiki** gist (operations, index/catalog, lint checklist—usable as **prompt ingredients**). See *External references* below.

## Scope

| In scope | Out of scope (unless explicitly added later) |
| -------- | -------------------------------------------- |
| Design: wiki **information architecture** for a codebase; **incremental** rebuild from changed files | Full UI wiki browser inside the app (may be a follow-up) |
| **Skill** contracts: build/update wiki, read wiki (path, depth, section) | Replacing semantic search; wiki is **complementary** |
| Integration hooks: task creation/refinement consumes wiki-derived **reference files** | Hosting the wiki as a public website |
| Change detection: leverage **filesystem / project file tracking** (and optionally git) to scope work | Perfect consistency proofs; human-grade editorial polish without review |

## Acceptance criteria

- **⏳** Plan is actionable: clear **phases**, **wiki layout** conventions, and **skill** responsibilities.
- **⏳** Incremental strategy is specified: what state is stored (e.g. last build id, per-file hashes), how **changed files** drive updates.
- **⏳** Sub-agent / orchestration story is explicit: **entry page**, navigation rules, and **output shape** for the task queue.
- **⏳** External patterns are summarized with pointers—not only internal opinion.

## External references (wiki build & read prompts)

**ℹ️** Scope: primary sources that spell out **operations**, **navigation**, and **checklists** you can lift into **wiki_build** / **wiki_read** system prompts. Prefer the **gists** below over thin blog summaries—they duplicate little of value beyond this plan.

### Karpathy — `llm-wiki` (gist, primary source)

**ℹ️** Full idea file (copy-paste friendly for agents): [gist.github.com/karpathy/442a6bf555914893e9891c11519de94f](https://gist.github.com/karpathy/442a6bf555914893e9891c11519de94f)

**Build / ingest (concrete flow from the gist)** — After a new source arrives, the maintainer: reads it; produces/updates a summary in the wiki; **updates `index.md`**; updates **entity and concept pages** across the wiki; **appends to a chronological log**. One source may touch many pages—encode that multi-page update expectation in **wiki_build** prompts.

**Schema / conventions** — The gist calls for a separate **schema** doc (it names `CLAUDE.md` / `AGENTS.md`-style) that defines structure, conventions, and workflows. For us, that role maps to **`docs/wiki/config.yaml`** plus a short **“wiki conventions”** section the build prompt must obey.

**`index.md` (catalog for readers and for query)** — Content-oriented catalog: each page with a **link**, **one-line summary**, optional metadata; **organized by category** (entities, concepts, sources, etc.); **updated on every ingest**. For **query**, the gist recommends: **read `index.md` first** to find relevant pages, then drill in—at moderate scale this can replace embedding-first RAG for navigation.

**`log.md` (optional but specified)** — Append-only timeline of ingests, queries, lint passes; use a **consistent heading prefix** so entries are grep-friendly (gist example pattern: `## [YYYY-MM-DD] ingest | …`).

**Lint (explicit checklist — suitable to paste into a lint sub-prompt)** — Contradictions between pages; **stale** claims superseded by newer material; **orphan** pages (no inbound links); important concepts **mentioned but lacking a page**; **missing cross-references**; **gaps** that could be filled (e.g. further search). The gist also frames lint as suggesting **new questions** and **new sources** to investigate.

### Wiki reading / query (same gist — Query operation)

**ℹ️** **Query**: search/read relevant wiki pages, **synthesize an answer with citations**; good answers can be **filed back** into the wiki as new pages so exploration compounds (maps to sub-agent **briefings** + optional “write finding to `docs/wiki/flows/` or similar”).

**ℹ️** **Optional tooling note in gist** — At larger scale, hybrid search over markdown (e.g. **qmd** — linked from the gist) is mentioned; we already have **semantic search** in-product—**wiki_read** prompts can combine “read `index.md` + follow links” with **codebase search** over `docs/wiki/` when the catalog grows.

### Optional extension (scale / lifecycle — not required for v1)

**💩** **LLM Wiki v2** — Extends Karpathy with production-oriented patterns (confidence/supersession, graph-typed edges, hybrid search when `index.md` exceeds ~100–200 pages, automation hooks). Use if **wiki_build** / **lint** prompts need a second tier of rigour: [gist.github.com/rohitg00/2067ab416f7bbe447c1977edaaa681e2](https://gist.github.com/rohitg00/2067ab416f7bbe447c1977edaaa681e2) (builds on the same Karpathy gist).

## Proposed behaviour

### Alignment with OLLMchat

- **✅** **Semantic search** (`libocvector`, `CodebaseSearchTool`, indexer): already chunks, embeds, and retrieves. Wiki maker **reuses the same file universe** (tracked files, languages, project roots) but writes **human-navigable** layers (overview → module → package → key types/flows).
- **⚠️** **Skills**: treat **wiki_build** (or split **wiki_ingest** / **wiki_refresh**) and **wiki_read** like other skills: schemas, orchestrator execution, outputs as **reference files** for tasks (same story as `1.23.*` task orchestration docs).
- **⚠️** **Task queue**: sub-agent runs **wiki_read** with a plan (“start at `docs/wiki/index.md`, follow *Architecture*, then *Tasks runner*”), then emits a **short markdown or JSON summary** attached to the task—parallel to multi-step codebase research.

### Incremental build (“what changed since last run”)

- **⚠️** **Primary signal**: files under project management already have **identity and change** information—reuse that to compute **dirty set** (added/changed/removed paths) without full-tree LLM passes every time.
- **💩** **Optional secondary signal**: `git diff` / `HEAD` commit SHA for teams that use git inside the project; useful for “since last wiki build” in CI or multi-dev scenarios. Product should work even when git is not the source of truth, as long as **filesystem state** is tracked.
- **⚠️** **Minimum persisted state**: wiki build **generation id** (monotonic or timestamp), plus **per-source fingerprint** (hash, mtime, or indexer’s own file version) to map wiki sections to files.
- **⚠️** **Patch strategy**: for each dirty file, **update or replace** the corresponding wiki leaf (and roll up **parent summaries** bottom-up: file → directory → subsystem). Unchanged subtrees **skip** LLM work.
- **ℹ️** **Cold start**: full pass over tracked code files (possibly batched by directory) to seed `docs/wiki/index.md` and top-level maps; same pipeline as incremental, with “everything dirty”.

### Wiki structure around a codebase (recommended)

**Base path:** `docs/wiki/`. Use a **fixed skeleton** so the reader (human or agent) always knows where to look. Adapt names to the repo’s layout.

- **⚠️** **Root `docs/wiki/index.md`**: one screen—product purpose, **how to read this wiki**, link to architecture map, conventions, “last built” metadata.
- **⚠️** **`docs/wiki/architecture/`**: system context diagram (mermaid optional), **major subsystems**, data flow, external dependencies; **stability** notes (what is stable vs in flux).
- **⚠️** **`docs/wiki/modules/`** (or `packages/`): one page per top-level source tree / Meson subproject; each lists **key entry points**, **owns what**, **depends on whom**.
- **⚠️** **`docs/wiki/flows/`**: cross-cutting behaviours (e.g. “task creation → refinement → runner”); link to **specific files** and symbols (paths + optional AST/ref id if we have them).
- **⚠️** **`docs/wiki/decisions/`** (optional): ADR-style short pages; link from architecture when relevant.
- **⚠️** **`docs/wiki/glossary.md`**: domain terms and acronyms used in code.
- **💩** **`docs/wiki/config.yaml`** (or `config.json`): `repo_notes` for LLM steer (“always document `liboccoder/Task` before UI”), optional **pinned pages**, **exclude_globs**, **priority_paths**. Keeps large repos from starving important corners.

**Linking rules**

- **⚠️** Every non-trivial page links **up** (parent) and **sideways** (related modules).
- **⚠️** File-level pages cite **path from repo root** and **last verified hash** or date—helps **lint** for drift.
- **💩** Optional: mirror semantic search—embed **paragraph-level** chunks of wiki for hybrid “navigate wiki + vector jump” later.

### Skill: create / update wiki (outline)

- **Inputs**: project id / root; optional “full rebuild”; optional max cost (files, tokens).
- **Steps**: compute dirty set; topologically order **directory rollups**; LLM updates leaves then parents; write markdown under **`docs/wiki/`** by default (commit policy: generated-only vs mixed human edits—**product decision**).
- **Outputs**: manifest JSON (paths written, errors, skipped), plus pointer to **`docs/wiki/index.md`**.

### Skill: read wiki (outline)

- **Inputs**: start path (default `docs/wiki/index.md`), **max depth** or **max pages**, optional focus query.
- **Steps**: deterministic read of markdown; optional LLM **compression** into a briefing.
- **Outputs**: markdown or JSON summary for **task reference_files**.

### Sub-agent loop (task prep)

- **⚠️** Orchestrator spawns a **research** pass: “Read wiki from `docs/wiki/index.md`; follow links until coverage of [feature X]; produce ≤N token summary with citations to wiki paths.”
- **⚠️** Result is **injected** into the task’s reference bundle before implementation tasks run—same pipeline as semantic search results.

## Risks and notes

- **⚠️** **Drift**: wiki can disagree with code; **lint** pass (stale links, stale hashes) and periodic **full refresh** options mitigate.
- **💩** **Cost**: LLM updates per change batch; cap batch size and prioritize dirty paths by **dependency** or **recency**.
- **ℹ️** **Human edit**: if `docs/wiki/` is committed, merges may conflict; document whether the tree is **generated-only**, **hand-maintained**, or **mixed**, and branch/PR rules if needed.
