## Overview

**Primary goal (bubblewrap path only):** When **`run_command`** runs inside **`bwrap`**, improve tool output so the model and user see **what actually happened**, not guesses:

1. **Network access reporting** — If networking was **off** for the run, say **clearly** that the command **tried** to use the network and **what** was observed (e.g. counts per syscall class from seccomp user-notify), instead of appending the generic “pass `network`: true” line on **every** failure.
2. **File access reporting** — When this run has a defined **allowed writable area** (project overlays + `/tmp` + approved roots, etc.), say when the command **tried** filesystem operations **outside** that area, with as much path/syscall detail as the kernel exposes — so the user/model can fix **`write_roots`** or expectations.

**Same document, supporting product work:** **Non-project UX + write roots** — default read-only where safe, verifiable directory roots instead of vague rw-`/`, permission copy — see **§ Non-project write roots** and **§ Non-project writable surfaces**.

Implementation direction discussed: **seccomp** (via **libseccomp** and/or **bubblewrap’s** built-in seccomp FD passing), wired into the same **`Bubble`** / **`bwrap`** path as today’s mounts and namespaces.

**Product intent (read carefully):** **Isolation and enforcement** (no network when disabled, read-only vs writable surfaces, project overlays, etc.) stay **bubblewrap’s job** — **`--unshare-net`**, **`--ro-bind` / `--bind`**, overlay args, **`/tmp`**, and related **`build_bubble_args`** behaviour. **Seccomp in this plan is not an additional policy layer to “enforce” those rules**; it exists **only** to **detect** interesting syscalls (with enough detail to be useful) and **report** them in tool output — **network access reporting** and **file access reporting** above — so the model and user get **ground-truth** text instead of heuristics. Prefer **`SCMP_ACT_NOTIFY`** with supervisor **record then allow** (syscall proceeds; bwrap / the kernel still applies the real restriction) or another **non-blocking** observation path; **do not** treat **`SCMP_ACT_TRAP`** / exit **159** as the shipping design — that **blocks** in the child and blurs “who enforced what.” Permission prompts and tool parameters remain separate **gates** for approval.

### Bubblewrap when available — fallback to legacy execution (**ℹ️** revised)

**ℹ️** Today **`Bubble.can_wrap()`** is **false** when **`FLATPAK_ID`** is set (Flatpak / app-in-container) **or** **`bwrap`** is not on **`PATH`** — see **`liboctools/RunCommand/Bubble.vala`**. There is **no** separate generic “am I in Docker?” probe; extend **`can_wrap()`** only if product adds explicit detection later.

**Preferred:** When **`Bubble.can_wrap()`** is **true**, **`run_command`** uses **`Bubble`** (bwrap). **Seccomp BPF** on this path exists to feed **network** and **file** access reporting (`--seccomp` / `--add-seccomp-fd`); **bubblewrap** still enforces isolation.

**Fallback:** When **`!Bubble.can_wrap()`**, **keep** the **existing** behaviour: **`Request.execute_tool_async()`** → **`execute_with_subprocess()`** — unsandboxed **`GLib.Subprocess`** with the **same permission / approval rules** as today (prompt when policy requires it). This matches **Flatpak**, **missing bubblewrap**, or any host where wrapping is unavailable without treating “no bwrap” as a hard failure for the whole feature.

**Trade-offs (document for users and seccomp design):**

- **No bubblewrap** ⇒ **no** kernel sandbox from bwrap ⇒ **seccomp attached via bwrap does not apply**. **Network/file syscall reporting** is **unavailable** on the fallback path unless a **separate** mechanism is added (e.g. `prctl(SECCOMP)` in the child — **not** planned here). UI/tool text may **note** “command ran without sandbox” when **`!can_wrap()`** so expectations stay honest.
- **Security posture:** Fallback is **weaker** than bwrap; **permission prompts** remain the gate (and should stay strict for non-project / network / writes per other sections).

**👎** Do **not** silently pretend fallback runs are sandboxed; **👍** do **retain** fallback so **Flatpak** and **no-bwrap** installs remain usable with **explicit** approval flow rather than a dead feature.

## Current behavior (codebase)

### Bubblewrap (`liboctools/RunCommand/Bubble.vala`)

- Root is `--ro-bind / /` unless `allow_write` requests a writable root (`--bind / /`).
- Network: `--unshare-net` when `allow_network` is false.
- Project mode: overlay mounts for project roots; optional playground bind under `$HOME/playground`.
- **No seccomp** is loaded today; there is **no** `libseccomp` dependency in `liboctools/meson.build`.

### Failure output hint (network)

On non-zero exit, stderr/stdout are combined and, if networking is disabled, a **fixed** line is appended:

```472:474:liboctools/RunCommand/Bubble.vala
		if (!this.allow_network) {
			final_fail_str += " - Note: Networking is disabled by default. Pass \"network\": true in the run_command arguments to enable it.";
		}
```

The same idea exists in the non-bwrap fallback in `Request.vala`. This fires for **any** failure when network is off, not only when the failure was caused by blocked networking—hence the desire for **ground-truth** signals when **bwrap** is used. On the **`!Bubble.can_wrap()`** fallback path, evidence-based hints remain **best-effort** until/unless another mechanism exists.

### Fallback when bubblewrap is unavailable (`liboctools/RunCommand/Request.vala`) — **ℹ️** retain

**ℹ️** Today, `execute_tool_async()` calls **`execute_with_subprocess()`** when **`!Bubble.can_wrap()`**:

```316:317:liboctools/RunCommand/Request.vala
			if (!Bubble.can_wrap()) {
				return yield this.execute_with_subprocess();
```

**Policy (revised):** **Keep** this branch. When **`Flatpak`** or **missing `bwrap`**, runs use **legacy subprocess** + **existing permission behaviour** (see **§ Bubblewrap when available**). **Seccomp / bwrap-only evidence** does not apply; optional UX can state that the run was **not** bubblewrapped.

### Permissions (`liboctools/RunCommand/Request.vala`)

- `network: true` always forces a permission prompt (`build_perm_question`).
- If `Bubble.can_wrap()`, there is an **active project**, and `allow_write` is not `"all"`, permission is **skipped** (sandbox is trusted for that case).
- Otherwise (including **no project**), simple commands still go through executable resolution and typically **still require** “Run command: …?” approval—this is the “prompt on all commands for non-project” pain point.

`allow_write` today: `"no" | "yes" | "project" | "all"` with logic tying `allow_write` on `Bubble` to project vs non-project in `execute_tool_async()`.

### Non-project writable surfaces (**⚠️** must verify — not “zero writable” today)

**⚠️** Product expectation stated in discussion: **non-project** sessions should have **no** writable areas (or only strictly intended ones). The **current tree** must be checked before claiming that; bwrap always adds **at least one** writable mount unrelated to project roots.

**ℹ️** `Request.execute_tool_async()` computes `allow_write` for `Bubble` (rw root only when project + `"all"`, or non-project + `"yes"`/`"all"`):

```325:327:liboctools/RunCommand/Request.vala
			// project: all = rw root (everywhere), yes/no = project only (ro root). non-project: all/yes = rw root (everywhere), no = ro root
			var aw = this.allow_write.strip().down();
			var allow_write = (project != null && aw == "all") || (project == null && (aw == "yes" || aw == "all"));
```

**ℹ️** For **no project** and default tool `allow_write` (`"project"` → not `yes`/`all`), **`allow_write` is false** → `Bubble` uses **`--ro-bind / /`**, not rw root.

**ℹ️** In **`Bubble.build_bubble_args`**, **regardless of project**, the same block always mounts a **writable** `/tmp`:

```263:273:liboctools/RunCommand/Bubble.vala
			if (this.allow_write) {
				args += "--bind";
				args += "/";
				args += "/";
			} else {
				args += "--ro-bind";
				args += "/";
				args += "/";
			}
			args += "--tmpfs";
			args += "/tmp";
```

- **ℹ️ Fact (tree):** Non-project + default write policy → **read-only host view** at `/`, **but `/tmp` is a writable tmpfs** (and `/dev/null` is bound writable for redirects — see `--dev-bind` below in the same function).
- **⚠️** **Verification tasks** (before shipping “no writable areas outside X” messaging or seccomp allowlists):
  1. Confirm **product** accepts writable **`/tmp`** (and **`/dev/null`**) for non-project, **or** change **bwrap** profile (e.g. smaller tmpfs, ro `/tmp` — each has UX/security tradeoffs). **Seccomp** here is for **observation/reporting**, not for replacing mount-based write control.
  2. Document the **canonical allowlist** of writable paths for **non-project** vs **project** modes (project adds overlay + playground bind; see `if (this.project != null)` playground block earlier in `build_bubble_args`).
  3. Only then define **file access reporting** (“outside allowed area”) — compare syscall/path evidence against that allowlist.

**👎** Do not assume non-project is “fully read-only filesystem” without enumerating **`--tmpfs /tmp`** (and dev) in docs and tests.

### Non-project write roots — **⚠️** target design (path-based, verifiable)

**⚠️** **Product requirement:** For **non-project** sessions, “writable” must mean **concrete directory roots**, not an opaque enum that maps to **rw `/`** without scrutiny. Each allowed root is a **top-level path** (absolute after normalization); **any subdirectory** under that root is writable **within the sandbox** (subject to bwrap `--bind` / mount plan).

**ℹ️** **Parameter shape (illustrative):** e.g. **`write_roots`** — a **string** containing **one or more paths**, separated by a **single documented delimiter** in the **tool schema / description visible to the LLM** (e.g. **`;`** — semicolon — keeps commas inside paths out of the way). **Backend** (`Request.vala` / `Bubble.vala`):

- Split on that delimiter, **trim** entries, reject empties.
- Resolve to **absolute** paths (`GLib.Path` / canonicalization rules TBD).
- **Deduplicate**; optionally **sort** for stable permission prompts.
- Map each root to **dedicated bwrap binds** (or a small set of overlays) so writes **cannot** escape those trees without leaving the sandbox story.

**⚠️** **LLM-facing tool description (required encouragement):** The **`run_command`** tool text shown to the model must **instruct** the assistant to:

- **List only the directories** where it **actually expects to write** (or create files under), not the whole filesystem.
- **Avoid** requesting broad roots such as **`/`** or the entire **home** unless truly necessary — **users will often reject** overly broad write permission.
- Use the **delimiter** spelled out in the schema (e.g. “separate multiple roots with **`;`**”) so the model’s output matches what the **parser** expects.

**ℹ️** **Backend vs LLM:** The **delimiter** in user-facing docs is **fixed by schema**; the OS may use different path separators **inside** each path (e.g. **`/`** on Linux). Tell the LLM explicitly: **paths use the host’s normal path rules**; **multiple roots** use **`;`** (or whatever delimiter is chosen) **between** paths only.

**⚠️** **Permission UX:** Presenting **resolved path roots** in the approval dialog (not just “yes write”) makes **user rejection** of over‑broad requests **auditable** and aligns with “verify non‑project isn’t rw‑everything by default.”

**👎** Do not ship non‑project **rw `/`** as the only “I need writes” knob without **named roots** + copy above — it is **not** verifiable and trains bad LLM habits.

## Goals

| Topic | What we want |
|------|--------------|
| **Network access reporting** | After a **bwrapped** run with networking off, if the process **attempted** network I/O, append a **clear, factual** line or block (e.g. “connect (1×), socket (2×)”) instead of only the blanket hint on failure. |
| **File access reporting** | After a **bwrapped** run, when the run has a defined **allowed writable set**, surface **attempted** writes or other sensitive FS operations **outside** that set (syscalls TBD: `openat` with create/write, `unlink`, `rename`, …). **Depends on** the same allowlist story as mounts — **§ Non-project writable surfaces** (`/tmp` is writable in almost all profiles). |

### Network access reporting (append to tool result text)

**Emit only if:** `Bubble.can_wrap () && !allow_network && network_syscall_events.length > 0` (shape of `network_syscall_events` is implementation-defined).

**Technical appendix** — build in Vala like:

```vala
const string RUN_CMD_NETWORK_REPORT_HEAD = """
---
Sandbox: networking was disabled for this run, but the command attempted network-related operations:
""";

// Example after formatting counts from notify / aggregation:
string summary = "connect (1×), socket (2×)";
string tail = """
To enable networking on a later run, pass "network": true in run_command (user approval required).
""";
string appendix = RUN_CMD_NETWORK_REPORT_HEAD + summary + "\n" + tail;
result += appendix;
```

**Rendered example** (what the user/model actually reads):

```text
---
Sandbox: networking was disabled for this run, but the command attempted network-related operations:
connect (1×), socket (2×)
To enable networking on a later run, pass "network": true in run_command (user approval required).
```

**Plain-language variant** (product toggle):

```vala
const string RUN_CMD_NETWORK_REPORT_PLAIN = """
---
Sandbox: this command tried to use the network while networking was off for this run.
To allow that next time, pass "network": true in run_command.
""";
```

**Fallback when `!Bubble.can_wrap ()`** — no syscall evidence; append at most:

```vala
const string RUN_CMD_NETWORK_REPORT_NO_BWRAP = """
---
Sandbox: this run did not use bubblewrap; detection of blocked network syscalls is unavailable.
""";
```

**Remove** the old blanket line when the network report appendix is present — today it is effectively:

```vala
const string LEGACY_NETWORK_HINT =
    " - Note: Networking is disabled by default. Pass \"network\": true in the run_command arguments to enable it.";
```

Do **not** append `LEGACY_NETWORK_HINT` when `appendix.length > 0` from network evidence. Do **not** emit this appendix when `allow_network` is **true**.

## Feasibility: seccomp and Vala

### libseccomp

- **Exists** and is the standard Linux API for building seccomp BPF programs: [seccomp/libseccomp](https://github.com/seccomp/libseccomp).
- It is **not** GObject-based; there is **no** widely shipped **Vala vapi** in the same way as `gtk+-3.0`. The codebase today has **zero** seccomp references.

### Vala integration options

1. **Small C helper** in-tree: wrap `seccomp_init`, `seccomp_rule_add`, `seccomp_export_bpf`, and optionally notify APIs. Vala links the `.c` file via meson. Minimal Vala `[CCode]` for the few entry points.
2. **Manual `.vapi`**: bind `libseccomp` C headers into Vala (more work up front, cleaner call sites). **Subplan:** [2.22.1.2 — GIR/`vapigen` spike, minimal vapi, validation tests](./2.22.1.2-seccomp-manual-vapi-and-validation.md).
3. **`Seccomp.Filter` in `vapi/seccomp.vapi`**: In-tree; still wire into **`Bubble`** only when **`can_wrap()`** (see subplan [2.22.1.2](./2.22.1.2-seccomp-manual-vapi-and-validation.md)).

**Conclusion**: Feasible, but **new build dependency** (`libseccomp`) and **explicit bindings or C glue** are required.

### Bubblewrap already accepts seccomp BPF

`bwrap --help` includes:

- `--seccomp FD` — load seccomp rules from FD (not repeatable).
- `--add-seccomp-fd FD` — load rules from FD (repeatable).

So the preferred integration is: **parent** builds a BPF program (libseccomp → exported FD), passes it into `bwrap`, and bubblewrap applies it to the **final** command inside the sandbox (verify against your installed `bwrap` version for exact semantics). This avoids injecting a custom wrapper binary as the shell **if** the filter can be fully described by BPF rules passed to bwrap.

## Design fork: TRAP + SIGSYS vs user notification

### Approach A: `SCMP_ACT_TRAP` + `SIGSYS` handler (from research notes)

- Filter actions: **TRAP** for syscalls of interest → kernel delivers **SIGSYS** to the **thread** that invoked the syscall.
- Problems for this app:
  - The signal runs in the **sandboxed child** (the process running `/bin/sh -c`), **not** in the OLLMchat parent. A Vala `sigaction` in the parent does **not** observe the child’s traps.
  - To handle traps in-process you need a **wrapper** that runs **inside** the sandbox before `exec` of the shell, or you accept **process death** on trap (bad UX).
  - Async-signal-safe logging: ring buffer or `write()` to a **pipe** whose read end is held by the parent—again requires a cooperating process inside the namespace or passing the pipe FD through bwrap (possible but fiddly).

**Verdict**: Possible but awkward; best if you control the innermost executable. **Not** the first choice if bwrap + parent-side APIs can avoid it.

### Approach B: `SECCOMP_RET_USER_NOTIF` (supervisor in parent)

- Rules use **notify** action; a **supervisor** (parent or sibling thread) reads events from a **seccomp notify fd**, inspects syscall number and arguments, then **allows / denies / kills** per `seccomp_notif_resp` — **for this product**, the intended response for **network- and file-observation** rules is **`SECCOMP_USER_NOTIF_FLAG_CONTINUE`** / allow after **recording** the event so **bwrap** (namespaces, mounts) remains the mechanism that actually **stops** traffic or **EROFS** writes; seccomp supplies **labels** (syscall class, path where available).
- Gives **structured** access to syscall args (e.g. `openat` path) without a signal handler in the child.
- Requires **Linux** with user notification support and more code (listener loop, response fd). libseccomp documents notify APIs.

**Verdict**: Stronger for **path-based** “tried to write outside X” stories; more implementation cost. **Matches** “seccomp = detection and reporting only” when paired with **allow-after-log**.

### Approach C: `SCMP_ACT_LOG` / audit

- Kernel logs violations; userspace reads audit logs. Heavier operationally; usually worse for a desktop app.

### Pragmatic phasing

1. **Phase 1 — network access reporting**: **`SCMP_ACT_NOTIFY`** on **network-related** syscalls (`socket`, `connect`, `sendto`, …—list must be maintained per arch); supervisor **aggregates**, then **allows** so the attempt hits **`--unshare-net`** (or succeeds if network is on). **Do not** rely on **TRAP + exit 159** for shipping — that **blocks** in the child and contradicts “seccomp only for detection/reporting.”
2. **Phase 2 — file access reporting**: Extend the same **notify** pattern with `openat`, `unlink`, `rename`, etc.; compare resolved paths against allowed roots for the **file-report appendix**; still **allow** after record where policy is “observe only” (mounts enforce writability).

## Changes by area (if proceeding)

### 1. Build / dependencies

- Add `dependency('libseccomp')` (or `cc.find_library('seccomp')` + headers) to `liboctools/meson.build`.
- Add C glue or vapi; export BPF fd for bwrap.

### 2. `Request.execute_tool_async` — bubblewrap vs fallback (**ℹ️** prerequisite)

- **ℹ️** When **`Bubble.can_wrap()`**, use **`Bubble`** as today; attach seccomp BPF when implemented.
- **ℹ️** When **`!Bubble.can_wrap()`**, **keep** **`execute_with_subprocess()`** fallback and existing permission flow; **do not** claim sandbox/seccomp guarantees for that path.

### 3. `Bubble.build_bubble_args` / exec path

- After creating the filter FD, append `--add-seccomp-fd` / `--seccomp` with the **dup’d** FD as required by `Subprocess.newv` (ensure FD is not closed prematurely; meson may need `FD_CLOEXEC` handling—exact pattern TBD).
- Define **per-mode** **observation** profiles (BPF still loaded via bwrap; **bwrap enforces**):
  - **Network allowed**: do not install notify rules on network syscalls (no spurious reporting), or install and **allow** immediately without accumulating.
  - **Network off**: notify on network syscalls → **record** for **network access report** → **allow** (actual block is **`--unshare-net`**).
  - **Filesystem**: notify on selected FS syscalls → **record** paths for **file access report** → **allow** (read-only / overlay rules enforced by **mounts**, not by seccomp deny).

### 4. Evidence reporting

- **After** `wait_async`, collect:
  - From notify queue, or
  - From pipe (if using a wrapper), or
  - From structured exit (if using kill + mapped errno—weak).
- Append to tool result **only when** evidence exists: e.g. “Attempted network-related syscalls while isolated: connect (1×), …” (exact copy from **§ Concrete delivery**) — wording should reflect **observation**, not “seccomp blocked,” unless product explicitly chooses otherwise.

### 5. Tool schema + permissions (`Request.vala`, `Tool.vala`)

- **Non-project**: adjust `build_perm_question()` so default read-only runs **skip** approval when `network` is false and **no** extra write roots are requested; **require** approval when:
  - `network` is true, or
  - **`write_roots`** (or equivalent) is **non-empty** after parse/trim — show **resolved absolute paths** in the prompt.
- **⚠️** Add **`write_roots`** (name TBD): **string**, delimiter-separated list of **directory roots** per **§ Non-project write roots**; **backend** splits, normalizes, validates; **`Bubble`** gains **per-root binds** instead of a single **`--bind / /`** for non-project write.
- **⚠️** Update **`Tool.vala`** long description / `@param` lines with **LLM coaching** (narrow paths, avoid `/`, use **`;`** between roots — match implementation).
- Deprecate or narrow **`allow_write: "yes"` / `"all"`** for **non-project** once **`write_roots`** exists — avoid duplicate/conflicting semantics (**exact migration** in implementation PR).

### 6. Security and compatibility

- **Flatpak / no `bwrap`**: `Bubble.can_wrap()` is **false** — **`execute_with_subprocess`** fallback remains; **seccomp-via-bwrap** is **N/A**. Surface **honest** copy if useful (“not sandboxed”), and rely on **permissions** for sensitive operations.
- **Architecture**: seccomp rules are architecture-specific; use libseccomp’s arch APIs or restrict to `x86_64`/`aarch64` explicitly in CI.
- **Maintenance**: syscall lists drift across kernels; prefer libseccomp constants and periodic review.

### 7. Future (optional): writable-mode history + “files touched” over time

**💩** Idea (user): If the user runs **`run_command` in writable modes** (e.g. broad write / project writes) **over an extended period** (e.g. a month), **track which files were modified** and surface or summarize that — useful for trust, review, or “what did commands change?”

**ℹ️** This is **not** required for the first seccomp/evidence milestone; treat as a **follow-on** if feasible.

**Why it is tricky**

- **Definition of “writable mode”** — Project runs use **overlay uppers** per root; non-project may use **rw root** or small binds. “Modified” might mean **host-visible** changes, **overlay upper files**, or **syscalls that opened paths for write** — three different datasets.
- **Where data lives today** — Overlay writes land under **`overlay_dir`** (see `Overlay` / `Bubble`); correlating those paths back to **real project paths** is already a mapping problem; **rw root** mode has **no** isolated upper — you would need **kernel-level** auditing (fanotify, eBPF, seccomp notify logs) or **periodic host scans** (expensive, racy).
- **Retention and privacy** — Storing a month of paths per project/session needs **storage policy**, **opt-in**, and **clear-user** deletion — overlaps with telemetry ethics.
- **Performance** — Continuous monitoring of all `openat`/`write` paths from notify can be heavy; batching and sampling need design.

**Possible directions (no commitment)**

- **💩** **Overlay-only audit** — When **`allow_write`** does **not** use full rw root, diff or inventory **`upper/`** after each run (or daily) — **bounded**, but **misses** rw-root mode entirely.
- **💩** **Seccomp notify log** — Append resolved paths (where the kernel provides them) to a **local ring buffer or sqlite** keyed by session — **feasible** if notify machinery exists for **file reporting** anyway; completeness depends on syscall coverage.
- **💩** **Explicit “audit this run”** flag — Simpler UX: only record when user opts in, reduces storage and surprise.

**👎** Do not block **network/file reporting** or the rest of this milestone on this idea; add a **separate plan** once notify/overlay story is stable.

## Summary

| Question | Answer |
|----------|--------|
| Is libseccomp “bound to Vala” already? | **No** in this repo; add C bindings or a small C helper. |
| Is the approach feasible? | **Yes** on Linux with bwrap + libseccomp; bwrap **`--seccomp` / `--add-seccomp-fd`** is the natural integration point. |
| Is `Seccomp.Filter` real in-repo? | **Yes** in **`vapi/seccomp.vapi`** (hand-maintained OO binding); seccomp integration still lands on the **bwrap** path only. |
| TRAP + SIGSYS in Vala parent? | **Misleading** for child syscalls; use **notify**, a **sandbox wrapper**, or **bwrap-applied** BPF with a design that reports back to the parent. |
| Does this fix non-project prompting alone? | **Partly**: permission logic changes are **orthogonal** to seccomp and can ship earlier; seccomp backs **evidence** and stricter write policy. |
| Non-bwrap fallback for `run_command`? | **Retain** when **`!Bubble.can_wrap()`** (Flatpak, missing **`bwrap`**, …): **`execute_with_subprocess`** + permissions; **no** seccomp-from-bwrap on that path; be **honest** about isolation (see **§ Bubblewrap when available**). |
| Non-project writes? | **⚠️** **Explicit directory roots** (`write_roots`-style), subdirs allowed; **LLM** coached to stay narrow; **`;`**-style list in schema; **not** vague rw-`/`-only (**§ Non-project write roots**). |
