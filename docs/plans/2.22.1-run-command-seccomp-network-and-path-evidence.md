# 2.22.1 Run command: evidence-based network hints, out-of-scope write detection, non-project UX

## Note on numbering

Plan **2.22** is already allocated to [2.22-write-file-tool.md](./2.22-write-file-tool.md). This document is **2.22.1** (same feature area: run_command / sandbox).

---

## Overview

Improve **run_command** behavior around:

1. **Network**: Replace the current heuristic (“if the command failed and networking is off, append a generic hint”) with **evidence** when the sandboxed process actually attempted disallowed network syscalls (or equivalent).
2. **Filesystem writes outside the allowed area**: Optionally report when the process attempted writes (or sensitive operations) **outside** the project directory / approved bind paths—for **non-project** sessions, especially paths outside the intended writable region. **Requires a defined “allowed writable” set** (see **§ Non-project writable surfaces (must verify)** below).
3. **Non-project UX + write policy**: Default **read-only** non-project runs **without** a permission prompt where safe. **Writable** non-project access must **not** rely on vague **`allow_write: "yes"` / `"all"`** alone — see **§ Non-project write roots (path-based, verifiable)**. **Security review** should be able to verify the policy is **not** “write anywhere by default” (no overly broad / zero‑trust‑breaking defaults).

Implementation direction discussed: **seccomp** (via **libseccomp** and/or **bubblewrap’s** built-in seccomp FD passing), possibly combined with permission and tool-parameter changes.

### Bubblewrap required — **⚠️** user strategy (no non-bwrap fallback)

**⚠️** **Remove** the **`run_command`** path that runs **`GLib.Subprocess`** without bubblewrap (`Request.execute_tool_async()` → `execute_with_subprocess()` when **`!Bubble.can_wrap()`**). **`run_command`** must **only** execute inside **bwrap** so sandboxing and future seccomp/evidence are guaranteed.

**⚠️** If **`bwrap`** is **not** on **`PATH`** (or bubblewrap is otherwise unavailable): **do not** run the command and **do not** return a normal tool result that looks like “the shell command failed” to the **LLM**. The **human** must fix the environment first (install **bubblewrap** — package names vary by distro, e.g. `bubblewrap` on Debian/Fedora).

**⚠️** **Stop the flow** at a **user-facing** error: clear copy that **run_command is unavailable until bubblewrap is installed**, with **no** spurious stderr/exit-code blob for the model to interpret. Avoid letting the LLM “retry” or “debug” a missing system dependency as if it were a bad shell command.

**ℹ️** **Flatpak:** `Bubble.can_wrap()` returns **false** when **`FLATPAK_ID`** is set (see `Bubble.vala`). Policy must be explicit: e.g. same “unavailable” UX with **Flatpak-specific** text (host install vs portal limitations), **not** silently falling back to unsandboxed subprocess.

**👎** Do not keep an “escape hatch” unsandboxed **`execute_with_subprocess`** for **`run_command`** once this ships — it undermines G1–G3 and user expectations.

## Current behavior (codebase)

### Bubblewrap (`liboctools/RunCommand/Bubble.vala`)

- Root is `--ro-bind / /` unless `allow_write` requests a writable root (`--bind / /`).
- Network: `--unshare-net` when `allow_network` is false.
- Project mode: overlay mounts for project roots; optional playground bind under `$HOME/playground`.
- **No seccomp** is loaded today; there is **no** `libseccomp` dependency in `liboctools/meson.build`.

### Failure output hint (network)

On non-zero exit, stderr/stdout are combined and, if networking is disabled, a **fixed** line is appended:

```472:474:liboctools/RunCommand/Bubble.vala
		if (!this.allow_network) {
			final_fail_str += " - Note: Networking is disabled by default. Pass \"network\": true in the run_command arguments to enable it.";
		}
```

The same idea exists in the non-bwrap fallback in `Request.vala`. This fires for **any** failure when network is off, not only when the failure was caused by blocked networking—hence the desire for **ground-truth** signals. **⏳** The non-bwrap fallback is **scheduled for removal** per **§ Bubblewrap required** — after that, this duplicate hint path may shrink to bwrap-only.

### Fallback when bubblewrap is unavailable (`liboctools/RunCommand/Request.vala`) — **⏳** remove

**ℹ️** Today, `execute_tool_async()` calls **`execute_with_subprocess()`** when **`!Bubble.can_wrap()`**:

```316:317:liboctools/RunCommand/Request.vala
			if (!Bubble.can_wrap()) {
				return yield this.execute_with_subprocess();
```

**⚠️** Replace this with a **hard stop** + **user-visible** “install bubblewrap” / “run_command unavailable” path — **no** subprocess execution, **no** tool success payload to the LLM that could be confused with a failed command.

### Permissions (`liboctools/RunCommand/Request.vala`)

- `network: true` always forces a permission prompt (`build_perm_question`).
- If `Bubble.can_wrap()`, there is an **active project**, and `allow_write` is not `"all"`, permission is **skipped** (sandbox is trusted for that case).
- Otherwise (including **no project**), simple commands still go through executable resolution and typically **still require** “Run command: …?” approval—this is the “prompt on all commands for non-project” pain point.

`allow_write` today: `"no" | "yes" | "project" | "all"` with logic tying `allow_write` on `Bubble` to project vs non-project in `execute_tool_async()`.

### Non-project writable surfaces (**⚠️** must verify — not “zero writable” today)

**⚠️** Product expectation stated in discussion: **non-project** sessions should have **no** writable areas (or only strictly intended ones). The **current tree** must be checked before claiming that; bwrap always adds **at least one** writable mount unrelated to project roots.

**ℹ️** `Request.execute_tool_async()` computes `allow_write` for `Bubble` (rw root only when project + `"all"`, or non-project + `"yes"`/`"all"`):

```325:327:liboctools/RunCommand/Request.vala
			// project: all = rw root (everywhere), yes/no = project only (ro root). non-project: all/yes = rw root (everywhere), no = ro root
			var aw = this.allow_write.strip().down();
			var allow_write = (project != null && aw == "all") || (project == null && (aw == "yes" || aw == "all"));
```

**ℹ️** For **no project** and default tool `allow_write` (`"project"` → not `yes`/`all`), **`allow_write` is false** → `Bubble` uses **`--ro-bind / /`**, not rw root.

**ℹ️** In **`Bubble.build_bubble_args`**, **regardless of project**, the same block always mounts a **writable** `/tmp`:

```263:273:liboctools/RunCommand/Bubble.vala
			if (this.allow_write) {
				args += "--bind";
				args += "/";
				args += "/";
			} else {
				args += "--ro-bind";
				args += "/";
				args += "/";
			}
			args += "--tmpfs";
			args += "/tmp";
```

- **ℹ️ Fact (tree):** Non-project + default write policy → **read-only host view** at `/`, **but `/tmp` is a writable tmpfs** (and `/dev/null` is bound writable for redirects — see `--dev-bind` below in the same function).
- **⚠️** **Verification tasks** (before shipping “no writable areas outside X” messaging or seccomp allowlists):
  1. Confirm **product** accepts writable **`/tmp`** (and **`/dev/null`**) for non-project, **or** change bwrap profile (e.g. smaller tmpfs, ro `/tmp`, or seccomp deny writes outside an explicit list — each has UX/security tradeoffs).
  2. Document the **canonical allowlist** of writable paths for **non-project** vs **project** modes (project adds overlay + playground bind; see `if (this.project != null)` playground block earlier in `build_bubble_args`).
  3. Only then define **G2** “outside allowed area” — compare syscall/path evidence against that allowlist.

**👎** Do not assume non-project is “fully read-only filesystem” without enumerating **`--tmpfs /tmp`** (and dev) in docs and tests.

### Non-project write roots — **⚠️** target design (path-based, verifiable)

**⚠️** **Product requirement:** For **non-project** sessions, “writable” must mean **concrete directory roots**, not an opaque enum that maps to **rw `/`** without scrutiny. Each allowed root is a **top-level path** (absolute after normalization); **any subdirectory** under that root is writable **within the sandbox** (subject to bwrap `--bind` / mount plan).

**ℹ️** **Parameter shape (illustrative):** e.g. **`write_roots`** — a **string** containing **one or more paths**, separated by a **single documented delimiter** in the **tool schema / description visible to the LLM** (e.g. **`;`** — semicolon — keeps commas inside paths out of the way). **Backend** (`Request.vala` / `Bubble.vala`):

- Split on that delimiter, **trim** entries, reject empties.
- Resolve to **absolute** paths (`GLib.Path` / canonicalization rules TBD).
- **Deduplicate**; optionally **sort** for stable permission prompts.
- Map each root to **dedicated bwrap binds** (or a small set of overlays) so writes **cannot** escape those trees without leaving the sandbox story.

**⚠️** **LLM-facing tool description (required encouragement):** The **`run_command`** tool text shown to the model must **instruct** the assistant to:

- **List only the directories** where it **actually expects to write** (or create files under), not the whole filesystem.
- **Avoid** requesting broad roots such as **`/`** or the entire **home** unless truly necessary — **users will often reject** overly broad write permission.
- Use the **delimiter** spelled out in the schema (e.g. “separate multiple roots with **`;`**”) so the model’s output matches what the **parser** expects.

**ℹ️** **Backend vs LLM:** The **delimiter** in user-facing docs is **fixed by schema**; the OS may use different path separators **inside** each path (e.g. **`/`** on Linux). Tell the LLM explicitly: **paths use the host’s normal path rules**; **multiple roots** use **`;`** (or whatever delimiter is chosen) **between** paths only.

**⚠️** **Permission UX:** Presenting **resolved path roots** in the approval dialog (not just “yes write”) makes **user rejection** of over‑broad requests **auditable** and aligns with “verify non‑project isn’t rw‑everything by default.”

**👎** Do not ship non‑project **rw `/`** as the only “I need writes” knob without **named roots** + copy above — it is **not** verifiable and trains bad LLM habits.

## Goals

| Goal | Description |
|------|-------------|
| G1 | After a run, if (and only if) the process **attempted** network I/O while isolated, surface a clear, factual message (e.g. syscall class or “connect/socket/… observed”) instead of a blanket hint. |
| G2 | Optionally surface **attempted** writes or dangerous FS operations **outside** allowed roots (exact policy TBD: openat with write flags, unlink, rename, etc.). **Depends on** a verified allowlist — **§ Non-project writable surfaces** shows `/tmp` is already writable in bwrap today. |
| G3 | **Non-project**: default = **read-only** sandbox; **writable** access = **explicit `write_roots` (or equivalent)** — list of **top-level directory paths** (subdirs allowed); **requires** user approval. **No** vague “write yes/all → rw /” as the only story — see **§ Non-project write roots**. Tool description must **coach the LLM** to request **narrow** roots and use the **documented list delimiter**. |
| G4 | **Bubblewrap mandatory** for `run_command`: remove unsandboxed fallback; **user-first** error when `bwrap` missing or Flatpak blocks it — **no** LLM-facing fake “command failed” output (see **§ Bubblewrap required**). |

## Feasibility: seccomp and Vala

### libseccomp

- **Exists** and is the standard Linux API for building seccomp BPF programs: [seccomp/libseccomp](https://github.com/seccomp/libseccomp).
- It is **not** GObject-based; there is **no** widely shipped **Vala vapi** in the same way as `gtk+-3.0`. The codebase today has **zero** seccomp references.

### Vala integration options

1. **Small C helper** in-tree: wrap `seccomp_init`, `seccomp_rule_add`, `seccomp_export_bpf`, and optionally notify APIs. Vala links the `.c` file via meson. Minimal Vala `[CCode]` for the few entry points.
2. **Manual `.vapi`**: bind `libseccomp` C headers into Vala (more work up front, cleaner call sites).
3. **Assume a `Seccomp.Filter` API**: That was illustrative pseudocode only; **do not** assume it exists until a vapi or wrapper is added.

**Conclusion**: Feasible, but **new build dependency** (`libseccomp`) and **explicit bindings or C glue** are required.

### Bubblewrap already accepts seccomp BPF

`bwrap --help` includes:

- `--seccomp FD` — load seccomp rules from FD (not repeatable).
- `--add-seccomp-fd FD` — load rules from FD (repeatable).

So the preferred integration is: **parent** builds a BPF program (libseccomp → exported FD), passes it into `bwrap`, and bubblewrap applies it to the **final** command inside the sandbox (verify against your installed `bwrap` version for exact semantics). This avoids injecting a custom wrapper binary as the shell **if** the filter can be fully described by BPF rules passed to bwrap.

## Design fork: TRAP + SIGSYS vs user notification

### Approach A: `SCMP_ACT_TRAP` + `SIGSYS` handler (from research notes)

- Filter actions: **TRAP** for syscalls of interest → kernel delivers **SIGSYS** to the **thread** that invoked the syscall.
- Problems for this app:
  - The signal runs in the **sandboxed child** (the process running `/bin/sh -c`), **not** in the OLLMchat parent. A Vala `sigaction` in the parent does **not** observe the child’s traps.
  - To handle traps in-process you need a **wrapper** that runs **inside** the sandbox before `exec` of the shell, or you accept **process death** on trap (bad UX).
  - Async-signal-safe logging: ring buffer or `write()` to a **pipe** whose read end is held by the parent—again requires a cooperating process inside the namespace or passing the pipe FD through bwrap (possible but fiddly).

**Verdict**: Possible but awkward; best if you control the innermost executable. **Not** the first choice if bwrap + parent-side APIs can avoid it.

### Approach B: `SECCOMP_RET_USER_NOTIF` (supervisor in parent)

- Rules use **notify** action; a **supervisor** (parent or sibling thread) reads events from a **seccomp notify fd**, inspects syscall number and arguments, then allows/denies/kills.
- Gives **structured** access to syscall args (e.g. `openat` path) without a signal handler in the child.
- Requires **Linux** with user notification support and more code (listener loop, response fd). libseccomp documents notify APIs.

**Verdict**: Stronger for **path-based** “tried to write outside X” stories; more implementation cost.

### Approach C: `SCMP_ACT_LOG` / audit

- Kernel logs violations; userspace reads audit logs. Heavier operationally; usually worse for a desktop app.

### Pragmatic phasing

1. **Phase 1 (network evidence)**: Notify or trap **network-related** syscalls (`socket`, `connect`, `sendto`, …—list must be maintained per arch). Prefer **notify** in parent if already building notify machinery; else **trap + process exit** is only acceptable if you switch those syscalls to **kill** with a distinct code—still not “evidence” in the parent. Simplest **parent-visible** evidence path: **user notif** or **pipe from a tiny LD_PRELOAD** (often rejected for security clarity).
2. **Phase 2 (paths)**: Extend the same mechanism with `openat`, `unlink`, `rename`, etc., and compare resolved paths against allowed roots (project roots, optional user-approved extra dirs).

## Changes by area (if proceeding)

### 1. Build / dependencies

- Add `dependency('libseccomp')` (or `cc.find_library('seccomp')` + headers) to `liboctools/meson.build`.
- Add C glue or vapi; export BPF fd for bwrap.

### 2. `Request.execute_tool_async` — bubblewrap gate (**⚠️** prerequisite)

- **⚠️** When **`!Bubble.can_wrap()`**, **throw** or return a **dedicated error** type the agent/UI maps to a **user** dialog — **not** `execute_with_subprocess()`. Message: install **bubblewrap** / unavailable on this platform.
- **⚠️** Ensure the **tool request does not complete** as a normal assistant-visible “command output” string that looks like a shell failure.

### 3. `Bubble.build_bubble_args` / exec path

- After creating the filter FD, append `--add-seccomp-fd` / `--seccomp` with the **dup’d** FD as required by `Subprocess.newv` (ensure FD is not closed prematurely; meson may need `FD_CLOEXEC` handling—exact pattern TBD).
- Define **per-mode** profiles:
  - **Network allowed**: do not notify/trap network syscalls (or allow them).
  - **Network denied**: notify/trap network syscalls and record events.
  - **Read-only non-project**: deny or notify writes outside allowlist mounts.

### 4. Evidence reporting

- **After** `wait_async`, collect:
  - From notify queue, or
  - From pipe (if using a wrapper), or
  - From structured exit (if using kill + mapped errno—weak).
- Append to tool result **only when** evidence exists: e.g. “Blocked network syscall: connect (1×)” instead of the current blanket string in `read_subprocess_output`.

### 5. Tool schema + permissions (`Request.vala`, `Tool.vala`)

- **Non-project**: adjust `build_perm_question()` so default read-only runs **skip** approval when `network` is false and **no** extra write roots are requested; **require** approval when:
  - `network` is true, or
  - **`write_roots`** (or equivalent) is **non-empty** after parse/trim — show **resolved absolute paths** in the prompt.
- **⚠️** Add **`write_roots`** (name TBD): **string**, delimiter-separated list of **directory roots** per **§ Non-project write roots**; **backend** splits, normalizes, validates; **`Bubble`** gains **per-root binds** instead of a single **`--bind / /`** for non-project write.
- **⚠️** Update **`Tool.vala`** long description / `@param` lines with **LLM coaching** (narrow paths, avoid `/`, use **`;`** between roots — match implementation).
- Deprecate or narrow **`allow_write: "yes"` / `"all"`** for **non-project** once **`write_roots`** exists — avoid duplicate/conflicting semantics (**exact migration** in implementation PR).

### 6. Security and compatibility

- **Flatpak**: `Bubble.can_wrap()` is false — **no** silent `execute_with_subprocess` fallback once **§ Bubblewrap required** is implemented; show **explicit unavailable** UX (and seccomp/evidence N/A there until a supported story exists).
- **Architecture**: seccomp rules are architecture-specific; use libseccomp’s arch APIs or restrict to `x86_64`/`aarch64` explicitly in CI.
- **Maintenance**: syscall lists drift across kernels; prefer libseccomp constants and periodic review.

### 7. Future (optional): writable-mode history + “files touched” over time

**💩** Idea (user): If the user runs **`run_command` in writable modes** (e.g. broad write / project writes) **over an extended period** (e.g. a month), **track which files were modified** and surface or summarize that — useful for trust, review, or “what did commands change?”

**ℹ️** This is **not** required for the first seccomp/evidence milestone; treat as a **follow-on** if feasible.

**Why it is tricky**

- **Definition of “writable mode”** — Project runs use **overlay uppers** per root; non-project may use **rw root** or small binds. “Modified” might mean **host-visible** changes, **overlay upper files**, or **syscalls that opened paths for write** — three different datasets.
- **Where data lives today** — Overlay writes land under **`overlay_dir`** (see `Overlay` / `Bubble`); correlating those paths back to **real project paths** is already a mapping problem; **rw root** mode has **no** isolated upper — you would need **kernel-level** auditing (fanotify, eBPF, seccomp notify logs) or **periodic host scans** (expensive, racy).
- **Retention and privacy** — Storing a month of paths per project/session needs **storage policy**, **opt-in**, and **clear-user** deletion — overlaps with telemetry ethics.
- **Performance** — Continuous monitoring of all `openat`/`write` paths from notify can be heavy; batching and sampling need design.

**Possible directions (no commitment)**

- **💩** **Overlay-only audit** — When **`allow_write`** does **not** use full rw root, diff or inventory **`upper/`** after each run (or daily) — **bounded**, but **misses** rw-root mode entirely.
- **💩** **Seccomp notify log** — Append resolved paths (where the kernel provides them) to a **local ring buffer or sqlite** keyed by session — **feasible** if notify machinery exists for G2 anyway; completeness depends on syscall coverage.
- **💩** **Explicit “audit this run”** flag — Simpler UX: only record when user opts in, reduces storage and surprise.

**👎** Do not block G1–G4 on this; add a **separate plan** once notify/overlay story is stable.

## Open questions

1. **Exact** non-project default: only `$HOME` read-only + `/tmp` tmpfs vs also allowing read of full `/` (current `--ro-bind / /`)—UX vs privacy. **Related:** Is **`/tmp` writable** acceptable for “no writes outside project,” or must non-project be tightened (see **§ Non-project writable surfaces**)?
2. Whether **notify** fd can be threaded through `GLib.Subprocess` + bwrap without FD leaks on all platforms you ship.
3. User-facing wording: technical (syscall names) vs plain language (“The command tried to open a network connection”).
4. Interaction with **`allow_write: "all"`** for projects vs non-project — reconcile with **`write_roots`** list (projects may keep overlay model; non-project uses **path roots** — **§ Non-project write roots**).
5. **Delimiter** — confirm **`;`** vs alternatives for LLM parsing (commas break if paths need comma — unlikely on Unix).
6. **Long-term file-change tracking** — product wants vs technical cost (see **§ Changes by area → 7**).
7. **Agent/UI contract** — exact error type / signal when `run_command` is blocked (missing bwrap) so the **session** surfaces install instructions **without** sending tool stderr to the model (**⚠️** user requirement).

## Summary

| Question | Answer |
|----------|--------|
| Is libseccomp “bound to Vala” already? | **No** in this repo; add C bindings or a small C helper. |
| Is the approach feasible? | **Yes** on Linux with bwrap + libseccomp; bwrap **`--seccomp` / `--add-seccomp-fd`** is the natural integration point. |
