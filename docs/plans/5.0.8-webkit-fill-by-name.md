# 5.0.8 — WebKit `fill` by field name (not press-ref)

---

## Purpose

- **🔷** Redesign **`fill`** so keys are **field names / attributes** (how forms are normally addressed), not ephemeral **press-ref ids** from the last a11y output.
- **🔷** v1 press-ref → text map is surprising and brittle (ids change every dump).
- **🔷** **Do not implement this plan now** — track here until fill UX is locked.
- **ℹ️** Classic tool description currently documents press-ref keys; update `Tool.parameter_description` when this lands.

---

## Current behaviour (v1)

- **ℹ️** `fill` is a JSON object: `{ "1": "typed text", … }` where keys are **stringified press-ref ids**.
- **ℹ️** `A11y.fill` resolves each key via `press_routes` / focus + keyboard (or `set_text_contents`).
- **ℹ️** Editable fields in References may show `= value` but have no stable **name** key in the dump.

---

## Proposed behaviour (locked later)

- **🔷** `fill` keys identify **form fields by name/attribute** (exact source of the name TBD in design lock — accessible name, AT-SPI name, HTML `name` if exposed by a11y, or a dump-side stable id).
- **🔷** Dump / References must **surface those keys** so the model can copy them (same idea as press-refs for buttons, but for fillables).
- **🔷** `fill` remains an **argument** on `action: press` (not a separate action) unless redesign says otherwise.
- **💩** Whether numeric press-ref keys stay accepted as a transition alias.
- **💩** Whether fill-only (no press on the same call) is allowed.
- **🚫** Do not solve fill by injecting page JS / DOM `querySelector` (parent **5.0** veto).

---

## Work when picked up

- **🔷** `⏳` Design lock: which attribute / name is the canonical fill key on each platform.
- **🔷** `⏳` A11y dump: emit fill keys on editable nodes (Content and/or References).
- **🔷** `⏳` `A11y.fill` + `Request` / `Tool` docs: map name → text; drop or soft-deprecate press-ref keys.
- **🔷** `⏳` Recipes / spike: search box fill without inventing press ids for the text field.
