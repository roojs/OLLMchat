## ℹ️ Background: deprecating the Ollama-specific API

If we plan to **deprecate** the Ollama-native API (and eventually remove it), we need a clear picture of what changes and how to handle them. This section outlines the fundamental shifts and options, with code blocks for review.

### ℹ️ 1. What “deprecation” means concretely

**Current (Ollama-native):** Call and response types are tied to Ollama’s JSON shapes.

```vala
// Client — chat
public async Response.Chat chat_execute(Call.Chat call) throws Error
{
    return yield call.send(call.messages, call.cancellable);
}

// Client — embed
var call = new Call.Embed(this.connection, model) { ... };
var result = yield call.exec_embed();   // returns Response.Embed
```

**Target (OpenAI-compatible as single path):** Same public surface where possible, but under the hood we use v1 endpoints and OpenAI-shaped types; Ollama-specific types are internal or removed.

```vala
// Client — chat (v1 path only, after deprecation)
public async Response.Chat chat_execute(Call.ChatCompletions call) throws Error
{
    return yield call.send(call.messages, call.cancellable);  // or exec()/exec_stream()
}

// Client — embed (v1 path only)
var call = new Call.Embeddings(this.connection, model) { ... };
var result = yield call.exec_embeddings();   // returns Response.EmbeddingList
```

So “deprecate Ollama API” means: **stop using `Call.Chat`/`Call.Embed` and `Response.Chat`/`Response.Embed` as the primary contract**, and standardise on `Call.ChatCompletions`/`Call.Embeddings` and `Response.ChatCompletion`/`Response.EmbeddingList` (or an abstraction over them).

---

### ℹ️ 2. Fundamental changes required

#### ℹ️ 2.1 Call type surface

Everything that today takes or returns `Call.Chat` or `Call.Embed` must move to the v1 call types (or an abstraction that can represent either during transition).

| Current type        | Used in / as                         | Change |
|---------------------|--------------------------------------|--------|
| `Call.Chat`         | Agent `chat()`, Factory `configure_tools(call)`, Client `chat_execute(call)` | Replace with `Call.ChatCompletions` or abstract “chat call” type. |
| `Call.Embed`        | Client `embed()` / `embed_array()`   | Replace with `Call.Embeddings`. |

Example — Agent today:

```vala
// Agent/Interface.vala
public abstract Call.Chat chat();

// Agent/Base.vala
protected OLLMchat.Call.Chat chat_call;
```

After deprecation we need either:

- Concrete v1 only: `Call.ChatCompletions chat_call` and `Call.ChatCompletions chat()`, or  
- Abstraction: e.g. an interface or base type “ChatCall” implemented by `Call.ChatCompletions` (and temporarily by `Call.Chat` if we keep both during migration).

#### ℹ️ 2.2 Response type surface

Everything that today takes or returns `Response.Chat` or `Response.Embed` must accept the v1 response shapes (or a common adapter).

| Current type        | Used in / as                                                                 | Change |
|---------------------|-------------------------------------------------------------------------------|--------|
| `Response.Chat`     | Signals `stream_chunk(…, Response.Chat)`, `message_completed(Response.Chat)`; Session `handle_stream_chunk` / `finalize_streaming`; UI, tools, `write_stats` | Use `Response.ChatCompletion` / `Response.ChatCompletionMessage` or adapter. |
| `Response.Embed`     | Client return; embed consumers (`embeddings`, `embed_to_floats`)               | Use `Response.EmbeddingList` and `data[].embedding`. |

Example — streaming and completion today:

```vala
// Agent/Base.vala
public signal void stream_chunk(string new_text, bool is_thinking, OLLMchat.Response.Chat response);
public signal void message_completed(OLLMchat.Response.Chat response);

// Response.Chat — Ollama shape
public Message message { get; set; }
public string done_reason { get; set; }
public int64 total_duration { get; set; }
public int eval_count { get; set; }
// ...
public string addChunk(Json.Object chunk)  // NDJSON chunk from /api/chat
```

After deprecation, streaming and “message completed” would be expressed in terms of:

- **Streaming:** SSE chunks as `Response.ChatCompletionChunk`; accumulated result as `Response.ChatCompletionMessage`.
- **Non-streaming:** `Response.ChatCompletion` with `choices[0].message` and `usage`.

So either:

- All downstream code is updated to use `Response.ChatCompletion` / `Response.ChatCompletionMessage` (and optional `Response.Usage`), or  
- We introduce an **adapter** that implements the same surface as `Response.Chat` (e.g. `message`, `get_summary()`, `eval_count`) by wrapping the v1 types.

#### ℹ️ 2.3 Request/response shape differences (reminder)

**Chat request:** Ollama uses `options` (nested), `think`, `keep_alive`, `format`; OpenAI uses flat `temperature`, `top_p`, `max_tokens`, `response_format`. We already have `Call.ChatCompletions` building the OpenAI body; deprecation means no code path builds the Ollama body for chat.

**Chat response:** Ollama has top-level `message`, `done`, `done_reason`, `eval_count`, `total_duration`, etc. OpenAI has `choices[0].message`, `choices[0].finish_reason`, `usage`. So any code that today does:

```vala
response.message.content
response.eval_count
response.get_summary()
```

must either get that from the v1 types (e.g. `completion.choices[0].message.content`, `completion.usage.completion_tokens`, and a new summary helper) or from an adapter that exposes the same API.

**Embed response:** Ollama has `embeddings` (array of arrays). OpenAI has `data[].embedding` and `data[].index`. So:

```vala
// Today
var vec = response.embeddings[0];

// After deprecation (v1 only)
var vec = response.data.get(0).embedding;
```

Call sites that assume `Response.Embed` need to be updated or an adapter must provide `embeddings` from `data[].embedding`.

#### ℹ️ 2.4 Streaming mechanics

Today, `Call.Chat` uses NDJSON and `Response.Chat.addChunk(Json.Object chunk)` to accumulate Ollama chunks:

```vala
// Call/Chat.vala (Ollama streaming)
response.addChunk(chunk);
// ...
this.stream_chunk(new_text, is_thinking, response);
this.agent.handle_stream_chunk(new_text, is_thinking, response);
```

With v1, `Call.ChatCompletions.exec_stream()` already reads SSE and accumulates deltas into `Response.ChatCompletionMessage`. So the fundamental change is:

- **Emit** `stream_chunk` / `handle_stream_chunk` from the ChatCompletions path, with either:
  - A common type (e.g. adapter that looks like `Response.Chat`), or  
  - New signatures that pass `Response.ChatCompletionMessage` (and possibly `Response.ChatCompletionChunk`) so UI/history/tools no longer depend on `Response.Chat`.

---

### ℹ️ 3. How to handle deprecation

Two main approaches, with code-level implications.

#### ℹ️ Option A: Adapter layer (recommended for incremental migration)

- Keep **public/signal APIs** in terms of `Response.Chat` and `Response.Embed` (and optionally `Call.Chat` as an alias or thin wrapper).
- Internally use only `Call.ChatCompletions` / `Call.Embeddings` and v1 response types.
- **Adapters** turn v1 → current shape where needed:
  - `Response.ChatCompletion` / `Response.ChatCompletionMessage` → something that quacks like `Response.Chat` (`message`, `get_summary()`, `eval_count` from `usage`, etc.).
  - `Response.EmbeddingList` → something that quacks like `Response.Embed` (`embeddings` from `data[].embedding`).

Example (conceptual):

```vala
// Adapter: OpenAI completion → legacy Response.Chat surface for signals/UI
class ChatCompletionAdapter : Object
{
    public Response.ChatCompletionMessage message { get; set; }
    public Response.Usage? usage { get; set; }
    public string get_summary() { /* from usage */ }
    public Message message_legacy { get; private set; }  // built from message
}
```

Then Agent/Session/UI still use `Response.Chat` in signatures; internally we build an adapter from `Response.ChatCompletionMessage` + `Response.Usage` and pass that as `Response.Chat` (or a new “ChatResponse” type that both `Response.Chat` and the adapter implement).

- **Pros:** Minimal change to Agent, History, Manager, Tool, UI. One place (Client + adapters) owns the v1 vs legacy mapping.  
- **Cons:** Two shapes in play; adapter must be kept in sync with what downstream expects from `Response.Chat` / `Response.Embed`.

#### ℹ️ Option B: Single type surface (full cutover)

- Change all signatures to use **only** v1 types: `Call.ChatCompletions`, `Response.ChatCompletion` / `Response.ChatCompletionMessage`, `Call.Embeddings`, `Response.EmbeddingList`.
- Update every call site (Agent, Session, Manager, Tool, ChatWidget, ChatView, libocvector, liboctools, examples, ollmapp) to read from the new types (e.g. `choices[0].message`, `usage`, `data[].embedding`).
- Remove or relegate `Call.Chat`, `Call.Embed`, `Response.Chat`, `Response.Embed` to legacy/deprecated and eventually delete.

Example:

```vala
// Agent/Base.vala — after cutover
public signal void stream_chunk(string new_text, bool is_thinking,
    OLLMchat.Response.ChatCompletionMessage message, OLLMchat.Response.Usage? usage);
public signal void message_completed(OLLMchat.Response.ChatCompletionMessage message,
    OLLMchat.Response.Usage? usage);
```

- **Pros:** One clear contract; no adapter; easier to support other OpenAI-compatible backends.  
- **Cons:** Large, coordinated change across all libraries and apps; every consumer of chat/embed must be updated.

#### ℹ️ Recommendation

- **Short term:** Use **Option A** so Phase 2 can “switch to v1 under the hood” without rewriting every consumer. Add a connection or config flag to use `Call.ChatCompletions`/`Call.Embeddings`; map results through adapters so existing `Response.Chat`/`Response.Embed`-based code keeps working.
- **Medium term:** Introduce a shared “chat response” abstraction (e.g. interface) implemented by both `Response.Chat` and the adapter (or by the v1 types directly), and migrate consumers to that abstraction so we can eventually drop the Ollama-shaped types.
- **Long term:** If we fully deprecate the Ollama API, move to **Option B** and remove adapters and legacy types.

### ℹ️ 4. Summary checklist (fundamental changes)

- [ ] **Call types:** Replace or abstract `Call.Chat` / `Call.Embed` so the default path uses `Call.ChatCompletions` / `Call.Embeddings`.
- [ ] **Response types:** Either adapt v1 → `Response.Chat`/`Response.Embed` (Option A) or change all signatures to v1 types (Option B).
- [ ] **Streaming:** Drive `stream_chunk` / `message_completed` (and Session/UI) from `Call.ChatCompletions.exec_stream()` and v1 chunk types, via adapter or direct v1 types.
- [ ] **Embed consumers:** Switch to `Response.EmbeddingList` and `data[].embedding`, or expose same surface via adapter.
- [ ] **Agent/Factory/Tools:** `chat()` and `configure_tools` accept or return v1 call type (or abstraction).
- [ ] **History/Manager/Session:** Handle stream and final message as v1 or adapter type.
- [ ] **UI (libollmchatgtk) and other libs:** Same as above; update or add adapter so `write_stats` and display logic work with the chosen response type.

### ℹ️ Phase A: Embedding (easiest)

**Why first:** No streaming, simple request/response. Call sites are fewer and response shape is a direct mapping (`data[].embedding` ↔ `embeddings[]`).

#### ℹ️ How Ollama embed response is handled (and why the shape)

The **Ollama** embed path uses a **single GObject** (`Response.Embed`) and stores the vectors as **nested collections** — not as a separate GObject per vector. `Json.gobject_from_data` / `Json.Serializable` do not natively handle `Gee.ArrayList<Gee.ArrayList<double?>>`, so the response is **manually (de)serialized** via overrides.

**Response type:** One GObject, one property for all vectors:

```vala
// Response/Embed.vala — Ollama shape
public class Embed : Base
{
    public string model { get; set; default = ""; }
    // Array of vectors; each vector is ArrayList<double?> (no per-vector GObject)
    public Gee.ArrayList<Gee.ArrayList<double?>> embeddings { get; set;
        default = new Gee.ArrayList<Gee.ArrayList<double?>>(); }
    public int64 total_duration { get; set; default = 0; }
    // ...
}
```

**Deserialization:** Custom `deserialize_property` for `"embeddings"` — JSON array-of-arrays is walked by hand and turned into `ArrayList<ArrayList<double?>>`. So the **vector data is not a GObject**; it’s “list of list of doubles” stored on the one Embed instance:

```vala
// Response/Embed.vala — custom deserialize for embeddings
case "embeddings":
    var embeddings_list = new Gee.ArrayList<Gee.ArrayList<double?>>();
    var array = property_node.get_array();
    for (int i = 0; i < array.get_length(); i++) {
        var inner_array = array.get_array_element(i);
        var embedding = new Gee.ArrayList<double?>();
        for (int j = 0; j < inner_array.get_length(); j++) {
            var val_node = inner_array.get_element(j);
            if (val_node.get_value_type() == typeof(double))
                embedding.add(val_node.get_double());
            if (val_node.get_value_type() == typeof(int))
                embedding.add((double)val_node.get_int());
        }
        embeddings_list.add(embedding);
    }
    value = Value(typeof(Gee.ArrayList));
    value.set_object(embeddings_list);
    return true;
```

**Call site:** `Call.Embed.exec_embed()` parses the raw JSON into `Response.Embed` (so the *response* is a GObject), then normalizes in place; consumers use `.embeddings` and helpers that take `Gee.ArrayList<double?>`:

```vala
// Call/Embed.vala — parse then normalize
var embed_obj = Json.gobject_from_data(typeof(Response.Embed), json_str, -1) as Response.Embed;
foreach (var embedding in embed_obj.embeddings) {
    this.normalize_embedding(embedding);
}
return embed_obj;
```

```vala
// libocvector/Database.vala, VectorBuilder, etc. — consumer contract
if (response.embeddings.size == 0) { ... }
int dim = (int)response.embeddings[0].size;
vector_batch.add(this.embed_to_floats(response.embeddings[0]));  // ArrayList<double?> → float[]
```

So: the **response** is a GObject; the **embedding vectors** are not modeled as GObjects (no “EmbeddingDatum” type). They’re stored as `Gee.ArrayList<Gee.ArrayList<double?>>` with custom (de)serialization because an ArrayList of doubles isn’t something `Json.gobject` maps sensibly by default.

**OpenAI (v1) shape:** We *do* have a per-item GObject: `Response.EmbeddingDatum` (with `embedding` = `ArrayList<double?>` and `index`). So `Response.EmbeddingList` is a GObject with `data: ArrayList<EmbeddingDatum>`. Each datum is still (de)serialized by hand for the `embedding` array (same “ArrayList of double” issue). The only structural difference is: v1 has one GObject per vector (with index); Ollama has one GObject for the whole response and raw nested lists for the vectors. For Phase A, we can either keep returning an adapter that exposes the old `.embeddings` (list of list of doubles) from `data[].embedding`, or change consumers to use `Response.EmbeddingList` and `data[i].embedding` directly.

#### ℹ️ Embed response JSON (Ollama vs OpenAI)

Raw response shapes for reference. Same underlying data (N vectors of floats); only structure and metadata differ.

**Ollama `/api/embed` response:**

```json
{
  "model": "embeddinggemma",
  "embeddings": [
    [0.010071029, -0.0017594862, 0.05007221, 0.04692972, ...]
  ],
  "total_duration": 14143917,
  "load_duration": 1019500,
  "prompt_eval_count": 8
}
```

**OpenAI-style `v1/embeddings` response (Ollama OpenAI-compat / other providers):**

```json
{
  "object": "list",
  "data": [
    {
      "object": "embedding",
      "embedding": [0.0023064255, -0.009327292, ..., -0.0028842222],
      "index": 0
    }
  ],
  "model": "text-embedding-ada-002",
  "usage": {
    "prompt_tokens": 8,
    "total_tokens": 8
  }
}
```

| | Ollama native | OpenAI-style |
|--|----------------|--------------|
| Vectors | `embeddings`: array of float[] | `data[].embedding`: float[] |
| Per-vector index | none | `data[].index` |
| Token info | `prompt_eval_count` | `usage.prompt_tokens` / `usage.total_tokens` |
| Timing | `total_duration`, `load_duration` | none |

**Conclusion:** Use only our existing `Response.Embed`. Add `prompt_tokens` and `total_tokens` (flattened, from v1 when present). For the v1 path, parse JSON and fill `embeddings` from `data[].embedding`; no `EmbeddingList`, no `EmbeddingDatum`, no `data` list — just our existing array-of-arrays with a few extra properties.

**Current vs proposed — response type:**

**Current (what exists):** New OpenAI-shaped type with `data` = list of EmbeddingDatum.

```
EmbeddingList {
    object  : string
    data    : Gee.ArrayList<EmbeddingDatum>   // one object per vector (embedding + index)
    model   : string
    usage   : Usage?
}
```

**Usage:** In the OpenAI API, `usage` is a standard nested object. We **flatten** `prompt_tokens` and `total_tokens` onto the parent (no nested `Usage` for this response).

**Proposed:** Use our **existing** `Response.Embed` as the only response type. Add two optional properties for v1; no new types, no `data` / ArrayList of per-vector objects. The API’s `object`, `embedding`, and `index` are pointless for us — we already have array of arrays.

```
Embed {                                         // our existing class — single response type
    model             : string
    embeddings        : Gee.ArrayList<Gee.ArrayList<double?>>   // unchanged; array of vectors
    total_duration    : int64
    load_duration     : int64
    prompt_eval_count : int
    prompt_tokens     : int                     // from v1 when present (flattened)
    total_tokens      : int                     // from v1 when present (flattened)
}
```

So: call `/v1/embeddings`, parse JSON, copy `data[].embedding` into `Embed.embeddings` (list of list of double), set `model`, set `prompt_tokens` / `total_tokens` when present. No `EmbeddingList`, no `EmbeddingDatum` — just our existing structure with a few extra properties.

### ✅ 1. Response/Embed.vala — add properties and parse v1 format

Add after `prompt_eval_count`:

```vala
public int prompt_tokens { get; set; default = 0; }
public int total_tokens { get; set; default = 0; }
```

**Extend `deserialize_property`** so Embed can parse both Ollama and v1 JSON. Keep existing `"embeddings"` case (Ollama). Add:

**Case `"data"`** (v1: array of `{ "embedding": number[], "index" }`):

```vala
case "data":
	var embeddings_list = new Gee.ArrayList<Gee.ArrayList<double?>>();
	var data_array = property_node.get_array();
	for (int i = 0; i < data_array.get_length(); i++) {
		var item = data_array.get_object_element(i);
		var emb_node = item.get_member("embedding");
		if (emb_node == null || emb_node.get_node_type() != Json.NodeType.ARRAY) {
			continue;
		}
		var inner = emb_node.get_array();
		var embedding = new Gee.ArrayList<double?>();
		for (int j = 0; j < inner.get_length(); j++) {
			var val_node = inner.get_element(j);
			if (val_node.get_value_type() == typeof(double)) {
				embedding.add(val_node.get_double());
			} else if (val_node.get_value_type() == typeof(int)) {
				embedding.add((double)val_node.get_int());
			}
		}
		embeddings_list.add(embedding);
	}
	this.embeddings = embeddings_list;
	value = Value(typeof(Gee.ArrayList));
	value.set_object(this.embeddings);
	return true;
```

**Case `"usage"`** (v1: `{ "prompt_tokens", "total_tokens" }`):

```vala
case "usage":
	var usage_obj = property_node.get_object();
	this.prompt_tokens = (int)usage_obj.get_int_member("prompt_tokens");
	this.total_tokens = (int)usage_obj.get_int_member("total_tokens");
	value = Value(typeof(int));
	value.set_int(0);
	return true;
```

So one Embed type works for both response shapes; no EmbeddingList or EmbeddingDatum. In `serialize_property`, exclude `prompt_tokens`/`total_tokens` from JSON if desired (or allow default).

### ✅ 2. libollmchat/Call/Embeddings.vala — string[] input, exec_embedding() returns Response.Embed

**Replace `input` + `input_array` (ArrayList) with a single `string[] input`.** Serialize `input` as the JSON "input" array when building the request.

**Replace `exec_embeddings()` with `exec_embedding()` returning Response.Embed.** Parse v1 (or Ollama) JSON directly into **Response.Embed** via `Json.gobject_from_data(typeof(Response.Embed), json_str, -1)`. Embed now deserializes both formats (see §1), so no EmbeddingList or EmbeddingDatum.

```vala
// Properties: drop input (string) and input_array (ArrayList). Use only:
public string[] input { get; set; default = new string[0]; }  // empty, not null; .length works, no null checks
public int dimensions { get; set; default = -1; }
// ... model, encoding_format, cancellable as needed

// serialize_property for "input": build Json.Array from this.input (string[]; default {} so no null check)
case "input":
	if (this.input.length == 0) {
		return null;
	}
	var arr = new Json.Array();
	foreach (var s in this.input) {
		arr.add_string_element(s);
	}
	var node = new Json.Node(Json.NodeType.ARRAY);
	node.init_array(arr);
	return node;

// exec_embedding() returns Response.Embed; Embed parses both Ollama and v1 JSON
public async Response.Embed exec_embedding() throws Error
{
	var bytes = yield this.send_request(true);
	var root = this.parse_response(bytes);
	if (root.get_node_type() != Json.NodeType.OBJECT) {
		throw new OllmError.FAILED("Invalid JSON response");
	}
	var generator = new Json.Generator();
	generator.set_root(root);
	var json_str = generator.to_data(null);
	var embed = Json.gobject_from_data(
		typeof(Response.Embed), json_str, -1) as Response.Embed;
	if (embed == null) {
		throw new OllmError.FAILED("Failed to deserialize embeddings response");
	}
	return embed;
}
```

## ℹ️ Notes

- **Call.Embeddings** is the direct entry point; no Client.embedding() in the new path. Callers instantiate Call.Embeddings, set `input` (string[]), call `exec_embedding()`, get Response.Embed.
- **Call.Embeddings** uses **string[] input** only (no ArrayList). **exec_embedding()** returns **Response.Embed**; EmbeddingList/data are not in the public flow.
- Response.Embed gets optional `prompt_tokens`/`total_tokens`. Chat is in plan 1.24.3; generate is in plan 1.24.2.
