# Plan 1.23.23: Task iterator behaviour, task creator scope, skill split, and Grep skill

## Status

🔄 **In progress** — Phase 0 done; Phases 1–9 pending.

## Coding standards (apply to all phases)

- **No nullable strings:** Do not use `string?` (nullable string). Use non-nullable `string` and the empty string `""` to mean “none” or “default” (e.g. optional suffix, optional fragment). All new and touched code in this plan must follow this.

## Purpose

Improve the skills pipeline by: (1) **supporting the new refinement process** (execution on tool outputs or on reference documents when no tools); (2) **file-based storage** for task list and per-task outputs with session layout; (3) **injecting completed tasks summary into the refine stage**; (4) running the task iterator after every step and output only new/uncompleted tasks; (5) limiting the task creator to initial tasks and adding "Recommended next steps"; (6) skill format (refinement vs execution — already in skills; Runner/prompts to match); (7) adding a Grep skill for detailed result locations.

## Phase 1: File-based storage for steps (priority)

**Issue:** Task list and per-task outputs are not yet persisted in a stable, referenceable way.

**Goal:**

- **Each step gets a filename:** `task://task-list.md`; per task `task://{name of task}.md` (e.g. `task://research-online-search-1.md`). Use `#` to refer to parts (e.g. `task://research-1.md#result-summary`).
- **Session layout:** Session file `history/2026/01/01/22-51-02.json`; task output files in `history/2026/01/01/22-51-02/{files}` (e.g. `task-list.md`, `research-online-search-1.md`). Each completed task's output is written so refinement and iteration can reference them.

**Scope:** Runner and persistence; session directory naming; writing task list and per-task output; resolving `task://` or session-relative paths when building precursor/refinement context.

| File | Layer |
|------|--------|
| `libollmchat/History/SessionBase.vala` | Session path API |
| `libollmchat/History/Session.vala` | Session path API |
| `libollmchat/History/SessionPlaceholder.vala` | Session path API |
| `libollmchat/History/EmptySession.vala` | Session path API |
| `liboccoder/Skill/Runner.vala` | Persistence + task:// resolution |
| `liboccoder/Task/List.vala` | Persistence hook |
| `liboccoder/Task/Details.vala` | Reference validation + write(suffix) |

### Phase 1: Summary of changes (bullets)

- **SessionBase.vala:** Add abstract (or virtual) method `task_dir()` returning the directory for task output files: `history_dir / to_path()` (e.g. `history/2026/01/01/22-51-02`). Subclasses implement or return `""`.
- **Session.vala:** Implement `task_dir()` as `Path.build_filename(manager.history_dir, to_path())`.
- **SessionPlaceholder.vala:** Implement `task_dir()` the same way (so loaded sessions have a task dir when Runner runs).
- **EmptySession.vala:** Implement `task_dir()` to return `""` (no task dir until converted to real Session).
- **Runner.vala:**  
  - Add `write()` (no arguments): ensure `session.task_dir()` exists; write `task_list.to_markdown()` to `task-list.md`. Call at start of `handle_task_list` and after each `run_task_list_iteration`.
  - In `reference_content(string href)`: if `href` has prefix `task://`, parse path and optional `#fragment` (fragment as non-nullable string, `""` when absent); resolve path under `session.task_dir()`; read file; if fragment non-empty, parse markdown and return that section; else return full content. Return `""` when session has no task dir (e.g. EmptySession) or file missing.
- **List.vala:** In `run_child`, after `yield t.post_evaluate()`, call `t.write()` so each completed task’s output is written via Details (write() has access to `this.slug()`).
- **Details.vala:** Add `write(string suffix = "")`: ensure session task dir exists (via runner.session.task_dir()); output filename = `this.slug() + suffix + ".md"` (Phase 1: call with default `""` so filename is `slug + ".md"`). Write this task’s `result_document.to_markdown()` to that file. *Later* (Phase 2): tool/ref results will need to be referenceable — we may need a map (e.g. task tool ids ⇒ TaskResult) and possibly a new class for TaskResult (individual tool or reference execution results); for Phase 1 just use suffix `""`. In `validate_references()`: accept `task://` scheme; validate target exists and referenced task is from an earlier section when applicable. In `reference_contents()`: when `link.scheme == "task"`, call `runner.reference_content(link.href)`.
- **Summaries:** Modify the completed-tasks summary (injected into refine stage) so it includes the `task://` URL for each completed task (e.g. `task://research-1.md` or `task://research-1.md#result-summary`). That way the refiner sees the reference link and can use it in References; resolution then pulls from the written files.

## Phase 2: Support new refinement process (priority)

**Issue:** The Runner and prompts must implement the refinement/execution flow that the skills now describe: refinement produces tool calls and/or fills in References; execution runs on the **output of those tools** or, when there are **no tools**, on the **reference documents** produced in refinement.

**Goal:**

- **When the skill has tool calls:** Refinement emits tool calls; Runner runs them; **execution receives the tool outputs** (e.g. combined or per-call) and produces Result summary + body sections. Execution runs **once per task** on the aggregated tool output for that task.
- **When the skill has no tool calls:** Refinement uses "what is needed" and **task list summary** to **fill in References** (e.g. links to prior task outputs, plan file, code sections). **Execution then runs on the reference documents** (resolved content of those References) as Precursor. No tool output; the "precursor" is the resolved References only.

**Scope:** Runner (how it invokes refinement vs execution, what it passes to execution), `task_execution.md`, and refinement prompt (must receive task list summary and produce References when no tools). Ensure skills that have no tools (e.g. plan_review, analyze_code) get References resolved and passed as precursor to execution.

**Done in skills (Runner/prompts to implement):** Refinement format (purpose of skill; receive what is needed + task list summary; fill in References or emit tool calls; avoid whole files). Execution receives single "output" (tool results or reference content). Task execution template: Result summary + body sections with descriptive titles; list sections as links; no Output References section.

**To do (concrete):**

- **Reference resolution (priority):** Prioritize resolution of **new task URLs** (e.g. `task://`, session task output paths like `history/…/task-name.md`) so that links to prior task outputs can be resolved. Without this, reference resolution cannot work for task outputs. Implement or extend existing logic to resolve task References (file paths, `#task-name-results`, **task:// and session task output paths first**) into concrete content for Precursor.
- **Refinement prompt:** Add a placeholder (e.g. `{task_list_summary}`) and ensure the Runner injects a summary of completed tasks so far (task name + Result summary, and section links) so the refiner can fill in References and/or decide tool calls.
- **Refinement output:** Parser must read the refined task’s **References** (and Skill call, Tool calls) from the refinement response; Runner stores that list of references on the task as a property (e.g. `task.references`) so execution can use them to build Precursor.
- **Runner — skills with tool calls:** Same as today (run tool calls, collect outputs) but **per task**: for this task only, run its tool calls, collect all tool outputs for that task, build Precursor from those outputs, invoke execution once for that task. No change to "we output all tool calls" — just scoped to a single task's calls and execution.
- **Runner — skills with no tool calls (key change):** After refinement, do **not** run any tools. **Run the executor on each reference**: resolve each of the task's References to content, then run the executor on each reference (e.g. once per reference document). This is the key change from "run once on combined precursor" to "run on each reference."
- **Detect “no tool calls”:** From refinement output (or skill definition), determine whether this task has tool calls; branch to “execution on tool outputs” vs "execution on each reference" accordingly.
- **Task execution template:** Confirm `task_execution.md` describes Precursor as “tool outputs + resolved References” (or “resolved References only” when no tools) and that executor output is Result summary + body sections with descriptive titles and section links; no Output References section.

## Phase 3: Completed tasks version for refine stage (priority)

**Issue:** Refinement needs a **version of the completed tasks** to fill in References and decide tool calls. Today this may not be injected consistently.

**Goal:** **Inject into the refine stage** a structured summary of completed tasks so far: task name/slug and **Result summary** (and section links) for each completed task. So the refiner can fill in References from prior outputs and use summaries to create tool calls (e.g. research_web_page: URLs from summaries).

**Scope:** Runner (build task list summary from current task list + session files if using file-based storage); refinement prompt placeholder `{task_list_summary}` or `{completed_tasks}`; format (e.g. markdown list of task name + Result summary + link to task output file/section).

## Phase 4: Task iterator — run after every step

**Issue:** The task list iteration (intermediary analyst) currently runs only after **all** tasks in a round are complete. This can delay discovery of new work and make the flow feel batch-oriented.

**Goal:** Run the **task iterator** after **every** completed task (or after every step), not only after the full task list for the round is done. This allows the pipeline to reassess goals and propose new or follow-up tasks as soon as one task finishes, keeping the plan responsive and incremental.

**Scope:** Runner (`liboccoder/Skill/Runner.vala`) and any orchestration that decides when to call `run_task_list_iteration()`. Today iteration is invoked once per round (e.g. after all tasks in the list have been executed); change to invoke it after each task completion (or each “step”), with the current task list (including that task’s Output) as input.

**Considerations:** May need to define “step” (single task vs. full section). Also ensure we don’t overwhelm the user with iteration prompts; optionally throttle or summarize.

## Phase 5: Task iterator — output only new or uncompleted tasks

**Issue:** The task list iteration prompt currently asks the LLM to output the **full** task list (all tasks with their Output lines) plus any new tasks. That is verbose and can obscure what actually changed.

**Goal:** The task iterator should output **only new or uncompleted tasks** (and possibly a short delta summary), not the entire list every time. Downstream (Runner, parser) must still be able to merge this with the existing task list so the full list stays consistent.

**Scope:** Prompt (`resources/skill-prompts/task_list_iteration.md`) and Runner/parser behaviour. The iteration response could take the form: “New/uncompleted tasks only” (e.g. new task sections or new tasks to append, plus any tasks that need re-run or are still uncompleted). The Runner merges these into `this.task_list` so the canonical list remains complete.

**Considerations:** Parser and task list representation must support “append/merge” rather than “replace entire list” when the iteration output is partial. May need a clear format for “tasks to add” vs “tasks still uncompleted” so the Runner can update state correctly.

## Phase 6: Task creator — only initial tasks; add “Recommended next steps”

**Issue:** The task creator currently produces a full coarse task list. There is no clear separation between “initial tasks we can run now” and “what to consider doing next.” Users and the system would benefit from an explicit “Recommended next steps” section that suggests what to consider doing next, without the task creator inventing those as full tasks yet.

**Goal:**

- **Task creator** produces **only the initial set of tasks** (the first round of work). It does **not** try to enumerate every possible follow-up task.
- After the task list (initial tasks), the task creator **outputs a section: “# Recommended next steps”** — short, high-level suggestions for what to consider doing next (e.g. “Consider running analysis on the findings,” “If the user approves, add implement_code tasks for the plan”). This is **advisory only**; the **task iterator** (run after every step) is responsible for actually proposing new tasks when goals are not yet complete.

**Scope:** Task creation prompt (`resources/skill-prompts/task_creation_initial.md`) and any runner/parser that consumes the task creator output. Ensure the output format includes **## Recommended next steps** (or equivalent) so it can be displayed to the user and optionally used as hints for the task iterator. No change to the task iterator’s responsibility: it still compares completed outputs to goals and adds new tasks when needed.

## Phase 7: Skill format — split into refinement part and execution part

**Issue:** Skills are currently single documents that mix “what the refiner should do” and “what the executor should do.” This can make it harder to maintain and to tune each phase separately.

**Goal:** **Split each skill into two clear parts:**

1. **Refinement part** — What the refiner does: which tool calls to emit, what arguments to pass, what to put in References, and how to fill in the Skill call. This is the “prepare this task for execution” behaviour.
2. **Execution part** — What the executor does: how to interpret Precursor and tool outputs, how to produce Result summary and Detail (and any skill-specific sections), and how to format output (e.g. AST links, Recommended next steps).

**Scope:** Skill file format and docs (`docs/skills-format.md`), and all skill files in `resources/skills/`. Each skill markdown can have two main sections (e.g. `## Refinement` and `## Execution`) so that the refinement prompt and execution prompt can pull the relevant part. Tool definitions and skill catalog (name, description) remain shared; only the body is split by phase.

**Considerations:** Refinement and execution prompts (e.g. `task_refinement.md`, `task_execution.md`) already receive the full skill document. We can either (a) keep one file with two sections and have the prompts instruct the LLM to focus on the relevant section, or (b) split into two files per skill (e.g. `analyze_codebase.refinement.md` and `analyze_codebase.execution.md`) and have the Runner pass the appropriate one. Option (a) is simpler and keeps one skill = one file.

## Phase 8: New skill: Grep tool (search + detailed result locations)

**Issue:** The codebase has a **codebase_search** (vector/semantic) tool and a **Grep** wrapped tool (`resources/wrapped-tools/grep.tool`) that runs `grep` with arguments. There is no **skill** that uses Grep and describes when to use it, and no standardized way to get **detailed information about result locations** (file, line, context) in a form that fits the skills pipeline.

**Goal:** Add a **new skill** that:

- Uses the **Grep** tool to search for patterns in the codebase (and optionally docs).
- **Returns detailed information about result locations:** not just “matches,” but file path, line numbers, surrounding context (e.g. a few lines before/after), and optionally symbol or scope (if derivable). Output should be markdown-friendly and usable as precursor for follow-up tasks (e.g. analyze_code, implement_code).

**Scope:**

- **Skill file:** New file in `resources/skills/` (e.g. `grep_code.md` or `search_pattern.md`). Description: use when you need to find exact text/pattern matches and get detailed location and context (file, line, snippet). Differentiate from `analyze_codebase` (semantic/codebase_search): Grep is for pattern/regex and precise locations; codebase_search is for “find code that does X.”
- **Tool:** The existing `Grep` wrapped tool (`resources/wrapped-tools/grep.tool`) runs `grep` with an array of arguments. The skill’s refinement step should emit **Grep** tool calls with appropriate arguments (e.g. `-r`, `-n`, pattern, path). The skill’s execution step should format the tool output into **Result summary** and **Detail** (or a “Grep results” section) with clear location info: file, line, and a short context snippet per match.
- **Output format:** Prefer the same markdown-friendly style as other skills (bullets, **KEY** value, fenced code where useful). Include enough detail so that “where did we find this?” is answerable and linkable (e.g. file path and line, or file path with line range).

**Considerations:** The Grep tool’s raw output is terminal grep output (line-oriented). The skill (or a small formatter in the tool path) may need to parse that and turn it into structured “result locations” with file, line, and context. If the wrapped tool does not currently return structured data, we may need to either (a) add a wrapper that parses grep output and returns markdown-friendly blocks, or (b) have the skill’s execution instructions tell the LLM how to parse and present the grep output. Option (a) gives consistent, parseable output for downstream tasks.

## Phase 9: Other items to address (from 1.23.22 and flow)

- **Goal fixed (no drift):** Original user goal must be fixed at the start and passed through unchanged; no LLM step (task creation, refinement, execution) should modify it. Inject the user's original prompt as "original prompt" in the task list; do not have the LLM generate or paraphrase it. (1.23.22 §3.)
- **Retry limits:** When searches (documentation or code) return nothing, limit retries (e.g. 2–3); then record "not found" or try fallback, do not retry indefinitely. Add to task creation / planner prompts. (1.23.22 §2.)
- **Code search target file:** Code search tool should support targeting a specific file or path scope so "search in file X" is possible. (1.23.22 §1.)
- **Validation:** Ensure skill exists (e.g. `skill_manager.validate(Task)`); validate reference targets (e.g. `#anchor`, `output:`, paths) before execution so refinement output is usable.
- **Task list summary content:** When building the "completed tasks" blob for refinement, include only Result summary text and section links (not full body) so the refiner gets a compact view; optionally link to `task://` or session file for full output.
