# Plan 1.24: OpenAI API chat and embed compatibility

## Overview

Move from Ollama-specific API calls to the OpenAI-compatible endpoints for **chat** and **embed**. Ollama supports the OpenAI API standard for chat (`/v1/chat/completions`) and embeddings (`/v1/embeddings`). Using these endpoints improves portability and allows the app to work with other OpenAI-compatible servers (e.g. OpenRouter, Together, or direct OpenAI) without changing call sites.

**Reference implementation:** `libollmchat/Call/Models.vala` â€” already uses the OpenAI-compatible pattern: `is_openai = true`, `url_endpoint = "v1/models"`, same connection URL as Ollama-native calls; `build_url()` strips `/api` and appends the endpoint. New chat/embed classes should follow this pattern.

**References:**
- [Ollama OpenAI compatibility](https://docs.ollama.com/api/openai-compatibility)
- [Ollama native /api/chat](https://docs.ollama.com/api/chat)
- OpenAI Chat Completions and Embeddings API (platform.openai.com)

## Rationale

- **Current:** We use Ollama-native endpoints: `/api/chat` and `/api/embed`. These are not part of the OpenAI API standard.
- **Target:** Use OpenAI-style endpoints: `/v1/chat/completions` and `/v1/embeddings`, following the same pattern as **`Call.Models`**: `is_openai = true` and `url_endpoint = "v1/chat/completions"` (or `"v1/embeddings"`). Then `Call.Base.build_url()` strips `/api` from the connection URL and appends the endpoint, so the same connection (e.g. `http://localhost:11434/api`) works for both Ollama-native and v1 endpoints.
- **Benefits:** One request/response shape for chat and embed across Ollama and other providers; easier to add or switch backends later; aligns with existing use of `v1/models` (see plan 1.17).

## Phased approach

- **Phase 1 (this plan):** Add **separate classes** in the **same namespace** (`OLLMchat.Call`, `OLLMchat.Response`) that represent the OpenAI API request/response shapes and perform the HTTP calls. These classes are **available** (e.g. for tests or direct use) but **not integrated** into Session, agent flow, history, or UI. Existing `Call.Chat`, `Call.Embed`, `Response.Chat`, `Response.Embed` remain unchanged and continue to be used by the app.
- **Phase 2 (later):** Integrate the OpenAI call path into Session/agent (e.g. choose OpenAI vs Ollama per connection, map OpenAI responses into existing UI/history if desired). Not in scope for Phase 1.

## New classes (OpenAI data standards, same namespace)

All new types live under existing namespaces: `OLLMchat.Call` and `OLLMchat.Response`.

### Call layer (OLLMchat.Call)

- **`Call.ChatCompletions`** â€” New class (separate from `Call.Chat`). Same pattern as **`Call.Models`**: `is_openai = true`, `url_endpoint = "v1/chat/completions"`. Builds request body in OpenAI form (model, messages, stream, max_tokens, temperature, top_p, response_format, tools). Handles SSE streaming and non-streaming. Returns OpenAI-shaped response types (see below). Subclasses `Call.Base`; implements its own request serialization and response parsing for the OpenAI spec.
- **`Call.Embeddings`** â€” New class (separate from `Call.Embed`). Same pattern as **`Call.Models`**: `is_openai = true`, `url_endpoint = "v1/embeddings"`. Request: model, input (string or array), optional dimensions, optional encoding_format. Returns `Response.EmbeddingList` (or equivalent) with `data[]` and `usage`.

### Response layer (OLLMchat.Response)

New classes that mirror the **OpenAI API response shapes** (so we can serialize/deserialize to and from the exact JSON). Existing `Response.Chat` and `Response.Embed` stay as-is for the Ollama pipeline.

**Chat completions:**

- **`Response.ChatCompletion`** â€” Top-level response: `id`, `object`, `created`, `model`, `choices` (array of `Response.ChatCompletionChoice`), optional `usage` (`Response.Usage`).
- **`Response.ChatCompletionChoice`** â€” One choice: `index`, `message` (`Response.ChatCompletionMessage`), `finish_reason`.
- **`Response.ChatCompletionMessage`** â€” `role`, `content`, optional `tool_calls` (array; structure matches OpenAI).
- **`Response.Usage`** â€” `prompt_tokens`, `completion_tokens`, `total_tokens` (optional fields as per API).

For streaming, either a dedicated type for a single SSE chunk (e.g. `Response.ChatCompletionChunk` with `choices[].delta`) or parse each SSE line into a known type: `Response.ChatCompletionChunk` (choices â†’ `ChatCompletionChunkChoice` with delta â†’ `ChatCompletionDelta`, finish_reason; delta has content?, role?, tool_calls? â†’ `ToolCallDelta`). Use `Json.gobject_from_data(typeof(Response.ChatCompletionChunk), payload, -1)` in `exec_stream()` and read from the typed object; still accumulate into `Response.ChatCompletionMessage`; plan can specify â€śstreaming returns same message type built from deltasâ€ť.

**Embeddings:**

- **`Response.EmbeddingList`** (or `Response.Embeddings` to avoid clash with call name) â€” Top-level: `object`, `data` (array of `Response.EmbeddingDatum`), `model`, optional `usage` (`Response.Usage`).
- **`Response.EmbeddingDatum`** â€” `embedding` (array of doubles), `index` (int). Matches OpenAI `data[].embedding` and `data[].index`.

Naming: if `Response.Embeddings` is used for the list, the call class can stay `Call.Embeddings`; both in same namespace is fine. Alternatively name the response type `Response.EmbeddingList` to avoid confusion with the call.

### No new namespace

All of the above live in `OLLMchat.Call` and `OLLMchat.Response`. No `OLLMchat.OpenAI` or similar.

## 1. Chat: API calls and differences

### 1.1 Endpoint and URL

- **Current:** `Call.Chat` sets `url_endpoint = "chat"`, `is_openai = false` â†’ `{base}/api/chat` (e.g. `http://localhost:11434/api/chat`).
- **Target (new classes):** Follow **`Call.Models`**: `Call.ChatCompletions` sets `is_openai = true`, `url_endpoint = "v1/chat/completions"`. Then `build_url()` strips `/api` from the connection URL and appends the endpoint â†’ `http://localhost:11434/v1/chat/completions`. Same connection URL works as for Models.

### 1.2 Request body (differences)

| Current (Ollama /api/chat)     | OpenAI /v1/chat/completions   | Notes |
|-------------------------------|------------------------------|--------|
| `model`                       | `model`                      | Same |
| `messages`                    | `messages`                   | Same array of `{ role, content }` (and tool_calls / tool results). Content can be string or array of parts (e.g. text + image_url). |
| `stream`                      | `stream`                     | Same |
| `options` (nested object with `temperature`, `top_p`, `num_ctx`, â€¦) | Top-level `temperature`, `top_p`, `max_tokens`, â€¦ | OpenAI uses flat params; no `num_ctx` (Ollama-specific; context size is via Modelfile/temp model). |
| `format` (string "json" or object) | `response_format` (e.g. `{ type: "json_object" }` or `{ type: "json_schema", json_schema: {...} }`) | Name and shape differ. |
| `think`                       | â€”                            | Ollama-only; thinking models may expose reasoning via message content or vendor-specific fields. For OpenAI compat we may omit or map if server supports. |
| `keep_alive`                  | â€”                            | Ollama-only; not in OpenAI. Omit for v1. |
| `tools`                       | `tools`                      | Same idea (array of tool definitions). |
| â€”                             | `tool_choice`                | OpenAI supports "none" / "auto" / specific tools. We donâ€™t use today; can add later. |
| â€”                             | `stop`, `seed`, `presence_penalty`, `frequency_penalty` | We can add where needed. |

**Planned request changes:**

- Add a path (e.g. connection flag or call-level flag) to send chat requests to `v1/chat/completions` with OpenAI-shaped JSON.
- Map our existing `Call.Options` (temperature, top_p, etc.) and `format` / `format_obj` into OpenAI top-level fields and `response_format`.
- Omit or gate Ollama-only fields (`think`, `keep_alive`, `options.num_ctx`) when using the OpenAI endpoint; document that custom context size still requires Modelfile/temp model when talking to Ollama.

### 1.3 Response body (non-streaming)

- **Current:** Single object: `model`, `created_at`, `message` (role, content, thinking?, tool_calls?), `done`, `done_reason`, `total_duration`, `load_duration`, `prompt_eval_count`, `prompt_eval_duration`, `eval_count`, `eval_duration`.
- **OpenAI:** `id`, `object`, `created`, `model`, `choices`: `[{ index, message: { role, content, tool_calls? }, finish_reason }]`, `usage`: `{ prompt_tokens, completion_tokens, total_tokens }`.

**Planned response handling (Phase 1):**

- Parse the response into the **new** OpenAI-shaped types: `Response.ChatCompletion`, `Response.ChatCompletionChoice`, `Response.ChatCompletionMessage`, `Response.Usage`. No conversion to existing `Response.Chat` or `Message` in Phase 1; that belongs in Phase 2 (integration).

### 1.4 Response body (streaming)

- **Current:** NDJSON chunks with `message` (role, content, thinking?, tool_calls?), `done`, `done_reason`, and timing fields. `Response.Chat.addChunk()` expects this shape (e.g. `chunk.get_object_member("message")`).
- **OpenAI:** SSE stream; each `data:` line is a JSON object with `choices`: `[{ index, delta: { content?, role?, tool_calls? }, finish_reason? }]`, `model`, `created`. Content and tool_calls are incremental (deltas).

**Planned streaming (Phase 1):**

- In the **new** `Call.ChatCompletions` class: consume SSE, parse each `data:` line as a chunk object with `choices[].delta`. Accumulate deltas into a single `Response.ChatCompletionMessage` (and optional `Response.Usage` if provided in a final chunk). Expose the accumulated message (and finish_reason) to the caller. No use of existing `Response.Chat.addChunk()` in Phase 1; the new call returns OpenAI-shaped types only.

### 1.5 API calls: new classes (chat)

- **New class `Call.ChatCompletions`** (separate from `Call.Chat`), same URL pattern as **`Call.Models`**:
  - `is_openai = true`, `url_endpoint = "v1/chat/completions"`.
  - Request body in OpenAI form: model, messages, stream, max_tokens, temperature, top_p, response_format, tools (no think, keep_alive, nested options).
  - Non-streaming: parse response into `Response.ChatCompletion` (choices[0].message, usage).
  - Streaming: read SSE, accumulate `choices[0].delta` into `Response.ChatCompletionMessage`, return that (and finish_reason when present).
- **Existing `Call.Chat`** unchanged; continues to use `/api/chat` and `Response.Chat` for the current app flow.

## 2. Embed: API calls and differences

### 2.1 Endpoint and URL

- **Current:** `Call.Embed` sets `url_endpoint = "embed"`, `is_openai = false` â†’ `{base}/api/embed`.
- **Target (new class):** Follow **`Call.Models`**: `Call.Embeddings` sets `is_openai = true`, `url_endpoint = "v1/embeddings"` â†’ `build_url()` yields `http://localhost:11434/v1/embeddings` with the same connection.

### 2.2 Request body (differences)

| Current (Ollama /api/embed) | OpenAI /v1/embeddings | Notes |
|-----------------------------|------------------------|--------|
| `model`                     | `model`                | Same |
| `input` (string or array of strings) | `input` (string or array of strings) | Same |
| `dimensions`                | `dimensions`           | Same (optional) |
| `options` (e.g. num_ctx)    | â€”                      | Ollama-only; omit for v1. |
| `keep_alive`                | â€”                      | Ollama-only; omit for v1. |
| `truncate`                  | â€”                      | Ollama-specific; omit for v1. |
| â€”                           | `encoding_format`      | e.g. "float" / "base64"; we can omit and use default. |

### 2.3 Response body (differences)

- **Current:** `model`, `embeddings`: array of arrays of numbers, `total_duration`, `load_duration`, `prompt_eval_count`. Our `Response.Embed` and `exec_embed()` normalize embeddings and return them in the same structure.
- **OpenAI:** `object`, `data`: array of `{ embedding: number[], index }`, `model`, `usage`: `{ prompt_tokens, total_tokens }`. No nanosecond timings.

**Planned response handling (Phase 1):**

- Parse into the **new** types: `Response.EmbeddingList` (or `Response.Embeddings`) with `data`: array of `Response.EmbeddingDatum` (each with `embedding`, `index`), and optional `usage`. Callers that need a flat list of vectors can read `data[].embedding`; normalization (if desired) can be done in `Call.Embeddings` before returning or left to the caller. No conversion to existing `Response.Embed` in Phase 1.

### 2.4 API calls: new classes (embed)

- **New class `Call.Embeddings`** (separate from `Call.Embed`), same URL pattern as **`Call.Models`**:
  - `is_openai = true`, `url_endpoint = "v1/embeddings"`.
  - Request body: `model`, `input` (string or array), optional `dimensions`, optional `encoding_format`.
  - Parse response into `Response.EmbeddingList` (or `Response.Embeddings`): `data[]` of `Response.EmbeddingDatum`, `model`, optional `usage`.
- **Existing `Call.Embed`** unchanged; continues to use `/api/embed` and `Response.Embed` for the current app flow.
