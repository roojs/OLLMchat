# 2.20.10 Faiss Vector Dump and Compare

**Status:** 📋 PLANNED

**Context:** Faiss search works in the test environment but fails in the live database. We need to compare vectors stored in both environments to determine whether the live vector DB is broken (vectors not stored correctly).

---

## Problem summary

- **Test:** `tests/test-vector-search.sh` indexes a single file (`liboccoder/Task/List.vala`), then runs a semantic search for "write" using a specific search (e.g. AST path `OLLMcoder.Task-List-write`). **It works.**
- **Live DB:** The same search fails.
- **Suspicion:** The vector DB in production is broken — vectors may not be stored correctly in Faiss.

---

## Next step: compare vectors between live and test

To validate the suspicion we need to:

1. **Pull vectors from Faiss** for the same logical entity (e.g. same file + AST path) in both:
   - live database (main data dir)
   - test database (e.g. `build/test/vector-test`)
2. **Compare** the raw vectors (or a stable dump) to see if they differ.

That requires a way to **dump a vector from a Faiss file** by AST path: resolve AST path → `vector_id` via SQL metadata, then dump that vector from Faiss. Run the dump twice (once on main DB, once on test DB) and diff the output.

---

## Deliverables

### 1. Method to dump a vector from Faiss by AST path

- **Location:** Can live in Search (e.g. `OLLMvector.Search`) or alongside it (e.g. a small dump helper that uses `vector_db` + `sql_db` for path→id resolution).
- Given folder + SQL DB + vector DB and an AST path: resolve to a single `vector_id` via `vector_metadata`, then reconstruct and return (or print) the vector.  
  Search already has `lookup_debug_vector_id()` for AST path → `vector_id`; similar logic can be reused or exposed.  
  `Index.reconstruct_vector(int64 vector_id)` already exists; output in a **stable, diff-friendly format** (e.g. one float per line, or JSON array).

### 2. CLI: single option to run the dump

- **Tool:** `oc-vector-search`.
- **Option:** `--dump-vector=AST_PATH`  
  Example (live uses default data dir; test uses explicit data dir):
  - `oc-vector-search <folder> --dump-vector=OLLMcoder.Task-List-write`
  - `oc-vector-search --data-dir=<test> <folder> --dump-vector=OLLMcoder.Task-List-write`
- Output: stable format (e.g. one float per line, or JSON) so that `diff live.txt test.txt` shows whether the stored vectors are identical.

### 3. Optional: test script step

- Extend `tests/test-vector-search.sh` (or add a small script) to dump the vector for the target AST path from the test DB and optionally compare against a pre-recorded baseline or a second dump from another data dir.

---

## Implementation notes

- **Existing code:**  
  - `OLLMvector.Index.reconstruct_vector(int64 vector_id)` — returns `float[]`.  
  - `Search.lookup_debug_vector_id()` — resolves `debug_ast_path` to `vector_id` using `vector_metadata` + folder’s `project_files`.  
  - `Search.exact_distance_string()` uses `vector_db.index.reconstruct_vector(vector_id)` for debug.
- **Output format:** Prefer something deterministic and easy to diff (e.g. sorted or fixed-order list of decimals with fixed precision, one per line, or a single JSON array). Avoid key-order or formatting noise.
- **Scope:** Focus on “dump one vector by AST path” and the single CLI flag; comparison/diff can be done externally (e.g. `diff`) or in a follow-up.

---

## Success criteria

- We can dump the same logical vector (by AST path) from two different data directories (live and test).
- The output format is stable and diff-friendly.
- Running the dump twice (main DB vs test DB) and diffing confirms whether the stored Faiss vectors for that entity are identical or not, informing the next fix (e.g. re-index live, or fix storage path).
