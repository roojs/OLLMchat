# Plan 1.24.3: OpenAI API — Chat (all libraries)

## Overview

Phase 2 integration of OpenAI-compatible **chat** for **all libraries** that use it. Parent plan: **1.24** (OpenAI API chat and embed compatibility). Ollama has `/api/chat`; OpenAI has `/v1/chat/completions` (same idea — messages in, assistant message out — but different request/response shape). This plan covers every chat usage across the codebase.

## Scope

**Method:** Chat only.

**Libraries:** libollmchat (Client, Agent, History, Tools), libollmchatgtk (ChatWidget, ChatView), libocvector (VectorBase), liboctools (EditMode, Child/Factory, ReadFile/Tool), examples (oc-test-cli), ollmapp (ollmchat-cli).

**Refer to plan 1.24 for:** rationale, API differences.

### Naming: Chat = Chat Completions

| Our name   | Ollama endpoint        | OpenAI endpoint              | Call class              | Response (Ollama)   | Response (OpenAI)        |
|------------|------------------------|------------------------------|-------------------------|---------------------|---------------------------|
| **Chat**   | `/api/chat`            | **`/v1/chat/completions`**    | **`Call.Chat`**         | **`Response.Chat`** | **`Call.ChatCompletions`** → `Response.ChatCompletion` / `Response.ChatCompletionMessage` (streaming) |

So we have two paths: **Ollama** uses `Call.Chat` and `Response.Chat`; **OpenAI-style** uses `Call.ChatCompletions` and returns `Response.ChatCompletion` (non-streaming) or an accumulated `Response.ChatCompletionMessage` (streaming). The existing classes are already implemented; this section and the following request/response comparison allow review for correctness.

## Request and response differences (chat): Ollama vs OpenAI

### Request

| Aspect | Ollama classic (`/api/chat`) | OpenAI-style (`/v1/chat/completions`) |
|--------|------------------------------|--------------------------------------|
| Endpoint | `POST /api/chat` | `POST /v1/chat/completions` |
| Input | `messages` (array of `{ role, content }`; content can be string or array of parts), optional `format` / `format_obj` | `messages` (same structure) |
| Model | `model` | `model` |
| Generation | Nested `options` (e.g. `num_predict`, `temperature`, `top_p`, `num_ctx`) | Flat: `max_tokens`, `temperature`, `top_p`, `stop`, `presence_penalty`, `frequency_penalty`, etc. |
| Format | `format` (string `"json"`) or `format` (object for schema) | `response_format` (e.g. `{ "type": "json_object" }` or `{ "type": "json_schema", "json_schema": {...} }`) |
| Tools | `tools` (array of tool definitions) | `tools`, optional `tool_choice` |
| Extra (Ollama) | `think`, `keep_alive`, `stream` | — |
| Extra (OpenAI) | — | `stream`, `seed`, `user` |

**Ollama request body example:**

```json
{
  "model": "llama3.2",
  "messages": [
    { "role": "user", "content": "Hello!" }
  ],
  "stream": true,
  "options": { "num_predict": 256, "temperature": 0.7, "top_p": 0.9 },
  "format": "json",
  "tools": []
}
```

**OpenAI-style request body example:**

```json
{
  "model": "llama3.2",
  "messages": [
    { "role": "user", "content": "Hello!" }
  ],
  "stream": true,
  "max_tokens": 256,
  "temperature": 0.7,
  "top_p": 0.9,
  "response_format": { "type": "json_object" },
  "tools": []
}
```

### Response (non-streaming)

| Aspect | Ollama classic | OpenAI-style |
|--------|----------------|--------------|
| Message | Top-level `message` (`role`, `content`, optional `thinking`, `tool_calls`) | `choices[0].message` (`role`, `content`, optional `tool_calls`) |
| Finish | `done`, `done_reason` | `choices[0].finish_reason` |
| Token counts | `prompt_eval_count`, `eval_count` | `usage.prompt_tokens`, `usage.completion_tokens`, `usage.total_tokens` |
| Timing | `total_duration`, `load_duration`, `eval_duration`, `prompt_eval_duration` | none |
| Thinking | `message.thinking` (Ollama extended) | none |
| Top-level | `model`, `created_at`, `message`, `done`, `done_reason`, timing, eval counts | `id`, `object`, `created`, `model`, `choices`, `usage` |

**Ollama response (non-streaming) example:**

```json
{
  "model": "llama3.2",
  "created_at": "2025-01-01T00:00:00.000Z",
  "message": {
    "role": "assistant",
    "content": "Hi there!",
    "tool_calls": []
  },
  "done": true,
  "done_reason": "stop",
  "total_duration": 12345678,
  "prompt_eval_count": 12,
  "eval_count": 8
}
```

**OpenAI-style response example:**

```json
{
  "id": "chatcmpl-xxx",
  "object": "chat.completion",
  "created": 1704067200,
  "model": "llama3.2",
  "choices": [
    {
      "index": 0,
      "message": {
        "role": "assistant",
        "content": "Hi there!",
        "tool_calls": []
      },
      "finish_reason": "stop"
    }
  ],
  "usage": { "prompt_tokens": 12, "completion_tokens": 8, "total_tokens": 20 }
}
```

### Response (streaming)

| Aspect | Ollama classic | OpenAI-style |
|--------|----------------|--------------|
| Protocol | NDJSON (one JSON object per line) | SSE (`data:` lines, JSON per line) |
| Chunk message | Top-level `message` (role, content, thinking?, tool_calls?); cumulative or delta per chunk | `choices[0].delta` (content?, role?, tool_calls?); incremental deltas |
| Chunk finish | `done`, `done_reason` on chunk | `choices[0].finish_reason` when present |
| Usage | Often in final chunk (prompt_eval_count, eval_count, etc.) | Sometimes in final chunk with `usage` |

**Ollama streaming chunk example (NDJSON line):**

```json
{"model":"llama3.2","message":{"role":"assistant","content":"Hi"},"done":false}
```

**OpenAI-style streaming chunk example (SSE data line):**

```json
{"id":"chatcmpl-xxx","object":"chat.completion.chunk","model":"llama3.2","choices":[{"index":0,"delta":{"content":"Hi"},"finish_reason":null}]}
```

## Options, required messages, and how ChatCompletions differs from Chat

### Options in ChatCompletions

- **Chat (Ollama)** sends a nested **`options`** object; the API expects e.g. `options: { num_predict, temperature, top_p, num_ctx, ... }`. So `Call.Chat` has `Call.Options options` and serializes it as a single nested object.
- **ChatCompletions (OpenAI)** currently has **no** `options` property; it has flat top-level `max_tokens`, `temperature`, `top_p` only. So today there is no fallback from a shared `Call.Options` for ChatCompletions (unlike Completions in the generate plan, which has `options` as fallback and flattens into top-level params).

If we want parity with the generate plan: add **`public Call.Options options { get; set; default = new Call.Options(); }`** to `Call.ChatCompletions` and, in serialization, for each param that exists both as a top-level property and in Options, **if the top-level property is set, send that; otherwise if options has a value, send from options** (with mapping e.g. `options.num_predict` → `max_tokens`). We do **not** send a key `"options"` in the JSON.

**Mapping Options → ChatCompletions request (if added):**

| Call.Options (options.) | ChatCompletions top-level / API key | Note |
|-------------------------|-------------------------------------|------|
| `num_predict`           | `max_tokens`                        | Use top-level if max_tokens >= 0, else options.num_predict |
| `temperature`           | `temperature`                       | Use top-level if >= 0, else options.temperature |
| `top_p`                 | `top_p`                             | Use top-level if >= 0, else options.top_p |
| `stop`                  | `stop`                              | Use top-level if non-empty, else options.stop |
| `presence_penalty`      | `presence_penalty`                  | Optional top-level or from options |
| `frequency_penalty`     | `frequency_penalty`                 | Optional top-level or from options |

### Required messages

- **OpenAI chat completions:** **`messages`** is required and must be a non-empty array. If we send an empty array or omit messages, the API will error.
- **Chat (Ollama):** In our code we typically send whatever messages the caller provides; the server may return an error if invalid.

**ChatCompletions:** We should **require messages** before sending: if `messages.size == 0` then throw `OllmError.INVALID_ARGUMENT("Messages are required for chat completions")`. **Chat** keeps current behaviour.

### How this differs from Chat

| Aspect | Chat (Ollama) | ChatCompletions (OpenAI) |
|--------|----------------|---------------------------|
| **Options** | Single nested `options` object; serialize when `options.has_values()`. | No `options` key. Today: flat params only (max_tokens, temperature, top_p). Optional: add options as fallback and flatten (as above). |
| **Format** | `format` (string or object). | `response_format` (object only). |
| **Messages** | No strict validation. | Required non-empty array; validate and throw before send. |
| **Endpoint** | `POST /api/chat` | `POST /v1/chat/completions` |
| **Streaming** | NDJSON; `Response.Chat.addChunk(Json.Object)`. | SSE; parse each `data:` as ChatCompletionChunk; accumulate deltas into ChatCompletionMessage. |

## Approach: adapter vs single type, reuse of Response.Chat

- **Option A (adapter):** Keep a single consumer-facing type **`Response.Chat`** in signals and APIs (Session, Agent, UI). When using the v1 path, parse the OpenAI response into `Response.ChatCompletion` (or streaming into `Response.ChatCompletionMessage`) and **adapt** it into `Response.Chat` (e.g. map `choices[0].message` → `message`, `usage` → prompt_eval_count/eval_count, finish_reason → done_reason). No change to downstream libraries’ types.
- **Option B (single type):** Switch all chat consumers to use OpenAI-shaped types (`Response.ChatCompletion` / `Response.ChatCompletionMessage`). Larger change across libollmchat, libollmchatgtk, libocvector, liboctools, examples, ollmapp.

**Recommendation:** Option A for Phase 2 — add v1 path in Client (build `Call.ChatCompletions`, call `exec()` or `exec_stream()`), then map the result into `Response.Chat` so that existing `stream_chunk(Response.Chat)`, `message_completed(Response.Chat)`, and history/UI code stay unchanged. The “Code that would need changing” table below assumes Option A unless we decide otherwise.

- **Response shape — flatten usage when adapting to Response.Chat:** When mapping from `Response.ChatCompletion` or from streaming `Response.ChatCompletionMessage` + usage, set `Response.Chat.prompt_eval_count` from `usage.prompt_tokens`, `Response.Chat.eval_count` from `usage.completion_tokens` (same pattern as generate plan: reuse existing fields; no new token properties on Response.Chat). Set `message` from `choices[0].message` or the accumulated message; set `done_reason` from `finish_reason`.
