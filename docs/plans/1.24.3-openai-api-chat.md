# Plan 1.24.3: OpenAI API ‚Äî Chat (all libraries)

## Overview

Integration of OpenAI-compatible **chat** for **all libraries** that use it. Parent plan: **1.24** (OpenAI API chat and embed compatibility). Ollama has `/api/chat`; OpenAI has `/v1/chat/completions` (same idea ‚Äî messages in, assistant message out ‚Äî but different request/response shape). This plan is split into **Phase 1** (updates to Call/Response classes only) and **Phase 2** (migration of code that uses the old chat method to the new style).

## Scope

**Method:** Chat only.

**Libraries:** libollmchat (Client, Agent, History, Tools), libollmchatgtk (ChatWidget, ChatView), libocvector (VectorBase), liboctools (EditMode, Child/Factory, ReadFile/Tool), examples (oc-test-cli), ollmapp (ollmchat-cli).

**Refer to plan 1.24 for:** rationale, API differences.

---

## `Call.Base` streaming hook (resolved)

**Location:** `libollmchat/Call/Base.vala` ‚Äî `handle_streaming_response(Soup.Message message)` drives `process_json_streaming` ‚Üí `process_json_chunk`, which parses each NDJSON line and invokes **`protected virtual void process_streaming_chunk(Json.Object chunk_obj)`**.

**Subclasses:** **`Call.Chat`**, **`Call.Create`**, and **`Call.Pull`** override **`process_streaming_chunk`** (no delegate, no callback argument on **`handle_streaming_response`**). A streaming response yields **many** chunks; each invocation of the virtual is one parsed object‚Äîthere is no single return value for the whole stream.

**1.24.3 / `Response.Chunk`:** **`Call.Chat`** may wrap **`chunk_obj`** in a **`Json.Node`** and **`Json.gobject_deserialize(typeof(Response.Chunk), ‚Ä¶)`** inside **`process_streaming_chunk`**, then **`addChunk`**, without further **`Call.Base`** API changes.

## Phase 1: Updates to Chat / Response classes

**Goal:** Extend existing classes so that the v1 chat completions API can be used and all responses are returned as **`Response.Chat`** (and **`Message`**). No new response types; no change yet to callers (Client, Agent, etc.).

**Deliverables:**

1. **Call.ChatCompletions** ‚Äî Add `options`, `format`, `format_obj`; serialize flat params with options fallback and `response_format` from format/format_obj. **`exec()`** parses v1 JSON into **`Response.Chat`** and returns it (require messages). **`exec_stream()`** builds a **`Response.Chat`**, feeds each SSE chunk into **`addChunk`**, and returns that **`Response.Chat`** (no `Response.ChatCompletion` / `Response.ChatCompletionMessage`).
2. **Response.Chat** ‚Äî Extend **`deserialize_property`** for v1 non-streaming (choices, usage, created). **addChunk(Response.Chunk)** ‚Äî both Ollama and v1 deserialize into **the same** **Chunk** (one class, deserialize_property handles both wire shapes). No Json.Object round-trip.
3. **Build** ‚Äî Add **`Response/Chunk.vala`** (see ¬ß1e) to **libollmchat/meson.build** and **docs/meson.build**. ChatCompletions and Response/Chat are already in meson.

Phase 1 does **not** wire the v1 path into Client, Agent, or UI; it only makes the classes ready.

## Phase 2: Migration ‚Äî code that uses the old chat method

**Goal:** Switch call sites that currently use **`Call.Chat`** (Ollama `/api/chat`) to use **`Call.ChatCompletions`** when the connection or configuration indicates v1, and have them receive **`Response.Chat`** unchanged (so no type change for Session, Agent, History, UI).

**Deliverables:**

- **Client:** Add v1 path (e.g. connection flag or API choice): build **`Call.ChatCompletions`**, call **`exec()`** or **`exec_stream()`**, get **`Response.Chat`**; keep same **`chat()`** / **`stream_chunk`** / **`message_completed`** contract.
- **Agent, History, Tools, libollmchatgtk, libocvector, liboctools, examples, ollmapp:** When v1 is used, construct **`Call.ChatCompletions`** instead of **`Call.Chat`**; exec still returns/streams into **`Response.Chat`**; no type changes in signals or handlers.

Concrete code for Phase 2 is **not** in this document; implement after Phase 1 is done and reviewed.

### Naming: Chat = Chat Completions

| Our name   | Ollama endpoint        | OpenAI endpoint              | Call class              | Response (both paths)   |
|------------|------------------------|------------------------------|-------------------------|--------------------------|
| **Chat**   | `/api/chat`            | **`/v1/chat/completions`**    | **`Call.Chat`** / **`Call.ChatCompletions`** | **`Response.Chat`** (reused; extended to accept v1 JSON/chunks) |

So we have two call classes but **one response type**: **Ollama** uses `Call.Chat` ‚Üí `Response.Chat`; **OpenAI-style** uses `Call.ChatCompletions` ‚Üí same **`Response.Chat`**. As with completions (plan 1.24.2), we **reuse the Ollama standard** `Response.Chat` and extend it to deserialize v1 response JSON and to handle v1 streaming chunks, so callers always receive `Response.Chat`.

### Response format (format / response_format)

Both APIs support a ‚Äúformat‚Äù for the reply (e.g. raw JSON, or a JSON schema). Names and shapes differ:

- **Ollama:** Top-level **`format`** ‚Äî either a **string** (e.g. `"json"`) or an **object** (e.g. for schema).
- **OpenAI:** Top-level **`response_format`** ‚Äî always an **object** (e.g. `{ "type": "json_object" }` or `{ "type": "json_schema", "json_schema": {...} }`).

Use the **same trick as in `Call.Chat`** (see `Chat.vala`): two variables on the call class:

1. **`format`** (string) ‚Äî for simple cases: `"json"` ‚Üí Ollama sends `"format": "json"`; for OpenAI we send `response_format: { "type": "json_object" }`.
2. **`format_obj`** (e.g. `Json.Object?`) ‚Äî for custom/schema: when set, Ollama gets that object as `format`, OpenAI gets it as `response_format` (same payload, different key).

So in **`Call.ChatCompletions`** we have both: **`format`** (string, optional) and **`format_obj`** (object, optional). On serialize: if `format_obj != null`, serialize it as **`response_format`**; else if `format == "json"` (or similar), emit `response_format: { "type": "json_object" }`; else omit. No separate ‚Äúresponse_format‚Äù property name in our type ‚Äî we reuse the same two variables as Chat and map to the correct key per API.

**Options for response_format (OpenAI):**

| Use case | OpenAI `response_format` |
|----------|---------------------------|
| Plain text (default) | Omit or `null` |
| JSON object | `{ "type": "json_object" }` |
| JSON schema | `{ "type": "json_schema", "json_schema": { "name": "...", "strict": true, "schema": {...} } }` |

Ollama accepts the same shapes under `format` (string or object).

### Response (non-streaming)

| Aspect | Ollama classic | OpenAI-style |
|--------|----------------|--------------|
| Message | Top-level `message` (`role`, `content`, optional `thinking`, `tool_calls`) | `choices[0].message` (`role`, `content`, optional `tool_calls`); delta has `reasoning_content` in chunks |
| Finish | `done`, `done_reason` | `choices[0].finish_reason` |
| Token counts | `prompt_eval_count`, `eval_count` | `usage.prompt_tokens`, `usage.completion_tokens`, `usage.total_tokens` |
| Timing | `total_duration`, `load_duration`, `eval_duration`, `prompt_eval_duration` | none |
| Thinking | `message.thinking` (Ollama) | **`reasoning_content`** in message/delta (v1); map to `thinking` on read; ignore `thinking` on write (do not send `reasoning_content`) |
| Top-level | `model`, `created_at`, `message`, `done`, `done_reason`, timing, eval counts | `id`, `object`, `created`, `model`, `choices`, `usage` |

**Ollama response (non-streaming) example:**

```json
{
  "model": "llama3.2",
  "created_at": "2025-01-01T00:00:00.000Z",
  "message": {
    "role": "assistant",
    "content": "Hi there!",
    "tool_calls": []
  },
  "done": true,
  "done_reason": "stop",
  "total_duration": 12345678,
  "prompt_eval_count": 12,
  "eval_count": 8
}
```

**OpenAI-style response example:**

```json
{
  "id": "chatcmpl-xxx",
  "object": "chat.completion",
  "created": 1704067200,
  "model": "llama3.2",
  "choices": [
    {
      "index": 0,
      "message": {
        "role": "assistant",
        "content": "Hi there!",
```

### Response (streaming)

| Aspect | Ollama classic | OpenAI-style |
|--------|----------------|--------------|
| Protocol | NDJSON (one JSON object per line) | SSE (`data:` lines, JSON per line) |
| Chunk message | Top-level `message` (role, content, thinking?, tool_calls?); cumulative or delta per chunk | `choices[0].delta` (content?, role?, **reasoning_content?**, tool_calls?); incremental deltas; map `reasoning_content` ‚Üí `thinking` on read |
| Chunk finish | `done`, `done_reason` on chunk | `choices[0].finish_reason` when present |
| Usage | Often in final chunk (prompt_eval_count, eval_count, etc.) | Sometimes in final chunk with `usage` |

**Ollama streaming chunk example (NDJSON line):**

```json
{"model":"llama3.2","message":{"role":"assistant","content":"Hi"},"done":false}
```

**OpenAI-style streaming chunk example (SSE data line):**

```json
{"id":"chatcmpl-xxx","object":"chat.completion.chunk","model":"llama3.2","choices":[{"index":0,"delta":{"content":"Hi"},"finish_reason":null}]}
```

## Classes and response approach (same as completions)

We **reuse the Ollama standard** `Response.Chat` for both paths and extend it to accept v1 data, same as we do with `Response.Generate` for completions. There is **one** response type for callers; v1 JSON is mapped into it in `Response.Chat` (deserialize and streaming). We do **not** use ChatCompletion, ChatCompletionChunk, or any other separate v1 response types‚Äîonly existing `Response.Chat` and `Message`, extended where necessary.

### Single response type: Response.Chat

- **Non-streaming:** `Call.ChatCompletions` receives v1 JSON. We parse it into **`Response.Chat`** (e.g. `Json.gobject_from_data(typeof(Response.Chat), json_str, -1)`). **`Response.Chat.deserialize_property`** is extended to handle v1 keys: **`choices`** ‚Üí set `message` from `choices[0].message`, `done_reason` from `choices[0].finish_reason`; **`usage`** ‚Üí set `prompt_eval_count` from `usage.prompt_tokens`, `eval_count` from `usage.completion_tokens`; **`created`** ‚Üí set `created_at` from unix timestamp. No new token properties; reuse existing fields like Generate.
- **Streaming:** **`Response.Chunk`** ‚Äî one class for both Ollama NDJSON and v1 SSE; **deserialize_property** branches on **usage** / **choices** (v1) and uses default for Ollama keys. **`Response.Chat.addChunk(Chunk)`** reads typed fields for both paths.

We **do not** introduce response types like ChatCompletion or ChatCompletionMessage for the public API. V1 non-streaming is mapped in **`Response.Chat.deserialize_property`**; v1 streaming uses the **same** **Chunk** class (deserialize_property handles v1 keys). We reuse **`Response.Chat`** and **`Message`** only.

## Options, required messages, and how ChatCompletions differs from Chat

### Options in ChatCompletions

- **Chat (Ollama)** sends a nested **`options`** object; the API expects e.g. `options: { num_predict, temperature, top_p, num_ctx, seed, ... }`. So `Call.Chat` has `Call.Options options` and serializes it as a single nested object.
- **ChatCompletions (OpenAI)** has no nested `options` key; the API expects **flat** top-level params: `max_tokens`, `temperature`, `top_p`, `stop`, `seed`, `presence_penalty`, `frequency_penalty`, etc.

Same pattern as **Completions** (generate plan): add **`public Call.Options options { get; set; default = new Call.Options(); }`** to `Call.ChatCompletions`. In serialization, for each param: if the top-level property is set, send that; otherwise if `options` has a value, send from options (with name mapping). We do **not** send a key `"options"` in the JSON.

**Mapping Options ‚Üí ChatCompletions request:**

| Call.Options (options.) | ChatCompletions top-level / API key | Note |
|-------------------------|-------------------------------------|------|
| `num_predict`           | `max_tokens`                        | Use top-level if set, else options.num_predict |
| `seed`                  | `seed`                              | Ollama: in options; OpenAI: top-level. Use top-level if set, else options.seed. |
| `temperature`           | `temperature`                       | Use top-level if >= 0, else options.temperature |
| `top_p`                 | `top_p`                             | Use top-level if >= 0, else options.top_p |
| `stop`                  | `stop`                              | Use top-level if non-empty, else options.stop |
| `presence_penalty`      | `presence_penalty`                  | Top-level or from options |
| `frequency_penalty`     | `frequency_penalty`                 | Top-level or from options |

### Required messages

- **OpenAI chat completions:** **`messages`** is required and must be a non-empty array. If we send an empty array or omit messages, the API will error.
- **Chat (Ollama):** In our code we typically send whatever messages the caller provides; the server may return an error if invalid.

**ChatCompletions:** We should **require messages** before sending: if `messages.size == 0` then throw `OllmError.INVALID_ARGUMENT("Messages are required for chat completions")`. **Chat** keeps current behaviour.

### How this differs from Chat

| Aspect | Chat (Ollama) | ChatCompletions (OpenAI) |
|--------|----------------|---------------------------|
| **Options** | Single nested `options` object; serialize when `options.has_values()`. | No `options` key; flat top-level params only; use same `Call.Options` as fallback (options.num_predict ‚Üí max_tokens, options.seed ‚Üí seed, etc.). |
| **Format** | Two variables: `format` (string), `format_obj` (object); serialize as top-level `format`. | Same two variables; serialize as top-level **`response_format`** (if format_obj set use it; else e.g. format == "json" ‚Üí `{ "type": "json_object" }`). |
| **Seed** | In `options.seed`. | Top-level `seed`. |
| **Messages** | No strict validation. | Required non-empty array; validate and throw before send. |
| **Endpoint** | `POST /api/chat` | `POST /v1/chat/completions` |
| **Streaming** | NDJSON ‚Üí **Chunk** ‚Üí **addChunk(Chunk)**. | SSE ‚Üí **Chunk** ‚Üí **addChunk** (same class). |

## Approach: same as completions ‚Äî reuse Response.Chat, extend for v1

We use the **same approach as plan 1.24.2 (completions)**: a single response type, extended to accept the v1 wire format.

- **Single response type:** **`Response.Chat`** (and **`Message`** for the message content) for both Ollama and v1. No ChatCompletion* or chunk types; we extend existing classes only.
- **Non-streaming:** When `Call.ChatCompletions` gets a v1 response, we parse it **into** `Response.Chat` (e.g. `Json.gobject_from_data(typeof(Response.Chat), json_str, -1)`). **`Response.Chat.deserialize_property`** is extended to handle v1 keys: `choices` ‚Üí `message`, `done_reason`; `usage` ‚Üí `prompt_eval_count`, `eval_count`; `created` ‚Üí `created_at`. Reuse existing fields; no new token properties.
- **Streaming:** **Chunk** (one class, both wire formats) + **addChunk(Chunk)**; no Json.Object for the chunk API.
- No change to downstream libraries‚Äô types; they keep using `Response.Chat`.

**1c. `exec()`** ‚Äî Replace the current body so it returns **`Response.Chat`**:

```vala
		public async Response.Chat exec() throws Error
		{
			if (this.messages.size == 0) {
				throw new OllmError.INVALID_ARGUMENT("Messages are required for chat completions");
			}
			var bytes = yield this.send_request(true);
			var root = this.parse_response(bytes);
			if (root.get_node_type() != Json.NodeType.OBJECT) {
				throw new OllmError.FAILED("Invalid JSON response");
			}
			var generator = new Json.Generator();
			generator.set_root(root);
			var json_str = generator.to_data(null);
			var chat = Json.gobject_from_data(
				typeof(Response.Chat), json_str, -1) as Response.Chat;
			if (chat == null) {
				throw new OllmError.FAILED("Failed to deserialize chat completion response");
```

### 1e. Chunk ‚Äî one class for both Ollama and v1

**One class `Response.Chunk`** (Json.Serializable) is used for **both** wire formats. Properties: **model**, **message**, **done**, **done_reason**, **prompt_eval_count**, **eval_count**, plus **choices** (Gee.ArrayList<Message>, for v1) and any duration fields **addChunk** uses.

**Deserialize behaviour:**
- **Ollama** NDJSON has **model**, **message**, **done**, **done_reason**, **prompt_eval_count**, **eval_count**, etc. ‚Äî no **usage** or **choices**. **default_deserialize_property** fills those keys.
- **V1** SSE has **model**, **usage**, **choices**. Custom **deserialize_property** for **"usage"** ‚Üí set **prompt_eval_count**, **eval_count**. For **"choices"**: loop over the array; for each choice read **finish_reason** and **delta**, deserialize **delta** as **Message**, add to **choices**; if **choices.size > 0** set **this.message = this.choices.get(0)** and **done** / **done_reason** from first choice. **default** ‚Üí **default_deserialize_property** (covers Ollama and v1 **model**).

**Refactor `Response.Chat.addChunk`:** **`addChunk(Response.Chunk chunk)`** ‚Äî read **chunk.model**, **chunk.message**, **chunk.done**, etc. **Ollama path:** parse NDJSON line ‚Üí **Json.gobject_deserialize(typeof(Chunk), node)** ‚Üí **addChunk(chunk)**. **V1 path:** parse SSE ‚Üí **Json.gobject_deserialize(typeof(Chunk), node)** ‚Üí **addChunk(chunk)**. We never serialize this class.

**Coding standards:** Follow **CODING_STANDARDS.md**: no underscore prefixes; use **this.** for instance members; brace style and early returns as specified. Do not use **??**; when **has_member** is true, assume the value is present and assign **get_string_member** (or equivalent) directly.

```vala
namespace OLLMchat.Response
{
	/**
	 * One stream chunk for both Ollama /api/chat NDJSON and v1 /v1/chat/completions SSE.
	 * Deserialize from either wire format; pass to Response.Chat.addChunk. Never serialized.
	 *
	 * @see Response.Chat.addChunk
	 */
	public class Chunk : Object, Json.Serializable
	{
		public string model { get; set; default = ""; }
		public Message message { get; set; default = new Message("assistant", ""); }
		public bool done { get; set; default = false; }
		public string done_reason { get; set; default = ""; }
		public int prompt_eval_count { get; set; default = 0; }
		public int eval_count { get; set; default = 0; }
		public Gee.ArrayList<Message> choices { get; set; default = new Gee.ArrayList<Message>(); }
		public int64 total_duration { get; set; default = 0; }
		public int64 load_duration { get; set; default = 0; }
		public int64 prompt_eval_duration { get; set; default = 0; }
		public int64 eval_duration { get; set; default = 0; }
		public string created_at { get; set; default = ""; }

		public override bool deserialize_property(
			string property_name, out Value value, ParamSpec pspec, Json.Node property_node)
		{
			switch (property_name) {
				case "usage": {
					var usage = property_node.get_object();
					if (usage.has_member("prompt_tokens")) {
						this.prompt_eval_count = (int) usage.get_int_member("prompt_tokens");
					}
					if (usage.has_member("completion_tokens")) {
						this.eval_count = (int) usage.get_int_member("completion_tokens");
					}
					value = Value(typeof(int));
					value.set_int(0);
					return true;
				}
				case "choices": {
					var array = property_node.get_array();
					for (var i = 0; i < array.get_length(); i++) {
						var choice_obj = array.get_object_element(i);
						if (choice_obj.has_member("finish_reason")) {
							this.done_reason = choice_obj.get_string_member("finish_reason");
							this.done = this.done_reason != "";
						}
						if (!choice_obj.has_member("delta")) {
							continue;
						}
						var delta_node = choice_obj.get_member("delta");
						var msg = Json.gobject_deserialize(typeof(Message), delta_node) as Message;
						if (msg != null) {
							this.choices.add(msg);
						}
					}
					if (this.choices.size > 0) {
						this.message = this.choices.get(0);
					}
					value = Value(typeof(Gee.ArrayList));
					value.set_object(this.choices);
					return true;
				}
				default:
					return default_deserialize_property(
						property_name, out value, pspec, property_node);
			}
		}
	}
}
```

```vala
		{
			this.new_content = "";
			this.new_thinking = "";

			this.load_duration = chunk.load_duration;
```

**Call.Chat.process_streaming_chunk** ‚Äî On each chunk: parse line to **Json.Node**, **`Json.gobject_deserialize(typeof(Response.Chunk), node)`** ‚Üí **`response.addChunk(stream_chunk)`** (no **Json.Object**). **After** addChunk, the same signal/callback flow as today must run so UI and agents see each chunk:
- **First chunk:** set **response.is_first_chunk = false**, emit **stream_start()**, and if **this.agent != null** call **this.agent.handle_stream_started()**.
- **Every chunk with content or done:** use **addChunk**‚Äôs return value (or **response.new_content** / **response.new_thinking** / **response.done**) to form **new_text** and **is_thinking**; skip emitting if **new_content** and **new_thinking** are both empty and **!response.done**. Then emit **this.stream_chunk(new_text, is_thinking, response)** and if **this.agent != null** call **this.agent.handle_stream_chunk(new_text, is_thinking, response)**.

So **process_streaming_chunk** is not only ‚Äúdeserialize ‚Üí addChunk‚Äù; it must also perform the **stream_start** (first chunk), **stream_chunk** signal, and **agent.handle_stream_chunk** (and **agent.handle_stream_started**) so Session, Manager, and tools receive chunk arrival as they do today.

```vala
		private void process_streaming_chunk(Response.Chunk stream_chunk)
		{
			var response = (Response.Chat) this.streaming_response;
			response.addChunk(stream_chunk);

			if (response.is_first_chunk) {
				response.is_first_chunk = false;
				this.stream_start();
				if (this.agent != null) {
					this.agent.handle_stream_started();
				}
			}
			if (response.new_thinking.length == 0 && response.new_content.length == 0 && !response.done) {
				return;
			}
			bool is_thinking = response.new_thinking.length > 0;
			string new_text = is_thinking ? response.new_thinking : response.new_content;
			this.stream_chunk(new_text, is_thinking, response);
			if (this.agent != null) {
				this.agent.handle_stream_chunk(new_text, is_thinking, response);
			}
		}
```
