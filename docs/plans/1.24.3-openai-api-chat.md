## Phase 1: Updates to Chat / Response classes

**Goal:** Extend existing classes so that OpenAI-style **chat completions** (`/v1/chat/completions`) can be used and all responses are still returned as **`Response.Chat`** (and **`Message`**). No new public response DTO types; Phase 1 does **not** switch Client, Agent, or UI to prefer **`ChatCompletions`** over **`Call.Chat`**—that is Phase 2.

**Deliverables:**

1. **`Call.ChatCompletions`** — Flat request JSON: **`Call.Options`** as fallback only (no nested **`options`** key in JSON); **`format`** / **`format_obj`** map to **`response_format`**. Validate non-empty **`messages`** before send. **`exec()`** returns **`Response.Chat`** for non-streaming. **`exec_stream()`** reads SSE **`data:`** lines, deserializes each line into **`Response.Chunk`**, **`addChunk`** on an accumulated **`Response.Chat`**. Implements **`ChatInterface.send_append`** for **`Response.Chat.reply`**.

2. **`Response.Chat`** — **`deserialize_property`** extended for v2 non-streaming: **`choices`**, **`usage`**, **`created`**. Constructor takes **`Call.ChatInterface`** so the same type works for Ollama **`Chat`** and v2 **`ChatCompletions`**.

3. **`Response.Chunk`** — One stream-line type: Ollama NDJSON chat lines and v2 SSE JSON both deserialize here; **`deserialize_property`** handles v2 **`usage`** and **`choices`** / **`delta`**.

4. **`Message`** — On deserialize, v2 **`reasoning_content`** maps to **`thinking`**; JSON key **`tool_calls`** (and Vala’s **`tool-calls`**) deserialize tool call arrays.

5. **`Call.Chat`** — Continues to stream via **`Call.Base`**: NDJSON → **`Response.Chunk`** → **`process_streaming_chunk`** → **`addChunk`**.

6. **Build** — **`Response/Chunk.vala`** listed in **`libollmchat/meson.build`** and **`docs/meson.build`** where applicable.

**Status:** **Done** — implemented in the repository; use **`.cursor/rules/CODING_STANDARDS.md`** for any follow-up edits. This plan section is descriptive only (no concrete Vala here).

## Approach: same as completions — reuse Response.Chat, extend for v2

Same pattern as **plan 1.24.2 (completions)**:

We **reuse** the existing **`Response.Chat`** and **`Message`** types for both the legacy Ollama **`/api/chat`** shape and the OpenAI-style completions shape. We do **not** add separate public types such as ChatCompletion or ChatCompletionMessage.

- **Non-streaming:** The v2 response object is deserialized into **`Response.Chat`**. **`deserialize_property`** maps **`choices[0].message`** into **`message`**, **`finish_reason`** into **`done_reason`**, **`usage`** into **`prompt_eval_count`** / **`eval_count`**, **`created`** (unix int) into **`created_at`**, reusing existing fields.

- **Streaming:** **`Response.Chunk`** absorbs both wire formats in **`deserialize_property`** (Ollama top-level **`message`** / **`done`** vs v2 **`choices`** / **`delta`** / **`usage`**). **`Response.Chat.addChunk`** performs the same accumulation regardless of source.

- **`Call.Base`:** NDJSON streaming is used by **`Call.Chat`** through **`handle_streaming_response`** → **`process_json_streaming`**. V2 chat streaming in Phase 1 is implemented as an SSE loop on **`ChatCompletions`** (**`exec_stream`**), not that NDJSON pipeline.

Downstream code keeps depending on **`Response.Chat`** only.

### 2) Refactor `Call.Chat` — one helper for “after addChunk, emit signals”

```vala
// NEW on Chat:
protected void emit_stream_progress_after_add_chunk(Response.Chat response)
{
	if (response.is_first_chunk) {
		response.is_first_chunk = false;
		this.stream_start();
		if (this.agent != null) this.agent.handle_stream_started();
	}
	if (response.new_thinking.length == 0 && response.new_content.length == 0 && !response.done)
		return;
	bool is_thinking = response.new_thinking.length > 0;
	string new_text = is_thinking ? response.new_thinking : response.new_content;
	this.stream_chunk(new_text, is_thinking, response);
	if (this.agent != null)
		this.agent.handle_stream_chunk(new_text, is_thinking, response);
}

// process_streaming_chunk becomes:
protected override void process_streaming_chunk(Response.Chunk stream_chunk)
{
	var response = (Response.Chat) this.streaming_response;
	response.addChunk(stream_chunk);
	this.emit_stream_progress_after_add_chunk(response);
}
```

### 3) `ChatCompletions` — override `execute_streaming` (SSE, not `handle_streaming_response` NDJSON)

```vala
// Pseudocode shape — copy body from current ChatCompletions.exec_stream() line loop,
// but use this.streaming_response (same as Chat.send), and after each addChunk:
private async Response.Chat execute_streaming() throws Error
{
	var response = (Response.Chat) this.streaming_response;
	// build_url(), get_request_body(), soup_message, send_async → DataInputStream
	while (true) {
		string? line = yield data_input.read_line_async(...);
		if (line == null) break;
		var trimmed = line.strip();
		if (!trimmed.has_prefix("data: ")) continue;
		var payload = trimmed.substring(6).strip();
		if (payload == "[DONE]") break;
		// parse JSON → Response.Chunk
		var chunk = Json.gobject_deserialize(typeof(Response.Chunk), root) as Response.Chunk;
		if (chunk == null) continue;
		response.addChunk(chunk);
		this.emit_stream_progress_after_add_chunk(response);
	}
	if (!response.done) response.done = true;
	// then same tool branch as Chat.execute_streaming: if done && tool_calls → yield toolsReply(response)
	return response;
}
```

*(Exact method visibility must match **`Chat`** — if **`execute_streaming` is private**, rename override to whatever **`Chat`** uses for the streaming entry point, or make **`Chat`**’s method **`protected virtual`**.)*

### Response (non-streaming)

| Aspect | Ollama classic | OpenAI-style |
|--------|----------------|--------------|
| Message | Top-level `message` (`role`, `content`, optional `thinking`, `tool_calls`) | `choices[0].message` (`role`, `content`, optional `tool_calls`); delta has `reasoning_content` in chunks |

### Response (streaming)

| Aspect | Ollama classic | OpenAI-style |
|--------|----------------|--------------|
| Protocol | NDJSON (one JSON object per line) | SSE (`data:` lines, JSON per line) |
| Chunk message | Top-level `message` (role, content, thinking?, tool_calls?); cumulative or delta per chunk | `choices[0].delta` (content?, role?, **reasoning_content?**, tool_calls?); incremental deltas; map `reasoning_content` → `thinking` on read |
| Chunk finish | `done`, `done_reason` on chunk | `choices[0].finish_reason` when present |
| Usage | Often in final chunk (prompt_eval_count, eval_count, etc.) | Sometimes in final chunk with `usage` |


## Classes and response approach (same as completions)

**Single public response surface:** **`Response.Chat`** and **`Message`** only—no parallel ChatCompletion* types for the app.

**Non-streaming v2:** One JSON object from **`/v1/chat/completions`** → **`Response.Chat`** via **`Json.Serializable`** / **`deserialize_property`** (**`choices`**, **`usage`**, **`created`**, etc.).

**Streaming:** **`Response.Chunk`** is the deserialized unit for one line (Ollama NDJSON or one SSE **`data:`** payload). **`Response.Chat.addChunk(Chunk)`** merges deltas into the running assistant message, **`new_content`**, **`new_thinking`**, **`done`**, and token counts as appropriate.

The tables below spell out request property layouts and field mapping for review when implementing or auditing serializers.

## Options, required messages, and how ChatCompletions differs from Chat

```text
# Chat (legacy /api/chat): nested JSON key "options" { num_predict, temperature, … }
# ChatCompletions (v2): NO "options" key — flatten to max_tokens, temperature, … (use Call.Options as fallback in code only)
# Same pattern as Call.Completions
```

### Mapping Call.Options → ChatCompletions JSON

| Call.Options (options.) | ChatCompletions top-level / API key | Note |
|-------------------------|-------------------------------------|------|
| `num_predict`           | `max_tokens`                        | Use top-level if set, else options.num_predict |
| `seed`                  | `seed`                              | Ollama: in options; OpenAI: top-level. Use top-level if set, else options.seed. |
| `temperature`           | `temperature`                       | Use top-level if >= 0, else options.temperature |
| `top_p`                 | `top_p`                             | Use top-level if >= 0, else options.top_p |
| `stop`                  | `stop`                              | Use top-level if non-empty, else options.stop |
| `presence_penalty`      | `presence_penalty`                  | Top-level or from options |
| `frequency_penalty`     | `frequency_penalty`                 | Top-level or from options |

### Required messages

```text
ChatCompletions: messages.size == 0 → throw OllmError.INVALID_ARGUMENT("Messages are required for chat completions")
```

### How this differs from Chat

| Aspect | Chat (Ollama) | ChatCompletions (OpenAI) |
|--------|----------------|---------------------------|
| **Options** | Single nested `options` object; serialize when `options.has_values()`. | No `options` key; flat top-level params only; use same `Call.Options` as fallback (options.num_predict → max_tokens, options.seed → seed, etc.). |
| **Format** | Two variables: `format` (string), `format_obj` (object); serialize as top-level `format`. | Same two variables; serialize as top-level **`response_format`** (if format_obj set use it; else e.g. format == "json" → `{ "type": "json_object" }`). |
| **Seed** | In `options.seed`. | Top-level `seed`. |
| **Messages** | No strict validation. | Required non-empty array; validate and throw before send. |
| **Endpoint** | `POST /api/chat` | `POST /v1/chat/completions` |
| **Streaming** | NDJSON → **Chunk** → **addChunk(Chunk)**. | SSE → **Chunk** → **addChunk** (same class). |
