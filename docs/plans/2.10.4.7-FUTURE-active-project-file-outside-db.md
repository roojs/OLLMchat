# 2.10.4.7 — Active project and file outside the database (FUTURE)

**Which project and file each window shows** lives in **UI session state** (agent config, window state file, or similar) — **not** in the files SQLite database and **not** on the daemon.

**Status:** PROPOSAL — **future**, after [`2.10.4.4`](done/2.10.4.4-DONE-projectmanager.md) ✅–[`2.10.4.6`](done/2.10.4.6-DONE-file.md) cutover and Meson V2 flip ([`done/2.10.4.21-DONE-meson-v2-cutover.md`](done/2.10.4.21-DONE-meson-v2-cutover.md)) ✅.

## Goal

Move **active project** and **active file** out of the daemon and SQLite (`is_active` flags) into **per-window UI state**, persisted **outside the database** (agent config / session JSON — see [`1.3.9`](1.3.9-agent-configuration-and-window-state.md)).

Each window's `V2/ProjectManager` holds `active_project` and `active_file` in memory; restore on startup reads saved paths from config, not from DB rows.

The daemon should own **files, folders, scan, index, and disk I/O**. It should **not** track which project or file the user is looking at, and should not persist “activation” in SQLite.

---

## Why (direction from 2.10.4.4 work)

Today (shipping `libocfiles`):

- `ProjectManager.activate_project` / `activate_file` set `is_active` on `Folder` / `File` rows and call `saveToDB`.
- `restore_active_state()` finds rows with `is_active = true`, then re-activates (including full directory scan on shipping PM).
- `Window` session restore pre-activates a project so `restore_active_state()` sees the right DB flags.

That made sense for a **single in-process** `ProjectManager` shared by one app instance. It does **not** fit:

- **Multiple windows** — each may show a different project; one global daemon `active_project` is wrong ([`2.10.4.0-summary`](2.10.4.0-summary.md) / archived parent already notes this as deferred).
- **Thin V2 client** — activation for UI is local state + Gtk signals; RPC to the daemon for “mark active in DB” is unnecessary latency and coupling.
- **Session restore** — belongs in agent/window config ([`1.3.9`](1.3.9-agent-configuration-and-window-state.md)), not in canonical file rows.

**Interim during 2.10.4.4–6:** V2 may still relay `ProjectManager.activate_project` / `File.activate` so the daemon mirrors shipping behaviour for cutover tests. Treat that as **temporary compatibility**, not the long-term wire shape.

**Near-term client shape (already aligned):**

- `restore_active_state()` → plain `void`, fire-and-forget via `activate_project.begin` / `activate_file.begin` (or drop restore entirely once agent config owns paths).
- V2 `activate_project` / `activate_file` already set **local** `active_project`, `active_file`, and emit signals **before** RPC — callers (`apply_manager_state`, dropdowns) only need in-memory manager properties, not daemon DB persistence.

## Target model

| Concern | Owner | Persistence |
| ------- | ----- | ----------- |
| Which project this **window** shows | `V2/ProjectManager.active_project` | Agent config / session JSON ([`1.3.9`](1.3.9-agent-configuration-and-window-state.md)) — `project_path`, `file_path` |
| Which file this **window** has open | `V2/ProjectManager.active_file` | Same — per agent or per window |
| Project list (paths, labels) | Daemon SQLite + `ProjectManager.load_projects_from_db` RPC | DB — **without** `is_active` |
| Scan on “open project” | Daemon — triggered by explicit RPC (e.g. `Folder.read_dir` or `ProjectManager.open_project` **without** activation semantics) | N/A |
| Background scan / index queue | Daemon | N/A |

```text
┌─────────────────┐     RPC: read/write/scan/list     ┌──────────────┐
│  Window A       │ ────────────────────────────────► │  ollmfilesd  │
│  PM.active_*    │     (no activate_*, no is_active) │  no active_* │
└─────────────────┘                                   └──────────────┘
┌─────────────────┐
│  Window B       │  separate active_project / file
│  PM.active_*    │
└─────────────────┘
```

## What to remove or change (when implementing)

### Daemon / SQLite

- Stop writing / reading **`is_active`** on project folders and files for UI purposes.
- Remove (or no-op) server-side `ProjectManager.active_project`, `ProjectManager.active_file` if duplicated on daemon PM.
- Migration: optional one-time clear of `is_active` columns; session paths move to agent config.

### Wire API

- **Remove** (or deprecate then remove):
  - `ProjectManager.activate_project` — if still needed for scan, rename/split to something that does **not** imply global activation (e.g. `ProjectManager.ensure_scanned` / `Folder.read_dir` only).
  - `File.activate` — drop; active file is client-only.
- **Do not** add `window_id` / `client_id` on activate — multi-window is solved by not having daemon activation at all.

### Client (`libocfiles/V2`)

- `activate_project` / `activate_file` become **local-only** (no RPC): set properties, flip in-memory flags if any, emit signals.
- Optional: trigger daemon scan via separate RPC when user opens a project (same as today’s scan side-effect, without DB activation).
- `restore_active_state()`: read paths from agent config (or session), set local `active_*`, `.begin()` scan if needed — **no** `get_active_project()` from `is_active` in loaded list.
- Remove deprecated reliance on `ProjectList.get_active_project()` / `ProjectFiles.get_active_file()` driven by DB `is_active`.

### UI / app

- `Window` session restore: stop pre-calling daemon `activate_project` for DB flags; restore from session/agent config into local PM.
- `AgentFactory` / `Skill/Factory` `initialize_widget`: load projects RPC → restore from config → `apply_manager_state` (chain off activation callbacks if scan is async).
- [`1.3.9`](1.3.9-agent-configuration-and-window-state.md): code-assistant agent config stores `project_path`, `file_path` (and cursor/scroll stay on `File` in memory or in config as planned).

### Shipping `libocfiles` (after cutover)

- Delete or stub activation DB writes in fat PM once all callers use V2 + config restore.

**Not in scope for this plan:** vector RPC, file watcher, Copyable — unchanged.

## Open questions (decide before phase B)

1. **Scan trigger** — on local `activate_project`, client calls `Folder.read_dir` RPC explicitly, or a single `ProjectManager.open_project` that only means “scan this path if stale”?
2. **`get_file_from_active_project`** — rename to path-based `get_file_at_path` everywhere; no “active project” in the name on daemon.
3. **Background worker** — today worker PM tracks `active_project` for memory only ([`2.10.1` done plan](done/2.10.1-DONE-codebase-search-background-scanning.md)); confirm daemon scan queue does not need client active project at all (path-driven only).
4. **Migration** — one release with both config restore and DB `is_active` fallback, or hard switch?

## Test plan (when implementing)

1. Two windows, different projects — each PM has correct `active_project` / `active_file`; daemon has no global active state.
2. Quit and restart — agent config restores project/file; no dependence on SQLite `is_active`.
3. `load_projects_from_db` — list projects without any row marked active.
4. Tools using `project_manager.active_project` — still work per-window (each tool run uses that window’s PM).
5. Remove / update `oc-test-files` `--test-db` restore path to use explicit paths, not DB activation flags.

## Summary

| Today | Target |
| ----- | ------ |
| `is_active` in SQLite | Paths in agent config / session state file |
| Daemon `activate_project` / `File.activate` | Client-local only; scan via separate RPC |
| `restore_active_state()` awaits DB + scan | Config-driven restore; fire-and-forget scan |
| One implicit global active project | Per-window `V2/ProjectManager` |

**Implement after** 2.10.4.4–6 cutover — use the interim RPC activation only to pass tests, not as the permanent design.
