# Plan 1.23.40: Refine stage — reference injection and template styles

**Coding standards:** `.cursor/rules/CODING_STANDARDS.md`.

## Status

Draft — design only; implementation not started.

---

## Read this first

- **What goes wrong today:** one “dump all references into refine” policy. Search-heavy skills need **links + tool scope** at refine; read/plan/implement skills need **resolved text**, **reference lists**, or **splits**. Same YAML shape for everyone does not fit.
- **Fix:** each skill picks an **`exec_type`** (one id), not ten flags — it selects **both** the refine prompt recipe and the exec runner (template registry maps `exec_type` → refine + exec templates).
- **Refine output:** **task definition** + **examine references** (list) + **shared references** (list, often **empty**) + **tool calls** (research only). Same wire shape everywhere; **shared** matters only when the stage is **multi-pass / split** — see **§ Refine: examine references** and Step 1.
- **Naming traps:**
  - `plan_apply_changes` → really **implement** (apply a plan to disk). Target name: `implement_plan`.
  - `plan_review` → really **review** (check the plan). Target name: `review_plan`.
- **Read-style analyze:** `exec_type: read_answer` — refine **only** picks **route**: **1-file** | **combined** | **separate** (one pass vs joint pass vs N passes). **Refine does not run the analysis** or load full files for it — **exec** resolves reads and runs the analysis. (Optional rename: `route_reads`.)
- **How this doc is organized:** by **pipeline step** (Refine → Exec → Post-exec), not by lifecycle stage. Under each step you get Research / Analyze / Plan / Implement / Review with pseudocode for that step only.

---

## Skill field (only)

```
skill:
  exec_type:              <one id>   # pipeline: tool_results | read_answer | file_output | review (+ registry maps refine + exec)
  template_style_version: <optional> # if wire format of an artifact changes
```

- **Refine** = model/runner produces:
  1. **Task definition** — refined task body and structured fields (no raw attachment dump on output).
  2. **Examine references** — refs **exec will read / use** this run (the work list).
  3. **Shared references** — only when **split or multi-pass**: common context prepended across passes, or global bucket. **Single-pass** stages (typical **plan**): **empty**.
  4. **Tool calls** — **research skills only** (`tool_search`); otherwise empty.
- **Exec** = **task_definition** + **examine_references** always; **shared_references** / **tool_calls** when that run uses them (same as Step 2 blocks). Resolves content for the model / runs tools — **not** a loose `references` field from refine.
- **Post-exec** = merge multiple exec outputs when the runner did **N** passes; skip when there was **one** pass (`file_output` skills = always one LLM exec, apply/merge happens in product code not a second LLM stage).

## Big picture: tool-based vs reference-based

- **Tool-based** (e.g. `research_*`)
  - Refine: **no full file or full fetched HTML bodies** — but **task items** may appear as **summaries only**; **file links** stay slices/metadata unless policy says more; **URLs** may use **partial** (title, host, known snippet / cached preview) when the system already has it — **not** a fresh full-page fetch. Real search hits and full reads come at **exec**.
- **Reference-based** (e.g. `plan_*`, `implement_*`, `review_*`, read-style `analyze_*`)
  - Refine: **examine_references** always; **shared_references** only when the recipe is **split / multi-pass** (+ **task definition**). **Not** a generic `references` blob on output.
  - Exec: reads those lists; no `read_file` tool on review skills.
- **Hybrid:** `analyze_web_page` — **partial** URL metadata at refine if available (same rule as **URLs** in **§ Expansion depth**); **full** page body only after **`web_fetch`** (or equivalent) in **exec**, then read like other analyze skills.

**Injection level** (lives *inside* the refine half of the chosen `exec_type`, not a global toggle):

- **Refine:** **`none`** or **`partial`** only. **Exec** expands to full bodies when needed (via **examine_references**). No default “full refine” tier in this doc.

**Refine input — `expanded_references`:** what gets **resolved into the refine prompt** under that rule. Not the same as **examine references** on output.

---

## Expansion depth (refine vs exec)

- **Partial expansion can apply at more than one point** — refine and exec are **not** required to use the same depth. Refine often gets **partial** material (metadata, slices, summaries); **exec** may **expand further** when resolving **examine_references** into the real prompt or tool inputs (still within caps). **Post-exec** does not expand refs again; it merges outputs.

- **Task items** (`task://`, deps): **summaries** on **expanded input** at refine (task summary text, short dependency lines, completed-task summaries). **Full** nested task bodies: **exec** when the run needs them — not a default refine injection.

- **File links:** partial at refine when policy allows (headers, counts, slices); **full** bodies when exec needs them for read / plan / implement.

- **URLs:** **partial** at refine when cheap or already known — e.g. **link text, title, description/snippet, OG preview** (policy-dependent). **No** live-fetched full HTML in the refine prompt; substantive page content → **exec** after fetch.

---

## Refine: examine references

Refine **examines** incoming attachments and decides what **exec** should look at. On **output**, refine does **not** pass through a vague `references` field — it emits structured fields. Two reference lists are a **unified wire shape**; **not every stage fills both**.

| List | Role |
| ---- | ---- |
| **examine_references** | Work list for **this** exec (paths, anchors, `task://`, URLs, search scope, per-unit targets when split). |
| **shared_references** | **Only when multi-pass / split:** common context across passes (standards, “common” block, global bucket). **Single-pass:** leave **empty**. |

- **Single-pass** — typical **`plan_manifest`**, **`implement` combined**, often **research**: everything goes in **examine_references**; **shared_references** = **empty**.
- **Multi-pass / split** — **`read_route`** with **separate**, **`implement` split**, **`review_split`**: **examine_references** = per-unit (or per-pass) targets; **shared_references** = optional common prepended / global refs.

- **Research:** refine emits **tool_calls**. **examine_references** = search scope; **shared_references** usually **empty** (or one shared scope).
- **Non-research:** **tool_calls** empty; scope from **examine_references** / **shared_references** per rules above.

## Pipeline by step (not by lifecycle stage)

Order of **steps** in the runner: **Refine** → **Exec** → **Post-exec**.

Under **each step** below: one pseudocode block covers Research → Analyze → Plan → Implement → Review.

---

### Step 1 — Refine (`task_refinement`)

Skill **`exec_type`** is on the skill (not produced here). Refine **output** (no loose `references` field): **task_definition** + **examine_references** + **shared_references** (empty when single-pass — see **§ Refine: examine references**) + **tool_calls** (research only).

**Refine input — `references`:** any combination of **task items** (`task://`, deps, related tasks), **file links** (paths, `file#anchor`), and **URLs** — use whichever subsets apply; not all three every time. **Expansion depth:** **§ Expansion depth (refine vs exec)** — refine stays **partial**; full reads in **exec**.

```text
research (tool_search):
  input:
    task
    references:
      task items
      file links
      urls
    expanded_references:
      partial: [research_codebase, research_docs, research_standards, research_online_search]
  output:
    task_definition:
      refined body
      refinement notes
    examine_references:
      scope / paths / targets for search (what exec tools will use)
    shared_references:
      optional shared scope
      often empty
    tool_calls:
      one or many

analyze (read_route):
  input:
    task
    references:
      task items
      file links
      urls
    expanded_references:
      partial: [analyze_code, analyze_docs, analyze_standards, analyze_web_page]
  output:
    task_definition:
      route: 1-file | combined | separate
    examine_references:
      per-unit refs exec will read (combined / separate / 1-file)
    shared_references:
      empty (1-file | combined) | optional common (separate)
    tool_calls:
      empty

plan (plan_manifest):
  input:
    task
    references:
      task items
      file links
      urls
    expanded_references:
      partial: [plan_create, plan_iterate, plan_code, plan_code_iterate]
  output:
    task_definition:
      refined body
    examine_references:
      refs exec will use to plan (files, task://, deps)
    shared_references:
      empty
    tool_calls:
      empty

implement (implement_combined | implement_split):
  input:
    task
    references:
      task items
      file links
      urls
    expanded_references:
      partial: [implement_code, implement_plan]
  output:
    task_definition:
      combined | split (global vs action)
    examine_references:
      action targets (edit units) when split; all relevant when combined
    shared_references:
      empty (combined) | global bucket / common (split)
    tool_calls:
      empty

review (review_split):
  input:
    task
    references:
      task items
      file links
      urls
    expanded_references:
      partial: [review_code, review_apply_standards, review_plan]
  output:
    task_definition:
      global vs singular layout
    examine_references:
      review targets (layout drives one pass vs N)
    shared_references:
      optional common (standards, plan) — typical when singular multi-pass
    tool_calls:
      empty
```

### Step 2 — Exec (`task_execution`)

Exec **inputs** come from refine **only** as structured fields — **no** separate refine output field named `references`. **task_definition** + **examine_references** always; **shared_references** and **tool_calls** only when that refine recipe uses them (see each block). Resolve from **examine_references** / **shared_references** per skill policy and caps (research: run **tool_calls**, tool hits — not full-file reads from refs).

```text
research (tool_results):
  input:
    task_definition
    examine_references
    shared_references
    tool_calls
  output:
    run tool_calls; scope from examine_references / shared_references
    artifact: hit lists (merge or display)
    exec_type: tool_results

analyze (read_answer):
  input:
    task_definition
    examine_references
    shared_references
    resolve to model prompt per policy (analyze_web_page → web_fetch here if needed)
  output:
    artifact: analysis text
    exec_type: read_answer

plan (file_output):
  input:
    task_definition
    examine_references
    build planning prompt
  output:
    artifact: plan markdown
    exec_type: file_output

implement_combined (file_output):
  input:
    task_definition
    examine_references
    resolve + approved plan if implement_plan
  output:
    artifact: structured bundle
    side_effect: output_executor.apply(bundle)
    exec_type: file_output

implement_split (file_output):
  input:
    task_definition
    examine_references
    shared_references
    resolve + approved plan if implement_plan
  output:
    artifact: structured bundle
    side_effect: output_executor.apply(bundle)
    exec_type: file_output

review (review):
  input:
    task_definition
    examine_references
    shared_references
    resolve refs (no read_file tool in skill)
  output:
    artifact: findings text
    exec_type: review
```

---

### Step 3 — Post-exec (`task_post_exec`)

```text
research:
  input:
    N tool result sets (if N runs)
  output:
    merged hits | pass-through if N=1

analyze:
  input:
    N analysis texts (if split/separate)
  output:
    merged analysis | pass-through if N=1

plan:
  input:
    single exec artifact
  output:
    unchanged; partial plan merge when persisting task (product code)

implement:
  input:
    bundle(s) after apply
  output:
    skip

review:
  input:
    N finding sets (if N passes)
  output:
    merged findings | pass-through if N=1
```

## Lifecycle phases (order of work)

- **research** — find candidates (`research_*`).
- **analyze** — read and answer (`analyze_*`). Not “search the web” (that is research).
- **plan** — markdown only, no writes (`plan_*`).
- **implement** — one exec bundle + **output executor** (`implement_*`). `plan_apply_changes` belongs **here** as `implement_plan`.
- **review** — check plan/code/standards (`review_*`). `plan_review` belongs **here** as `review_plan`.

## `exec_type` (lookup)

Stored on the skill. Registry picks **refine template** + **exec runner** from this one id (and skill kind where needed).

| exec_type | Refine recipe (registry) | Exec behavior | Typical skills |
| --------- | ------------------------ | ------------- | -------------- |
| `tool_results` | `tool_search` — scope in tool args; **partial** expanded refs at refine only | run refine’s **tool_calls** + collect hits | `research_*` |
| `read_answer` | `read_route` — route only (**1-file** / **combined** / **separate**); optional **common** via **shared_references**; **no** full analysis at refine | **exec:** resolve reads + model → analysis | `analyze_*` |
| `file_output` | `plan_manifest` / `implement_combined` / `implement_split` (registry) | **plan** + **implement_combined:** `task_definition` + **examine** only; **implement_split:** + **shared** | `plan_*`, `implement_*` |
| `review` | `review_split` — global vs singular | model → findings; resolved refs only | `review_*` |

**Migration**

- One reference-heavy default → per-skill `exec_type`.
- Many YAML toggles → one `exec_type` enum + template registry + optional `template_style_version`.
- Implement via tools → `exec_type: file_output` + output executor + implement refine variant.
