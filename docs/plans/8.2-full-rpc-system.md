## Purpose

End-to-end RPC stack beyond the bin socket cutover (**8.1**):

- **🔷** **JSON line mode** on the same transport — detect `{` as first byte, read newline-delimited JSON, decode via **`Bin.Json`** without wire `*type` identifiers (“auto JSON”).
- **🔷** **Type hint tables** — method prefix and polymorphic-array hints so auto JSON can resolve param and result element types.
- **🔷** **HTTP transport** — request/response over HTTP using bin objects internally, JSON on the wire (OpenAI-style calls).
- **🔷** **Replace `Json.Serializable` in `libollmchat`** — chat/API HTTP bodies built from bin types + auto JSON encode, not hand-rolled `serialize_property`.
- **🔷** **OpenAI-compatible server** — local chat backend exposing OpenAI HTTP **and** bin RPC (client/server).
- **🔷** **Session resumption** — reconnect after mobile/network drops; carry stream/session state in server memory.
- **🔷** **TLS + client certificate registration** — SSL as session manager; post-auth client cert for identity on reconnect.

**Suggested order (phases below):** 1 → 2 → 3 → 4 → 5 → 6 → 7. Later phases depend on earlier ones; sub-plans may split out per phase when implementing.

---

## Current behaviour

- **ℹ️** Server accepts a connection, sets up **`Connection`**, **does not write** until the client sends the first message (bin `Request` today).
- **ℹ️** Client **`connect()`** opens socket, then sends **`Daemon.hello`** as first bin `Request`.
- **ℹ️** **`Bin.Json`** (`libocrpc/Bin/Json.vala`) — test/tooling bridge; every object needs `*type` meta; nested objects and object arrays require explicit element aliases.
- **ℹ️** **`Request.register(name, target, param_type)`** — maps handler singleton → param **`GLib.Type`**; **not** keyed by full method string (`File.read` vs `Folder.list`).
- **ℹ️** **`libollmchat`** — HTTP to Ollama/OpenAI uses **`Json.Serializable`** overrides on **`Call.*`**, **`Response.*`**, **`Settings.Connection`** (separate from RPC bin stack).
- **ℹ️** RPC assumes a **persistent** socket; no session reattach after disconnect.
- **ℹ️** No TLS or client-cert flow on RPC transports today.

## Phase 1 — JSON line mode on `Connection`

### Goal

- **🔷** One accepted client may speak **bin** or **JSON line** on the same socket.
- **🔷** **Detection:** first received byte `{` (0x7B) → JSON line mode for that connection; otherwise bin mode (existing path).
- **🔷** JSON mode read loop: accumulate bytes until `\n`, parse one JSON object per line, hand to **`Bin.Json`**.
- **🔷** JSON mode write loop: encode replies as one JSON object + `\n` per message (responses and notifications).

### Connection changes (conceptual)

- **🔷** `⏳` **`Connection`** — defer bin `parse()` until transport mode is known (peek first byte, or buffer first chunk).
- **🔷** `⏳` **`Connection`** — `json_mode` flag per connection; separate read path for line accumulation vs bin byte stream.
- **🔷** `⏳` After JSON decode → same **`Request.dispatch()`** / **`reply()`** path as bin (GObject graph in memory is identical).

### Client symmetry

- **🔷** `⏳` **`Client`** — optional JSON line mode (send `{...}\n` instead of bin root object); used for debug tooling and HTTP-adjacent clients before Phase 3.

### Risks

- **💩** `⏳` Mixed mode on one connection after first byte is fixed — probably **🚫** no mid-stream mode switch (first byte is permanent for connection lifetime).
- **💩** `⏳` Bin mode first byte can never be `{` — verify bin root encoding never emits 0x7B as first octet (true for current type-byte layout).

## Phase 2 — Auto JSON (`Bin.Json` without `*type`)

### Goal

- **🔷** **`Bin.Json`** gains an **auto JSON** mode (name TBD: `auto_json`, `strip_types`, etc.).
- **🔷** **Decode (inbound):** do **not** require `*type` on objects.
  - Root line is always a **`Request`** — inject `*type: "Request"` (or equivalent) before bin overlay.
  - **`param`** — resolve type from **hint table** (Phase 2b), then overlay JSON properties onto that GObject type.
  - Nested objects — infer from **property `ParamSpec`** / registered field types on the parent GObject (existing bin polymorphic rules where they work).
- **🔷** **Encode (outbound):** omit `*type` (and other `*` meta keys) from generated JSON when auto JSON flag is set.
- **🔷** Polymorphic **`Response.result`** arrays — element type from **result hint table** (Phase 2b), not from per-element `*type` in JSON.

### Why hints are required

- **🔷** Auto JSON cannot rely on `*type` in the payload.
- **🔷** **`Request.param`** — JSON has `"param": { "path": "..." }` with no type tag; server must know `File.*` → `FileParams`, `Daemon.hello` → `DaemonParams`, etc.
- **🔷** **`Response.result`** — root object arrays (`fetch_files` → `[File, …]`) need a declared element type when JSON elements lack `*type`.

### Phase 2a — `Bin.Json` auto mode API

- **🔷** `⏳` Flag on **`Json.write`** / **`Json.parse`** (or session object) — `auto_json: bool`.
- **🔷** `⏳` When `auto_json` and encoding: skip `*type`, `*array`, and other `*` meta members in output.
- **🔷** `⏳` When `auto_json` and decoding: accept objects without `*type`; caller supplies root/param/element hints.

### Phase 2b — Hint tables

- **🔷** `⏳` **Method → param type** map.
  - Key: method prefix or pattern (e.g. `File.` → `FileParams`, `Folder.` → `FolderParams`, `Daemon.hello` → `DaemonParams`).
  - Lookup at JSON decode: `request.method` → param `GLib.Type` → inject `param.*type` before **`Bin.Json`** overlay.
  - **ℹ️** Extends / complements **`Request.param_types`** (handler singleton → param type); may need **method-string** index for JSON path.
- **🔷** `⏳` **Method → result element type** map (responses with object arrays).
  - Key: `method` or `method` + `is_array` flag (e.g. `File.fetch_files` → `File`).
  - Used when encoding/decoding `Response.result` in auto JSON.
- **🔷** `⏳` Registration lives alongside existing **`rpc_register()`** / **`Request.register()`** (same init pass as wire types).

### Polymorphic pitfalls

- **🔷** `⏳` Audit every **`bin_read_prop`** / **`bin_write_prop`** override that assumes wire `*type` or JIT type bytes — list properties, graph omits, `Gee.ArrayList` paths.
- **🔷** `⏳` Stdio test scripts (`tests/rpc/*.script.in`) may keep explicit `*type` (non-auto mode) for regression; add parallel auto-JSON fixtures.

## Phase 3 — HTTP transport (bin inside, JSON outside)

### Goal

- **🔷** RPC-style **request/response over HTTP** — one POST (or GET where appropriate) per call, body = one JSON line/object, response = JSON (not a persistent socket).
- **🔷** Internal pipeline: HTTP body → auto JSON decode → **`Request`** → handler → **`Response`** → auto JSON encode → HTTP body.
- **🔷** Reuse **`Bin.Json`** + hint tables; **no** parallel HTTP DTO types.

### Scope

- **🔷** `⏳` **`libocrpc`** HTTP client — `Client` variant or transport adapter: URL + TLS options, no `IOChannel` read loop (single round-trip per `call()`).
- **🔷** `⏳` HTTP server hook — for daemons or future OpenAI-compat server (Phase 5); map route → `method` string + body parse.
- **💩** `⏳` Mapping table: HTTP path ↔ RPC `method` (e.g. `POST /v1/chat/completions` ↔ `Chat.completions` internal method) — define when Phase 4/5 specs land.

### Out of scope for Phase 3 alone

- **🚫** SSE/streaming HTTP for chat tokens — separate follow-up (may share session id from Phase 6).
- **🚫** Full OpenAI schema compatibility — Phase 5.

## Phase 4 — `libollmchat`: bin + auto JSON instead of `Json.Serializable`

### Goal

- **🔷** Remove (or drastically shrink) **`Json.Serializable`** on **`Call.Chat*`**, **`Call.Embeddings`**, **`Response.*`** used for HTTP wire format.
- **🔷** Build request/response as **`Bin.Serializable`** wire types (new or migrated from existing call objects).
- **🔷** HTTP send path: GObject → **`Bin.Json`** (auto JSON) → HTTP body.
- **🔷** HTTP receive path: HTTP body → auto JSON → GObject → existing exec/stream handling.

### Motivation

- **🔷** One serialization stack for socket RPC, HTTP OpenAI calls, tests, and future bin RPC to a remote chat server.
- **🔷** Drop duplicate `serialize_property` / `deserialize_property` maintenance in **`libollmchat`**.

### Work items

- **🔷** `⏳` Inventory all **`Json.Serializable`** implementers in **`libollmchat`** used on the wire (not config/session persistence).
- **🔷** `⏳` Define wire types + **`rpc_register()`** / hint entries for chat, embeddings, models, errors.
- **🔷** `⏳` Switch **`Call.*.exec()`** / streaming paths to bin encode for outbound JSON.
- **🔷** `⏳` Sub-plan likely required — large surface (`Call/Chat.vala`, `Response/*`, streaming deltas).

### Dependency

- **ℹ️** Phase 2 (auto JSON + hints) and Phase 3 (HTTP transport) must be far enough along to exercise real chat payloads.

## Phase 5 — OpenAI-compatible server (dual HTTP + bin RPC)

### Goal

- **🔷** Server process exposing:
  - **OpenAI-compatible HTTP** (`/v1/chat/completions`, `/v1/embeddings`, …) for standard clients.
  - **Bin RPC** (socket or TLS) for **OLLMchat** remote mode (e.g. phone app → desktop/server).
- **🔷** Same handler core — HTTP and bin JSON line/bin modes converge on **`Request.dispatch()`**.
- **🔷** Local inference via existing **`libollmchat`** / future **8.0** local backend behind the same API.

### Client scenarios

- **🔷** Desktop **`ollmchat`** — local socket to `ollmfilesd` (today); optional remote bin RPC to a chat server.
- **🔷** Mobile (**`9.0`**) — HTTP and/or bin over TLS to user’s server; no local daemon.

### Work items

- **🔷** `⏳` Route map: OpenAI JSON shapes ↔ internal RPC methods (may differ from literal `Chat.completions` naming).
- **🔷** `⏳` Auth boundary (ties to Phase 7 for remote).
- **🔷** `⏳` Streaming: SSE or chunked HTTP vs bin notifications — design sub-plan.

## Phase 6 — Session resumption (mobile / flaky networks)

### Problem

- **🔷** Bin RPC today assumes **always-on** connection — unsuitable for mobile (OS kills socket, network changes).
- **🔷** Chat streaming and JIT bin type registration are **connection-scoped** — reconnect must restore session context.

### Goal

- **🔷** **Session id** assigned at hello (or first authenticated call).
- **🔷** Server holds **in-memory session state** — partial streams, pending notifications, registered JIT keys if needed.
- **🔷** **Reconnect handshake** — client presents `session_id` (+ auth from Phase 7); server reattaches or rejects.
- **🔷** **Resume streaming** — continue pushing deltas for in-flight chat/tool work after reconnect.

### Work items

- **🔷** `⏳` Extend **`Daemon.hello`** (or new `Session.resume`) with `session_id`, `client`, protocol version.
- **🔷** `⏳` Session table on server — TTL, max clients, cleanup on shutdown.
- **🔷** `⏳` Client **`connect()`** — persist `session_id`; retry reconnect with backoff.
- **💩** `⏳` Optional persistence to sqlite for long-lived sessions — confirm if in-memory only is enough for v1.

### Dependency

- **ℹ️** Phase 7 auth should identify the client before trusting session reattach.

## Phase 7 — TLS and client certificate registration

### Goal

- **🔷** Encrypted transport for remote RPC (phone ↔ server, internet-facing daemon).
- **🔷** **First connection:** TLS with server auto-generated / configured cert; application-level auth over RPC (method TBD).
- **🔷** **After auth:** server **registers a client certificate**, returns it to the client (or CSR flow).
- **🔷** **Subsequent connections:** client presents client cert; server maps cert → session identity (replaces or supplements token in Phase 6).

### Flow (conceptual)

1. Client → TLS → server (server cert only).
2. Client → RPC auth call (credentials / token / pairing code).
3. Server → issue client cert (OpenSSL/GnuTLS “registration” or custom CA sign).
4. Client disconnects, reconnects with **client cert** → server recognizes peer without repeating full auth.

### Work items

- **🔷** `⏳` TLS wrapper on **`TcpListen`** / **`Client`** (not Unix socket).
- **🔷** `⏳` Cert storage paths under `~/.local/share/ollmchat/` (or configurable).
- **🔷** `⏳` RPC methods: `Auth.*` or `Daemon.register_client` — exact API in sub-plan.
- **💩** `⏳` Certificate rotation and revocation — v2 if needed.

### Notes

- **ℹ️** User referenced SSL “registration” — map to concrete API (GnuTLS `gnutls_certificate_allocate_credentials` / OpenSSL equivalent) during implementation spike.
- **🚫** Mutual TLS alone without app-level pairing is **not** sufficient for first-time trust — keep explicit auth step before issuing client cert.

## Cross-cutting: transport mode summary

| Mode | First byte / framing | Type tags on wire | Typical use |
| ---- | -------------------- | ----------------- | ----------- |
| **Bin** (today) | Type byte / reg token | Bin type bytes + JIT names | `ollmfilesd` Unix socket, TCP RPC |
| **JSON line** | `{` … `\n` | `*type` meta (tests) | Debug, migration, stdio harness |
| **Auto JSON line** | `{` … `\n` | None — hint tables | HTTP bodies, OpenAI compat, mobile |
| **HTTP** | Request/response | Auto JSON body | Stateless calls, OpenAI clients |
| **TLS + bin/JSON** | Same as above inside TLS | Same | Remote OLLMchat, internet daemon |

**🚫** Do not maintain a fourth parallel serializer — all paths go through **`Bin.Serializable`** + **`Bin.Json`**.
