## Boundary

**🔷 `libocvector2` owns:**

- FAISS + embed API
- **`SQT/VectorMetadata.vala`** — moved from `libocfiles` (same class name, same table)
- **`VectorBuilder`**, **`Search`** — bodies from `libocvector`; use **`OLLMvector2.SQT.VectorMetadata`**

**🔷 `file_id`** on each row is an opaque **`int64`** (daemon's `filebase.id`). No **`File`** type in the lib.

**🔷 `libocvector2` does not know about:** `File`, `Folder`, `ProjectManager`, `filebase` schema, tree-sitter, snippet file reads, scan queues, tools, RPC.

**🔷 Consumers (when wired — not Phase A):**

- **`ollmfilesd`** only — indexer, search assembly, `vector.*` RPC server side

```text
Today:   libocfiles/SQT/VectorMetadata  +  libocvector → libocfiles
Phase A: libocvector2/SQT/VectorMetadata  (library exists; nothing wired yet)
Later:   ollmfilesd → libocvector2
         libocfiles  → no VectorMetadata, no link to ocvector2 (RPC to daemon)
         libocvector → retired (untouched until that cutover)
```

**ℹ️** `VectorMetadata` lived in **`libocfiles`** historically to break a circular dep ([`1.23.18`](done/1.23.18-DONE-tail-end-issues.md)). With **`libocvector2`**, **`libocfiles`** stops owning vector rows entirely — it is **not** a downstream library of `libocvector2`.

## Who needs `VectorMetadata`?

**🔷 Vector stack only:**

- **`libocvector2`** (`VectorBuilder`, `Search`)
- **`ollmfilesd`** vector pipeline (after companion plan wires it)

**🚫 Not consumers of `libocvector2`:**

- **`libocfiles`** — never links `--pkg=ocvector2`. Today it still has its own copy for legacy paths; at cutover that file is **removed**, not repointed.
- **`libocvector`** — stays on `OLLMfiles.SQT.VectorMetadata` until **retirement**; no interim migration in Phase A.
- **`examples`**, **`ollmapp`** — wait for daemon + app cutover.

**🔷 What moves off `libocfiles` at cutover** (daemon / RPC side, not this plan):

- **`Folder.project_description()`** → **`vector.project.describe`** RPC (V2 client relay)
- **`to_summary(..., keymap)`** → daemon or tool builds keymap for display
- **`DeleteManager.on_cleanup`** → daemon calls **`VectorMetadata.cleanup_all_deleted`**

**🚫** No second metadata type, no **`MetadataRow`** rename, no daemon SQL copy under `ollmfilesd/SQT/`.

## Daemon (companion plan — not Phase A)

`ollmfilesd` follows the **same relay pattern** as file RPC: app/V2 client → daemon → `libocvector2`.

```text
ollmfilesd/Vector/CodeTree.vala              ← Indexing/Tree.vala
ollmfilesd/Vector/DocumentationTree.vala
ollmfilesd/Vector/Analysis.vala              extends VectorBase
ollmfilesd/Vector/Indexer.vala
ollmfilesd/Vector/ScanQueue.vala
ollmfilesd/Vector/SearchResult.vala          snippet: file_id → path → lines (daemon only)
```

Pipeline: parse → analyze → **`VectorBuilder.process_elements`** → **`file.last_vector_scan`**.

**🚫** Not in companion stub yet: `CodebaseSearchTool`, `ollmapp` tool wiring.
