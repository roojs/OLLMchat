# 2.22.1.2 Seccomp: GIR/`vapigen` first, minimal `.vapi`, validation tests

**Next (implementation deltas):** [2.22.1.3 — seccomp reporting after bwrap](./2.22.1.3-seccomp-reporting-after-bwrap.md)

## Purpose

Ship **enough** [libseccomp](https://github.com/seccomp/libseccomp) surface in Vala to build a BPF program and pass an FD to **bubblewrap** (`--seccomp` / `--add-seccomp-fd`), matching the direction in **§ Feasibility: seccomp and Vala** of the parent plan.

**End state for OLLMchat:** a **minimal `.vapi`** covering only what we need (e.g. `seccomp_init`, `seccomp_rule_add`, `seccomp_export_bpf`, `seccomp_release`, architecture helpers as required) so call sites in `Bubble.vala` stay readable. **How we get there:** try **generated GIR → `vapigen` → hand edits** first (see below); if that stalls, fall back to **fully manual** vapi or a **small C helper**.

Non-goals for this sub-milestone: full libseccomp coverage, GObject-style bindings, or committing to **notify** vs **TRAP** vs **LOG** — only prove **build + linked API + observable syscall events** in controlled tests.

**Parent alignment:** In [2.22.1](./2.22.1-run-command-seccomp-network-and-path-evidence.md), **libseccomp + bwrap** apply only on the **`Bubble.can_wrap()`** path. The **`execute_with_subprocess`** fallback when **`!can_wrap()`** (e.g. Flatpak, no **`bwrap`**) keeps **prior permission behaviour** and **does not** load seccomp via bubblewrap.

## VAPI / binding generators (research)

### Recommended spike order

1. **`g-ir-scanner`** against **standard headers** (and/or a tiny **stub `.c`** that `#include`s `seccomp.h` so the scanner sees real declarations — some setups use headers-only; others need a compile unit for macros and typedefs).
2. **`vapigen`** the resulting **`.gir`** into a **first-draft `.vapi`** (see [Generating a VAPI with GObject Introspection](https://docs.vala.dev/developer-guides/bindings/generating-a-vapi-with-gobject-introspection.html)).
3. **Hand-edit** the vapi: fix namespaces, `[CCode]`, pointer/`ref`/`unowned` mistakes, missing constants, and **trim** APIs we do not need.

If step 1–3 produces more churn than a short manual file, **stop** and commit to **fully manual `.vapi`** or **in-tree C glue** (parent plan options).

### Tool summary

| Tool / path | Role | Fit for libseccomp |
|---------------|------|---------------------|
| **`g-ir-scanner`** | Parses C headers/sources and writes **`.gir`**. Often used for GObject libraries, but **can** be aimed at ordinary headers (optional `.metadata`, flags such as **`--accept-unprefixed`** where appropriate) — **worth trying first** for a bootstrap GIR. | **Try first.** Expect **cleanup**: opaque structs, enums, macro constants, and includes may need fixes or a thin `.c` shim. |
| **`vapigen`** | Generates `.vapi` from `.gir`. Ships with Vala toolchains. | **Good second step** after a usable GIR; output usually needs **manual edits**. |
| **Manual `.vapi`** | Hand-written `[CCode]` blocks, `extern` functions, enums/constants as needed. | **Fallback** if generator output is unwieldy; still fine for the **minimal** subset we ship. |
| **Small in-tree `.c` helper** | Wrap a few functions; Vala calls C only. | Fallback if both GIR and manual vapi are painful. |

**Conclusion:** Treat **`g-ir-scanner` → `vapigen` → hand-edit`** as the **first** workflow; fall back to **manual vapi** when generated bindings fight us (bad macros, incomplete GIR, or excessive churn).

### In-repo spike (first step)

**Location:** [`contrib/seccomp-gir/`](../contrib/seccomp-gir/README.md)

- **`g-ir-scanner --header-only`** on `/usr/include/seccomp.h` → **`Seccomp-2.5.gir`** (namespace version matches local libseccomp **2.5.x**).
- **`vapigen`** writes a **reference** **`seccomp-vapigen-raw.vapi`** (via **`./contrib/seccomp-gir/regenerate.sh`**); **`sed`** fixes header names. **`vapi/seccomp.vapi`** is **hand-maintained**, object-oriented:**`Seccomp.Filter`** wraps **`scmp_filter_ctx`** (**`new`**, **`seccomp_release`** as destructor); ctx-taking APIs are **methods**, not loose **`init`/`release`**.
- **`test-minimal`** uses **`new Seccomp.Filter (...)`** with **`valac --vapidir=…/vapi --pkg seccomp`**.
- **Meson PoC** (built when **`libseccomp`** is linkable and **`vapi/seccomp.vapi`** is present; **`cc.find_library('seccomp')`** + **`gio-2.0`**): **`examples/oc-test-seccomp-spike`** (**Vala**, linking **`vapi/seccomp-fd-pass.c`** for post-fork-safe **`SCM_RIGHTS`** via **`Seccomp.pass_unix_fd` / `Seccomp.receive_unix_fd`**) — child installs **`SCMP_ACT_NOTIFY`** on **`openat`** (file mode) or **`socket`/`connect`** (network mode), **`seccomp_load`**, passes **notify fd** to parent (**`SCM_RIGHTS`**), syncs, then **`execvp("bwrap", …)`** (no **`--seccomp`** BPF in this PoC: the filter is inherited after load). Parent **`GLib.Thread`** runs **`seccomp_notify_receive` / `seccomp_notify_respond`** with **`SECCOMP_USER_NOTIF_FLAG_CONTINUE`** (record + allow). Modes: **`file`** — inner **`echo … > /root/silly.txt`**; **`network`** — **`--unshare-net`** + **`python3`** socket one-liner. Pass when stdout shows **`REPORT:`** lines and final **`OK:`** summary for the mode.

**Regenerate:** `./contrib/seccomp-gir/regenerate.sh` (uses **`SECCOMP_GIR_VERSION`** / **`SECCOMP_HEADER`** overrides if needed).

## Minimal API surface (illustrative)

Exact names depend on headers and arch; keep the **first** `.vapi` revision to **export BPF fd** only:

- **Init / teardown:** `seccomp_init`, `seccomp_release` (and any `_reset` if we use it).
- **Rules:** `seccomp_rule_add` / `seccomp_rule_add_exact` for a **small** syscall set used in validation (e.g. `openat`, `connect`, `socket`).
- **Export:** `seccomp_export_bpf` (or `seccomp_gen_bpf` — verify against installed `pkg-config --modversion libseccomp` and `seccomp.h`).
- **Arch:** use libseccomp’s arch resolution if rules must be arch-specific (`seccomp_arch_native`, etc.).

**Pin** supported build architectures in docs/CI (**x86_64**, **aarch64**) to avoid untested syscall number drift.

- **Namespace:** e.g. `namespace Seccomp { ... }` with `[CCode (cheader_filename = "seccomp.h", cprefix = "SCMP_", lower_case_cprefix = "seccomp_")]` patterns **only where accurate** — match existing repo conventions for other manual bindings.

## Validation tests (must demonstrate behavior)

These are **integration-style** checks: **bubblewrap** applies the sandbox; **seccomp** instruments **specific** syscalls so we can assert **parent-visible** evidence. **`oc-test-seccomp-spike`** uses **`SCMP_ACT_NOTIFY`** + **continue** (aligned with **parent 2.22.1**: **notify → record → allow** for **network and file access reporting**). **`test-minimal`** and **`vapi/seccomp.vapi`** still prove **Vala + `Seccomp.Filter` + `export_bpf`** for the **`bwrap --seccomp`** path planned in **`liboctools`**.

### Test 1 — Filesystem write under RO root (`echo` redirect to forbidden path)

**Intent:** With **`--ro-bind / /`** (or equivalent read-only root), running  
`echo test > /root/silly.txt`  
should **attempt** a write **outside** allowed writable mounts. Seccomp rules should include **filesystem write path** syscalls we care about (at minimum **`openat`** with create/write semantics — align with kernel behavior for `>` redirection).

**Procedure (sketch):**

1. **`oc-test-seccomp-spike file`** — NOTIFY on **`openat`**, parent prints **`REPORT:`** per event, responds **continue**, then **`execvp` `bwrap`** with RO root and inner `echo … > /root/silly.txt`.
2. **Assert:** stdout shows **`openat`** in **`REPORT:`** lines and exits **0** with **`OK:`** (see spike source for exact strings).

**Note:** Read-only bind causes **EPERM/EACCES** before seccomp in some setups; the test still validates **our filter is loaded** and ** syscall instrumentation works**. If ordering masks seccomp, narrow the scenario (e.g. writable `/tmp` only vs forbidden path under RO) so seccomp remains the **observable** signal — **iterate in implementation**.

### Test 2 — Network isolation (`python3` socket with net unshared)

**Intent:** With **`--unshare-net`**, a command that creates an IPv4 stream socket should hit **`socket`** (and optionally **`connect`**). Seccomp instrumentation should **detect** and **report** those syscalls.

**Procedure (sketch):**

1. **`oc-test-seccomp-spike network`** — NOTIFY on **`socket`** and **`connect`**, **`bwrap`** with **`--unshare-net`**, inner **`python3 -c '…socket.socket…'`**.
2. **Assert:** stdout shows **`REPORT:`** lines naming **`socket`** (and **`connect`** if exercised) and exits **0** with **`OK:`**.
